## 1. 已确认做得正确的部分

### 1.1 用户主体与项目授权边界清楚

`users.py` 和 `user_service.py` 的总体口径是正确的：

```text
User = 账号主体
Authorization = 用户 × 项目的授权记录
项目内等级 / 授权设备数 / 到期时间归属 Authorization
```

`users.py` 的文件头已经明确接口职责和 Admin / Agent 权限口径，Admin 可操作所有用户，Agent 只能操作自己创建的用户。
`user_service.py` 的文件头也明确用户等级、设备数、到期时间不再以 User 为主口径，而归属 Authorization。

### 1.2 路由层 response_model 基本完整

`users.py` 中用户创建、列表、详情、更新、授权创建、授权更新、授权升级预览、授权升级等接口都有显式 `response_model`。这对 OpenAPI 契约治理是好事。

### 1.3 Agent 自动生成密码被禁止

`update_user_password()` 中明确：

```text
管理员可 auto_generate，并一次性返回 generated_password。
代理不能 auto_generate 获取明文，只能提交 new_password。
```

这个安全边界正确。

### 1.4 tester 授权只允许管理员设置

`grant_authorization()` 中：

```text
body.user_level == "tester" and admin is None → 403
```

这避免代理售卖测试级别。

## 2. P0 问题：代理项目授权过期后仍可能给用户开通项目

### 状态：已确认，P0

在 `grant_authorization()` 中，代理给用户开通项目授权时，只检查了：

```text
AgentProjectAuth.agent_id == agent.id
AgentProjectAuth.project_id == body.game_project_id
AgentProjectAuth.status == "active"
```

没有检查：

```text
AgentProjectAuth.valid_until is None
或 AgentProjectAuth.valid_until > now
```

这意味着如果代理的项目授权已经过期，但状态仍是 `active`，代理仍可能继续给用户开通该项目授权。当前代码片段可见该查询只检查 `status == "active"`。

### 风险

```text
代理项目准入/项目授权已经到期
↓
代理仍可继续给用户开通该项目
↓
项目准入系统与用户授权系统边界被绕过
```

### 修复要求

建议新增统一 helper，例如：

```python
async def _assert_agent_can_authorize_project(
    *,
    agent_id: int,
    project_id: int,
    db: AsyncSession,
) -> AgentProjectAuth:
    now = datetime.now(timezone.utc)
    result = await db.execute(
        select(AgentProjectAuth).where(
            AgentProjectAuth.agent_id == agent_id,
            AgentProjectAuth.project_id == project_id,
            AgentProjectAuth.status == "active",
        )
    )
    auth = result.scalar_one_or_none()
    if not auth:
        raise HTTPException(status_code=403, detail="代理没有该项目授权权限")
    if auth.valid_until is not None and _ensure_aware(auth.valid_until) <= now:
        raise HTTPException(status_code=403, detail="代理项目授权已过期")
    return auth
```

然后在所有用户授权、升级、续费入口复用。

## 3. P0 问题：代理可以通过 PATCH 免费延长用户授权到期时间

### 状态：已确认，P0

`AuthorizationUpdateRequest` 允许传：

```text
valid_until
```

Schema 中 `valid_until` 是可选字段。

`update_authorization()` 对 Agent 做了部分限制：

```text
代理不能改 user_level
代理不能改 status
代理不能减少 authorized_devices
```

但后面无论 Admin 还是 Agent，只要请求体包含 `valid_until`，就执行：

```python
auth.valid_until = body.valid_until
```

可见代码中 `valid_until` 更新没有对 Agent 禁止，也没有触发续费扣点。

### 风险

代理可以对自己创建的用户调用：

```http
PATCH /api/users/{user_id}/authorizations/{auth_id}
```

请求体：

```json
{
  "valid_until": "2099-12-31T00:00:00Z"
}
```

当前逻辑下可能直接延长授权期限，**不扣点、不生成 AuthorizationChargeSnapshot、不写续费流水**。

这和计费文档中的规则冲突。`代理点数计费与返点规则.md` 明确把授权扣点列为代理开通授权的前置财务规则，要求存在项目授权、项目定价、授权设备数、有效期和可用点数等前置条件。

### 修复要求

建议短期强制禁止代理通过普通 PATCH 修改 `valid_until`：

```python
if admin is None and agent is not None:
    if "valid_until" in body.model_fields_set:
        raise HTTPException(
            status_code=status.HTTP_403_FORBIDDEN,
            detail="代理不能直接修改授权到期时间，请走续费/升级专用接口",
        )
```

长期应新增“续费专用接口”，走扣点快照与账务流水。

## 4. P1 问题：代理升级授权时没有重新检查代理项目授权是否仍有效

### 状态：已确认，P1

`preview_authorization_upgrade()` 和 `upgrade_authorization()` 都只做：

```text
_get_user_or_404
_assert_can_access_user
_get_authorization_or_404
_get_project_or_404
```

没有看到对当前代理的 `AgentProjectAuth` 做：

```text
status == active
valid_until 未过期
```

`upgrade_authorization()` 中 Agent 升级设备数会扣点，但前置权限只依赖“这个用户是代理创建的”，而不是“代理当前仍拥有该项目授权”。

### 风险

```text
代理过去拥有项目 A 授权
↓
代理给用户开通过项目 A
↓
后来代理项目授权到期/被停用
↓
代理仍可能继续给该用户升级项目 A 设备数
```

这会造成项目授权边界和后续升级链路不一致。

### 修复要求

在以下函数中统一调用“代理项目授权有效性检查”：

```text
grant_authorization()
update_authorization()  # 如果仍允许代理增加设备
preview_authorization_upgrade()
upgrade_authorization()
```

## 5. P1 问题：升级与更新入口职责重叠

### 状态：已确认，P1

当前存在两条代理增加设备数路径：

```text
PATCH /api/users/{user_id}/authorizations/{auth_id}
POST  /api/users/{user_id}/authorizations/{auth_id}/upgrade
```

`update_authorization()` 里如果 Agent 增加 `authorized_devices`，会计算并扣点。
同时 `upgrade_authorization()` 也是专用升级设备数接口，支持 `append / average` 两种模式。

Knowledge 文档《授权升级续费与均分方案》明确把升级作为专用端点处理：

```text
POST /api/users/{user_id}/authorizations/{auth_id}/upgrade
```

并区分追加模式和平均模式。

### 风险

两条路径都能“增加设备数”，容易出现：

```text
计费规则不一致
前端误用 PATCH
测试覆盖分散
后续续费/升级规则难以维护
```

### 建议

短期建议：

```text
Agent 禁止通过 PATCH 修改 authorized_devices。
Agent 只能走 /upgrade。
Admin 可以继续用 PATCH 直接改。
```

也就是：

```python
if admin is None and agent is not None:
    if body.authorized_devices is not None:
        raise HTTPException(
            status_code=403,
            detail="代理不能直接修改授权设备数，请走升级专用接口",
        )
```

除非你明确决定保留 PATCH 作为“简化升级入口”，否则不建议双轨。
