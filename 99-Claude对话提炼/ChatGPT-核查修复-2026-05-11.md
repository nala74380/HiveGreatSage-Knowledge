## 1. 运维部署核查

### 1.1 Nginx 与 API service 监听方式仍不匹配

**状态：未修复，P0。**

当前 `deploy/nginx.conf` 仍然写：

```nginx
location /api/ {
    proxy_pass http://unix:/var/run/hive-verify.sock;
}
```

也就是 Nginx 期待 API 监听 Unix socket。

但 `deploy/systemd/hive-verify-api.service` 当前 `ExecStart` 使用 Gunicorn，并绑定：

```text
--bind 0.0.0.0:8000
```

这是 TCP 8000，不是 `/var/run/hive-verify.sock`。

**结论：**
如果按当前两个文件直接部署，`/api/` 很可能 502。

**必须二选一：**

```text
方案 A：nginx 改为 proxy_pass http://127.0.0.1:8000;
方案 B：systemd 改为 --bind unix:/var/run/hive-verify.sock
```

---

### 1.2 service 使用 Gunicorn，但 requirements 缺 gunicorn

**状态：未修复，P0。**

`hive-verify-api.service` 的 `ExecStart` 使用：

```text
/opt/hive-verify/venv/bin/gunicorn
```



但 `requirements.txt` 当前只有：


没有 `gunicorn`。

**结论：**
如果部署机只执行 `pip install -r requirements.txt`，API service 可能启动失败。

**必须二选一：**

```text
方案 A：requirements.txt 增加 gunicorn
方案 B：systemd 改回 uvicorn ExecStart
```

### 1.3 nginx.conf 仍缺 `/admin/api/` 代理

**状态：未修复，P0。**

当前 `nginx.conf` 只有：

```text
location /api/
location /admin/
location /updates/
```

没有：

```text
location /admin/api/
```



但 Verify 的管理端接口大量使用 `/admin/api/*`。如果 Nginx 只配置 `/admin/` 静态 alias，`/admin/api/auth/login` 很可能被静态前端接管，导致管理后台登录失败。

**结论：未闭合。**

建议加在 `/admin/` 前面：

```nginx
location /admin/api/ {
    proxy_pass http://127.0.0.1:8000;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

如果最终选 Unix socket，就同步换成 socket。
