## 3. 本轮确认的风险问题

### P0 / P1：账务幂等键仍不能防“业务级重复扣点”

**状态：已确认风险。**

当前授权扣点账本的 idempotency key 是：

```text
authorization:{authorization_id}:{snapshot.id}:charged
authorization:{authorization_id}:{snapshot.id}:credit
```

这个设计能避免同一条 ledger key 重复，但因为 `snapshot.id` 每次扣点都会新生成，所以它**不能防止同一个业务请求被重复执行后生成新的 snapshot 并再次扣点**。

也就是说，它是“账本行唯一键”，不是“业务请求幂等键”。

风险场景：

```text
代理点击授权升级
请求超时
前端重试
后端再次生成新的 AuthorizationChargeSnapshot
再次扣点
```

**建议修复：**

引入业务级幂等键，例如：

```text
authorization_create:{authorization_id}
authorization_upgrade:{authorization_id}:{request_id}
authorization_renew:{authorization_id}:{request_id}
delete_refund:{user_id}:{snapshot_id}
```

或者要求前端/调用方传 `Idempotency-Key`，后端落到 `AccountingDocument` 或独立幂等表。

---

### P1：钱包 `status / risk_status` 存在，但写操作没有硬约束

`AccountingWallet` 模型有：

```text
status: active / locked / closed
risk_status: normal / watch / restricted / frozen
```

并且数据库层有 check constraint。

但本轮核查到的充值、授信、冻结、解冻、扣点路径中，没有看到统一的：

```text
wallet.status == active
wallet.risk_status 不为 frozen / restricted
```

校验。

这意味着当前 `wallet.status` 和 `risk_status` 更像展示字段，而不是业务硬约束。

风险场景：

```text
钱包已 locked / closed，仍可能被充值、授信、扣点。
钱包 risk_status=frozen，仍可能继续给用户授权扣点。
```

**建议修复：**

增加统一 helper：

```python
_assert_wallet_can_mutate(wallet, operation)
```

按操作区分：

```text
recharge：locked 是否允许，需定规则
credit：locked 是否允许，需定规则
consume：locked / closed / frozen 必须禁止
refund：closed 是否允许系统返点，需定规则
freeze / unfreeze：仅 active/locked 是否允许，需定规则
```

### P1：对账可以发现差异，但缺少“差异修复闭环”

当前对账服务能生成 `AccountingReconciliationRun` 和 `AccountingReconciliationItem`，能标记 `normal / abnormal`，并记录差异详情。

但我没有看到：

```text
异常项转调账申请
异常项标记 pending_review / fixed
异常项生成 reversal / manual_adjust
差异修复审批流
```

虽然模型里 `AccountingReconciliationItem.status` 支持 `pending_review / fixed`，但服务层目前主要是生成和查询。 

**建议：**

后续至少补一个闭环：

```text
abnormal item
→ 创建 AccountingAdjustmentRequest
→ 审批
→ posted manual_adjust / reversal document
→ item.status = fixed
```
