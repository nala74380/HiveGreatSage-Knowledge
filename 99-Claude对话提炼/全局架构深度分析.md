## 三、安卓脚本 — 完整登录流程

### 登录后需要做什么（14步流程）

```
Step 1  显示启动 UI（账号/密码/IP/随机延时选项）
Step 2  校验输入
Step 3  分辨率适配（Screen.init）
Step 4  代理/VPN 启动（如需）
Step 5  OCR 初始化（TomatoOCR）
Step 6  网络验证登录（Verify.login → POST /api/auth/login）
Step 7  写入登录时间戳
Step 8  热更新检查（Updater.check_and_update）
Step 9  释放图片资源（Finder.extract_assets）
Step 10 拉取云端参数（CloudSync.pull_params → GET /api/params/get）
Step 11 启动心跳（Heartbeat.start → 每30秒 POST /api/device/heartbeat）
Step 12 连接 PC 中控（CommLan.connect，可选）
Step 13 看门狗初始化（ErrorHandler.init）
Step 14 随机延时（游戏前，模拟真实人类操作）
Step 15 调度器 + 游戏主逻辑（game_main.run）
```

### 二次登录免填密码
- ✅ 已实现：登录时保存 username，密码加密（XOR+Base64，密钥=设备指纹）存 KV
- ✅ 二次启动时 UI 预填账号，密码框显示 "●●●●●●"（已保存）
- ✅ `ensure_token()` 自动用保存的账号密码重新登录

### 密码加密方案
- 算法：XOR + Base64，密钥由设备指纹派生（不同设备加密结果不同）
- 目的：防止本地文件 hive_kv.json 明文泄露，不是强加密
- 实现：`framework/crypto.lua`

### 延时位置调整
- **修改前**：在 Step 3（验证前），意义不大
- **修改后**：在 Step 14（游戏前），模拟真实人类从登录到开始操作的时间差
  - 效果：不同设备随机延时（1-60秒），避免所有设备同时开始操作被检测

## 四、PC 中控 — 显示设计原则

### 设备列表列设计（重新规划）

**去掉序列号**（20字符，信息密度低，用户看不懂，占宝贵列宽）

**推荐列设计**：

```
复选 | 编号 | 角色 | 状态 | 激活 | 等级 | 战力 | 区服 | 当前任务 | 心跳 | 备注
  36 |  80  |  70  |  80  |  65  |  60  |  90  |  55  |  stretch |  90  | 110
```

**数据来源映射**：
- 编号  → `device_meta.json` alias（本地）
- 角色  → `device_meta.json` role（本地）
- 状态  → Redis 心跳 status 字段（running/idle/error/offline）
- 激活  → `device_meta.json` activated（本地）
- 等级  → `game_data.level`（云端，安卓上报）
- 战力  → `game_data.combat_power`（云端，安卓上报）
- 区服  → `game_data.server`（云端，安卓上报）
- 当前任务 → `game_data.current_task`（云端，安卓上报）
- 心跳  → `last_seen`（云端，Redis 或 DB）
- 备注  → `device_meta.json` note（本地）

### 全局设置重构方向

**原来**：全局设置弹窗里有"脚本参数"（这是每台设备配置，不应在全局）

**正确分层**：

```
右键菜单 → 编辑设备
  ├── 基本信息（只读）：指纹/状态/心跳
  ├── 设备标签（可编辑）：编号/角色/备注
  ├── 脚本参数（此设备专属）：从云端 /api/params/get 加载
  └── 游戏账号（此设备关联的游戏账号）

全局设置（⚙）
  ├── 服务器连接（API地址/UUID）
  ├── 同步设置（刷新间隔）
  ├── 游戏账号数据库（连接设置）
  ├── ADB 设置
  ├── 热更新设置
  └── 日志级别
```

### 问题 4.1 — 游戏账号数据库

**需求**：中控需要为每台设备（或每个账号）配置游戏账号信息（游戏账号名、密码、角色等）。

**两种模式**：

**模式 A：本地 SQLite 数据库**
```python
# config/game_accounts.db
# 表结构：
CREATE TABLE game_account (
    id          INTEGER PRIMARY KEY,
    game_name   TEXT,         -- 游戏账号名
    game_pass   TEXT,         -- 加密存储
    server      TEXT,         -- 所在区服
    role_name   TEXT,         -- 角色名
    device_fp   TEXT,         -- 绑定的设备指纹（可多对一）
    level       INTEGER,      -- 等级（手动填写或从游戏同步）
    notes       TEXT,
    created_at  DATETIME
);
```
- 优点：本地管理，无网络依赖
- 缺点：每台 PC 各自维护，不便于协同

**模式 B：云端 Verify 系统存储**
```
新增 game_account 表（hive_game_{code} 库）
用户在 PC 中控填写，POST /api/game/accounts/ 上传云端
安卓脚本从 /api/params/get 下发中获取游戏账号
```
- 优点：统一管理，多台 PC 共享
- 缺点：游戏账号敏感，需加密传输

**建议**：Phase 2 先用模式 A（本地 SQLite），Phase 3 迁移到模式 B（云端加密存储）。

## 五、自动化目标 — 各系统能力边界与协作

### 每个系统的能力边界

```
安卓脚本（Lua/懒人精灵）
  能做：
    ✓ 手机屏幕操作（点击/滑动/截图/找图/OCR）
    ✓ 游戏内数据采集（通过 UI 读取/截图识别）
    ✓ 单设备游戏自动化（日常/副本/采集等）
    ✓ 定时上报数据到云端
    ✓ 接收 PC 中控指令（WebSocket）
  不能做：
    ✗ 复杂计算（Lua 性能有限）
    ✗ 跨设备协调（单个脚本只知道自己的状态）
    ✗ 访问本地文件系统外的数据库
    ✗ 运行 Python/Java 等高级语言代码

PC 中控（Python/PySide6）
  能做：
    ✓ ADB 指令（激活、TCP/IP 模式、截图）
    ✓ 协调多台安卓设备（通过 WebSocket 下发指令）
    ✓ 本地数据库（SQLite）读写
    ✓ 复杂计算（Python）
    ✓ 可视化监控（设备状态表格）
    ✓ 文件操作（下载/解压/安装 APK）
  不能做：
    ✗ 跨互联网同步（依赖 Verify 云端）
    ✗ 跨地区多 PC 协调（需要云端中继）
    ✗ 7×24 小时稳定运行（PC 会关机、网络不稳定）

网络验证系统（Python/FastAPI/PostgreSQL）
  能做：
    ✓ 全平台数据持久化（所有设备、用户、参数）
    ✓ 跨设备状态共享（Redis 实时状态）
    ✓ 定时任务（Celery Beat）
    ✓ 跨 PC 协调（多个 PC 中控都连接同一个 Verify）
    ✓ 7×24 稳定运行（服务器持续在线）
    ✓ 数据分析和报表
  不能做：
    ✗ 直接操控安卓设备（无 ADB）
    ✗ 实时低延迟操作（HTTP 轮询有延迟）

其他工具（Python/ADB/OpenCV）
  能做：
    ✓ 批量 ADB 操作（不需要打开 PC 中控 GUI）
    ✓ CV 图片处理（模板提取、坐标标注）
    ✓ 自动化脚本（cron 定时执行）
    ✓ 数据导入导出（Excel/CSV）
  不能做：
    ✗ 实时监控（无 UI）
    ✗ 用户交互（命令行工具）
```

### 任务分工矩阵

```
任务类型                 | 执行者          | 触发者          | 数据存储
─────────────────────────|────────────────|────────────────|──────────────────
游戏内操作               | 安卓脚本        | 安卓脚本自主     | Redis→云端DB
采集游戏数据             | 安卓脚本        | 安卓脚本自主     | POST /api/device/heartbeat
激活设备                 | PC中控（ADB）   | 用户手动触发     | device_binding.imsi
协调组队（同LAN）         | PC中控（WS服务）| PC中控          | 本地（不上云）
协调组队（跨LAN/跨PC）   | Verify云端调度   | Celery任务      | PostgreSQL
参数下发                 | Verify           | PC中控修改保存  | PostgreSQL→安卓拉取
热更新                   | Verify存储+下发 | 管理员上传       | 本地文件/S3
价格监控                 | 安卓脚本采集     | 脚本自动触发     | POST /api/market/price
供需匹配                 | Verify算法       | 用户发单        | PostgreSQL
批量ADB操作              | Tools脚本        | 管理员手动       | 本地日志
CV图片处理               | Tools脚本        | 开发时手动       | 本地文件
```

## 六、当前最重要的欠缺项（按优先级）

### P0 — 阻塞完整自动化

1. **设备激活状态统计**（`device_binding` 缺 `game_project_id`）
   - 现在无法区分"哪个项目的设备激活了"
   - 迁移：加 `game_project_id` 字段

2. **游戏账号数据库**（PC中控侧）
   - 没有地方管理"游戏账号名/密码/角色"
   - 目前脚本参数只能配置游戏逻辑参数，不含账号信息

3. **安卓脚本 game_data 字段标准化**
   - 心跳上报的 `game_data` 是自由 JSONB
   - 需要定义标准字段：`level / combat_power / server / current_task / game_account`
   - 没有标准化，PC中控无法通用地显示这些数据

4. **PC中控设备列表优化**
   - 序列号列占位却无用
   - 游戏数据字段（等级/战力/区服/当前任务）未显示

### P1 — 影响运营效率

5. **批量设备操作 API**
   - 目前"批量设置"是本地操作，不能通过云端批量推送参数变更
   - 需要：`POST /admin/api/batch/params` → 向多个用户推送参数

6. **任务状态回传**
   - 安卓脚本执行完某个任务后，应回传任务状态
   - 目前 `game_data.current_task` 是心跳上报的，延迟 30 秒
   - 需要：任务完成时立即 POST 到 `/api/device/event`（事件推送，不等心跳）

7. **PC中控账号管理面板**
   - 右键菜单"编辑设备"应包含脚本参数和游戏账号（目前脚本参数在全局设置里）

### P2 — 长期完善

8. **跨设备任务协调 API**（Verify 云端）
   - `/api/task/demand` 需求发布
   - `/api/task/assignment` 任务分配
   - 安卓脚本主动查询："有没有任务需要我做？"

9. **官网监控 → 自动触发脚本更新**
   - 游戏更新公告 → 自动停止脚本 → 等待新版本 → 自动恢复

10. **PC中控"无头模式"**
    - PC 可以在无显示器的服务器上运行（只有 ADB 管理功能，无 GUI）
    - 用于纯服务器环境部署
