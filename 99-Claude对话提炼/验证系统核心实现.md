# 2026-04-22 验证系统核心实现

## 对话目标

完成 HiveGreatSage-Verify 的核心功能实现，打通从登录 → 心跳上报 → Redis 缓冲 → Celery 批量落库的完整数据流。

---

## 关键结论

### 1. Windows 开发环境 asyncpg SSL 问题（影响所有 create_async_engine 调用）

**症状**：`ConnectionRefusedError: [WinError 1225]` 或 `socket.gaierror: [Errno 11001] getaddrinfo failed`

**根因**：asyncpg 默认尝试 SSL 握手，开发环境 PostgreSQL 未启用 SSL，握手被拒绝；`localhost` 在部分 Windows 网络栈下 DNS 解析失败。

**修复**：
- 所有 `create_async_engine` 调用加入 `connect_args={"ssl": False}`
- `.env` 中所有 `localhost` 改为 `127.0.0.1`（PostgreSQL 端口 15432，Redis 端口 16379）
- 影响文件：`app/database.py`、`scripts/init_data.py`

### 2. Celery 任务必须使用 NullPool（解决 Event loop is closed）

**症状**：第二次任务执行时报 `RuntimeError: Event loop is closed` 和 `'NoneType' object has no attribute 'send'`

**根因**：`asyncio.run()` 每次执行后**销毁事件循环**。SQLAlchemy 的 QueuePool 缓存了绑定到旧事件循环的 asyncpg 连接，下次任务执行时 `pool_pre_ping` 尝试复用旧连接，触发异常。

**修复**：Celery 任务内所有 `create_async_engine` 调用使用 `poolclass=NullPool`，每次查询新建连接、用完立即关闭。Redis 客户端同样在任务内按需创建（`aioredis.Redis.from_url()`），不复用模块级连接池。

```python
from sqlalchemy.pool import NullPool

def _make_task_engine(url):
    return create_async_engine(
        url,
        poolclass=NullPool,
        connect_args={"ssl": False},
    )
```

### 3. DATABASE_GAME_PREFIX 拼接导致库名重复

**症状**：`database "hive_game_game_001" does not exist`

**根因**：`.env` 中 `DATABASE_GAME_PREFIX` 以 `hive_game_` 结尾，拼接 `code_name="game_001"` 后变成 `hive_game_game_001`，实际库名是 `hive_game_001`。

**修复**：凡需要构造游戏库 URL 的地方，改用 `make_url(DATABASE_MAIN_URL).set(database=project.db_name)`，用 GameProject 表中的 `db_name` 字段（`hive_game_001`）替换，而非拼接 `code_name`。影响文件：`scripts/init_data.py`、`tasks/heartbeat_flush.py`。

### 4. 认证模块架构修复（选项 A）

**问题**：`routers/auth.py` 承载全部业务逻辑（违反三层分离架构），`refresh` 接口用 SCAN 全表扫描。

**修复**：
- 新建 `services/auth_service.py`，迁移全部业务逻辑
- `routers/auth.py` 精简为薄路由层（只做参数提取 + 限流 + 调用 service）
- 新建 `core/dependencies.py`，`get_current_user` 依赖独立管理
- Refresh Token 新增反查索引（`rt_lookup:{sha256(rt)[:32]}` → `{user_id}:{jti}`），从 SCAN O(n) 变为 O(1) 查找
- 登录失败日志使用独立 Session 写入，防止主事务回滚导致审计数据丢失

### 5. 游戏库模型与 Alembic 隔离

**设计**：`GameBase` 与主库 `Base` 完全独立（两个不同的 `DeclarativeBase`），`migrations/env.py` 只注册 `Base.metadata`，游戏表不经过 Alembic。

**建表方式**：`GameBase.metadata.create_all(bind=engine)`，在 `scripts/init_data.py` 和 `/admin/api/projects/{id}/init-db` 接口中调用。

### 6. 心跳落库说明（total_devices=0 属正常）

Celery 任务只落库 Redis 中 TTL 未过期的数据（TTL=120s）。如果心跳是在 Celery 启动前上报的，Redis key 可能在 Celery 首次运行前已过期，`total_devices=0` 是正常行为。正确测试流程：先启动 Celery Worker + Beat，再上报心跳。
