# 用户管理页面推理、风险项与改进计划 V1.2 完整合并修正版

> 本文用于替代 V1.0 与 V1.1 单独版本。
>
> V1.0 内容完整，但早期交互原型中曾把新增设备、续费、升级等级、停用冻结、启用恢复等复杂动作直接放到超级列表行操作列。
>
> V1.1 修正了“超级列表按钮保持原样，复杂功能放入编辑抽屉”的方向，但文档过度简化，并且 Vue 原型里出现了新的设计失衡：**管理员编辑抽屉没有代理端完善**。
>
> V1.2 的结论是：
>
> ```text
> 1. 超级列表操作按钮保持现有结构，不堆复杂功能。
> 2. 新增或优化功能统一进入“编辑”按钮后的编辑抽屉。
> 3. 管理员编辑抽屉必须比代理端更完整，而不是更弱。
> 4. 代理端编辑抽屉是管理员端能力的受限子集。
> 5. 保留 V1.0 的完整风险项、推理链路和改进计划。
> ```

# 0. V1.2 关键修正

## 0.1 超级列表不做按钮堆叠

最终口径：

```text
用户超级列表的行操作按钮保持原样。
```

管理员端行操作建议保持：

```text
详情
编辑
停用 / 启用
删除
```

代理端行操作建议保持：

```text
详情
编辑
停用 / 启用
```

不建议在超级列表行操作列直接增加：

```text
授权新项目
新增设备
老设备续费
升级等级
停用并冻结权益
启用并恢复权益
查看扣点
查看冻结权益
```

这些动作需要项目上下文、扣点预览、冻结状态、设备状态和账务信息，应该进入编辑抽屉。

```text

9. 老设备续费

12. 启用授权
13. 停用并冻结权益
14. 启用并恢复权益
15. 删除用户并清算返点
16. 批量删除并清算返点
17. 设备绑定查看
18. 设备解绑
19. 扣点快照
20. 返点记录
```

# 4. V1.2 新增风险项

## R001：管理员编辑抽屉弱于代理端

```text
问题：
V1.1 原型中管理员编辑抽屉没有代理端完善。

风险：
管理员端作为治理面板，反而无法完整管理授权、设备、账务、冻结、审计。

修正：
管理员端编辑抽屉必须包含代理端所有授权动作，并额外包含设备绑定、账务清算、审计、创建代理治理、删除清算等能力。
```

## R002：超级列表按钮堆叠风险

```text
问题：
早期原型把新增设备、续费、升级等级等按钮放到列表行。

风险：
页面拥挤、上下文不足、误操作概率高。

修正：
复杂动作进入编辑抽屉。
```

## R003：Agent 删除用户返点风险

```text
业务口径：
管理员删除用户账户时返点。
Agent 不应删除用户账户并触发返点。

建议：
后端 soft_delete_user 对 Agent 返回 403。
```

## R004：停用 / 启用 / 冻结权益模型缺失

```text
问题：
当前只支持 Authorization.status=suspended。
缺少冻结权益、启用恢复、删除清算的状态机。

建议：
新增 AuthorizationFreezeRecord。
```

# 三、已确认业务口径

## 3.1 User 与 Authorization 的边界

已确认当前系统口径：

```text
User 是账号主体。
Authorization 是“用户 × 项目”的授权记录。
项目等级、授权设备数、项目到期时间全部归属 Authorization。
```

这意味着：

```text
1. 用户本身不再持有全局等级。
2. 同一用户可在 A 项目是 normal，在 B 项目是 vip。
3. 同一用户在不同项目可以有不同设备数和不同到期时间。
4. 用户数量只作为统计展示，不再作为代理配额硬约束。
5. 代理商业约束由项目准入、项目授权、授权扣点、点数余额和风险状态控制。
```

---

## 3.2 管理员端用户管理口径

管理员端应允许：

```text
1. 查看所有用户。
2. 创建用户。
3. 编辑用户基础状态。
4. 修改用户密码。
5. 自动生成用户密码，并一次性查看明文。
6. 给用户授权新项目。
7. 管理调整项目授权。
8. 停用项目授权。
9. 删除用户账户。
10. 批量删除用户。
11. 查看设备绑定。
12. 解绑设备。
13. 查看代理创建者详情。
14. 查看授权扣点与返点记录。
15. 查看冻结权益记录（待新增）。
```

管理员端授权调整原则：

```text
1. 管理员操作默认不扣代理点数。
2. 管理员修改授权必须写 audit_log。
3. 是否写 0 点 AccountingDocument，暂定为后续决策项。
4. 管理员删除用户账户时，才触发删除用户返点清算。
```

## 3.3 代理端用户管理口径

代理端应允许：

```text
1. 查看自己创建的用户。
2. 创建用户。
3. 编辑自己用户状态。
4. 修改自己用户密码。
5. 给用户授权新项目。
6. 编辑项目授权，但不能免费绕过扣点。
7. 停用项目授权。
8. 启用自己停用且存在冻结权益的项目授权。
9. 新增设备。
10. 老设备续费。
11. 升级项目等级。
12. 查看自己的余额、流水、扣点、返点说明。
13. 查看自己可售卖的项目目录。
```

代理端不应允许：

```text
1. 批量删除用户。
2. 触发删除用户返点。
3. 删除用户账户并清算返点。
4. 给用户设置 tester 等级。
5. 设置无限设备。
6. 设置永久授权。
7. 绕过扣点直接延长到期时间。
8. 绕过扣点直接增加设备数。
9. 绕过扣点直接升级等级。
10. 使用已过期或未授权项目给用户授权。
```

## 3.4 Agent 停用项目授权规则

已确认新业务规则：

```text
Agent 停用项目授权：
  1. 不进行返点。
  2. 不把点数退回代理余额。
  3. 需要冻结该授权对应的剩余权益 / 剩余价值。
  4. 停用后不再继续消耗授权权益。
  5. 停用后用户不能继续使用该项目授权。
  6. 后台管理员删除对应账户时，才按删除用户返点规则进行清算。
```

该动作不是退款，也不是删除，而是：

```text
停用项目授权 + 冻结剩余权益 + 停止继续消耗权益。
```

---

## 3.5 Agent 启用项目授权规则

已确认采用方案 A：

```text
Agent 启用已停用授权 = 恢复被冻结的剩余权益，不再次扣点。
```

推导规则：

```text
1. 停用时记录剩余权益，例如 remaining_hours。
2. 启用时恢复剩余权益。
3. 启用不扣代理点数。
4. 启用后新的 valid_until = 启用时间 + frozen_remaining_hours。
5. 启用前必须校验代理仍拥有该项目有效授权。
6. 启用前必须校验用户、项目、代理状态仍允许继续服务。
```

---

## 3.6 Admin 删除用户账户返点规则

已确认业务口径：

```text
后台管理员删除对应账户时进行返点。
Agent 停用项目授权不返点。
Agent 不应通过删除用户触发返点。
```

建议最终收口为：

```text
Agent 不删除用户账户，只能停用用户或停用项目授权。
Admin 删除用户账户，才触发删除用户返点清算。
```

```text
agent_suspend_user()
```

# 七、完整风险项清单

## 7.1 P0 / P1 高优先级风险

### R001：Agent 批量删除权限未收口

## 10.1 已确认

```text
1. 用户管理页面主方向正确：User 主体与 Authorization 项目授权已经分离。
```

```text
  - 用户名只读
```

```text
7. 代理不能批量删除。
```

# 四、对 V1.0 风险项的修正

## 4.1 不再将“Agent 可见编辑授权”本身视为错误

V1.0 中曾将“Agent 看到编辑授权”视为高风险。根据最新业务口径，修正为：

```text
Agent 可以编辑项目授权。
但 Agent 的编辑不能是普通字段直改。
Agent 的编辑应进入“授权动作面板”。
```

因此风险从：

```text
Agent 可见编辑按钮
```

修正为：

```text
Agent 编辑项目授权的功能语义需要拆分。
```

---

## 4.2 不再建议在列表行操作列新增动作

V1.0 原型中曾将新增设备、续费、升级等级等按钮直接放在用户行操作列。根据最新反馈修正：

```text
不在超级列表行操作列新增这些按钮。
它们全部收纳到“编辑”抽屉内。
```
