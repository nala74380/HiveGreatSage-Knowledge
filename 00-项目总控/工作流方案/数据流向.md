本文从以下文档拆出：


本文只负责说明项目工作流中"信息、文档、代码、图、对外资料"的流向边界，不负责说明工具职责细节；工具职责见 [[00-项目总控/工作流方案/工具分工]]。

---

## 一、核心原则

```text
数据只从上往下流。
真相源单一。
不允许多个位置各存一半。
不允许把 AI 对话当长期存储。
```

项目工作流中的两个核心真相源：

```text
Obsidian = 知识真相源
GitHub = 代码真相源
```

## 三、各流向说明

### 3.1 想法 → AI Project

用途：

```text
1. 把模糊问题变成结构化问题。
2. 推演架构方案。
3. 识别风险。
4. 生成草稿或候选方案。
```

限制：

```text
AI Project 不是事实源。
AI 输出必须经过人工判断和必要的代码/文档验证。
```

---

### 3.2 AI Project → Obsidian

用途：

```text
1. 把对话结论沉淀为项目知识。
2. 把已确认决策写入决策日志。
3. 把未确认问题写入待决策清单。
4. 把风险写入风险清单或治理清单。
```

禁止：

```text
1. 把完整 AI 对话原文当作主文档。
2. 把未验证推断写成已确认事实。
3. 让结论只停留在对话窗口里。
```

---

### 3.3 Obsidian → Cursor / PyCharm

用途：

```text
1. 以 Obsidian 中的架构、接口契约、规范指导编码。
2. 开发前先读当前模块文档。
3. 代码实现后反向更新实现状态、测试基线、追踪矩阵。
```

限制：

```text
编辑器不是知识库。
不要在代码仓库里长期维护项目治理文档。
```

---

### 3.4 Cursor / PyCharm → GitHub

用途：

```text
1. 代码通过 commit + push 进入 GitHub。
2. GitHub 记录变更历史。
3. 代码状态以 GitHub 最新分支为准。
```

要求：

```text
1. 提交前看 diff。
2. Commit message 写清楚变更范围。
3. 业务变更需能追溯到文档、决策或任务。
4. 涉及实现状态的文档，必须精读对应代码后再更新。
```

---

### 3.5 Obsidian → draw.io / Excalidraw

用途：

```text
1. Obsidian 文档中产生画图需求。
2. Excalidraw 用于推演草图。
3. draw.io 用于正式架构图、ER 图、时序图。
4. 图导出后嵌入或链接回 Obsidian。
```

限制：

```text
图不能脱离文档单独成为事实源。
图一旦过时，必须在对应文档中标记。
```

---

### 3.6 Obsidian → Google Drive

用途：

```text
1. 对外共享。
2. 用户手册、产品说明、交付材料。
3. 定稿文档的镜像。
```

禁止反向编辑：

```text
Google Drive 修改 → 再搬回 Obsidian
```

除非做一次明确的受控迁移，并记录来源、目标和差异。

## 四、禁止的数据流

### 4.1 Google Drive 反向覆盖 Obsidian

```text
禁止：Google Drive 里改了文档，再手工覆盖 Obsidian。
```

风险：

```text
1. 两边版本不一致。
2. Obsidian 真相源被破坏。
3. 后续不知道哪个版本是最新。
```

---

### 4.2 AI 对话长期替代 Obsidian

```text
禁止：把 Claude / ChatGPT 对话窗口当长期记忆。
```

风险：

```text
1. 新窗口无法继承完整上下文。
2. 旧结论难以搜索。
3. 容易重复讨论同一问题。
4. 容易把 AI 推断误认为项目事实。
```

---

### 4.3 编辑器内写主文档

```text
禁止：在代码仓库中散落项目治理文档，绕过 Obsidian。
```

例外：

```text
1. 代码仓库自身 README。
2. 开发者运行说明。
3. 与代码强绑定的注释或接口说明。
```

长期架构、决策、规范必须回到 Obsidian。

---

### 4.4 多工具同时编辑同一事实

```text
禁止：Obsidian、Google Drive、AI 对话、代码注释各写一套不同结论。
```

治理原则：

```text
1. 一个事实只能有一个主真相源。
2. 其他位置只能引用、镜像或摘要。
3. 若发现冲突，必须标记来源并回到真相源修正。
```

---

## 五、数据流向检查清单

提交或迁移前检查：

```text
□ 当前内容属于知识、代码、图、对外资料，还是临时推演？
□ 是否放在正确的真相源？
□ 是否存在另一个位置维护同一结论？
□ 是否需要从 AI 对话提炼到 Obsidian？
□ 是否需要从 Obsidian 同步到 GitHub 代码实现？
□ 是否需要从 GitHub 代码状态反向更新 Obsidian 追踪矩阵？
□ 是否有 Google Drive 反向覆盖 Obsidian 的风险？
```

## 六、当前治理结论

已确认：

```text
1. Obsidian 是非代码内容真相源。
2. GitHub 是代码和版本真相源。
3. AI Project 是推演区，不是存储区。
4. Google Drive 是镜像层，不是真相源。
5. 拆分文档时，新文档必须保留来源说明并回写治理清单。
```

待确认：

```text
1. 是否需要单独建立 Google Drive 镜像目录规范。
2. 是否需要把对外文档交付流程单独拆成一份文档。
3. 是否需要把图文件存放规范拆入各子系统目录。
```
