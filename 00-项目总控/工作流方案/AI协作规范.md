## 一、当前定位

AI Project，包括 Claude Project、ChatGPT Project、Cursor Chat 等，是项目的：

```text
推演区
审查区
草稿区
排错辅助区
```

它不是：

```text
知识真相源
代码真相源
长期存储区
最终事实来源
```

核心原则：

```text
AI 负责辅助推演，不负责替你确认事实。
AI 输出必须进入 Obsidian 或 GitHub 后，才成为可追踪项目资产。
```

---

## 二、AI 适合负责什么

AI 适合承担：

```text
1. 架构设计推演。
2. 模块边界审查。
3. 风险识别。
4. 文档草稿生成。
5. 代码框架草稿生成。
6. Bug 排查思路整理。
7. 接口、数据库、测试清单的结构化检查。
8. 对已有文档和代码做交叉复核。
```

AI 不适合直接承担：

```text
1. 最终事实确认。
2. 生产级代码自动合并。
3. 未读代码情况下断言实现状态。
4. 未运行测试情况下断言已通过。
5. 真实密钥、Token、授权码存储。
6. 长期项目记忆。
```

## 三、AI 对话窗口组织规则

窗口命名规则：

```text
[模块编号]-[主题]-[日期]
```

示例：

```text
02-PC中控-通信协议设计-0413
03-安卓脚本-OCR方案选型-0415
00-工作流-整体复盘-0420
01-Verify-热更新三端联调-0512
```

禁止：

```text
1. 一个窗口同时讨论多个无关模块。
2. 一个窗口长期堆积所有项目问题。
3. 不给上下文就让 AI 猜。
4. 把旧窗口中的结论当作自动可继承记忆。
5. 在对话中反复修补，不回写 Obsidian。
```

建议：

```text
1. 一个窗口只服务一个主题。
2. 跨模块问题可以开专门窗口，但结论要分别回写对应模块文档。
3. 长窗口结束后必须提炼结论。
4. 重要结论进入 Obsidian，不留在聊天窗口里。
```

## 四、新对话标准输入

每次开启新对话，应提供：

```text
1. 当前任务目标。
2. 当前范围。
3. 当前非目标。
4. 相关 Obsidian 文档摘要。
5. 相关 GitHub 代码路径或文件。
6. 已确认信息。
7. 待确认信息。
8. 希望 AI 执行的动作。
```

推荐开场模板：

```markdown
## 当前目标

## 范围

## 非目标

## 已确认信息

## 待确认信息

## 相关文档

## 相关代码

## 希望你做什么
```

## 五、给 AI 大局观的方法

AI 需要大局观，但不能让它凭空补全大局观。

推荐投喂顺序：

```text
1. 项目总大纲。
2. 当前子系统 README。
3. 当前子系统架构设计。
4. 当前任务相关接口契约或测试清单。
5. 当前问题的上下文摘要。
```

对于蜂巢·大圣项目，推荐优先投喂：

```text
00-项目总控/项目总大纲.md
00-项目总控/治理批次记录.md
00-项目总控/文档代码测试追踪矩阵.md
当前子系统 README.md
当前子系统核心真相源文档
```

## 六、AI 输出使用规则

AI 输出分四类处理：

| 类型 | 处理方式 |
|---|---|
| 已确认事实 | 必须有文档、代码、测试或用户明确输入支撑 |
| 推断 | 必须标记"推断"，不能写成事实 |
| 建议方案 | 必须标记"建议方案"，不能直接当决策 |
| 未知 / 待确认 | 必须保留不确定性，进入待确认清单 |

禁止：

```text
1. 把 AI 猜测写成已确认事实。
2. 把源码存在写成端到端已闭环。
3. 把未运行测试写成测试已通过。
4. 为了让回答完整而补不存在的文件、接口、字段。
5. 为了迎合用户而放弃边界判断。
```

## 七、AI 安全风险识别

协作中重点识别以下风险。

### 7.1 hallucination / 幻觉

表现：

```text
AI 编造不存在的文件、接口、函数、字段、历史结论。
```

处理：

```text
必须回到 GitHub 或 Obsidian 精读验证。
```

### 7.2 confabulation / 编造式补全

表现：

```text
信息不完整时，AI 把碎片补成完整故事。
```

处理：

```text
拆成已确认、推断、待确认三类，不允许完整化叙事。
```

### 7.3 sycophancy / 迎合

表现：

```text
用户提出不合理设计时，AI 为了显得有帮助而附和。
```

处理：

```text
必须优先指出边界、风险和替代方案。
```

### 7.4 specification gaming / 规格投机

表现：

```text
表面满足文字要求，但没有解决真实维护问题。
```

处理：

```text
回到项目目标：长期维护、清晰边界、可验证、可回退。
```

### 7.5 deception / 欺骗性表达

表现：

```text
未完成说成已完成，未验证说成已验证。
```

处理：

```text
严格使用：已确认 / 待确认 / 推断 / 建议方案。
```

## 八、AI 对代码任务的要求

涉及代码时，AI 必须遵守：

```text
1. 先明确目标、范围、非目标。
2. 先读对应文档和代码。
3. 不知道的路径必须搜索或标记未知。
4. 不直接大改核心结构，除非已有重构计划。
5. 不在旧垃圾结构上继续堆补丁。
6. 修改后必须说明测试方式和回退方式。
```

对蜂巢·大圣当前治理阶段，尤其要求：

```text
1. 先治理文档真相源。
2. 再对照代码。
3. 再进入重构或实现。
4. 不把文档治理和代码实现混在同一结论里。
```

---

## 九、AI 对文档任务的要求

涉及文档治理时，AI 必须遵守：

```text
1. 该删除的删除。
2. 该合并的合并。
3. 需要拆分的先建立对应文件夹，再逐步拆分。
4. 每次治理必须回写治理清理清单。
5. 原文档在拆分完成前保留为主入口。
6. 拆分完成后，原文档改为导航页或过时跳转页。
```

文档处置必须说明：

```text
1. 来源文件。
2. 目标文件。
3. 处置类型：保留 / 删除 / 合并 / 拆分 / 改名。
4. 当前状态。
5. 回退方式。
```

## 十、AI 协作收尾要求

每次有价值的 AI 协作结束后，必须至少沉淀一类结果：

```text
1. 决策 → 决策日志。
2. 未决问题 → 待决策清单。
3. 文档治理 → 文档治理清理清单。
4. 代码状态 → 文档代码测试追踪矩阵。
5. 测试结论 → 测试基线或联调清单。
6. 对话摘要 → 99-Claude对话提炼 或对应模块文档。
```

未沉淀的 AI 对话，不视为项目资产。

## 十一、当前治理结论

已确认：

```text
1. AI 是推演区，不是真相源。
2. AI 输出必须经过证据分级。
3. 关键实现状态必须精读文档与代码。
4. 未测试内容不能写成已验证。
5. 文档拆分必须先建立对应文件夹。
```

```text
1. 是否需要把 AI 协作开场模板放入 _模板 目录。
2. 是否需要建立单独的 AI 输出审查清单。
3. 是否需要建立 Cursor 专用代码审查规范。
```
