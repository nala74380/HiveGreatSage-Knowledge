# 日常工作循环

## 来源说明

本文从以下文档拆出：

```text
00-项目总控/工作流方案.md
```

来源章节：

```text
三、日常工作循环（可持续节奏）
```

本文只负责说明每天或每个工作单元如何推进项目，不负责说明工具职责细节；工具职责见 [[00-项目总控/工作流方案/工具分工]]，数据流向见 [[00-项目总控/工作流方案/数据流向]]。

---

## 一、核心原则

```text
先看真相源，再推演，再执行，最后沉淀。
```

每个工作单元必须避免两类问题：

```text
1. 只和 AI 讨论，不沉淀到 Obsidian。
2. 只改代码，不回写文档、决策、风险和测试状态。
```

日常工作循环的目标不是"多做事"，而是让每次推进都能留下可追踪、可复用、可回退的项目资产。

---

## 二、标准工作单元

一个典型工作单元建议控制在 2-4 小时，分为四个阶段：

```text
Phase 1：准备
Phase 2：推演
Phase 3：执行
Phase 4：收尾
```

其中 Phase 4 最关键，不能省略。

---

## 三、Phase 1：准备

建议时间：15 分钟。

目标：确认今天要解决什么，不让任务范围蔓延。

步骤：

```text
1. 打开 Obsidian。
2. 查看 00-项目总控/项目总大纲.md。
3. 查看对应子系统 README 或架构文档。
4. 查看待决策清单、风险清单或追踪矩阵。
5. 写下本轮具体目标，控制在 1-2 个。
```

输出物：

```text
1. 本轮目标。
2. 本轮范围。
3. 本轮非目标。
4. 需要读取的文档和代码路径。
```

禁止：

```text
1. 不看当前文档就直接问 AI。
2. 一次工作单元塞入多个无关模块。
3. 不区分目标和非目标。
```

---

## 四、Phase 2：推演

建议时间：30-60 分钟。

目标：用 AI 辅助把方案想清楚，但不把 AI 输出直接当事实。

步骤：

```text
1. 开启 AI 对话。
2. 贴入相关文档摘要或当前事实源。
3. 明确本次目标和非目标。
4. 讨论方案、风险、边界、实现顺序。
5. 对 AI 输出中的不确定点做标记。
```

需要特别关注：

```text
1. 是否存在 hallucination：编造不存在的文件、接口、字段。
2. 是否存在 sycophancy：为了迎合用户而附和不合理设计。
3. 是否存在 specification gaming：看似满足字面要求，但没有解决真实问题。
4. 是否存在 deception 风险：把未完成说成已完成。
```

输出物：

```text
1. 候选方案。
2. 风险点。
3. 待确认项。
4. 推荐执行顺序。
```

---

## 五、Phase 3：执行

建议时间：60-120 分钟。

目标：在正确工具中执行，不把执行过程变成新的混乱来源。

执行类型：

```text
代码任务：进入 Cursor / PyCharm，并最终提交到 GitHub。
文档任务：进入 Obsidian / Knowledge 仓库，并最终提交到 GitHub。
图形任务：使用 Excalidraw / draw.io，并回链到 Obsidian。
治理任务：更新治理清单、批次记录、追踪矩阵。
```

执行规则：

```text
1. 同一个仓库不要多个编辑器同时打开。
2. 同一个文件不要多处并行修改。
3. 代码变更必须能追踪到文档、决策、问题或测试。
4. 文档变更必须能追踪到来源、目标和治理记录。
5. 遇到卡住的问题，带错误信息回到 AI 推演，而不是盲改。
```

---

## 六、Phase 4：收尾

建议时间：15-30 分钟。

目标：把本轮工作变成可追踪的项目资产。

收尾动作：

```text
1. Git commit + push。
2. 更新 Obsidian 文档。
3. 更新决策日志或待决策清单。
4. 更新问题追踪、风险清单或治理清单。
5. 更新文档代码测试追踪矩阵。
6. 标记哪些内容已确认、哪些仍待验证。
```

文档沉淀规则：

```text
AI 对话结论 → 99-Claude对话提炼 或对应模块文档
架构变更 → 对应架构设计.md
接口变更 → 接入契约 / API 路由清单 / OpenAPI 相关文档
实现状态变化 → 实现状态对照表 / 文档代码测试追踪矩阵
测试结果 → 测试基线 / 三端联调清单
重要决策 → 决策日志
未决问题 → 待决策清单
治理动作 → 文档治理清理清单
```

禁止：

```text
1. 只提交代码，不更新相关文档。
2. 只更新文档，不记录来源和状态。
3. 把源码存在写成已闭环。
4. 把未运行测试写成已验证。
```

---

## 七、为什么 Phase 4 最重要

过去"修修补补"的核心问题通常不是单次代码写得不好，而是知识没有沉淀。

如果每次 AI 对话后不做收尾，会出现：

```text
1. 结论留在对话窗口里，后续难以复用。
2. 新对话无法继承完整上下文。
3. 同一个问题反复讨论。
4. 文档与代码越来越漂移。
5. 用户以为已经确认，实际上只是讨论过。
```

Phase 4 相当于项目会议纪要和版本治理。即使只写 3 条要点，也比完全不沉淀更可靠。

## 八、工作单元检查清单

开始前：

```text
□ 是否明确本轮目标？
□ 是否明确范围和非目标？
□ 是否读取当前真相源文档？
□ 是否知道要修改哪些文件？
□ 是否知道哪些内容需要代码验证？
```

执行中：

```text
□ 是否避免同时修改多个无关模块？
□ 是否记录了新发现的风险？
□ 是否标记了待确认项？
□ 是否避免把 AI 输出直接写成事实？
```

收尾时：

```text
□ 是否 commit + push？
□ 是否更新 Obsidian？
□ 是否更新决策日志或待决策清单？
□ 是否更新治理清单或追踪矩阵？
□ 是否明确哪些已确认、哪些待验证？
```

## 九、工作单元输出模板

每个较大的工作单元结束后，建议至少留下以下记录：

```markdown
## 工作单元记录

### 目标

### 范围

### 非目标

### 已完成

### 已确认

### 待确认

### 风险

### 变更文件

### 测试 / 验证

### 下一步
```

该模板可以放入具体治理文件、对话提炼文件或子系统任务记录中，不强制每次单独新建文件。

## 十、当前治理结论

已确认：

```text
1. 日常工作必须从 Obsidian 真相源开始。
2. AI 只负责推演，不负责长期存储。
3. 代码和文档都必须进入 GitHub 形成版本历史。
4. 每个工作单元必须有收尾沉淀。
5. 未验证内容必须标记为待确认或待验证。
```

待确认：

```text
1. 是否需要为每个工作单元建立固定任务编号。
2. 是否需要在 00-项目总控 下建立统一任务看板。
3. 是否需要把工作单元记录模板移入 _模板 目录。
```
