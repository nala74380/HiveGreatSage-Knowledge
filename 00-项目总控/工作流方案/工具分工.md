# 工具分工

## 来源说明

本文从以下文档拆出：

```text
00-项目总控/工作流方案.md
```

来源章节：

```text
一、工具角色定义（六个位置，各司其职）
```

本文只负责说明各工具的长期职责边界，不负责记录历史搭建步骤。

---

## 一、总原则

```text
每个工具只做它最擅长的事。
职责不重叠。
数据流向单一且可追溯。
```

工具分工的核心目标：

```text
1. Obsidian 管知识。
2. GitHub 管代码。
3. AI 工具负责推演和辅助，不负责长期存储。
4. 编辑器负责执行，不负责项目记忆。
5. Google Drive 只做镜像和对外共享，不做真相源。
6. 图形工具只画必要图，不做装饰性图。
```

---

## 二、Obsidian — 知识真相源

### 2.1 它是什么

Obsidian 是整个项目的知识大脑，是所有非代码内容的唯一权威存储地。

### 2.2 它负责

```text
1. 项目总大纲及所有子模块架构文档。
2. 每次 AI 对话后的结论提取。
3. 技术方案的定稿版本。
4. 问题追踪清单。
5. AI 安全术语表与表达纪律。
6. 决策日志。
7. 待决策清单。
8. 文档治理记录。
```

### 2.3 它不负责

```text
1. 不存代码，代码去 GitHub。
2. 不存正式对外文档，对外文档去 Google Drive。
3. 不做实时多人协作，Obsidian 是本地优先工具。
4. 不保存真实密钥、授权码、Token、数据库密码。
```

### 2.4 操作习惯

```text
1. 每次 AI 对话结束后，把有价值结论写入 Obsidian。
2. 使用双链把相关文档串起来。
3. 每份文档必须有 YAML frontmatter。
4. 文档状态必须明确：草稿 / 定稿 / 过时。
5. 涉及实现状态时，必须回到 GitHub 代码与测试交叉验证。
```

## 三、Claude Project / ChatGPT Project — 主推演区

### 3.1 它是什么

AI Project 是设计思考、技术推演、方案审查、风险识别的会议室，不是长期存储位置。

### 3.2 它负责

```text
1. 架构设计讨论和推演。
2. 技术选型分析。
3. 文档草稿撰写与修改。
4. Bug 排查思路讨论。
5. 代码框架草案生成。
6. 风险审查与边界检查。
```

### 3.3 它不负责

```text
1. 不是知识存储，结论必须搬去 Obsidian。
2. 不是代码仓库，代码必须进入 GitHub。
3. 不应依赖 AI 记住所有历史。
4. 不应把 AI 输出直接当作事实。
```

### 3.4 AI 对话窗口规则

```text
窗口命名规则：[模块编号]-[主题]-[日期]

示例：
02-PC中控-通信协议设计-0413
03-安卓脚本-OCR方案选型-0415
00-工作流-整体复盘-0420
```

禁止：

```text
1. 一个窗口讨论多个无关模块。
2. 不提供背景就让 AI 猜。
3. 在对话中长期修修补补而不沉淀到 Obsidian。
4. 把未验证的 AI 输出写成已确认事实。
```

### 3.5 每次新对话标准流程

```text
1. 从 Obsidian 复制相关模块当前文档。
2. 明确本次目标。
3. 如果接续之前讨论，贴上上次结论摘要。
4. 对话推演。
5. 结束后提炼结论。
6. 写入 Obsidian。
7. 必要时更新决策日志、待决策清单或治理清单。
```

## 四、GitHub — 代码真相源

### 4.1 它是什么

GitHub 是所有代码和 Obsidian 知识库版本历史的权威存储地。

### 4.2 它负责

```text
1. 所有项目代码的版本管理。
2. Obsidian Knowledge 仓库的版本管理。
3. 代码级变更历史追踪。
4. 分支管理。
5. Pull Request / Issue / Commit 历史。
```

### 4.3 仓库边界

```text
HiveGreatSage-Knowledge       Obsidian 知识库
HiveGreatSage-Verify          网络验证系统
HiveGreatSage-PCControl       PC 中控框架
HiveGreatSage-AndroidScript   安卓端懒人精灵脚本框架
HiveGreatSage-Monitor         官网更新监控
HiveGreatSage-Tools           其他工具
```

### 4.4 分支原则

```text
main          稳定版本，只合并测试通过的代码或已确认文档
中间分支       用于日常开发、治理或功能验证
feature/xxx   功能分支
fix/xxx       修复分支
```

### 4.5 提交前检查

```text
1. 提交前看 diff。
2. commit message 写清楚做了什么。
3. 文档变更要能追溯到治理清单或决策日志。
4. 代码变更要能追溯到测试或明确的待验证项。
```

## 五、Cursor vs PyCharm — 编码分工

### 5.1 核心原则

```text
同一个仓库不要同时用多个编辑器打开。
同一个文件不要两个工具同时改。
```

### 5.2 推荐分工

| 维度 | Cursor | PyCharm |
|---|---|---|
| 适合做 | AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码 | Python 大型项目、调试、重构、数据库工具 |
| 主要项目 | AndroidScript、Tools、小工具、快速实验 | Verify、PCControl、Monitor |
| 优势 | 从 0 写、脚本类工作、轻量重构 | 调试器、数据库工具、Python 项目级重构 |

### 5.3 当前建议

```text
Cursor:
  - HiveGreatSage-AndroidScript
  - HiveGreatSage-Tools
  - 任何需要快速 AI 辅助原型的新代码

PyCharm:
  - HiveGreatSage-Verify
  - HiveGreatSage-PCControl
  - HiveGreatSage-Monitor
```

## 六、Google Drive — 文档镜像层

### 6.1 它是什么

Google Drive 是对外共享和协作展示窗口，不是真相源。

### 6.2 它负责

```text
1. 对外共享文档。
2. 产品说明、用户手册等正式交付文档。
3. 参考资料同步。
4. Obsidian 定稿内容的镜像。
```

### 6.3 它不负责

```text
1. 不作为日常编辑位置。
2. 不作为代码存储。
3. 不与 Obsidian 双向编辑。
```

### 6.4 数据流向

```text
Obsidian 定稿 → Google Drive 镜像
```

禁止反向：

```text
Google Drive 修改 → 再搬回 Obsidian
```

除非明确做一次受控迁移。

## 七、Excalidraw + draw.io — 图形工具分工

### 7.1 核心原则

```text
只画有用的图，不画装饰性图。
```

### 7.2 推荐分工

| 工具 | 用途 | 原因 |
|---|---|---|
| Excalidraw | 快速草图、头脑风暴、架构初始构想、流程草稿 | 低摩擦，适合嵌入 Obsidian |
| draw.io | 正式系统架构图、数据库 ER 图、通信时序图 | 结构化，适合技术文档 |

### 7.3 只保留两类图

```text
类型A：系统架构图
  工具：draw.io
  用途：说明子系统连接、数据流、部署边界
  存放：对应 Obsidian 模块文件夹

类型B：思维草图
  工具：Excalidraw
  用途：讨论时快速画关系、流程、备选方案
  存放：Obsidian Vault 中
```

禁止：

```text
1. 纯装饰流程图。
2. 已经在代码或文档说清楚的重复图。
3. 长期无人维护的漂亮图。
```

## 八、边界总表

| 工具 | 是什么 | 负责 | 不负责 |
|---|---|---|---|
| Obsidian | 知识真相源 | 架构、决策、规范、治理、沉淀 | 代码、真实密钥、对外镜像 |
| AI Project | 推演会议室 | 方案推演、审查、草稿、排错 | 长期存储、事实源 |
| GitHub | 代码与版本真相源 | 代码、版本、提交历史 | 业务文档推演 |
| Cursor | AI 编码辅助 | Lua、小工具、快速原型 | 长期项目记忆 |
| PyCharm | Python 工程 IDE | Verify、PCControl、Monitor | 文档真相源 |
| Google Drive | 对外镜像 | 共享、交付、备份 | 日常编辑、真相源 |
| Excalidraw | 草图工具 | 快速推演图 | 正式架构图 |
| draw.io | 正式图工具 | 架构图、ER 图、时序图 | 临时脑暴 |
