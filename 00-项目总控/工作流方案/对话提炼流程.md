# 对话提炼流程

## 来源说明

本文从以下文档拆出：

```text
00-项目总控/工作流方案.md
```

来源章节：

```text
五、对话提炼模板
```

本文只负责说明 AI 对话结束后的结论提炼、沉淀位置、模板字段和质量要求。AI 协作规则见 [[00-项目总控/工作流方案/AI协作规范]]，日常工作循环见 [[00-项目总控/工作流方案/日常工作循环]]。

---

## 一、当前定位

对话提炼不是保存聊天记录，而是把 AI 对话中有长期价值的内容转化为项目资产。

项目资产包括：

```text
1. 已确认事实。
2. 架构决策。
3. 待确认问题。
4. 风险点。
5. 文档治理动作。
6. 代码实现状态。
7. 测试或联调结论。
8. 下一步任务。
```

不属于项目资产的内容：

```text
1. AI 的寒暄。
2. 重复的推理过程。
3. 未经验证的完整叙事。
4. 临时猜测。
5. 与项目无关的泛泛建议。
```

---

## 二、提炼时机

以下情况必须做对话提炼：

```text
1. 产生了架构决策。
2. 产生了模块边界调整。
3. 产生了接口、数据库、测试、部署相关结论。
4. 发现了风险或待决策项。
5. 进行了文档治理、合并、拆分、删除。
6. 进行了代码精读或实现状态判断。
7. 一次对话跨越多个步骤，后续需要继续接力。
```

以下情况可以不单独建提炼文档：

```text
1. 只是一次很小的格式修正。
2. 只是执行已经明确的单一步骤。
3. 结论已直接写入对应真相源文档。
```

但即使不单独建提炼文档，也必须把关键结果写入对应位置。

---

## 三、提炼目标位置

根据内容类型决定写入位置：

| 内容类型 | 目标位置 |
|---|---|
| 架构结论 | 对应子系统 `架构设计.md` |
| 项目级原则 | `00-项目总控/项目总大纲.md` 或相关规范 |
| 已定决策 | `00-项目总控/决策日志.md` |
| 未决问题 | `00-项目总控/待决策清单.md` |
| 文档治理动作 | `00-项目总控/文档治理清理清单.md` |
| 代码状态 | `00-项目总控/文档代码测试追踪矩阵.md` |
| 测试结论 | 对应测试基线或联调清单 |
| 对话摘要 | `99-Claude对话提炼/` 或对应模块沉淀文档 |

## 四、标准提炼模板

每次 AI 对话结束后，如需单独沉淀，可使用以下模板：

```markdown
# [日期] [模块] [主题]

## 对话目标
<!-- 这次对话想解决什么 -->

## 范围
<!-- 本次实际覆盖哪些内容 -->

## 非目标
<!-- 本次明确不处理什么 -->

## 关键结论
<!-- 3-5 条最重要的结论，用自己的话写 -->
1.
2.
3.

## 已确认信息
<!-- 有文档、代码、测试、用户输入支撑的信息 -->

## 推断
<!-- 合理但尚未验证的推断，必须明确标记 -->

## 待确认 / 待验证
<!-- 需要后续精读、测试、联调或用户确认的内容 -->

## 代码产出
<!-- 产出了什么代码？在哪个仓库、分支、commit？ -->

## 文档产出
<!-- 更新或新增了哪些文档？ -->

## 风险点
<!-- 包括 AI 输出风险、架构风险、实现风险、联调风险 -->

## 下一步
<!-- 基于这次对话，下一步要做什么 -->

## 对话链接
<!-- Claude / ChatGPT 对话 URL；没有则写"无" -->
```

## 五、提炼质量标准

### 5.1 必须区分证据等级

提炼时必须区分：

```text
已确认
推断
建议方案
待确认
未知
```

禁止把以下内容写成"已确认"：

```text
1. 只是 AI 推理出来的内容。
2. 没有读代码就判断的实现状态。
3. 没有跑测试就判断的测试结论。
4. 没有实机验证就判断的端到端闭环。
```

---

### 5.2 必须保留来源

每条重要结论至少应该能追溯到：

```text
1. 某个 Obsidian 文档。
2. 某个 GitHub 文件。
3. 某个测试结果。
4. 用户明确输入。
5. 本次对话链接。
```

如果没有来源，应标记为：

```text
推断 / 待确认
```

---

### 5.3 必须压缩而不是复制

对话提炼不是粘贴完整聊天记录。

推荐压缩方式：

```text
1. 删除寒暄。
2. 删除重复推理。
3. 保留结论。
4. 保留理由。
5. 保留风险。
6. 保留下一步。
```

## 六、AI 输出风险标记

对话中发现下列风险时，应在"风险点"中明确标注：

| 风险 | 标记方式 |
|---|---|
| 编造不存在的内容 | `hallucination / 幻觉风险` |
| 信息不完整却补成完整故事 | `confabulation / 编造式补全风险` |
| 为迎合用户放弃准确性 | `sycophancy / 迎合风险` |
| 字面满足但偏离真实目标 | `specification gaming / 规格投机风险` |
| 未完成说成已完成 | `deception / 欺骗性表达风险` |

相关术语见：

```text
00-项目总控/AI安全术语表_中文简体.md
```

## 七、不同类型对话的提炼重点

### 7.1 架构推演类

必须提炼：

```text
1. 最终建议方案。
2. 被否决方案和原因。
3. 模块边界。
4. 数据流向。
5. 风险与回退方式。
```

### 7.2 代码排错类

必须提炼：

```text
1. 错误现象。
2. 根因判断。
3. 已验证的修复。
4. 未验证的假设。
5. 相关代码路径。
6. 后续测试方式。
```

### 7.3 文档治理类

必须提炼：

```text
1. 来源文档。
2. 处置类型：删除 / 合并 / 拆分 / 保留 / 改名。
3. 目标文档或目录。
4. 是否已回写治理清单。
5. 是否存在旧双链风险。
```

### 7.4 测试联调类

必须提炼：

```text
1. 测试范围。
2. 测试环境。
3. 已通过项。
4. 失败项。
5. 待复测项。
6. 不能宣称已闭环的原因。
```

## 八、对话提炼后的回写规则

对话提炼完成后，必须根据内容回写对应文件。

```text
决策 → 决策日志
待决策 → 待决策清单
实现状态 → 文档代码测试追踪矩阵
文档治理 → 文档治理清理清单
测试结论 → 测试基线或联调清单
规范变化 → 编码规范或对应规范文档
```

如果只写了 `99-Claude对话提炼/xxx.md`，但没有更新主真相源，仍视为治理未完成。

## 九、文件命名建议

对话提炼文件建议使用：

```text
YYYY-MM-DD_模块_主题.md
```

示例：

```text
2026-05-12_00项目总控_工作流方案拆分.md
2026-05-12_01网络验证系统_热更新链路精读.md
2026-05-12_03安卓脚本_SHA256校验方案.md
```

注意：

```text
1. 会议纪要和对话提炼可以带日期。
2. 主真相源文档不要带日期。
3. 日期文档不能长期替代主文档。
```

## 十、当前治理结论

已确认：

```text
1. 对话提炼是知识沉淀，不是聊天记录备份。
2. 提炼后的结论必须回写主真相源。
3. 未验证内容必须显式标记为推断或待确认。
4. 日期类对话提炼不能替代稳定命名的主文档。
```

```text
1. 是否需要把本模板复制到 `_模板/对话提炼模板.md`。
2. 是否需要建立 `99-Claude对话提炼/README.md`。
3. 是否需要统一旧对话提炼文件命名。
```
