## 一、我需要同时用多个 AI 工具吗？

建议方案：

```text
可以同时使用，但必须明确分工。
```

推荐边界：

```text
ChatGPT / DeepSeek / Claude / Cursor Chat 等 AI 工具:
  - 架构推演
  - 文档治理
  - 风险审查
  - 方案比较
  - 对话提炼

Cursor Chat（代码辅助）:
  - 代码草案
  - 小范围重构
  - 局部代码解释
  - 快速脚本辅助
```

禁止：

```text
1. 两个 AI 同时改同一份事实源。
2. 两个 AI 各自生成一套不同结论，不回到 Obsidian 合并。
3. 把 AI 的上下文记忆当作长期项目记忆。
```

## 二、Obsidian 和 Google Drive 哪个是最终版本？

已确认原则：

```text
Obsidian 是知识真相源。
Google Drive 是对外镜像层。
```

正确流向：

```text
Obsidian 定稿 → Google Drive 镜像
```

禁止反向：

```text
Google Drive 修改 → 覆盖 Obsidian
```

例外：

```text
如果必须从 Google Drive 回迁内容，必须作为一次受控迁移记录：
1. 来源文档
2. 目标文档
3. 差异说明
4. 是否覆盖原内容
5. 回退方式
```

## 三、代码说明写在 Obsidian 还是 GitHub README？

建议方案：按职责拆开。

```text
Obsidian:
  - 架构
  - 决策
  - 接口契约
  - 风险
  - 测试基线
  - 文档治理

GitHub README:
  - 如何安装
  - 如何运行
  - 如何测试
  - 仓库目录说明
  - 开发者快速入口
```

原则：

```text
长期项目知识进 Obsidian。
代码仓库本身的运行说明进 README。
```

## 四、AI 对话要不要全部保存？

不建议全部保存。

应该保存的是：

```text
1. 关键结论。
2. 已确认事实。
3. 决策理由。
4. 风险点。
5. 待确认事项。
6. 下一步。
```

不应该保存的是：

```text
1. 寒暄。
2. 重复推理。
3. 临时猜测。
4. 已被主文档吸收的中间过程。
```

对话提炼规则见：

```text
00-项目总控/工作流方案/对话提炼流程.md
```

## 五、Daily Notes 要不要用？

当前不建议作为主工作方式。

原因：

```text
蜂巢·大圣项目按模块组织，不按日期组织。
Daily Notes 容易让项目知识散落到日期文档里。
```

可接受用法：

```text
1. 作为临时工作记录。
2. 作为每日流水。
3. 作为后续提炼素材。
```

禁止：

```text
1. 用 Daily Notes 替代模块文档。
2. 把决策长期留在日期文档。
3. 把接口契约或架构结论写在日期文档后不迁移。
```

## 六、为什么不直接把所有拆分文件放在 00-项目总控 根目录？

因为这样会让根目录继续膨胀。

已确认治理规则：

```text
拆分前必须先建立对应文件夹，再逐步拆分。
禁止直接在父目录下散落多个拆分文件。
```

原因：

```text
1. 保持 00-项目总控 根目录清晰。
2. 让拆分来源和目标可追踪。
3. 方便原文档最终改为导航页。
4. 避免治理文件本身制造新混乱。
```

## 七、什么时候可以删除原文档？

满足以下条件后，才考虑删除或改为过时跳转：

```text
1. 有效内容已全部迁移。
2. 目标文件已经创建。
3. README 已记录拆分关系。
4. 文档治理清理清单已记录。
5. 原文档中的旧双链风险已处理。
6. Git 历史可回退。
```

对于 `工作流方案.md`，当前已执行的做法是：

```text
已完成拆分并删除。
正式工作流入口为 00-项目总控/工作流方案/README.md。
```

不再保留独立的导航页文件。

原因：

```text
1. 有效内容已迁移到 00-项目总控/工作流方案/ 正式目录。
2. 正式入口已由 00-项目总控/工作流方案/README.md 承接。
3. 旧双链已完成收口，不再依赖 00-项目总控/工作流方案.md。
4. Git 历史可回退。
5. 继续保留旧导航页会制造双入口风险。
```

## 八、什么时候需要精读代码？

凡涉及"实现状态"的判断，都必须精读代码。

例如：

```text
1. 某接口是否存在。
2. 某字段是否已实现。
3. 某测试是否覆盖。
4. PCControl 是否调用 Verify。
5. AndroidScript 是否完成 SHA-256 校验。
6. 热更新是否真正闭环。
```

不能只凭文档判断：

```text
源码存在 ≠ 已运行验证
测试文件存在 ≠ 测试已通过
单端代码存在 ≠ 三端联调已闭环
```

证据等级应参考：

```text
00-项目总控/治理批次记录.md
00-项目总控/文档代码测试追踪矩阵.md
```

## 九、什么时候创建新文档，什么时候合并旧文档？

建议判断：

```text
同一主题，多份散落文档 → 合并
一份文档承担多个职责 → 拆分
内容已被主文档吸收 → 删除或过时跳转
仍有入口价值但正文已迁移 → 导航页
```

禁止：

```text
1. 为了看起来整齐而过度拆分。
2. 为了省事而把多个职责塞进一个文档。
3. 把历史记录伪装成当前规范。
4. 把日期补丁文档当成长期真相源。
```

## 十、如何避免文档越治理越乱？

必须遵守四条：

```text
1. 每次只处理一个职责块。
2. 每次治理都回写治理清理清单。
3. 原文档在迁移完成前保留。
4. 完成后根据入口价值、双链风险和正式 README 承接情况，选择导航页、过时跳转页或删除，并必须回写文档治理清理清单。
```

并且每份新文档必须有：

```text
1. YAML frontmatter。
2. 来源说明。
3. 当前定位。
4. 待确认项。
5. 关联双链。
```

## 十一、当前治理结论

已确认：

```text
1. 工作流问题多数来自职责边界不清。
2. Obsidian 与 GitHub 必须分别作为知识真相源和代码真相源。
3. AI 对话必须提炼，不能长期停留在聊天窗口。
4. 历史搭建步骤不能伪装成当前强制规范。
5. 大文档拆分完成后，原文档可按导航页、过时跳转页或删除三类处理；是否删除取决于有效内容是否迁移、旧双链是否收口、是否有正式入口承接和 Git 历史是否可回退。
```
