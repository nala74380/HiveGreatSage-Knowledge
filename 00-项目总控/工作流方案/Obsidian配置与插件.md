# Obsidian配置与插件

## 来源说明

本文从以下文档拆出：

```text
00-项目总控/工作流方案.md
```

来源章节：

```text

```

本文记录 Obsidian 插件的选型建议、使用边界和治理原则。
本文不追踪当前本机 / 当前 Vault 是否已经安装某个插件。
插件是否实际安装属于用户本地环境事实，不作为本轮文档治理检查项。

Obsidian 的项目角色见 [[00-项目总控/工作流方案/工具分工]]，数据流向见 [[00-项目总控/工作流方案/数据流向]]。

## 一、当前定位

Obsidian 是蜂巢·大圣项目的知识真相源。

插件配置必须服务于：

```text
1. 文档长期维护。
2. 双链关系管理。
3. 模板复用。
4. 任务追踪。
5. 知识库版本控制。
6. 必要的架构草图沉淀。
```

插件配置不应服务于：

```text
1. 花哨美化。
2. 多余仪表盘。
3. 复杂到需要额外维护的自动化。
4. 替代 GitHub、代码仓库或测试系统。
```

---

## 二、插件选择原则

```text
只装有用的。
能不用插件解决的，不优先上插件。
插件必须降低维护成本，而不是增加维护成本。
```

判断一个插件是否值得安装：

```text
1. 是否服务于长期项目治理？
2. 是否能减少重复劳动？
3. 是否能提高文档可追踪性？
4. 是否容易迁移或停用？
5. 是否会制造新的真相源？
```

如果插件会制造新的孤岛数据，不建议使用。

## 三、核心推荐插件

| 插件           | 用途           | 必要性  | 使用边界                                             |
| ------------ | ------------ | ---- | ------------------------------------------------ |
| Obsidian Git | 知识库版本控制与备份   | 核心推荐 | 不检查当前是否已安装；仅说明适用场景；只负责提交 Knowledge，不替代 GitHub 审查 |
| Excalidraw   | 快速草图、脑图、架构初稿 | 核心推荐 | 不检查当前是否已安装；图不能单独成为事实源；只画推演草图，不替代正式架构文档           |

### 3.1 Obsidian Git

用途：

```text
1. 给 Knowledge 仓库提供版本历史。
2. 防止误删、误改。
3. 让 Obsidian 文档变化进入 GitHub。
```

建议配置：

```text
1. 手动提交优先。
2. 自动提交可以开启，但频率不要过高。
3. 每次大治理前先手动 commit。
4. 大批量删除、合并、拆分必须单独 commit。
```

注意：

```text
Obsidian Git 不是代码审查工具。
重要治理仍然要在文档治理清理清单中记录。
```

### 3.2 Excalidraw

用途：

```text
1. 临时架构草图。
2. 流程关系梳理。
3. 和 AI 讨论前整理思路。
4. 复杂模块的视觉化推演。
```

限制：

```text
1. 不画纯装饰图。
2. 不让图单独成为事实源。
3. 正式架构图仍建议使用 draw.io 或稳定文档描述。
4. 图一旦过时，必须在对应文档中标记。
```

## 四、推荐插件

| 插件 | 用途 | 何时使用 |
|---|---|---|
| Kanban | 任务看板 | 当任务数量明显增多，纯表格不够用时 |
| Templater | 模板快速创建 | 当对话提炼、架构设计、问题追踪开始重复时 |
| Dataview | 跨文档查询 | 文档数量稳定超过 20-30 篇且 YAML 字段规范后 |

### 4.1 Kanban

适合管理：

```text
待做
进行中
卡住
待验证
已完成
```

不建议一开始就过度设计任务系统。

个人项目阶段，优先使用：

```text
00-项目总控/待决策清单.md
00-项目总控/文档治理清理清单.md
00-项目总控/文档代码测试追踪矩阵.md
```

任务规模变大后，再考虑 Kanban。

### 4.2 Templater

适合模板：

```text
1. 对话提炼模板。
2. 架构设计模板。
3. 问题追踪模板。
4. 测试记录模板。
5. 文档治理记录模板。
```

使用边界：

```text
模板只负责降低重复输入成本。
模板不能替代人工判断。
```

### 4.3 Dataview

适合查询：

```text
1. 列出所有状态为草稿的文档。
2. 列出所有待确认项。
3. 列出所有关联某个模块的文档。
4. 列出版本低于指定标准的文档。
```

启用前提：

```text
1. 大多数文档已有 YAML frontmatter。
2. 字段命名基本统一。
3. 文档状态已经规范为 草稿 / 定稿 / 过时。
```

不要过早启用复杂查询，否则会增加维护成本。

## 五、不建议安装的插件或功能

当前阶段不建议：

```text
1. Calendar。
2. Daily Notes。
3. 花哨主题。
4. 大量美化插件。
5. 复杂 Dashboard。
6. 与项目治理无关的自动化插件。
```

原因：

```text
1. 蜂巢·大圣项目按模块组织，不按日期组织。
2. Daily Notes 容易让项目知识散落到日期文档里。
3. 主题和美化插件不提升维护性。
4. Dashboard 容易变成第二套维护对象。
```

例外：

```text
如果某天需要会议纪要或每日工作记录，可以用日期文档，但不能让日期文档替代主真相源。
```

## 六、Vault 基础结构建议

当前建议使用模块化结构：

```text
HiveGreatSage/
├── 00-项目总控/
├── 01-网络验证系统/
├── 02-PC中控框架/
├── 03-安卓脚本框架/
├── 04-官网监控/
├── 05-工具与拓展/
├── 06-游戏适配/
├── 99-Claude对话提炼/
└── _模板/
```

注意：

```text
1. 项目治理文档优先放 00-项目总控。
2. 子系统真相源放对应子系统目录。
3. 对话提炼可以有日期，但不能替代稳定命名的主文档。
4. 模板放 _模板，不混入主真相源目录。
```

## 七、模板目录建议

建议逐步建立：

```text
_模板/
├── 对话提炼模板.md
├── 架构设计模板.md
├── 问题追踪模板.md
├── 测试记录模板.md
└── 文档治理记录模板.md
```

当前待确认：

```text
1. _模板 目录是否已经存在。
2. 对话提炼模板是否已有旧版本。
3. 是否需要从 对话提炼流程.md 复制模板到 _模板。
```

在未确认前，不直接创建或覆盖模板文件。

## 八、配置治理规则

Obsidian 插件配置也属于项目资产，但不一定全部进入文档。

必须记录的配置：

```text
1. 影响知识库结构的配置。
2. 影响 Git 提交行为的配置。
3. 影响模板生成的配置。
4. 影响双链和查询规则的配置。
```

不必记录的配置：

```text
1. 纯个人显示偏好。
2. 字体大小。
3. 主题颜色。
4. 快捷键偏好。
```

---

## 九、插件变更流程

新增插件前：

```text
1. 说明为什么需要。
2. 说明替代方案。
3. 说明是否会制造新真相源。
4. 说明停用后的影响。
```

新增插件后：

```text
1. 更新本文。
2. 如果影响工作流，更新相关工作流文档。
3. 如果影响模板，更新 _模板。
4. 如果影响文档结构，更新文档治理清理清单。
```

```text
1. Obsidian Git 和 Excalidraw 属于核心推荐插件。
2. Kanban、Templater、Dataview 属于条件推荐插件。
3. Calendar、Daily Notes、花哨主题、复杂 Dashboard 当前不建议。
4. 项目应按模块组织，不按日期组织。
5. 日期类文档只能作为提炼或记录，不能替代主真相源。
6. 本文只维护插件选型建议与使用边界，插件是否实际安装属于用户本地环境事实，不作为文档治理检查项。
```
