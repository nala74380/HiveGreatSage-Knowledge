### 2.2 正文结构规范

```text
# 一级标题（每个文档只有一个，作为文档标题）
## 二级标题（章节）
### 三级标题（小节）
#### 四级标题（谨慎使用，过深说明结构需要拆分）
```

正文规则：

```text
1. 段落之间用一个空行分隔。
2. 标题前后各留一个空行。
3. 列表与段落之间留一个空行。
4. 单文档层级过深时，优先拆分文档，不继续堆四级标题。
```

### 2.3 代码块规范

所有代码块必须标注语言。

正确示例：

```python
def hello():
    pass
```

```lua
function hello()
end
```

```sql
SELECT * FROM user;
```

```text
纯文本内容、目录结构、ASCII 图等。
```

```bash
ls -la
```

禁止使用无语言标注的裸代码块。

### 2.4 表格规范

```text
1. 表格前后各留一个空行。
2. 列头和数据行的分隔线必须完整。
3. 单元格内尽量不要换行；如必须多行，用 `<br>`。
4. 表格用于状态、清单、矩阵；长篇解释不要塞进表格。
```

示例：

```markdown
| 列1 | 列2 | 列3 |
|---|---|---|
| 值1 | 值2 | 值3 |
```

### 2.5 双链规范

每个文档必须至少有：

```text
1. 一个父节点引用。
2. 一个相关文档引用。
```

可放在 YAML 的 `关联文档` 字段或正文第一段。孤岛文档禁止长期存在。

双链写法：

```markdown
- [[项目总大纲]]
- [[项目总大纲|总大纲]]
- [[01-网络验证系统/架构设计]]
```

推荐面包屑：

```markdown
← 返回 [[00-项目总控/项目总大纲]] | 父节点: [[02-PC中控框架/架构设计]]
```

### 2.6 文档状态流转

```text
草稿 → 定稿 → 过时
  ↑      │
  └──────┘ 内容需要修订时，新版本回到草稿
```

状态规则：

```text
草稿：内容仍在推演，不作为最终依据。
定稿：内容已经稳定，是当前真相源。
过时：历史上有价值，但当前方案已被替代。
```

过时文档不直接删除，除非符合治理批次中的删除标准。过时文档顶部必须注明被哪个新文档替代。

### 2.8 拆分前置文件夹规则

拆分文档时，必须先建立对应文件夹，再逐步拆分。禁止直接在父目录下散落多个新文件。

推荐流程：

```text
1. 在原文档所在模块下创建对应文件夹。
2. 先创建 README.md，说明拆分范围、来源文档、目标结构。
3. 再逐步迁移子文档。
4. 原文档保留为入口，直到迁移完成。
5. 迁移完成后，原文档改为导航页或过时跳转页。
```

示例：

```text
00-项目总控/工作流方案.md

拆分前先建立：
00-项目总控/工作流治理/
├── README.md
├── 工具分工.md
├── AI协作规范.md
├── 对话提炼流程.md
└── 历史搭建指南.md
```

### 2.9 符号使用规范

| 符号 | 含义 | 用法 |
|---|---|---|
| `✓` | 确认 / 完成 | 决策已确认、任务已完成 |
| `✗` | 禁止 / 错误 | 反面示例、禁止做法 |
| `□` | 待做 | 未完成任务 |
| `⚠️` | 警告 / 注意 | 重要提醒 |
| `→` | 流向 | 数据流或步骤顺序 |
| `←` | 引用关系 | A ← B 表示 A 被 B 引用 |
| `◄──` | 单向连接 | WebSocket 客户端连接服务端 |
| `◄─►` | 双向连接 | 对等通信，谨慎使用 |

### 2.9.1 语义型 Emoji（允许）

以下 emoji 具有明确语义，用于项目总览、路线图、状态总览：

| emoji | 含义 | 适用场景 |
|---|---|---|
| ✅ | 已完成 / 已确认 | Phase 状态、决策确认 |
| 🔲 | 待实现 / 待处理 | 路线图、待办清单 |
| ⚠️ | 风险 / 注意 | 注意事项、风险提示 |
| ❌ | 禁止 / 错误 | 不推荐做法、禁用的内容 |

使用规则：
1. 同一含义使用同一 emoji，不混用 ✅ 和 ✓
2. 承担状态、风险、结果、提示等明确语义
3. 项目总大纲、路线图、状态总览允许使用

### 2.9.2 禁用符号

禁止无语义装饰型 emoji：🎉 🔥 💡 ⭐ 等仅用于美化或情绪表达的符号。

### 2.10 时间格式

唯一允许的日期格式：

```text
YYYY-MM-DD
```

正确：

```text
2026-04-16
```

错误：

```text
2026/04/16
2026年4月16日
16/04/2026
Apr 16, 2026
```

时间戳格式：

```text
YYYY-MM-DD HH:MM:SS
```

### 2.11 文档审查机制

为了防止文档漂移：

```text
1. 每次架构级变更必须同步更新总大纲。
2. 决策当天必须登记到决策日志或待决策清单。
3. 新文档必须有父节点双链。
4. 每轮治理必须记录删除、合并、拆分、保留结论。
5. 涉及代码实现状态时，必须按治理批次记录中的证据等级判断。
```

## 五、注释规范

### 5.1 原则

```text
1. 注释说明“为什么”，不说明“是什么”。
2. 复杂逻辑必须注释。
3. 不注释显而易见的代码。
```

反例：

```python
count += 1  # 将 count 加 1
```

正例：

```python
count += 1  # 重试次数用完后不再尝试，避免无限循环
```

### 5.2 TODO / FIXME / HACK 标记

```python
# TODO: 后续需要实现 Token 自动刷新 —— 2026-04-16
# FIXME: 当设备数超过 200 时列表渲染卡顿 —— 2026-04-20
# HACK: 临时绕过懒人精灵 OCR 的编码问题 —— 2026-05-01
```

统一格式：

```text
# 标记: 说明 —— 日期
```

### 5.3 Python 函数文档字符串

```python
def login(self, username: str, password: str) -> LoginResult:
    """用户登录验证。

    调用云端 API 进行身份验证，成功后保存 Token 和用户信息。

    Returns:
        LoginResult: 包含 success、token、user_info、error_message

    Raises:
        NetworkError: 网络连接失败时抛出
    """
```

## 八、目录和文件组织原则

```text
原则1：一个文件只做一件事。
原则2：相关文件放在一起，按功能模块分目录。
原则3：测试文件和源码文件对应。
原则4：配置文件不硬编码在代码中。
原则5：敏感信息不提交到 Git。
原则6：文档拆分必须先建对应文件夹，再逐步拆分。
```

示例：

```text
core/auth/auth_manager.py → tests/test_auth_manager.py
```

`.gitignore` 至少包含：

```text
config/local.yaml
logs/
*.pyc
__pycache__/
.env
*.exe
*.lrj
```

## 九、错误处理规范

### 9.1 Python

错误示例：

```python
try:
    result = api.get_devices()
except:
    pass
```

正确示例：

```python
try:
    result = api.get_devices()
except httpx.TimeoutException:
    logger.warning("获取设备列表超时，将使用缓存数据")
    result = cache.get("device_list")
except httpx.HTTPStatusError as e:
    if e.response.status_code == 401:
        self.token_expired.emit()
    else:
        logger.error(f"获取设备列表失败: {e}")
        raise
```

### 9.2 Lua

错误示例：

```lua
local data = http_get(url)
process(data)
```

正确示例：

```lua
local ok, data = pcall(http_get, url)
if not ok then
    log_error("请求失败: " .. tostring(data))
    return nil, "network_error"
end
if not data then
    log_error("返回数据为空")
    return nil, "empty_response"
end
process(data)
```

## 十一、安全规范

```text
1. 密码 / Token 不明文存储。
2. API 通信使用 HTTPS。
3. 数据库密码、API 密钥放在 local.yaml 或 .env。
4. local.yaml 和 .env 必须进入 .gitignore。
5. 日志中不打印密码、完整 Token、真实授权码。
6. SQL 必须使用 ORM 或参数化查询，禁止拼接 SQL 字符串。
7. Obsidian 普通文档中禁止保存真实密钥、授权码、Token、数据库密码。
```

敏感信息索引统一放在：

```text
00-项目总控/密钥与授权码索引.md
```

该索引只记录名称、用途、保管位置、负责人、轮换周期，不记录真实值。

## 十二、规范执行检查清单

### 12.1 代码提交前检查

```text
□ 文件头是否完整。
□ 文件头版本号是否更新。
```

```text

  □ 版本是 V主.次.修 格式。

正文:
  □ 一级标题只有一个。
  □ 代码块都标注语言。
  □ 表格格式完整。
  □ 双链引用路径正确。
  □ 状态符号使用一致。

  □ 文件名不含空格、版本号、日期。
```
