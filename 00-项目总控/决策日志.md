## D001 — 2026-04-13 — 工作流整体方案

**背景**
过去 5 个月修修补补积累了大量问题。决定全新重构工作流，让"我 + 工作流软件 + AI 智能体"进入可循环的状态。

**选项**
- A：继续修修补补
- B：全新重构工作流

修修补补不可能解决根本问题，AI 风险难以避免。重构才能建立稳定的知识沉淀和 AI 协作流程。

## D002 — 2026-04-13 — 工具链六角分工

**背景**
手上有 Obsidian / Claude Project / Google Drive / GitHub / Cursor / PyCharm / Excalidraw / draw.io 等多个工具，职责模糊。

**选项**
- A：每个工具想用就用
- B：明确每个工具的唯一职责，不重叠

**结论**
选 B — 六角分工：
- Obsidian = 知识真相源
- Claude Project = 主推演区
- GitHub = 代码真相源
- Cursor / PyCharm = 编辑器分工
- Google Drive = 对外镜像层
- Excalidraw / draw.io = 只画两类图

**理由**
每个工具只做最擅长的事，数据流向单一可追溯。

## D003 — 2026-04-13 — 核心架构原则

**背景**
项目总大纲需要定下不可动摇的架构底线。

**选项**
原先的理解 vs 修正后的理解。

**结论**
三条原则：
1. 一个游戏 = 一套独立的中控 + 一套独立的脚本
2. 设备数据通过云端，不走本地（组队模块例外）
3. 网络验证系统是中枢

**理由**
避免做成"一个程序包含所有游戏"的全家桶；数据走云端保证跨 PC 可见；中枢架构让子系统关系清晰。

**关联**
[[项目总大纲]] 核心架构原则

## D004 — 2026-04-13 — 后端技术选型

**背景**
单个游戏项目可能有 10,000+ 安卓设备，心跳上报约 333 次/秒。需要选后端框架。

**选项**
- A：Python + FastAPI
- B：Go
- C：Python + Django

**结论**
选 A — Python + FastAPI。

**理由**
- FastAPI 性能足够扛 10,000 设备级并发
- 和 PC 中控、官网监控统一 Python 技术栈
- 学习资源丰富，文档质量高
- 不选 Go 避免学两门语言；不选 Django 因为它是同步框架

**关联**
[[项目总大纲]] 2.2 后端技术选型

## D005 — 2026-04-13 — 数据库选型

**选项**
- A：MySQL
- B：PostgreSQL
- C：SQLite

**结论**
选 B — PostgreSQL。

**理由**
- JSONB 类型适合设备运行数据（不同游戏字段不同）
- WITH RECURSIVE 适合多级代理递归查询
- Python 生态支持成熟（asyncpg / SQLAlchemy）

## D006 — 2026-04-13 — 部署双模式

**背景**
有两种部署场景：家庭 VPS（可能用宽带）+ 云服务商（大客户独立部署）。

**结论**
两种模式都必须支持，代码层兼容：
- 模式 A：家庭 VPS + FRP 内网穿透 + 云端中转（低配 1C1G）
- 模式 B：云服务商独立部署（2-4C 4-8G）

**理由**
通过配置外置让两套代码统一，降低维护成本。

## D010 — 2026-04-14 — 文件存储双模式

**选项**
- A：只用服务器本地文件系统
- B：只用对象存储（S3/MinIO）
- C：双模式，代码层通过存储抽象统一接口

**结论**
选 C。

**理由**
家庭 VPS 适合本地文件系统（简单）；云部署适合对象存储（易扩展）。抽象层让两种模式共用一套代码。

## D011 — 2026-04-14 — 仓库策略：独立仓库模式

**背景**
每个游戏的 PC 中控和安卓脚本需要基于框架基座独立修改。有两种组织方式。

**选项**
- A：分支模式（从基座拉分支）
- B：独立仓库模式（从基座 fork 成独立仓库）

**结论**
选 B — 独立仓库模式。

**理由**
每个游戏完全隔离，互不干扰。代价是框架基座更新时需要各仓库手动 cherry-pick / merge，但可接受。

## D012 — 2026-04-16 — PC↔安卓局域网通信方案

**背景**
组队功能需要 PC 和安卓在同一局域网内直连。原规划的"UDP 广播 → 安卓监听"方案在懒人精灵环境下不可行（懒人精灵无监听端口能力）。

**选项**
- A：用户手动在安卓脚本填入 PC 的 IP
- B：UDP 广播自动发现（不可行 — 懒人精灵限制）
- C：二维码扫描配对
- D：云端中转配对

**结论**
以方案 A 为主，WebSocket 协议（PC 当服务端，安卓当客户端 startWebSocket 主动连）。B/D 作为 Phase 2+ 扩展。C 被否决（工作室几百上千台手机不可能逐个扫码）。

**理由**
- 懒人精灵只有 WebSocket 客户端能力（startWebSocket），无法监听端口
- 工作室一般使用固定局域网 IP，填一次永久用
- PC 中控显示内网 IP + 复制按钮，降低配置门槛
