## D003 — 2026-04-13 — 核心架构原则

**背景**
项目总大纲需要定下不可动摇的架构底线。

**选项**
原先的理解 vs 修正后的理解。

**结论**
三条原则：
1. 一个游戏 = 一套独立的中控 + 一套独立的脚本
2. 设备数据通过云端，不走本地（组队模块例外）

**理由**
避免做成"一个程序包含所有游戏"的全家桶；数据走云端保证跨 PC 可见；中枢架构让子系统关系清晰。

## D012 — 2026-04-16 — PC↔安卓局域网通信方案

**背景**
组队功能需要 PC 和安卓在同一局域网内直连。原规划的"UDP 广播 → 安卓监听"方案在懒人精灵环境下不可行（懒人精灵无监听端口能力）。

**选项**
- A：用户手动在安卓脚本填入 PC 的 IP
- B：UDP 广播自动发现（不可行 — 懒人精灵限制）
- C：二维码扫描配对
- D：云端中转配对

**结论**
以方案 A 为主，WebSocket 协议（PC 当服务端，安卓当客户端 startWebSocket 主动连）。B/D 作为 Phase 2+ 扩展。C 被否决（工作室几百上千台手机不可能逐个扫码）。

**理由**
- 懒人精灵只有 WebSocket 客户端能力（startWebSocket），无法监听端口
- 工作室一般使用固定局域网 IP，填一次永久用
- PC 中控显示内网 IP + 复制按钮，降低配置门槛

## D013 — 2026-04-22 — 游戏库 URL 派生方式（双前缀 Bug 修复）

**背景**
`DATABASE_GAME_PREFIX` 配置值已含 `hive_game_` 后缀，若直接与 `code_name="game_001"` 拼接，会得到 `hive_game_game_001`（双前缀 Bug），导致游戏库连接失败。

**选项**
- A：直接字符串拼接 `f"{DATABASE_GAME_PREFIX}{code_name}"`（有 Bug）
- B：用 `make_url` 从主库 URL 派生，只替换数据库名

**结论**
选 B。统一规则：`db_name = "hive_" + code_name`，再用 `make_url(DATABASE_MAIN_URL).set(database=db_name)` 构造 URL。

**理由**
- `hive_` 前缀在命名上表示「属于蜂巢平台的库」，游戏库名为 `hive_game_001`，不应重复 `game_`
- `make_url` 方式完全复用主库的 host/port/user/password，避免配置冗余

**影响文件**
`app/database.py`、`scripts/init_data.py`、`tests/test_params.py`

## D014 — 2026-04-22 — 集成测试 conftest 三层 patch 模式

**背景**
集成测试中多处副作用绕过了 FastAPI DI 直接操作模块级对象，导致跨测试事件循环崩溃（`Event loop is closed`、`MissingGreenlet`）。

**结论**
`client` fixture 必须做三层 patch：

| 层级 | 目标对象 | 原因 |
|------|---------|------|
| 1 | `app.dependency_overrides[get_main_db/get_redis]` | FastAPI DI 覆盖，每测试独立 Session/Redis |
| 2 | `app.database._main_session_factory` + `app.services.auth_service._main_session_factory` | `_write_login_log` 通过 `from...import` 持有旧引用，DI 覆盖对它无效 |
| 3 | `app.routers.auth.incr_rate_limit` | 测试共用同一 IP，32 用例超出 10次/分钟限流 |

**关键陷阱**
`from app.database import _main_session_factory` 在模块加载时绑定旧引用，patch `app.database` 的属性不影响已绑定的模块变量，必须同时 patch 使用方模块自身的引用。

**影响文件**
`tests/conftest.py` V1.0.5

**关联**
[[99-Claude对话提炼/测试基础设施规范]]

## D015 — 2026-04-24 — 测试中游戏库操作改用原生 asyncpg

**背景**
在 pytest session loop（Windows ProactorEventLoop）中，SQLAlchemy asyncpg dialect 对 `hive_game_001` 的 SCRAM-SHA-256 认证持续返回 `InvalidPasswordError`，但原生 `asyncpg.connect()` 完全正常。

**根因**
Windows ProactorEventLoop + SQLAlchemy asyncpg dialect 在 pytest-asyncio session scope fixture 中存在兼容问题，与账号权限无关（直接 asyncpg 连接验证通过）。

**结论**
测试中凡是需要直接操作游戏库的地方（如 `seed_params` fixture），一律使用原生 `asyncpg.connect()`，不用 SQLAlchemy engine。

**影响文件**
`tests/test_params.py` V1.0.2（`seed_params` fixture）

**关联**
[[99-Claude对话提炼/测试基础设施规范]]

## D016 — 2026-04-24 — pg_hba.conf 认证方式统一为 scram-sha-256

**背景**
`hive_user` 密码以 `SCRAM-SHA-256` 格式存储，但 `pg_hba.conf` TCP 规则为 `md5`，导致 `hive_user` 通过 TCP 连接任何数据库均失败（含 `hive_game_001`）。

**结论**
将 `/etc/postgresql/16/main/pg_hba.conf` 中：
```text
host    all    all    0.0.0.0/0    md5
```
改为：
```text
host    all    all    0.0.0.0/0    scram-sha-256
```
重载配置：`SELECT pg_reload_conf();`

**理由**
新版 PostgreSQL 默认使用 scram-sha-256，比 md5 更安全，认证格式需与密码存储格式一致。

**一次性配置，不需要重复操作。**

**关联**
[[99-Claude对话提炼/测试基础设施规范]]

## D015 — 2026-04-26 — Vue Router 单一父路由策略

**背景**
`sharedRoutes` 和 `adminRoutes` 两文件均以 `path:'/'` 注册父路由，Vue Router 4 只匹配第一棵树，导致后子路由（`users/:id`、`login-logs`、`updates` 等）全部 404。

**结论**
所有登录后页面路由合并进单一 `path:'/'` 父路由，第二级子路由通过 `meta.requiresAdmin` 区分权限。

**设计原则**
不能将同一 `path` 的父路由拆分到两个数组里再展开合并，路由必须在单一地方完整定义。

**关联**
`frontend/src/router/index.js`

## D017 — 2026-04-27 — 组队任务执行通信节奏

**背景**
安卓脚本通过 WebSocket 接收 PC中控下发的组队任务指令。任务可能连续，多台设备需要协调执行。需要确定"连续任务"的推进节奏由谁驱动。

**选项**
- A：PC中控驱动 — 设备完成一步后上报，PC中控收到后再下发下一步。安卓侧只负责执行单步并回报结果，自身不做连续推进决策。
- B：安卓自驱动 — PC中控下发完整任务包，安卓按序执行多个步骤，只在关键节点上报状态。

**结论**
暂选 A — PC中控驱动。B 作为 Phase 2+ 优化选项保留。

**理由**
- PC中控掌握所有设备的全局状态，由它驱动节奏更利于组队协调
- 安卓侧实现简单清晰：收指令 → 执行 → 上报，三段式无歧义
- WebSocket 消息量较大，但局域网带宽充裕可接受
- B 方案在网络抖动场景下更稳定，留作后续优化方向

**关联**
[[03-安卓脚本框架/架构设计]] comm_lan.lua 消息协议
