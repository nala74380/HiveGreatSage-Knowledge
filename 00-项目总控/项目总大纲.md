## 〇、项目定位与核心架构原则

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：

- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控**框架**（每个游戏基于框架独立修改、独立打包）
- 一个安卓端自动化脚本**框架**（每个游戏基于框架独立修改、独立打包）
- 辅助工具（官网监控、素材提取等）

### ⚠️ 核心架构原则（v2 修正）

```
原则1：一个游戏 = 一套独立的中控 + 一套独立的脚本

  ✗ 错误理解：一个中控程序动态加载所有游戏
  ✓ 正确理解：中控框架是"基座"，每个游戏基于基座修改后独立打包

  这意味着：
  · PC中控框架仓库 = 公共基座代码 + 游戏A分支 + 游戏B分支（或独立仓库）
  · 安卓脚本框架仓库 = 公共基座代码 + 游戏A分支 + 游戏B分支（或独立仓库）
  · 用户下载的是"游戏A专用中控.exe" + "游戏A专用脚本.lrj"
  · 不同游戏的中控/脚本之间互不干扰

原则2：设备数据通过云端（网络验证系统），不走本地

  ✗ 错误理解：设备列表和运行数据存在 PC 本地
  ✓ 正确理解：设备数据存在网络验证系统的云端数据库中
    换了 PC 登录同一账号 → 数据还在
    安卓设备上报数据 → 发到云端 → PC 中控从云端拉取显示

  唯一例外：组队调度模块 — 现阶段仅支持局域网内
    原因：实时通讯需求 + 成本考虑
    PC 和安卓手机必须在同一局域网内才能组队

原则3：网络验证系统是"中枢"

  所有子系统都围绕网络验证系统运转：
  · PC中控 → 登录验证、拉取设备数据、下载参数配置、热更新
  · 安卓脚本 → 登录验证、上报运行数据、热更新
  · 官网监控 → 推送变更通知（可选集成）
```

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是"一个程序包含所有游戏"的全家桶
- 不涉及对任何在运营游戏的讨论

### 平台的商业模型概要

```
管理员
  └── 创建/管理 代理 L1（一级代理）
        └── 代理 L1 可创建/管理 代理 L2（二级代理）
              └── 代理 L2 可创建/管理 代理 L3 ...（支持多级）
                    └── 任意级代理都可创建/管理 用户（5级）
                          └── 用户购买 游戏项目授权
                                └── 使用 该游戏专用PC中控 + 该游戏专用安卓脚本

用户不可自行注册，必须由代理或管理员创建。
代理无权创建"项目测试用户"级别（仅管理员可创建）。
```

## 二、子系统1：网络验证系统（中枢）

**代号：** Verify **仓库：** HiveGreatSage-Verify **编辑器：** PyCharm

### 2.1 规模与性能需求

```
关键数据：
  · 单个游戏项目可能有 10,000+ 台安卓设备同时运行脚本
  · 单个 VIP/SVIP 客户可能有 3,000+ 台设备
  · 多个游戏项目并行 → 总设备数可能远超 10,000

这意味着网络验证系统需要承受的并发压力：
  · 登录验证：集中在脚本启动时段（可能有几千台同时启动）
  · 心跳/状态上报：10,000+ 设备按固定间隔上报（如每30秒一次）
    → 最坏情况约 333 次/秒的持续写入
  · 数据查询：PC中控拉取设备状态/数据（频率较低，但可能聚合大量数据）
  · 热更新下载：版本发布时可能有大量设备同时请求下载
```

### 2.2 后端技术选型（分析与建议）

```
你的情况：
  · PC中控用 Python (PySide 6) → 后端也用 Python 可以统一技术栈
  · 数据库已选 PostgreSQL → Python 生态对 PostgreSQL 支持成熟
  · 你需要从初学者角度能学会部署

选型建议：Python + FastAPI

理由：
  1. 性能足够：
     FastAPI 基于 ASGI 异步架构，在 2025 年基准测试中
     性能接近 Go 和 Node.js，能够高效处理上万并发连接。
     10,000 设备的心跳上报（~333次/秒）对 FastAPI 来说不是问题。

  2. 技术栈统一：
     PC中控（Python）、后端（Python）、官网监控（Python）
     你只需要深入学一门语言，降低学习和维护成本。

  3. 学习资源丰富：
     FastAPI 官方文档质量极高，有中文社区，
     从入门到部署的教程非常完整。

  4. PostgreSQL 集成：
     使用 SQLAlchemy（ORM）+ asyncpg（异步驱动）
     或 Tortoise ORM，都是成熟方案。

  5. 自动生成 API 文档：
     FastAPI 自带 Swagger UI，开发调试时极其方便。

不选 Go 的原因：
  · Go 性能确实更高，但 10,000 设备级别 FastAPI 完全能扛
  · 你要同时学 Go + Python 两门语言，精力分散
  · PC 中控和后端无法共享代码
  · 等未来规模真的到了 FastAPI 扛不住再考虑迁移也不迟

不选 Django 的原因：
  · Django 是同步框架，高并发处理不如 FastAPI
  · Django 更适合传统网站，FastAPI 更适合 API 服务

技术栈确认：
  ✓ 语言：Python 3.11+
  ✓ Web框架：FastAPI
  ✓ ASGI服务器：Uvicorn（开发）/ Uvicorn + Gunicorn（生产）
  ✓ 数据库：PostgreSQL
  ✓ ORM：SQLAlchemy 2.0（async 模式）+ asyncpg
  ✓ 数据校验：Pydantic v2（FastAPI 内置）
  ✓ 缓存/消息：Redis
      用途1：设备心跳缓存（10,000+ 设备高频上报，先写 Redis 再批量落库）
      用途2：Token 存储 / 会话管理
      用途3：Celery 任务队列的 Broker
  ✓ 任务队列：Celery + Redis
      用途：热更新通知推送、批量数据处理等异步任务
  ✓ 管理后台前端：Vue 3（前后端分离）
      后端提供 RESTful API，前端 Vue 独立部署
  ✓ 文件存储：双模式支持
      模式A：服务器本地文件系统（VPS自建场景，简单直接）
      模式B：对象存储（S3/MinIO，云部署场景，易扩展）
      代码层通过存储抽象层统一接口，部署时配置选择使用哪种
```

### 2.3 部署方案

```
方案A：VPS 自建（家庭宽带场景）

  问题：家庭宽带通常没有公网 IP 或公网 IP 会变
  解决方案：使用内网穿透中转

  架构：
    [家庭VPS] ←── FRP/内网穿透 ──→ [云端中转服务器（最低配）]
                                        ↑
                                    公网域名指向这里
                                        ↑
                              [PC中控 / 安卓脚本 连接]

  云端中转服务器只负责流量转发，不跑业务逻辑，
  所以可以选最便宜的云服务器（1核1G就够）。
  真正的验证系统、数据库跑在家庭VPS上。

  优点：节省服务器成本
  缺点：依赖家庭网络稳定性、需要维护穿透服务

方案B：云服务商部署（大客户场景）

  适用场景：某个 VIP/SVIP 客户有 3,000+ 设备
  该客户可能需要独立部署一套验证系统

  架构：
    [云服务器] ← 该客户专用
    · 独立的 FastAPI 实例
    · 独立的 PostgreSQL 数据库
    · 独立的域名/子域名

  好处：大客户的数据隔离、性能独占
  需要的服务器配置（3000设备预估）：
    · CPU: 2-4核
    · 内存: 4-8GB
    · 存储: 50GB+（取决于数据量）
    · 带宽: 看热更新包大小和更新频率

两种方案必须都能部署：
  □ 需要编写统一的部署文档/脚本
  □ 使用 Docker 容器化可以让两种方案的部署步骤一致
    （但这是 Phase 2+ 的事，MVP 阶段先手动部署）
  □ 配置文件外置（数据库地址、域名等），不硬编码
```

### 2.5 数据库设计要点

**数据库A — 用户管理库（所有游戏项目共用，PostgreSQL）**

```
核心表：

  admin（管理员表）
    id, username, password_hash, created_at, status

  agent（代理表）— 支持多级代理的树形结构
    id, username, password_hash,
    parent_agent_id → agent.id（上级代理，NULL=顶级代理直属管理员）
    level（代理层级：1=一级代理，2=二级代理...）
    created_by_admin_id / created_by_agent_id
    max_users（可创建用户数上限）
    status, created_at

  user（用户表）
    id, username, password_hash,
    user_level（枚举：trial/normal/vip/svip/tester）
    created_by_agent_id → agent.id（哪个代理创建的）
    created_by_admin（布尔，是否管理员直接创建）
    status, created_at, expired_at

  authorization（授权表）— 用户 × 游戏项目
    id, user_id → user.id,
    game_project_id（游戏项目标识）
    valid_from, valid_until
    status

  device_binding（设备绑定表）
    id, user_id → user.id,
    device_fingerprint（安卓设备指纹）
    bound_at, status

  login_log（登录日志表）
    id, user_id, device_fingerprint, ip, login_at, client_type(pc/android)

关键约束：
  · user_level = 'tester' 时，created_by_admin 必须为 true
  · 代理只能看到并管理自己创建的 + 下级代理创建的用户
  · 代理无法越级操作（L2代理不能管L1代理的直属用户）
```

**数据库B — 游戏项目库（每个游戏项目独立，PostgreSQL）**

```
为什么独立：
  · 每个游戏的安卓端采集数据结构完全不同
  · 每个游戏的 PC 中控需要显示/设置的内容完全不同
  · 项目下线时可以直接归档/删除该库，不影响其他项目

每个游戏项目库包含：
  project_config    → 该游戏的全局配置
  device_runtime    → 安卓设备上报的实时运行数据
                      （这是 PC 中控拉取并显示的数据来源）
  script_params     → 该游戏可配置的脚本参数定义 + 用户的参数值
  version_record    → PC包/安卓包的版本历史
  custom_*          → 因游戏而异的自定义表

device_runtime 表设计思路：
  · 由安卓脚本定时上报（如每30秒）
  · 包含：user_id, device_id, 上报时间, 运行状态, 游戏数据(JSON)
  · PC中控按 user_id 查询自己所有设备的最新数据
  · 游戏数据字段用 JSONB 类型，不同游戏内容不同
```

### 2.6 多级代理的权限模型

```
代理树形结构示例：

  管理员
  ├── 代理A（L1）
  │   ├── 代理A1（L2）
  │   │   ├── 代理A1a（L3）
  │   │   │   └── 用户...
  │   │   └── 用户...
  │   ├── 代理A2（L2）
  │   │   └── 用户...
  │   └── 用户...（代理A直属用户）
  └── 代理B（L1）
      └── 用户...

权限规则：
  · 每个代理能看到：自己直属的用户 + 所有下级代理 + 下级代理的用户
  · 代理不能看到：同级代理的数据、上级代理的数据
  · 管理员能看到：所有
  · 代理能做的操作范围由上级设定（如最多创建多少用户、创建什么级别的用户）

技术实现思路：
  · 查询下级代理：递归查询 parent_agent_id（PostgreSQL 支持 WITH RECURSIVE）
  · 查询可见用户：先查出自己 + 所有下级代理的 agent_id 列表 → 再查 user 表
```

### 2.7 热更新机制

```
热更新目录结构（服务端）：
  hot_update/
  ├── 游戏项目A/
  │   ├── pc/
  │   │   ├── latest.json          ← {version, file_list, checksums, download_url}
  │   │   └── packages/
  │   │       └── v1.0.1/
  │   └── android/
  │       ├── latest.json
  │       └── packages/
  │           └── v1.0.1.lrj       ← 懒人精灵热更新包
  └── 游戏项目B/
      └── ...

安卓端热更新流程：
  脚本启动 → 登录验证通过 → 请求 latest.json → 比对本地版本
  → 如有更新 → 下载 .lrj → 校验完整性 → 调用 installLRPkg → 重载脚本

PC端热更新流程：
  中控启动 → 登录验证通过 → 请求 latest.json → 比对本地版本
  → 如有更新 → 下载更新包 → 校验 → 解压覆盖 → 提示重启
  □ PC 端更新后如何安全重启自身？→ 待推演
```

### 2.8 待推演的决策点


### 3.1 "框架基座 + 独立仓库"的工作模式（已确认）

```
中控框架不是一个"成品程序"，而是一个"项目模板 + 公共代码库"。

仓库策略：独立仓库模式 ✓

  HiveGreatSage-PCControl          ← 框架基座（公共代码库）
  HiveGreatSage-PCControl-GameA    ← 游戏A的中控（从基座 fork）
  HiveGreatSage-PCControl-GameB    ← 游戏B的中控（从基座 fork）

  优点：每个游戏的中控完全隔离，互不干扰，各自独立版本管理
  代价：框架基座更新时需要手动同步到各游戏仓库

开发流程：
  1. 框架基座仓库包含所有游戏共用的代码（验证、数据同步、热更新等）
  2. 适配新游戏时：
     · fork 框架基座 → 创建 HiveGreatSage-PCControl-GameX
     · 在新仓库中修改/新增该游戏的定制 UI 和业务逻辑
     · 独立打包为 "游戏X中控.exe"
  3. 框架基座有更新时：
     · 在基座仓库打 tag 标记版本
     · 各游戏仓库按需 cherry-pick 或 merge 基座更新
     · 不强制所有游戏同步更新——稳定运行的游戏可以暂不合并

用户下载的是 "游戏A中控 v1.2.0.exe"，不是通用中控。
```

## 九、AI 协作安全守则（本项目专用）

基于《AI安全术语整理》：

|风险|在本项目中的表现|应对措施|
|---|---|---|
|**Hallucination**|编造懒人精灵 API、编造 PySide 6 不存在的方法|所有 API 必须对照官方文档验证|
|**Sycophancy**|"这个方案完全没问题" 但实际有坑|每个方案追问"最大风险是什么"|
