## 〇、项目定位与核心架构原则

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：

- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控**框架**（每个游戏基于框架独立修改、独立打包）
- 一个安卓端自动化脚本**框架**（每个游戏基于框架独立修改、独立打包）
- 跨账号智能调度能力（账号分工 + 供需匹配 + 跨服数据）
- 辅助工具（官网监控、素材提取等）

### ⚠️ 核心架构原则

```text
原则1：一个游戏 = 一套独立的中控 + 一套独立的脚本
  中控框架是"基座"，每个游戏基于基座修改后独立打包。
  用户下载的是"游戏A专用中控.exe" + "游戏A专用脚本.lrj"，不是通用程序。

原则2：设备数据通过云端（网络验证系统），不走本地
  安卓设备上报数据 → 云端存储 → PC 中控从云端拉取显示。
  换 PC 登录同一账号，数据还在。
  唯一例外：组队调度模块 — 仅支持同一局域网内（详见 7.3）。

原则3：网络验证系统是"中枢"
  所有子系统都围绕网络验证系统运转。
```

详见 [[决策日志]] D003。

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是"一个程序包含所有游戏"的全家桶
- 不涉及对任何在运营游戏的讨论

### 平台的商业模型

```text
管理员
  └── 创建/管理 代理 L1（一级代理）
        └── 代理 L1 可创建/管理 代理 L2（二级代理）
              └── 代理 L2 可创建/管理 代理 L3 ...（支持多级）
                    └── 任意级代理都可创建/管理 用户（5级）
                          └── 用户购买 游戏项目授权
                                └── 使用 该游戏专用PC中控 + 该游戏专用安卓脚本

用户不可自行注册，必须由代理或管理员创建。
代理无权创建"项目测试用户"（tester）级别，仅管理员可。
```

多级代理实现详见 [[01-网络验证系统/代理管理方案]]（已存在）。

## 二、子系统1：网络验证系统（中枢）

**代号：** Verify
**仓库：** HiveGreatSage-Verify
**编辑器：** PyCharm
**深入文档：** [[01-网络验证系统/架构设计]]

### 2.1 规模与性能需求

```text
关键数据：
  · 单个游戏项目可能有 10,000+ 台安卓设备同时运行脚本
  · 单个 VIP/SVIP 客户可能有 3,000+ 台设备
  · 多个游戏项目并行 → 总设备数可能远超 10,000

并发压力：
  · 登录验证：集中在脚本启动时段
  · 心跳上报：10,000+ 设备 / 每 30 秒一次 ≈ 333 次/秒持续写入
  · 数据查询：PC 中控定时拉取
  · 热更新下载：版本发布时集中请求
```

### 2.3 部署方案（双模式，已确认 D006）

**模式 A：家庭 VPS 自建**
```text
[家庭 VPS] ←── FRP/内网穿透 ──→ [云端中转服务器（1C1G）]
                                         ↑
                                     公网域名
                                         ↑
                           [PC中控 / 安卓脚本 连接]
```

**模式 B：云服务商部署（大客户独立部署）**
```text
[云服务器 2-4C 4-8G]
  · 独立的 FastAPI 实例
  · 独立的 PostgreSQL + Redis
  · 独立的域名/子域名
```

**两种模式代码必须兼容**：通过配置外置（数据库地址、域名等）实现，不硬编码。

**代号：** PCControl

**编辑器：** PyCharm
**GUI 框架：** PySide 6
**深入文档：** [[02-PC中控框架/架构设计]]

### 3.1 "框架基座 + 独立仓库"模式（已确认 D011）

```text
HiveGreatSage-PCControl          ← 框架基座（公共代码库）
HiveGreatSage-PCControl-GameA    ← 游戏A的中控（从基座 fork）
HiveGreatSage-PCControl-GameB    ← 游戏B的中控（从基座 fork）
```

每个游戏的中控完全隔离，独立打包为 `.exe`。框架基座更新时各游戏仓库按需 cherry-pick / merge，不强制同步。

## 七、跨子系统设计

### 7.1 项目管理视角：一个新游戏怎么接入

```text
Step 1 — 网络验证系统注册新游戏项目
  管理员操作，系统自动创建独立 PostgreSQL 数据库 + 热更新目录

Step 2 — 开发 PC 中控（该游戏版本）
  fork HiveGreatSage-PCControl → HiveGreatSage-PCControl-GameX
  修改 game/ 目录下的定制内容 → 打包 .exe

Step 3 — 开发安卓脚本（该游戏版本）
  fork HiveGreatSage-AndroidScript → HiveGreatSage-AndroidScript-GameX
  编写 ui_map.lua + tasks/*.lua → 打包 .lrj 或 APK

Step 4 — 热更新发布
  上传 PC 包和安卓包 → 更新 latest.json

Step 5 — 授权用户
  代理/管理员为用户添加该游戏项目的授权
```

详见 [[06-游戏适配/_模板_游戏适配文档]]（已存在）。

### 7.2 各子系统 API 端点

```text
网络验证系统统一 API（✅ 已实现 / 🔲 待实现）：

✅ /api/auth/login            ← PC 中控和安卓脚本都调用
✅ /api/auth/verify-token     ← Token 有效性校验
✅ /api/auth/refresh          ← Token 刷新
✅ /api/auth/logout           ← 登出
✅ /api/auth/me               ← 获取当前用户信息
✅ /api/auth/revoke-all      ← 踢出所有设备（token_version 已落地）

✅ /api/device/heartbeat      ← 安卓脚本定时调用（上报状态+数据）
✅ /api/device/list           ← PC 中控调用（获取设备状态列表）
✅ /api/device/data           ← PC 中控调用（获取设备运行数据详情）
✅ /api/device/imsi           ← IMSI 上传（Fernet 加密 + hash 反查已落地）

✅ /api/params/get            ← 安卓脚本调用（获取用户设置的参数）
✅ /api/params/set            ← PC 中控调用（保存用户修改的参数）

✅ /api/update/check          ← PC 和安卓都调用（检查新版本）
✅ /api/update/download       ← 获取签名下载 URL
✅ /admin/api/updates/        ← 管理员上传 + 分块SHA-256 + 审计

/api/market/price-report      ← 跨服价格上报（Phase 2+）
/api/market/price-overview    ← 跨服价格查询（Phase 2+）

/api/task/demand              ← 发布需求单（Phase 2+）
/api/task/assignment          ← 获取分配的任务（Phase 2+）

✅ /api/admin/*               ← 管理后台 Web API（基础实现）
✅ /api/agent/*               ← 代理后台 Web API（基础实现）
✅ /admin/api/devices/        ← Admin/Agent 分权设备监控 + 黑名单校验
```

所有 API 都需要携带：
- `Authorization: Bearer {token}` — 身份验证
- JWT payload 中已含 `project`（game_project.code_name），无需手动传 game_project_id

### 7.3 跨账号智能调度（Phase 2+）

本平台不仅支持单设备单账号独立运行，还支持账号间的角色分工与协作。

**四个核心场景：**

1. **老号带新号（轮转带队）**
2. **账号角色系统（角色分工）** — captain / power / farmer / newbie
3. **供需匹配（战力号 + 打工号协作）**
4. **跨服价格监控**

**建议实现顺序：** 场景4 → 场景2基础 → 场景1 → 场景3

详见 [[02-PC中控框架/跨账号智能调度分析]]。

### 7.4 PC ↔ 安卓局域网通信（组队场景专用）

- **PC 中控** = WebSocket 服务端，监听 8889 端口
- **安卓脚本** = WebSocket 客户端，通过 `startWebSocket` 主动连接
- **配对方式** = 用户手动在安卓脚本配置中填入 PC 的内网 IP

详细方案见 [[02-PC中控框架/PC安卓局域网通信方案]]。

## 九、AI 协作安全守则（本项目专用）

基于 [[00-项目总控/AI安全术语表]]：

| 风险 | 在本项目中的表现 | 应对措施 |
|------|----------------|---------|
| Hallucination | 编造懒人精灵 API、编造 PySide 6 不存在的方法 | 对照 [[03-安卓脚本框架/懒人精灵函数]] 等官方文档验证 |
| Sycophancy | "这个方案完全没问题" 但实际有坑 | 每个方案追问"最大风险是什么" |
| Specification Gaming | 代码字面满足要求但逻辑不对 | 每段代码读懂再用 |
| Confabulation | 补全了你没给的上下文 | 每次对话给足上下文 |
| Mirage Reasoning | 推理看似合理但前提错误 | 检查前提假设 |

**硬规则：** AI 输出（ChatGPT / DeepSeek / Claude）与实测不一致 → 以源码、测试、实机日志、运行结果为准；不确定内容必须标为[待确认]；修正结论记录到 [[决策日志]]。
