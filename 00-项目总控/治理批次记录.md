## 5. 文档治理处置标准

### 5.1 删除标准

满足任一条件，进入删除候选：

```text
1. 内容已经被主文档完整吸收。
2. 文件只是日期版补丁，无独立长期价值。
3. 文件内容与当前代码或当前决策冲突。
4. 文件是空壳、占位、标题党。
5. 文件包含敏感明文，不应留在知识库。
6. 文件只是 AI 对话原文，未提炼成项目知识。
7. 文件与另一文件高度重复，且没有新增事实。
```

删除前必须记录：

```text
00-项目总控/文档治理清理清单.md
```

建议记录字段：

```text
文件路径 | 处置 | 原因 | 是否迁移有效内容 | 迁移到 | 回退方式 | 执行状态
```

### 5.2 合并标准

满足任一条件，进入合并候选：

```text
1. 多份文档讲的是同一主题。
2. 一份是旧版，一份是新版。
3. 一份是方案，一份是补丁，一份是复核结论。
4. 多份文档分别记录同一功能的不同阶段。
5. 文件名不同，但实际都在描述同一个模块。
```

合并后的主文档必须使用稳定命名，不使用日期补丁命名。

### 5.3 拆分标准

满足任一条件，进入拆分候选：

```text
1. 单文档超过一个模块边界。
2. 单文档同时包含架构、接口、数据库、测试、实现计划。
3. 单文档既描述 Verify，又描述 PCControl，又描述 AndroidScript。
4. 单文档中“已确认 / 待确认 / 建议方案 / 实现计划”混在一起。
5. 一份文档已经变成后续多轮开发的总包，不利于执行。
```

拆分原则：按职责拆，不按篇幅拆。

## 6. 精读要求

### 6.1 精读顺序

涉及当前实现状态的文档，必须按以下顺序治理：

```text
1. 精读 Obsidian 真相源文档。
2. 精读 Verify 对应 Router / Service / Schema / Model。
3. 精读 PCControl 对应 API Client / 调用模块 / UI 或任务模块。
4. 精读 AndroidScript 对应 Lua 模块。
5. 精读测试文件或测试清单。
6. 再更新文档代码测试追踪矩阵。
```

### 6.2 证据等级

```text
D0 = 仅有文档描述，未读代码。
D1 = 已精读 Obsidian 真相源文档。
C1 = 已精读对应源码，确认模块/接口/字段存在。
T1 = 已精读测试文件或测试清单。
R1 = 已运行单仓测试。
E2 = 已完成跨端联调。
M1 = 已完成真实 PC / 安卓实机验证。
```

状态判断不得越级：

```text
D1 + C1 + T1 只能说明“文档、源码、测试存在或有覆盖”。
未达到 R1 / E2 / M1 前，不得写“已闭环”“已可发布”。
```
