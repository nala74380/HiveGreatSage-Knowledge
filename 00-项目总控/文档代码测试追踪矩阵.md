# 文档代码测试追踪矩阵

> 目的：把 Obsidian 文档、Verify 源码、PCControl 源码、AndroidScript 源码和测试证据放到同一张追踪表中，防止“文档说已完成、代码未实现”或“代码已改、文档漂移”。

## 一、状态与证据等级

### 1.1 当前状态

```text
待精读       = 尚未完整读对应文档和代码。
源码已确认   = 已精读代码，确认模块/接口/字段存在。
待运行验证   = 源码和测试存在，但尚未启动依赖实际跑通。
待三端联调   = 需要 Verify + PCControl + AndroidScript 联调。
待实机验证   = 需要真实 PC exe 或懒人精灵环境验证。
高风险       = 当前存在明确缺口或未闭环风险。
已闭环       = 文档、源码、测试、联调或实机验证均已完成。
```

### 1.2 证据等级

```text
D0 = 仅有文档描述，未读代码。
D1 = 已精读 Obsidian 真相源文档。
C1 = 已精读对应源码，确认模块/接口/字段存在。
T1 = 已精读测试文件或测试清单。
R1 = 已运行单仓测试。
E2 = 已完成跨端联调。
M1 = 已完成真实 PC / 安卓实机验证。
```

> 判定规则：没有 R1 / E2 / M1 时，不允许把能力写成“已闭环”。

## 三、热更新链路精读摘要

### 3.1 已精读文档

```text
01-网络验证系统/热更新端到端测试清单.md
```

文档结论：

```text
1. 热更新不是单接口测试，而是 Verify 管理端、Verify 客户端、PCControl、AndroidScript 的端到端链路。
2. 当前源码层面已确认存在热更新模块。
3. 主库/游戏库 version_record 真相源已统一。
4. 在完成 PCControl / AndroidScript 下载、校验、安装实测前，不应标记热更新端到端可用。
```

### 3.2 已精读 Verify 源码

```text
app/routers/update.py
app/services/update_service.py
app/routers/update_admin.py
tests/test_update.py
tests/test_update_admin.py
```

当前确认：

```text
1. /api/update/check 存在，要求 client_type=pc/android，current_version=MAJOR.MINOR.PATCH。
2. /api/update/download 存在，返回 download_url / expires_at / version / checksum_sha256。
3. update_service.py 从主库 GameProject + VersionRecord 查 active 版本。
4. Redis key 为 update:latest:{game_project_code}:{client_type}。
5. update_admin.py 上传路径为 /admin/api/updates/{project_id}/{client_type}。
6. update_admin.py 分块写临时文件并计算 SHA-256，限制 500MB。
7. update_admin.py 写主库 VersionRecord，并删除 Redis 缓存。
8. tests/test_update.py 已写主库 version_record。
9. tests/test_update_admin.py 已使用 project_id 路径。
```

### 3.3 已精读 PCControl 源码

```text
core/api_client/update_api.py
core/updater/update_checker.py
core/updater/update_downloader.py
core/updater/update_installer.py
```

当前确认：

```text
1. PCControl 已封装 /api/update/check。
2. PCControl 已封装 /api/update/download。
3. update_checker.py 将 Verify 返回的 current_version 作为 new_version 使用。
4. update_downloader.py 会下载包并校验 SHA-256。
5. SHA-256 不一致时删除临时文件并失败退出。
6. update_installer.py 开发模式直接解压覆盖并 os.execv 重启。
7. exe 模式生成 updater.bat，由 bat 解压覆盖并重启 exe。
```

风险：

```text
exe 模式覆盖自身文件仍需真实打包验证。
```

### 3.4 已精读 AndroidScript 源码

```text
脚本/framework/updater.lua
```

当前确认：

```text
1. AndroidScript 登录后可调用 /api/update/check?client_type=android。
2. check 响应中 current_version 被视为服务端 active 版本。
3. AndroidScript 可调用 /api/update/download?client_type=android。
4. 下载成功后调用 installLrPkg(save_path)。
5. installLrPkg 使用小写 r，不是 installLRPkg。
6. 本地版本号不在安装前写入，避免安装失败误判已更新。
7. force_update 失败时会 exitScript 阻断继续运行。
```

明确缺口：

```text
当前 updater.lua 明确记录：收到 checksum_sha256，但当前懒人精灵环境尚未接入 SHA-256 文件校验，待补强。
```

## 五、矩阵维护规则

1. 新增能力时，必须同时填写文档、代码、测试三类证据。
2. 未精读代码时，状态只能写“待精读”。
3. 源码已存在但未运行，只能写“源码已确认”或“待运行验证”。
4. 涉及 PCControl / AndroidScript 的能力，未三端联调前不能写“已闭环”。
5. 涉及真实设备、懒人精灵、exe 自更新的能力，未实机验证前不能写“可发布”。
6. 文档与代码冲突时，先以源码确认当前实现，再判断源码是否合理；不能盲目改文档迎合源码。
