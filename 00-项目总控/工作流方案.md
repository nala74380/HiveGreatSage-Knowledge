# 蜂巢·大圣 (Hive-GreatSage) 工作流重构方案

> **文档性质：** 工作流唯一参考文档，后续所有工具配置以此为准 
> **适用阶段：** 项目全生命周期（6个月~多年） 
> **核心原则：** 每个工具只做它最擅长的事，职责不重叠，数据流向单一且可追溯

---

## 一、工具角色定义（六个位置，各司其职）

### 1. Obsidian — 知识真相源（Knowledge Truth）

**它是什么：** 整个项目的"大脑"，所有非代码内容的唯一权威存储地。

**它负责：**

- 项目总大纲及所有子模块的架构文档
- 每次 Claude 对话后的**结论提取**（不是对话记录，是提炼后的决策/知识）
- 技术方案的定稿版本（草稿在 Claude 里推演，定稿搬到 Obsidian）
- 问题追踪清单（什么问题出现了、怎么解决的、当前状态）
- AI安全术语表（你已有的那份，作为项目级参考）
- 版本决策日志（为什么选A不选B，当时的理由是什么）

**它不负责：**

- 不存代码（代码去 GitHub）
- 不存正式对外文档（对外文档去 Google Drive）
- 不做实时协作（Obsidian 是本地优先工具）

**Obsidian 库结构建议：**

```text

**关键操作习惯（必须养成）：**

- 每次 Claude 对话结束后，花 5-15 分钟把结论写入 Obsidian 对应位置
- 使用 `双链` 把相关文档串起来，形成知识网络
- 文档头部统一格式：`状态: 草稿/定稿/过时` + `最后更新: 日期` + `关联Claude对话: 链接`

### 2. Claude Project — 主推演区

**它是什么：** 设计思考和技术推演的"会议室"，不是存储的地方。

**它负责：**

- 架构设计的讨论和推演（"这个方案行不行？有什么坑？"）
- 代码框架的原型草稿（先在这里讨论清楚，再到 Cursor 里写）
- 技术选型分析（"用 SQLite 还是 MySQL？为什么？"）
- 文档草稿的撰写和修改
- Bug 排查思路讨论

**它不负责：**

- 不是知识存储（结论搬去 Obsidian）
- 不是代码仓库（代码搬去 GitHub）
- 不期望 Claude 记住所有历史（每次对话要给足上下文）

**Claude Project 窗口组织策略：**

```text
蜂巢·大圣 Project 内的对话窗口规划：

窗口命名规则：[模块编号]-[主题]-[日期]
例如：
  02-PC中控-通信协议设计-0413
  03-安卓脚本-OCR方案选型-0415
  00-工作流-整体复盘-0420

不要做的事：
  ✗ 一个窗口里讨论3个不同模块 → 上下文混乱
  ✗ 不提供背景就问问题 → Claude 会 hallucinate（你的术语表第一条）
  ✗ 在对话里"修修补补" → 这正是你说的过去5个月的教训
```

**每次开启新对话的标准流程：**

```text
第1步：从 Obsidian 复制相关模块的当前架构文档（给 Claude 大局观）
第2步：明确本次对话的目标（"这次我要解决 XXX 问题"）
第3步：如果是接续之前的讨论，贴上上次的结论摘要
第4步：对话进行
第5步：对话结束 → 提炼结论 → 写入 Obsidian → 标记对话链接
```

**给 Claude 大局观的方法：**

你说得对，这个项目需要给 AI 很强的项目大局观。实际操作就是——

在 Claude Project 的 **Project Knowledge**（项目知识库）中上传/粘贴以下内容：

- `项目总大纲.md`（让 Claude 知道整个项目的全貌）
- `AI安全术语表.md`（让 Claude 遵守你的术语规范）
- 当前正在工作的模块的架构设计文档

每当架构文档有更新，同步更新 Project Knowledge 中的版本。

### 3. GitHub — 代码真相源（Code Truth）

**它是什么：** 所有代码的唯一权威存储地。

**它负责：**

- 所有项目代码的版本管理
- 代码级别的变更历史追踪
- 分支管理（开发/测试/发布）

**仓库结构建议：**

```text
组织或个人账户下创建以下仓库：

HiveGreatSage-Verify          ← 网络验证系统（服务端）
HiveGreatSage-PCControl        ← PC中控框架
HiveGreatSage-AndroidScript    ← 安卓端懒人精灵脚本框架
HiveGreatSage-Monitor          ← 官网更新监控
HiveGreatSage-Tools            ← 其他工具（ADB+CV等）

每个仓库的分支策略：
  main        ← 稳定版本，只合并测试通过的代码
  dev         ← 日常开发
  feature/xxx ← 功能分支（完成后合并到 dev）
```

**GitHub Desktop 的角色：**

- 作为 Git 的图形界面，降低 Git 操作门槛
- 提交前先看 diff，确认改了什么
- 每次提交写清楚 commit message（和 Obsidian 的决策日志对应）

### 4. Cursor vs PyCharm — 编码分工

**核心原则：不重叠。同一个文件不要两个编辑器同时打开。**

|维度|Cursor|PyCharm|
|---|---|---|
|适合做|AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码|Python 大型项目、调试、重构、数据库工具|
|用在哪个子项目|安卓脚本框架（Lua）、小工具、快速实验|PC中控框架（Python）、网络验证系统（后端）、官网监控|
|为什么|Cursor 的 AI 能力适合"从0写"和脚本类工作|PyCharm 的调试器、数据库工具、项目级重构更强|

**具体分工：**

```text
Cursor 负责打开的项目：
  - HiveGreatSage-AndroidScript（Lua 脚本）
  - HiveGreatSage-Tools（小工具）
  - 任何需要"用 AI 帮写"的新代码

PyCharm 负责打开的项目：
  - HiveGreatSage-Verify（Python 后端 + 数据库）
  - HiveGreatSage-PCControl（Python GUI）
  - HiveGreatSage-Monitor（Python 爬虫/监控）

规则：不要两个编辑器打开同一个仓库。选定了就固定下来。
```

### 5. Google Drive — 文档镜像层（不是真相源）

### 5. Google Drive — 本地工作区实时镜像层 + AI 读取通道

**它是什么：**  
Google Drive 是本地 Obsidian Vault 和本地项目目录的实时同步镜像层，用于让 AI、其他设备或备份系统读取“当前本地真实文件状态”。

**它负责：**

- 实时同步本地 Obsidian / Knowledge 文档
- 实时同步本地项目目录中的当前文件状态
- 作为 ChatGPT / Claude 等 AI 工具读取本地文件的入口
- 作为本地工作区的备份层
- 存放需要共享或临时查看的资料
- 辅助发现 GitHub 中没有的未提交文件、草稿、运行痕迹和本地状态

**它不负责：**

- 不作为代码版本控制工具
- 不替代 GitHub 的 commit、branch、diff、rollback 能力
- 不用来判断哪些代码文件“应该进入正式项目”
- 不把 `.git`、`.idea`、`.env`、`logs`、`.pytest_cache`、`celerybeat-schedule.*` 等本地运行痕迹视为正式项目内容

**数据流向：**

```text
本地 Obsidian / 本地项目目录
        ↓ 实时同步
Google Drive
        ↓
AI 读取 / 多设备查看 / 备份 / 临时共享

### 6. Excalidraw + draw.io — 只画两类图

**核心原则：只画有用的图，不画好看但没人看的图。**

|工具|画什么|为什么用它|
|---|---|---|
|Excalidraw|快速草图、头脑风暴、架构初始构想、流程草稿|手绘风格，低摩擦，随想随画，嵌入 Obsidian|
|draw.io|正式系统架构图、数据库ER图、通信时序图|结构化、可导出、适合技术文档|

**只画这两类图：**

```text
类型A — 系统架构图（draw.io）
  用途：各子系统之间如何连接、数据怎么流
  什么时候画：架构设计定稿时
  存放：导出 PNG/SVG → Obsidian 对应模块文件夹

类型B — 思维草图（Excalidraw）
  用途：讨论时快速画关系、流程、备选方案对比
  什么时候画：和 Claude 讨论时，或自己理思路时
  存放：Excalidraw 文件直接存在 Obsidian 库里（有插件支持）

不画的东西：
  ✗ 纯装饰性的流程图
  ✗ 已经在代码/文档里说清楚的东西
  ✗ 给自己看的没必要正式化的图
```

## 二、数据流向总图（谁产出 → 谁消费）

```text
                    ┌──────────────────┐
                    │   你的大脑/想法    │
                    └────────┬─────────┘
                             │
                             ▼
                    ┌──────────────────┐
                    │  Claude Project   │ ← 推演、讨论、草稿
                    │  （会议室）        │
                    └────────┬─────────┘
                             │ 提炼结论
                             ▼
              ┌──────────────────────────────┐
              │        Obsidian              │ ← 知识真相源
              │  （项目的大脑/唯一权威）       │
              └──┬──────────┬────────────┬───┘
                 │          │            │
    ┌────────────▼──┐  ┌───▼──────┐  ┌──▼────────────┐
    │ Cursor/PyCharm │  │ draw.io  │  │ Google Drive   │
    │ （写代码）      │  │（正式图） │  │（对外镜像）    │
    └───────┬────────┘  └──────────┘  └───────────────┘
            │ commit + push
            ▼
    ┌──────────────────┐
    │     GitHub        │ ← 代码真相源
    │  （代码的权威）    │
    └──────────────────┘
```

**关键规则：数据只从上往下流，不要反向。**

- 不要在 Google Drive 里改了文档再搬回 Obsidian（容易冲突）
- 不要在 Claude 对话里当存储（对话会丢失上下文）
- 不要在编辑器里写文档（文档写在 Obsidian）

## 三、日常工作循环（可持续节奏）

### 一个典型工作单元（2-4小时）

```text
┌─────────────────────────────────────────────────────┐
│  Phase 1：准备（15分钟）                              │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 打开 Obsidian → 看项目总大纲 → 确定今天做什么  │ │
│  │ 2. 看对应模块文档的当前状态                       │ │
│  │ 3. 写下今天的具体目标（1-2个，不要贪多）           │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 2：推演（30-60分钟）                            │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 开 Claude 对话，贴入相关文档 + 今天的目标      │ │
│  │ 2. 和 Claude 讨论方案、排查问题、写代码草稿       │ │
│  │ 3. 对话中如果发现 Claude 输出可疑 → 用术语表判断  │ │
│  │    是 hallucination 还是 sycophancy？标记存疑     │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 3：执行（60-120分钟）                           │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 在 Cursor/PyCharm 中写代码                     │ │
│  │ 2. 测试、调试                                     │ │
│  │ 3. 遇到卡住的问题 → 回 Claude 讨论（带错误信息）  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 4：收尾（15-30分钟）← 最重要但最容易跳过的！    │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. Git commit + push（写清楚做了什么）             │ │
│  │ 2. Obsidian 更新：                                │ │
│  │    - 对话结论 → 99-Claude对话提炼/                │ │
│  │    - 架构变更 → 对应模块文档                      │ │
│  │    - 新问题 → 问题追踪清单                        │ │
│  │    - 重要决策 → 决策日志                          │ │
│  │ 3. 标记模块文档状态（草稿/进行中/定稿）            │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 为什么 Phase 4 是最重要的

你过去5个月"修修补补"的核心问题，大概率不是代码写得不好，而是**知识没有沉淀**。

每次和 Claude 对话完，有价值的结论留在了对话窗口里，下次开新对话又要从头来。这就像开了100次会但从来不写会议纪要——每次都在重新讨论同样的问题。

Phase 4 就是"写会议纪要"的环节。哪怕只花10分钟，写3条要点，也比不写强100倍。

## 四、你可能真正需要补充的（实用，不花哨）

### 需要补充的

|工具/习惯|为什么需要|具体建议|
|---|---|---|
|**任务追踪方式**|你有那么多子模块，需要知道"哪个在做、哪个卡住"|不需要买 Jira。在 Obsidian 里建一个 `任务看板.md`，用 Kanban 插件或者纯文本表格。字段只需要：任务名、状态（待做/进行/卡住/完成）、所属模块、下一步|
|**Obsidian Git 插件**|给你的知识库也加版本控制，防止误改|安装 Obsidian Git 插件，设置每天自动提交到一个私有 GitHub 仓库|
|**固定的对话提炼模板**|让 Phase 4 变成机械化动作，降低阻力|见下方模板|

### 不需要补充的

|经常被推荐但你不需要的|为什么不需要|
|---|---|
|Notion / Confluence|你已经有 Obsidian，再加一个知识库只会分散内容|
|Trello / Linear / Jira|个人项目用 Obsidian 看板足够，上系统反而增加维护成本|
|Docker（现阶段）|等代码稳定后再容器化，现在加只增加复杂度|
|CI/CD（现阶段）|等有稳定的测试用例后再搭，现阶段手动部署够了|
|更多 AI 工具|你已经有 Claude + Cursor，再加 Copilot/GPT 会增加信息冲突风险|

## 五、对话提炼模板

每次 Claude 对话结束后，在 Obsidian 的 `99-Claude对话提炼/` 下新建文件，用以下模板：

```markdown
# [日期] [模块] [主题]

## 对话目标
<!-- 这次对话想解决什么 -->

## 关键结论
<!-- 3-5 条最重要的结论，用自己的话写 -->
1.
2.
3.

## 代码产出
<!-- 这次对话产出了什么代码？已经提交到哪个仓库的哪个分支？ -->

## 存疑/待验证
<!-- Claude 的输出中哪些你不确定是否正确？用 AI 安全术语标注 -->
<!-- 例如：Claude 说 XX 方案性能更好 → 可能是 sycophancy，需要实测验证 -->

## 下一步
<!-- 基于这次对话，下一步要做什么 -->

## 对话链接
<!-- Claude 对话的 URL -->
```

## 六、Obsidian 必装插件清单（只装有用的）

|插件|用途|必要性|
|---|---|---|
|**Obsidian Git**|知识库版本控制 + 自动备份|必装|
|**Excalidraw**|在 Obsidian 里直接画草图|必装（替代单独开 Excalidraw）|
|**Kanban**|任务看板视图|推荐|
|**Templater**|模板快速创建（对话提炼、架构设计等）|推荐|
|**Dataview**|跨文档查询（如"列出所有状态=卡住的任务"）|等文档积累到20+再装|

不需要装的：Calendar、Daily Notes（你的项目是按模块组织的，不是按日期）、花哨的主题和美化插件。

## 七、从零搭建的具体步骤（按顺序做）

### Step 1：创建 Obsidian Vault（30分钟）

```text
1. 打开 Obsidian → 创建新 Vault → 命名 "HiveGreatSage"
2. 按照上面的目录结构手动创建文件夹
3. 在 00-项目总控/ 下创建：
   - 项目总大纲.md（先写一个简短版本，后面在 Claude 里推演完善）
   - 工作流方案.md（就是本文档的内容）
   - AI安全术语表.md（你已有的那份，转成 Markdown 格式）
   - 决策日志.md（空文件，写好表头）
4. 在 _模板/ 下创建对话提炼模板（用上面给的模板）
```

### Step 2：安装 Obsidian 插件（15分钟）

```text
1. 设置 → 第三方插件 → 关闭安全模式
2. 搜索安装：Obsidian Git, Excalidraw, Kanban, Templater
3. Obsidian Git 配置：
   - 在 GitHub 上创建私有仓库 "HiveGreatSage-Knowledge"
   - 在 Vault 根目录 git init → 添加远程仓库
   - 插件设置：自动提交间隔 = 30分钟 或手动提交
4. Templater 配置：
   - 模板文件夹 = _模板
```

### Step 3：创建 GitHub 仓库（20分钟）

## 八、常见问题预判

**Q: Claude 对话中产出的代码，直接复制到编辑器吗？** A: 复制后必须自己读一遍、理解、测试。Claude 产出的代码可能存在 hallucination（编造不存在的API）或 specification gaming（字面上满足了你的要求但实际逻辑不对）。特别是懒人精灵相关的 API，Claude 的训练数据中可能不包含准确信息。

**Q: Obsidian 和 Google Drive 的内容重复了怎么办？** A: 以 Obsidian 为准。Google Drive 里的版本如果更旧，覆盖掉。不要出现两个地方各存一半的情况。

**Q: 一个问题涉及多个模块怎么办？** A: 在 Claude 里可以讨论跨模块问题（比如 PC中控和安卓脚本的通信协议），但提炼结论时要分别写入两个模块的文档，并用 `双链` 互相指向。

**Q: 画图到底什么时候画？** A: 只在两种情况下画：(1) 你自己理不清关系需要可视化思考时 → Excalidraw 草图；(2) 架构设计定稿需要一张正式参考图时 → draw.io。其他时候用文字描述就够了。

**Q: 这个工作流本身需要多久适应？** A: 给自己两周时间。前几天会觉得"收尾阶段写 Obsidian 好麻烦"，但坚持一周后你会发现之前讨论过的东西再也不用重新问了。
