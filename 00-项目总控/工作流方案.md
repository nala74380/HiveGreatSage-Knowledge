## 〇、项目定位与边界

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：
- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控程序（局域网内控制与信息展示）
- 一个安卓端自动化脚本框架（基于懒人精灵高级版）
- 辅助工具（官网监控、素材提取等）

平台设计为**可适配不同 MMORPG 游戏**——更换游戏时，核心框架不变，只替换游戏适配层。

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是纯工具集合，而是有完整用户体系、授权体系、热更新体系的平台
- 不涉及对任何在运营游戏的讨论（所有开发均以框架和模拟数据进行）

### 平台的商业模型概要

```
管理员
  └── 创建/管理 代理（分销/合伙人）
        └── 代理创建/管理 用户（5级）
              └── 用户购买 游戏项目授权
                    └── 使用 PC中控 + 安卓脚本
```

用户不可自行注册，必须由代理或管理员创建。

### 子系统1：网络验证系统

**代号：** Verify
**仓库：** HiveGreatSage-Verify
**编辑器：** PyCharm

#### 1.1 功能模块清单

| 模块 | 功能概述 | 优先级 |
|------|---------|--------|
| **用户认证** | 登录验证、Token管理、设备绑定 | P0 |
| **用户管理** | 5级用户体系(试用/普通/VIP/SVIP/项目测试)，增删改查 | P0 |
| **代理管理** | 代理注册、分销层级、权限范围、业绩统计 | P0 |
| **游戏项目管理** | 项目注册、授权绑定、配置下发 | P0 |
| **热更新管理** | 按项目创建文件夹、版本管理、PC/安卓分别更新 | P1 |
| **数据收集** | 接收安卓端上报的游戏运行数据 | P1 |
| **管理后台** | Web 管理界面（管理员+代理各自的面板） | P1 |

#### 1.2 数据库设计要点

**数据库A — 用户管理库（所有游戏项目共用）**

```
核心表：
  管理员表        admin
  代理表          agent          → 关联 admin
  用户表          user           → 关联 agent
  授权表          authorization  → 关联 user + 游戏项目
  设备绑定表      device_bindng  → 关联 user
  登录日志表      login_log

用户5级体系：
  级别1: 试用用户      trial       → 时间受限，功能受限
  级别2: 普通用户      normal      → 基础功能
  级别3: VIP用户       vip         → 扩展功能
  级别4: SVIP用户      svip        → 全功能
  级别5: 项目测试用户  tester      → 仅管理员可创建，代理无权

关键约束：
  · 用户只能由代理或管理员创建，无注册入口
  · 不同游戏项目需要独立授权
  · 代理只能管理自己创建的用户
```

**数据库B — 游戏项目库（每个游戏项目独立）**

```
为什么独立：
  · 不同游戏从安卓端采集的数据结构完全不同
  · 不同游戏在PC中控需要显示/设置的内容完全不同
  · 独立数据库便于游戏项目的新增和下线

每个游戏项目库包含：
  项目配置表      project_config    → 该游戏的全局配置
  脚本参数表      script_params     → PC中控可设置的脚本参数定义
  运行数据表      runtime_data      → 安卓端上报的运行数据
  版本记录表      version_record    → PC包/安卓包版本历史
  自定义表...     custom_*          → 因游戏而异
```

#### 1.3 热更新机制

```
热更新目录结构（服务端）：
  hot_update/
  ├── 游戏项目A/
  │   ├── pc/
  │   │   ├── latest_version.json
  │   │   └── packages/
  │   └── android/
  │       ├── latest_version.json    → 包含版本号、文件列表、校验值
  │       └── packages/
  │           └── v1.0.1.lrj         ← 懒人精灵热更新包格式
  └── 游戏项目B/
      └── ...

流程：
  客户端启动 → 请求 latest_version.json → 比对本地版本
  → 如有更新 → 下载新包 → 校验 → 安装（安卓端调用 installLRPkg）
```

#### 2.3 游戏适配层设计思路

```
核心问题：PC中控怎么做到"换个游戏就能换界面"？

方案思路：
  · 每个游戏项目提供一份"适配描述文件"（JSON/YAML/Lua表）
  · 描述文件定义：
    - 这个游戏需要哪些设置项（参数名、类型、默认值、取值范围）
    - 这个游戏需要显示哪些数据（字段名、展示格式）
    - 这个游戏需要哪些操作按钮
  · PC中控根据描述文件动态生成 UI
  · 安卓端脚本也读同一份描述文件 → 双端自动对齐

优点：新增游戏不需要改中控代码，只需要写一份新的适配描述文件
```

#### 2.4 PC ↔ 安卓通信

```
场景：PC 和安卓设备在同一局域网内

通信内容（方向）：
  PC → 安卓：
    · 脚本启动/停止/暂停指令
    · 脚本参数更新
    · 组队指令（如"你去A地图，他去B地图"）

  安卓 → PC：
    · 设备状态心跳（在线/离线/运行中/异常）
    · 运行数据上报（游戏内采集的数据）
    · 日志推送
    · 异常告警

待确认的决策点：
  □ 通信协议：TCP Socket / UDP / HTTP 局域网服务？
  □ 数据格式：JSON？Protobuf？（考虑懒人精灵 Lua 的解析能力）
  □ 设备发现机制：UDP 广播？mDNS？手动配置 IP？
  □ 安卓端谁来开 Socket？懒人精灵脚本能否开 Socket 服务？
    还是需要一个安卓端的辅助 App 来桥接？
```

#### 2.5 待确认/待推演的决策点

```
□ GUI 框架选型：PyQt / Tkinter / 其他？
□ 是否需要做成可安装程序（exe）？打包方式？
□ 多游戏项目切换的 UI 方案：Tab 页 / 下拉选择 / 项目列表？
□ PC 端也需要接入热更新 → 更新后如何重启自身？
```

#### 3.3 游戏适配层设计思路

```
新增游戏时的工作量：
  1. 创建 游戏X/ 文件夹
  2. 编写 config.lua（参数定义 — 要和PC中控的适配描述文件对齐）
  3. 截取 UI 模板图片 → resources/
  4. 编写 ui_map.lua（界面识别逻辑：哪个界面是什么、怎么判断）
  5. 编写 tasks/（具体任务逻辑：日常任务、副本、采集等）
  6. 在 PC中控 侧编写对应的适配描述文件
  7. 在 网络验证系统 注册新游戏项目 + 创建对应数据库

框架核心层不需要改动 → 这就是"框架"的意义
```

  通过 ADB 连接安卓设备，使用 CV（计算机视觉）从游戏画面中

  生成可直接用于脚本自动化的模板图片和坐标数据。

大致流程：
  1. ADB 截图 → 传到 PC
  2. PC 端 CV 处理（OpenCV）→ 边缘检测 / 模板标注
  3. 导出：模板图片 + 坐标配置文件
  4. 导入到对应游戏的 resources/ 目录

这个工具可以大幅减少手动截图+标注的工作量。
```

## 三、跨子系统的关键设计问题

### 3.1 游戏适配机制（全局）

```
核心矛盾：
  同一套框架适配不同游戏，但每个游戏的一切都不同：
  UI 不同、操作流程不同、数据结构不同、参数不同。

解决思路 — "适配描述 + 适配代码"双层机制：

  适配描述文件（JSON/YAML/Lua表）：
    · 声明这个游戏有哪些参数、哪些数据字段、哪些UI面板
    · PC中控和安卓脚本共用同一份定义 → 天然对齐
    · 存放在网络验证系统 → 热更新时一起下发

  适配代码（Lua / Python）：
    · 实现这个游戏的具体操作逻辑
    · 安卓端：Lua 代码在 游戏X/ 目录下
    · PC端：Python 代码在 adapters/游戏X/ 目录下

新增游戏的完整清单：
  □ 编写适配描述文件
  □ 安卓端：游戏X/ 下编写 Lua 适配代码 + 截图资源
  □ PC端：adapters/游戏X/ 下编写 Python 适配代码
  □ 网络验证系统：注册游戏项目 + 创建数据库 + 上传描述文件
  □ 官网监控：添加监控配置
  □ 热更新：打包首个版本
```

### 3.2 接入流程（项目管理视角）

```
问题：那套游戏项目的中控要怎么接入？那套安卓脚本要怎么接入？

标准接入流程：

Step 1 — 网络验证系统注册
  管理员在后台创建新游戏项目 → 系统自动：
    · 生成项目ID
    · 创建该游戏的独立数据库
    · 创建热更新目录结构
    · 初始化默认配置

Step 2 — 编写适配层
  开发者编写：
    · 适配描述文件 → 上传到验证系统
    · 安卓端 Lua 适配代码 → 打包为 .lrj
    · PC端 Python 适配代码 → 打包（方式待定）

Step 3 — 首次发布
  通过热更新系统发布首版：
    · 上传安卓端 .lrj 包
    · 上传 PC端更新包
    · 设置版本号

Step 4 — 授权与分发
  管理员/代理为用户添加该游戏项目的授权
  用户在 PC中控 或 安卓脚本中看到新项目 → 下载 → 使用
```

## 〇、项目定位与边界

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：
- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控程序（局域网内控制与信息展示）
- 一个安卓端自动化脚本框架（基于懒人精灵高级版）
- 辅助工具（官网监控、素材提取等）

平台设计为**可适配不同 MMORPG 游戏**——更换游戏时，核心框架不变，只替换游戏适配层。

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是纯工具集合，而是有完整用户体系、授权体系、热更新体系的平台
- 不涉及对任何在运营游戏的讨论（所有开发均以框架和模拟数据进行）

### 平台的商业模型概要

```
管理员
  └── 创建/管理 代理（分销/合伙人）
        └── 代理创建/管理 用户（5级）
              └── 用户购买 游戏项目授权
                    └── 使用 PC中控 + 安卓脚本
```

用户不可自行注册，必须由代理或管理员创建。

#### 1.2 数据库设计要点

**数据库A — 用户管理库（所有游戏项目共用）**

```
核心表：
  管理员表        admin
  代理表          agent          → 关联 admin
  用户表          user           → 关联 agent
  授权表          authorization  → 关联 user + 游戏项目
  设备绑定表      device_bindng  → 关联 user
  登录日志表      login_log

用户5级体系：
  级别1: 试用用户      trial       → 时间受限，功能受限
  级别2: 普通用户      normal      → 基础功能
  级别3: VIP用户       vip         → 扩展功能
  级别4: SVIP用户      svip        → 全功能
  级别5: 项目测试用户  tester      → 仅管理员可创建，代理无权

关键约束：
  · 用户只能由代理或管理员创建，无注册入口
  · 不同游戏项目需要独立授权
  · 代理只能管理自己创建的用户
```

**数据库B — 游戏项目库（每个游戏项目独立）**

```
为什么独立：
  · 不同游戏从安卓端采集的数据结构完全不同
  · 不同游戏在PC中控需要显示/设置的内容完全不同
  · 独立数据库便于游戏项目的新增和下线

每个游戏项目库包含：
  项目配置表      project_config    → 该游戏的全局配置
  脚本参数表      script_params     → PC中控可设置的脚本参数定义
  运行数据表      runtime_data      → 安卓端上报的运行数据
  版本记录表      version_record    → PC包/安卓包版本历史
  自定义表...     custom_*          → 因游戏而异
```

  · 每个游戏项目提供一份"适配描述文件"（JSON/YAML/Lua表）
  · 描述文件定义：
    - 这个游戏需要哪些设置项（参数名、类型、默认值、取值范围）
    - 这个游戏需要显示哪些数据（字段名、展示格式）
    - 这个游戏需要哪些操作按钮
  · PC中控根据描述文件动态生成 UI
  · 安卓端脚本也读同一份描述文件 → 双端自动对齐

优点：新增游戏不需要改中控代码，只需要写一份新的适配描述文件
```

### 3.1 游戏适配机制（全局）

```
核心矛盾：
  同一套框架适配不同游戏，但每个游戏的一切都不同：

解决思路 — "适配描述 + 适配代码"双层机制：

  适配描述文件（JSON/YAML/Lua表）：
    · 声明这个游戏有哪些参数、哪些数据字段、哪些UI面板
    · PC中控和安卓脚本共用同一份定义 → 天然对齐

  适配代码（Lua / Python）：
    · 实现这个游戏的具体操作逻辑
    · 安卓端：Lua 代码在 游戏X/ 目录下
    · PC端：Python 代码在 adapters/游戏X/ 目录下

  □ 编写适配描述文件
  □ 安卓端：游戏X/ 下编写 Lua 适配代码 + 截图资源
  □ PC端：adapters/游戏X/ 下编写 Python 适配代码
  □ 网络验证系统：注册游戏项目 + 创建数据库 + 上传描述文件
  □ 官网监控：添加监控配置
  □ 热更新：打包首个版本
```

## 六、AI 协作安全守则（本项目专用）

基于《AI安全术语整理》，在本项目中具体化为：

| 风险 | 在本项目中的表现 | 应对措施 |
|------|----------------|---------|
| **Hallucination** | Claude 编造懒人精灵 API、编造不存在的函数 | 所有 API 调用必须对照懒人精灵官方文档验证 |
| **Sycophancy** | Claude 说"这个方案完全没问题"但实际有坑 | 对每个方案追问"这个方案的最大风险是什么？" |
| **Specification Gaming** | Claude 的代码字面上满足要求但逻辑不对 | 每段代码必须自己读懂再用，不能直接复制 |
| **Confabulation** | Claude 补全了你没给的上下文细节 | 每次对话开始时给足上下文（从 Obsidian 复制） |
| **Mirage Reasoning** | Claude 的推理链看起来合理但前提是错的 | 关注推理的前提假设是否成立，而不只看结论 |

**一条硬规则：** 凡是 Claude 说的和你实际测试结果不一致的，以实际测试为准。把不一致记录到决策日志，标注"AI 输出与实测不符"。

### 2. Claude Project — 主推演区

**它是什么：** 设计思考和技术推演的"会议室"，不是存储的地方。

**它负责：**
- 架构设计的讨论和推演（"这个方案行不行？有什么坑？"）
- 代码框架的原型草稿（先在这里讨论清楚，再到 Cursor 里写）
- 技术选型分析（"用 SQLite 还是 MySQL？为什么？"）
- 文档草稿的撰写和修改
- Bug 排查思路讨论

**它不负责：**
- 不是知识存储（结论搬去 Obsidian）
- 不是代码仓库（代码搬去 GitHub）
- 不期望 Claude 记住所有历史（每次对话要给足上下文）

**Claude Project 窗口组织策略：**

```
蜂巢·大圣 Project 内的对话窗口规划：

窗口命名规则：[模块编号]-[主题]-[日期]
例如：
  02-PC中控-通信协议设计-0413
  03-安卓脚本-OCR方案选型-0415
  00-工作流-整体复盘-0420

不要做的事：
  ✗ 一个窗口里讨论3个不同模块 → 上下文混乱
  ✗ 不提供背景就问问题 → Claude 会 hallucinate（你的术语表第一条）
  ✗ 在对话里"修修补补" → 这正是你说的过去5个月的教训
```

**每次开启新对话的标准流程：**

```
第1步：从 Obsidian 复制相关模块的当前架构文档（给 Claude 大局观）
第2步：明确本次对话的目标（"这次我要解决 XXX 问题"）
第3步：如果是接续之前的讨论，贴上上次的结论摘要
第4步：对话进行
第5步：对话结束 → 提炼结论 → 写入 Obsidian → 标记对话链接
```

**给 Claude 大局观的方法：**

你说得对，这个项目需要给 AI 很强的项目大局观。实际操作就是——

在 Claude Project 的 **Project Knowledge**（项目知识库）中上传/粘贴以下内容：
- `项目总大纲.md`（让 Claude 知道整个项目的全貌）
- `AI安全术语表.md`（让 Claude 遵守你的术语规范）
- 当前正在工作的模块的架构设计文档

每当架构文档有更新，同步更新 Project Knowledge 中的版本。

### 4. Cursor vs PyCharm — 编码分工

**核心原则：不重叠。同一个文件不要两个编辑器同时打开。**

| 维度 | Cursor | PyCharm |
|------|--------|---------|
| 适合做 | AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码 | Python 大型项目、调试、重构、数据库工具 |
| 用在哪个子项目 | 安卓脚本框架（Lua）、小工具、快速实验 | PC中控框架（Python）、网络验证系统（后端）、官网监控 |
| 为什么 | Cursor 的 AI 能力适合"从0写"和脚本类工作 | PyCharm 的调试器、数据库工具、项目级重构更强 |

**具体分工：**

```
Cursor 负责打开的项目：
  - HiveGreatSage-AndroidScript（Lua 脚本）
  - HiveGreatSage-Tools（小工具）
  - 任何需要"用 AI 帮写"的新代码

PyCharm 负责打开的项目：
  - HiveGreatSage-Verify（Python 后端 + 数据库）
  - HiveGreatSage-PCControl（Python GUI）
  - HiveGreatSage-Monitor（Python 爬虫/监控）

规则：不要两个编辑器打开同一个仓库。选定了就固定下来。
```

### 6. Excalidraw + draw.io — 只画两类图

**核心原则：只画有用的图，不画好看但没人看的图。**

| 工具 | 画什么 | 为什么用它 |
|------|--------|-----------|
| Excalidraw | 快速草图、头脑风暴、架构初始构想、流程草稿 | 手绘风格，低摩擦，随想随画，嵌入 Obsidian |
| draw.io | 正式系统架构图、数据库ER图、通信时序图 | 结构化、可导出、适合技术文档 |

**只画这两类图：**

```
类型A — 系统架构图（draw.io）
  用途：各子系统之间如何连接、数据怎么流
  什么时候画：架构设计定稿时
  存放：导出 PNG/SVG → Obsidian 对应模块文件夹

类型B — 思维草图（Excalidraw）
  用途：讨论时快速画关系、流程、备选方案对比
  什么时候画：和 Claude 讨论时，或自己理思路时
  存放：Excalidraw 文件直接存在 Obsidian 库里（有插件支持）

不画的东西：
  ✗ 纯装饰性的流程图
  ✗ 已经在代码/文档里说清楚的东西
  ✗ 给自己看的没必要正式化的图
```

## 三、日常工作循环（可持续节奏）

### 一个典型工作单元（2-4小时）

```
┌─────────────────────────────────────────────────────┐
│  Phase 1：准备（15分钟）                              │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 打开 Obsidian → 看项目总大纲 → 确定今天做什么  │ │
│  │ 2. 看对应模块文档的当前状态                       │ │

│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 2：推演（30-60分钟）                            │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 开 Claude 对话，贴入相关文档 + 今天的目标      │ │
│  │ 2. 和 Claude 讨论方案、排查问题、写代码草稿       │ │
│  │ 3. 对话中如果发现 Claude 输出可疑 → 用术语表判断  │ │
│  │    是 hallucination 还是 sycophancy？标记存疑     │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 3：执行（60-120分钟）                           │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 在 Cursor/PyCharm 中写代码                     │ │
│  │ 2. 测试、调试                                     │ │
│  │ 3. 遇到卡住的问题 → 回 Claude 讨论（带错误信息）  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 4：收尾（15-30分钟）← 最重要但最容易跳过的！    │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. Git commit + push（写清楚做了什么）             │ │
│  │ 2. Obsidian 更新：                                │ │
│  │    - 对话结论 → 99-Claude对话提炼/                │ │
│  │    - 架构变更 → 对应模块文档                      │ │
│  │    - 新问题 → 问题追踪清单                        │ │
│  │    - 重要决策 → 决策日志                          │ │
│  │ 3. 标记模块文档状态（草稿/进行中/定稿）            │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 为什么 Phase 4 是最重要的

你过去5个月"修修补补"的核心问题，大概率不是代码写得不好，而是**知识没有沉淀**。

每次和 Claude 对话完，有价值的结论留在了对话窗口里，下次开新对话又要从头来。这就像开了100次会但从来不写会议纪要——每次都在重新讨论同样的问题。

Phase 4 就是"写会议纪要"的环节。哪怕只花10分钟，写3条要点，也比不写强100倍。

## 八、常见问题预判

**Q: Claude 对话中产出的代码，直接复制到编辑器吗？**
A: 复制后必须自己读一遍、理解、测试。Claude 产出的代码可能存在 hallucination（编造不存在的API）或 specification gaming（字面上满足了你的要求但实际逻辑不对）。特别是懒人精灵相关的 API，Claude 的训练数据中可能不包含准确信息。

**Q: Obsidian 和 Google Drive 的内容重复了怎么办？**
A: 以 Obsidian 为准。Google Drive 里的版本如果更旧，覆盖掉。不要出现两个地方各存一半的情况。

**Q: 一个问题涉及多个模块怎么办？**
A: 在 Claude 里可以讨论跨模块问题（比如 PC中控和安卓脚本的通信协议），但提炼结论时要分别写入两个模块的文档，并用 `[[双链]]` 互相指向。

**Q: 画图到底什么时候画？**
A: 只在两种情况下画：(1) 你自己理不清关系需要可视化思考时 → Excalidraw 草图；(2) 架构设计定稿需要一张正式参考图时 → draw.io。其他时候用文字描述就够了。

**Q: 这个工作流本身需要多久适应？**
A: 给自己两周时间。前几天会觉得"收尾阶段写 Obsidian 好麻烦"，但坚持一周后你会发现之前讨论过的东西再也不用重新问了。

## 〇、项目定位与边界

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：
- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控程序（局域网内控制与信息展示）
- 一个安卓端自动化脚本框架（基于懒人精灵高级版）
- 辅助工具（官网监控、素材提取等）

平台设计为**可适配不同 MMORPG 游戏**——更换游戏时，核心框架不变，只替换游戏适配层。

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是纯工具集合，而是有完整用户体系、授权体系、热更新体系的平台
- 不涉及对任何在运营游戏的讨论（所有开发均以框架和模拟数据进行）

### 平台的商业模型概要

```
管理员
  └── 创建/管理 代理（分销/合伙人）
        └── 代理创建/管理 用户（5级）
              └── 用户购买 游戏项目授权
                    └── 使用 PC中控 + 安卓脚本
```

用户不可自行注册，必须由代理或管理员创建。

#### 1.2 数据库设计要点

**数据库A — 用户管理库（所有游戏项目共用）**

```
核心表：
  管理员表        admin
  代理表          agent          → 关联 admin
  用户表          user           → 关联 agent
  授权表          authorization  → 关联 user + 游戏项目
  设备绑定表      device_bindng  → 关联 user
  登录日志表      login_log

用户5级体系：
  级别1: 试用用户      trial       → 时间受限，功能受限
  级别2: 普通用户      normal      → 基础功能
  级别3: VIP用户       vip         → 扩展功能
  级别4: SVIP用户      svip        → 全功能
  级别5: 项目测试用户  tester      → 仅管理员可创建，代理无权

关键约束：
  · 用户只能由代理或管理员创建，无注册入口
  · 不同游戏项目需要独立授权
  · 代理只能管理自己创建的用户
```

**数据库B — 游戏项目库（每个游戏项目独立）**

```
为什么独立：
  · 不同游戏从安卓端采集的数据结构完全不同
  · 不同游戏在PC中控需要显示/设置的内容完全不同
  · 独立数据库便于游戏项目的新增和下线

每个游戏项目库包含：
  项目配置表      project_config    → 该游戏的全局配置
  脚本参数表      script_params     → PC中控可设置的脚本参数定义
  运行数据表      runtime_data      → 安卓端上报的运行数据
  版本记录表      version_record    → PC包/安卓包版本历史
  自定义表...     custom_*          → 因游戏而异
```

#### 1.3 热更新机制

```
热更新目录结构（服务端）：
  hot_update/
  ├── 游戏项目A/
  │   ├── pc/
  │   │   ├── latest_version.json
  │   │   └── packages/
  │   └── android/
  │       ├── latest_version.json    → 包含版本号、文件列表、校验值
  │       └── packages/
  │           └── v1.0.1.lrj         ← 懒人精灵热更新包格式
  └── 游戏项目B/
      └── ...

流程：
  客户端启动 → 请求 latest_version.json → 比对本地版本
  → 如有更新 → 下载新包 → 校验 → 安装（安卓端调用 installLRPkg）
```

#### 2.3 游戏适配层设计思路

```
核心问题：PC中控怎么做到"换个游戏就能换界面"？

方案思路：
  · 每个游戏项目提供一份"适配描述文件"（JSON/YAML/Lua表）
  · 描述文件定义：
    - 这个游戏需要哪些设置项（参数名、类型、默认值、取值范围）
    - 这个游戏需要显示哪些数据（字段名、展示格式）
    - 这个游戏需要哪些操作按钮
  · PC中控根据描述文件动态生成 UI
  · 安卓端脚本也读同一份描述文件 → 双端自动对齐

优点：新增游戏不需要改中控代码，只需要写一份新的适配描述文件
```

#### 2.4 PC ↔ 安卓通信

```
场景：PC 和安卓设备在同一局域网内

通信内容（方向）：
  PC → 安卓：
    · 脚本启动/停止/暂停指令
    · 脚本参数更新
    · 组队指令（如"你去A地图，他去B地图"）

  安卓 → PC：
    · 设备状态心跳（在线/离线/运行中/异常）
    · 运行数据上报（游戏内采集的数据）
    · 日志推送
    · 异常告警

待确认的决策点：
  □ 通信协议：TCP Socket / UDP / HTTP 局域网服务？
  □ 数据格式：JSON？Protobuf？（考虑懒人精灵 Lua 的解析能力）
  □ 设备发现机制：UDP 广播？mDNS？手动配置 IP？
  □ 安卓端谁来开 Socket？懒人精灵脚本能否开 Socket 服务？
    还是需要一个安卓端的辅助 App 来桥接？
```

#### 2.5 待确认/待推演的决策点

```
□ GUI 框架选型：PyQt / Tkinter / 其他？
□ 是否需要做成可安装程序（exe）？打包方式？
□ 多游戏项目切换的 UI 方案：Tab 页 / 下拉选择 / 项目列表？
□ PC 端也需要接入热更新 → 更新后如何重启自身？

## 三、跨子系统的关键设计问题

### 3.1 游戏适配机制（全局）

```
核心矛盾：
  同一套框架适配不同游戏，但每个游戏的一切都不同：
  UI 不同、操作流程不同、数据结构不同、参数不同。

解决思路 — "适配描述 + 适配代码"双层机制：

  适配描述文件（JSON/YAML/Lua表）：
    · 声明这个游戏有哪些参数、哪些数据字段、哪些UI面板
    · PC中控和安卓脚本共用同一份定义 → 天然对齐
    · 存放在网络验证系统 → 热更新时一起下发

  适配代码（Lua / Python）：
    · 实现这个游戏的具体操作逻辑
    · 安卓端：Lua 代码在 游戏X/ 目录下
    · PC端：Python 代码在 adapters/游戏X/ 目录下

新增游戏的完整清单：
  □ 编写适配描述文件
  □ 安卓端：游戏X/ 下编写 Lua 适配代码 + 截图资源
  □ PC端：adapters/游戏X/ 下编写 Python 适配代码
  □ 网络验证系统：注册游戏项目 + 创建数据库 + 上传描述文件
  □ 官网监控：添加监控配置
  □ 热更新：打包首个版本
```

### 3.2 接入流程（项目管理视角）

```
问题：那套游戏项目的中控要怎么接入？那套安卓脚本要怎么接入？

标准接入流程：

Step 1 — 网络验证系统注册
  管理员在后台创建新游戏项目 → 系统自动：
    · 生成项目ID
    · 创建该游戏的独立数据库
    · 创建热更新目录结构
    · 初始化默认配置

Step 2 — 编写适配层
  开发者编写：
    · 适配描述文件 → 上传到验证系统
    · 安卓端 Lua 适配代码 → 打包为 .lrj
    · PC端 Python 适配代码 → 打包（方式待定）

Step 3 — 首次发布
  通过热更新系统发布首版：
    · 上传安卓端 .lrj 包
    · 上传 PC端更新包
    · 设置版本号

Step 4 — 授权与分发
  管理员/代理为用户添加该游戏项目的授权
  用户在 PC中控 或 安卓脚本中看到新项目 → 下载 → 使用
```

## 〇、项目定位与边界

### 项目是什么

蜂巢·大圣是一个**多游戏适配的自动化平台**，由以下部分组成：
- 一套网络验证与用户管理后端（云端）
- 一个 PC 端中控程序（局域网内控制与信息展示）
- 一个安卓端自动化脚本框架（基于懒人精灵高级版）
- 辅助工具（官网监控、素材提取等）

平台设计为**可适配不同 MMORPG 游戏**——更换游戏时，核心框架不变，只替换游戏适配层。

### 项目不是什么

- 不是针对某一款游戏的一次性脚本
- 不是纯工具集合，而是有完整用户体系、授权体系、热更新体系的平台
- 不涉及对任何在运营游戏的讨论（所有开发均以框架和模拟数据进行）

### 平台的商业模型概要

```
管理员
  └── 创建/管理 代理（分销/合伙人）
        └── 代理创建/管理 用户（5级）
              └── 用户购买 游戏项目授权
                    └── 使用 PC中控 + 安卓脚本
```

用户不可自行注册，必须由代理或管理员创建。

#### 1.2 数据库设计要点

**数据库A — 用户管理库（所有游戏项目共用）**

```
核心表：
  管理员表        admin
  代理表          agent          → 关联 admin
  用户表          user           → 关联 agent
  授权表          authorization  → 关联 user + 游戏项目
  设备绑定表      device_bindng  → 关联 user
  登录日志表      login_log

用户5级体系：
  级别1: 试用用户      trial       → 时间受限，功能受限
  级别2: 普通用户      normal      → 基础功能
  级别3: VIP用户       vip         → 扩展功能
  级别4: SVIP用户      svip        → 全功能
  级别5: 项目测试用户  tester      → 仅管理员可创建，代理无权

关键约束：
  · 用户只能由代理或管理员创建，无注册入口
  · 不同游戏项目需要独立授权
  · 代理只能管理自己创建的用户
```

**数据库B — 游戏项目库（每个游戏项目独立）**

```
为什么独立：
  · 不同游戏从安卓端采集的数据结构完全不同
  · 不同游戏在PC中控需要显示/设置的内容完全不同
  · 独立数据库便于游戏项目的新增和下线

每个游戏项目库包含：
  项目配置表      project_config    → 该游戏的全局配置
  脚本参数表      script_params     → PC中控可设置的脚本参数定义
  运行数据表      runtime_data      → 安卓端上报的运行数据
  版本记录表      version_record    → PC包/安卓包版本历史
  自定义表...     custom_*          → 因游戏而异
```

#### 2.3 游戏适配层设计思路

```
核心问题：PC中控怎么做到"换个游戏就能换界面"？

方案思路：
  · 每个游戏项目提供一份"适配描述文件"（JSON/YAML/Lua表）
  · 描述文件定义：
    - 这个游戏需要哪些设置项（参数名、类型、默认值、取值范围）
    - 这个游戏需要显示哪些数据（字段名、展示格式）
    - 这个游戏需要哪些操作按钮
  · PC中控根据描述文件动态生成 UI
  · 安卓端脚本也读同一份描述文件 → 双端自动对齐

优点：新增游戏不需要改中控代码，只需要写一份新的适配描述文件

### 3.1 游戏适配机制（全局）

```
核心矛盾：
  同一套框架适配不同游戏，但每个游戏的一切都不同：

  适配描述文件（JSON/YAML/Lua表）：

    · 实现这个游戏的具体操作逻辑
    · 安卓端：Lua 代码在 游戏X/ 目录下
    · PC端：Python 代码在 adapters/游戏X/ 目录下

  □ 编写适配描述文件
  □ 安卓端：游戏X/ 下编写 Lua 适配代码 + 截图资源
  □ PC端：adapters/游戏X/ 下编写 Python 适配代码
  □ 网络验证系统：注册游戏项目 + 创建数据库 + 上传描述文件
  □ 官网监控：添加监控配置

```

## 六、AI 协作安全守则（本项目专用）

基于《AI安全术语整理》，在本项目中具体化为：

| 风险 | 在本项目中的表现 | 应对措施 |
|------|----------------|---------|
| **Hallucination** | Claude 编造懒人精灵 API、编造不存在的函数 | 所有 API 调用必须对照懒人精灵官方文档验证 |
| **Sycophancy** | Claude 说"这个方案完全没问题"但实际有坑 | 对每个方案追问"这个方案的最大风险是什么？" |
| **Specification Gaming** | Claude 的代码字面上满足要求但逻辑不对 | 每段代码必须自己读懂再用，不能直接复制 |
| **Confabulation** | Claude 补全了你没给的上下文细节 | 每次对话开始时给足上下文（从 Obsidian 复制） |
| **Mirage Reasoning** | Claude 的推理链看起来合理但前提是错的 | 关注推理的前提假设是否成立，而不只看结论 |

**一条硬规则：** 凡是 Claude 说的和你实际测试结果不一致的，以实际测试为准。把不一致记录到决策日志，标注"AI 输出与实测不符"。

### 2. Claude Project — 主推演区

**它是什么：** 设计思考和技术推演的"会议室"，不是存储的地方。

**它负责：**
- 架构设计的讨论和推演（"这个方案行不行？有什么坑？"）
- 代码框架的原型草稿（先在这里讨论清楚，再到 Cursor 里写）
- 技术选型分析（"用 SQLite 还是 MySQL？为什么？"）
- 文档草稿的撰写和修改
- Bug 排查思路讨论

**它不负责：**
- 不是知识存储（结论搬去 Obsidian）
- 不是代码仓库（代码搬去 GitHub）
- 不期望 Claude 记住所有历史（每次对话要给足上下文）

**Claude Project 窗口组织策略：**

```
蜂巢·大圣 Project 内的对话窗口规划：

窗口命名规则：[模块编号]-[主题]-[日期]
例如：
  02-PC中控-通信协议设计-0413
  03-安卓脚本-OCR方案选型-0415
  00-工作流-整体复盘-0420

不要做的事：
  ✗ 一个窗口里讨论3个不同模块 → 上下文混乱
  ✗ 不提供背景就问问题 → Claude 会 hallucinate（你的术语表第一条）
  ✗ 在对话里"修修补补" → 这正是你说的过去5个月的教训
```

**每次开启新对话的标准流程：**

```
第1步：从 Obsidian 复制相关模块的当前架构文档（给 Claude 大局观）
第2步：明确本次对话的目标（"这次我要解决 XXX 问题"）
第3步：如果是接续之前的讨论，贴上上次的结论摘要
第4步：对话进行
第5步：对话结束 → 提炼结论 → 写入 Obsidian → 标记对话链接
```

**给 Claude 大局观的方法：**

你说得对，这个项目需要给 AI 很强的项目大局观。实际操作就是——

在 Claude Project 的 **Project Knowledge**（项目知识库）中上传/粘贴以下内容：
- `项目总大纲.md`（让 Claude 知道整个项目的全貌）
- `AI安全术语表.md`（让 Claude 遵守你的术语规范）
- 当前正在工作的模块的架构设计文档

每当架构文档有更新，同步更新 Project Knowledge 中的版本。

### 4. Cursor vs PyCharm — 编码分工

**核心原则：不重叠。同一个文件不要两个编辑器同时打开。**

| 维度 | Cursor | PyCharm |
|------|--------|---------|
| 适合做 | AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码 | Python 大型项目、调试、重构、数据库工具 |
| 用在哪个子项目 | 安卓脚本框架（Lua）、小工具、快速实验 | PC中控框架（Python）、网络验证系统（后端）、官网监控 |
| 为什么 | Cursor 的 AI 能力适合"从0写"和脚本类工作 | PyCharm 的调试器、数据库工具、项目级重构更强 |

**具体分工：**

```
Cursor 负责打开的项目：
  - HiveGreatSage-AndroidScript（Lua 脚本）
  - HiveGreatSage-Tools（小工具）
  - 任何需要"用 AI 帮写"的新代码

PyCharm 负责打开的项目：
  - HiveGreatSage-Verify（Python 后端 + 数据库）
  - HiveGreatSage-PCControl（Python GUI）
  - HiveGreatSage-Monitor（Python 爬虫/监控）

规则：不要两个编辑器打开同一个仓库。选定了就固定下来。
```

### 6. Excalidraw + draw.io — 只画两类图

**核心原则：只画有用的图，不画好看但没人看的图。**

| 工具 | 画什么 | 为什么用它 |
|------|--------|-----------|
| Excalidraw | 快速草图、头脑风暴、架构初始构想、流程草稿 | 手绘风格，低摩擦，随想随画，嵌入 Obsidian |
| draw.io | 正式系统架构图、数据库ER图、通信时序图 | 结构化、可导出、适合技术文档 |

**只画这两类图：**

```
类型A — 系统架构图（draw.io）
  用途：各子系统之间如何连接、数据怎么流
  什么时候画：架构设计定稿时
  存放：导出 PNG/SVG → Obsidian 对应模块文件夹

类型B — 思维草图（Excalidraw）
  用途：讨论时快速画关系、流程、备选方案对比
  什么时候画：和 Claude 讨论时，或自己理思路时
  存放：Excalidraw 文件直接存在 Obsidian 库里（有插件支持）

不画的东西：
  ✗ 纯装饰性的流程图
  ✗ 已经在代码/文档里说清楚的东西
  ✗ 给自己看的没必要正式化的图
```

## 三、日常工作循环（可持续节奏）

### 一个典型工作单元（2-4小时）

```
┌─────────────────────────────────────────────────────┐
│  Phase 1：准备（15分钟）                              │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 打开 Obsidian → 看项目总大纲 → 确定今天做什么  │ │
│  │ 2. 看对应模块文档的当前状态                       │ │
│  │ 3. 写下今天的具体目标（1-2个，不要贪多）           │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 2：推演（30-60分钟）                            │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 开 Claude 对话，贴入相关文档 + 今天的目标      │ │
│  │ 2. 和 Claude 讨论方案、排查问题、写代码草稿       │ │
│  │ 3. 对话中如果发现 Claude 输出可疑 → 用术语表判断  │ │
│  │    是 hallucination 还是 sycophancy？标记存疑     │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 3：执行（60-120分钟）                           │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 在 Cursor/PyCharm 中写代码                     │ │
│  │ 2. 测试、调试                                     │ │
│  │ 3. 遇到卡住的问题 → 回 Claude 讨论（带错误信息）  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 4：收尾（15-30分钟）← 最重要但最容易跳过的！    │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. Git commit + push（写清楚做了什么）             │ │
│  │ 2. Obsidian 更新：                                │ │
│  │    - 对话结论 → 99-Claude对话提炼/                │ │
│  │    - 架构变更 → 对应模块文档                      │ │
│  │    - 新问题 → 问题追踪清单                        │ │
│  │    - 重要决策 → 决策日志                          │ │
│  │ 3. 标记模块文档状态（草稿/进行中/定稿）            │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 为什么 Phase 4 是最重要的

你过去5个月"修修补补"的核心问题，大概率不是代码写得不好，而是**知识没有沉淀**。

每次和 Claude 对话完，有价值的结论留在了对话窗口里，下次开新对话又要从头来。这就像开了100次会但从来不写会议纪要——每次都在重新讨论同样的问题。

Phase 4 就是"写会议纪要"的环节。哪怕只花10分钟，写3条要点，也比不写强100倍。

## 四、你可能真正需要补充的（实用，不花哨）

### 需要补充的

| 工具/习惯 | 为什么需要 | 具体建议 |
|-----------|-----------|---------|
| **任务追踪方式** | 你有那么多子模块，需要知道"哪个在做、哪个卡住" | 不需要买 Jira。在 Obsidian 里建一个 `任务看板.md`，用 Kanban 插件或者纯文本表格。字段只需要：任务名、状态（待做/进行/卡住/完成）、所属模块、下一步 |

### 不需要补充的

| 经常被推荐但你不需要的 | 为什么不需要 |
|----------------------|-------------|
| Notion / Confluence | 你已经有 Obsidian，再加一个知识库只会分散内容 |
| Trello / Linear / Jira | 个人项目用 Obsidian 看板足够，上系统反而增加维护成本 |
| Docker（现阶段） | 等代码稳定后再容器化，现在加只增加复杂度 |
| CI/CD（现阶段） | 等有稳定的测试用例后再搭，现阶段手动部署够了 |
| 更多 AI 工具 | 你已经有 Claude + Cursor，再加 Copilot/GPT 会增加信息冲突风险 |

## 八、常见问题预判

**Q: Claude 对话中产出的代码，直接复制到编辑器吗？**
A: 复制后必须自己读一遍、理解、测试。Claude 产出的代码可能存在 hallucination（编造不存在的API）或 specification gaming（字面上满足了你的要求但实际逻辑不对）。特别是懒人精灵相关的 API，Claude 的训练数据中可能不包含准确信息。

**Q: Obsidian 和 Google Drive 的内容重复了怎么办？**
A: 以 Obsidian 为准。Google Drive 里的版本如果更旧，覆盖掉。不要出现两个地方各存一半的情况。

**Q: 一个问题涉及多个模块怎么办？**
A: 在 Claude 里可以讨论跨模块问题（比如 PC中控和安卓脚本的通信协议），但提炼结论时要分别写入两个模块的文档，并用 `[[双链]]` 互相指向。

**Q: 画图到底什么时候画？**
A: 只在两种情况下画：(1) 你自己理不清关系需要可视化思考时 → Excalidraw 草图；(2) 架构设计定稿需要一张正式参考图时 → draw.io。其他时候用文字描述就够了。

**Q: 这个工作流本身需要多久适应？**
A: 给自己两周时间。前几天会觉得"收尾阶段写 Obsidian 好麻烦"，但坚持一周后你会发现之前讨论过的东西再也不用重新问了。

## 九、这套工作流解决了什么

| 过去的问题 | 现在的解法 |
|-----------|-----------|
| 修修补补写了5个月的垃圾代码 | 先在 Claude 推演清楚方案 → 再写代码，不边写边改 |
| AI 产出了大量错误内容，分不清真假 | AI安全术语表 + 对话提炼时标注存疑点 |

# 蜂巢·大圣 (Hive-GreatSage) 工作流重构方案

> **核心原则：** 每个工具只做它最擅长的事，职责不重叠，数据流向单一且可追溯

## 一、工具角色定义（六个位置，各司其职）

### 1. Obsidian — 知识真相源（Knowledge Truth）

**它是什么：** 整个项目的"大脑"，所有非代码内容的唯一权威存储地。

**它负责：**
- 项目总大纲及所有子模块的架构文档
- 每次 Claude 对话后的**结论提取**（不是对话记录，是提炼后的决策/知识）
- 技术方案的定稿版本（草稿在 Claude 里推演，定稿搬到 Obsidian）
- 问题追踪清单（什么问题出现了、怎么解决的、当前状态）
- AI安全术语表（你已有的那份，作为项目级参考）
- 版本决策日志（为什么选A不选B，当时的理由是什么）

**它不负责：**
- 不存代码（代码去 GitHub）
- 不存正式对外文档（对外文档去 Google Drive）
- 不做实时协作（Obsidian 是本地优先工具）

### 2. Claude Project — 主推演区

**它是什么：** 设计思考和技术推演的"会议室"，不是存储的地方。

**它负责：**
- 架构设计的讨论和推演（"这个方案行不行？有什么坑？"）
- 代码框架的原型草稿（先在这里讨论清楚，再到 Cursor 里写）
- 技术选型分析（"用 SQLite 还是 MySQL？为什么？"）
- 文档草稿的撰写和修改
- Bug 排查思路讨论

**它不负责：**
- 不是知识存储（结论搬去 Obsidian）
- 不是代码仓库（代码搬去 GitHub）
- 不期望 Claude 记住所有历史（每次对话要给足上下文）

**Claude Project 窗口组织策略：**

```
蜂巢·大圣 Project 内的对话窗口规划：

窗口命名规则：[模块编号]-[主题]-[日期]
例如：
  02-PC中控-通信协议设计-0413
  03-安卓脚本-OCR方案选型-0415
  00-工作流-整体复盘-0420

不要做的事：
  ✗ 一个窗口里讨论3个不同模块 → 上下文混乱
  ✗ 不提供背景就问问题 → Claude 会 hallucinate（你的术语表第一条）
  ✗ 在对话里"修修补补" → 这正是你说的过去5个月的教训
```

**每次开启新对话的标准流程：**

```
第1步：从 Obsidian 复制相关模块的当前架构文档（给 Claude 大局观）
第2步：明确本次对话的目标（"这次我要解决 XXX 问题"）
第3步：如果是接续之前的讨论，贴上上次的结论摘要
第4步：对话进行
第5步：对话结束 → 提炼结论 → 写入 Obsidian → 标记对话链接
```

**给 Claude 大局观的方法：**

你说得对，这个项目需要给 AI 很强的项目大局观。实际操作就是——

在 Claude Project 的 **Project Knowledge**（项目知识库）中上传/粘贴以下内容：
- `项目总大纲.md`（让 Claude 知道整个项目的全貌）
- `AI安全术语表.md`（让 Claude 遵守你的术语规范）
- 当前正在工作的模块的架构设计文档

每当架构文档有更新，同步更新 Project Knowledge 中的版本。

### 4. Cursor vs PyCharm — 编码分工

**核心原则：不重叠。同一个文件不要两个编辑器同时打开。**

| 维度 | Cursor | PyCharm |
|------|--------|---------|
| 适合做 | AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码 | Python 大型项目、调试、重构、数据库工具 |
| 用在哪个子项目 | 安卓脚本框架（Lua）、小工具、快速实验 | PC中控框架（Python）、网络验证系统（后端）、官网监控 |
| 为什么 | Cursor 的 AI 能力适合"从0写"和脚本类工作 | PyCharm 的调试器、数据库工具、项目级重构更强 |

**具体分工：**

```
Cursor 负责打开的项目：
  - HiveGreatSage-AndroidScript（Lua 脚本）
  - HiveGreatSage-Tools（小工具）
  - 任何需要"用 AI 帮写"的新代码

PyCharm 负责打开的项目：
  - HiveGreatSage-Verify（Python 后端 + 数据库）
  - HiveGreatSage-PCControl（Python GUI）
  - HiveGreatSage-Monitor（Python 爬虫/监控）

规则：不要两个编辑器打开同一个仓库。选定了就固定下来。
```

### 6. Excalidraw + draw.io — 只画两类图

**核心原则：只画有用的图，不画好看但没人看的图。**

| 工具 | 画什么 | 为什么用它 |
|------|--------|-----------|
| Excalidraw | 快速草图、头脑风暴、架构初始构想、流程草稿 | 手绘风格，低摩擦，随想随画，嵌入 Obsidian |
| draw.io | 正式系统架构图、数据库ER图、通信时序图 | 结构化、可导出、适合技术文档 |

**只画这两类图：**

```
类型A — 系统架构图（draw.io）
  用途：各子系统之间如何连接、数据怎么流
  什么时候画：架构设计定稿时
  存放：导出 PNG/SVG → Obsidian 对应模块文件夹

类型B — 思维草图（Excalidraw）
  用途：讨论时快速画关系、流程、备选方案对比
  什么时候画：和 Claude 讨论时，或自己理思路时
  存放：Excalidraw 文件直接存在 Obsidian 库里（有插件支持）

不画的东西：
  ✗ 纯装饰性的流程图
  ✗ 已经在代码/文档里说清楚的东西
  ✗ 给自己看的没必要正式化的图
```

## 三、日常工作循环（可持续节奏）

### 一个典型工作单元（2-4小时）

```
┌─────────────────────────────────────────────────────┐
│  Phase 1：准备（15分钟）                              │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 打开 Obsidian → 看项目总大纲 → 确定今天做什么  │ │
│  │ 2. 看对应模块文档的当前状态                       │ │
│  │ 3. 写下今天的具体目标（1-2个，不要贪多）           │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 2：推演（30-60分钟）                            │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 开 Claude 对话，贴入相关文档 + 今天的目标      │ │
│  │ 2. 和 Claude 讨论方案、排查问题、写代码草稿       │ │
│  │ 3. 对话中如果发现 Claude 输出可疑 → 用术语表判断  │ │
│  │    是 hallucination 还是 sycophancy？标记存疑     │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 3：执行（60-120分钟）                           │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 在 Cursor/PyCharm 中写代码                     │ │
│  │ 2. 测试、调试                                     │ │
│  │ 3. 遇到卡住的问题 → 回 Claude 讨论（带错误信息）  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 4：收尾（15-30分钟）← 最重要但最容易跳过的！    │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. Git commit + push（写清楚做了什么）             │ │
│  │ 2. Obsidian 更新：                                │ │
│  │    - 对话结论 → 99-Claude对话提炼/                │ │
│  │    - 架构变更 → 对应模块文档                      │ │
│  │    - 新问题 → 问题追踪清单                        │ │
│  │    - 重要决策 → 决策日志                          │ │
│  │ 3. 标记模块文档状态（草稿/进行中/定稿）            │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 为什么 Phase 4 是最重要的

你过去5个月"修修补补"的核心问题，大概率不是代码写得不好，而是**知识没有沉淀**。

每次和 Claude 对话完，有价值的结论留在了对话窗口里，下次开新对话又要从头来。这就像开了100次会但从来不写会议纪要——每次都在重新讨论同样的问题。

Phase 4 就是"写会议纪要"的环节。哪怕只花10分钟，写3条要点，也比不写强100倍。

## 四、你可能真正需要补充的（实用，不花哨）

### 需要补充的

| 工具/习惯 | 为什么需要 | 具体建议 |
|-----------|-----------|---------|
| **任务追踪方式** | 你有那么多子模块，需要知道"哪个在做、哪个卡住" | 不需要买 Jira。在 Obsidian 里建一个 `任务看板.md`，用 Kanban 插件或者纯文本表格。字段只需要：任务名、状态（待做/进行/卡住/完成）、所属模块、下一步 |
| **Obsidian Git 插件** | 给你的知识库也加版本控制，防止误改 | 安装 Obsidian Git 插件，设置每天自动提交到一个私有 GitHub 仓库 |
| **固定的对话提炼模板** | 让 Phase 4 变成机械化动作，降低阻力 | 见下方模板 |

### 不需要补充的

| 经常被推荐但你不需要的 | 为什么不需要 |
|----------------------|-------------|
| Notion / Confluence | 你已经有 Obsidian，再加一个知识库只会分散内容 |
| Trello / Linear / Jira | 个人项目用 Obsidian 看板足够，上系统反而增加维护成本 |
| Docker（现阶段） | 等代码稳定后再容器化，现在加只增加复杂度 |
| CI/CD（现阶段） | 等有稳定的测试用例后再搭，现阶段手动部署够了 |
| 更多 AI 工具 | 你已经有 Claude + Cursor，再加 Copilot/GPT 会增加信息冲突风险 |

## 八、常见问题预判

**Q: Claude 对话中产出的代码，直接复制到编辑器吗？**
A: 复制后必须自己读一遍、理解、测试。Claude 产出的代码可能存在 hallucination（编造不存在的API）或 specification gaming（字面上满足了你的要求但实际逻辑不对）。特别是懒人精灵相关的 API，Claude 的训练数据中可能不包含准确信息。

**Q: Obsidian 和 Google Drive 的内容重复了怎么办？**
A: 以 Obsidian 为准。Google Drive 里的版本如果更旧，覆盖掉。不要出现两个地方各存一半的情况。

**Q: 一个问题涉及多个模块怎么办？**
A: 在 Claude 里可以讨论跨模块问题（比如 PC中控和安卓脚本的通信协议），但提炼结论时要分别写入两个模块的文档，并用 `[[双链]]` 互相指向。

**Q: 画图到底什么时候画？**
A: 只在两种情况下画：(1) 你自己理不清关系需要可视化思考时 → Excalidraw 草图；(2) 架构设计定稿需要一张正式参考图时 → draw.io。其他时候用文字描述就够了。

**Q: 这个工作流本身需要多久适应？**
A: 给自己两周时间。前几天会觉得"收尾阶段写 Obsidian 好麻烦"，但坚持一周后你会发现之前讨论过的东西再也不用重新问了。

# 蜂巢·大圣 (Hive-GreatSage) 工作流重构方案

> **文档性质：** 工作流唯一参考文档，后续所有工具配置以此为准
> **适用阶段：** 项目全生命周期（6个月~多年）
> **核心原则：** 每个工具只做它最擅长的事，职责不重叠，数据流向单一且可追溯

## 一、工具角色定义（六个位置，各司其职）

### 1. Obsidian — 知识真相源（Knowledge Truth）

**它是什么：** 整个项目的"大脑"，所有非代码内容的唯一权威存储地。

**它负责：**
- 项目总大纲及所有子模块的架构文档
- 每次 Claude 对话后的**结论提取**（不是对话记录，是提炼后的决策/知识）
- 技术方案的定稿版本（草稿在 Claude 里推演，定稿搬到 Obsidian）
- 问题追踪清单（什么问题出现了、怎么解决的、当前状态）
- AI安全术语表（你已有的那份，作为项目级参考）
- 版本决策日志（为什么选A不选B，当时的理由是什么）

**它不负责：**
- 不存代码（代码去 GitHub）
- 不存正式对外文档（对外文档去 Google Drive）
- 不做实时协作（Obsidian 是本地优先工具）

### 2. Claude Project — 主推演区

**它是什么：** 设计思考和技术推演的"会议室"，不是存储的地方。

**它负责：**
- 架构设计的讨论和推演（"这个方案行不行？有什么坑？"）
- 代码框架的原型草稿（先在这里讨论清楚，再到 Cursor 里写）
- 技术选型分析（"用 SQLite 还是 MySQL？为什么？"）
- 文档草稿的撰写和修改
- Bug 排查思路讨论

**它不负责：**
- 不是知识存储（结论搬去 Obsidian）
- 不是代码仓库（代码搬去 GitHub）
- 不期望 Claude 记住所有历史（每次对话要给足上下文）

**Claude Project 窗口组织策略：**

```
蜂巢·大圣 Project 内的对话窗口规划：

窗口命名规则：[模块编号]-[主题]-[日期]
例如：
  02-PC中控-通信协议设计-0413
  03-安卓脚本-OCR方案选型-0415
  00-工作流-整体复盘-0420

不要做的事：
  ✗ 一个窗口里讨论3个不同模块 → 上下文混乱
  ✗ 不提供背景就问问题 → Claude 会 hallucinate（你的术语表第一条）
  ✗ 在对话里"修修补补" → 这正是你说的过去5个月的教训
```

**每次开启新对话的标准流程：**

```
第1步：从 Obsidian 复制相关模块的当前架构文档（给 Claude 大局观）
第2步：明确本次对话的目标（"这次我要解决 XXX 问题"）
第3步：如果是接续之前的讨论，贴上上次的结论摘要
第4步：对话进行
第5步：对话结束 → 提炼结论 → 写入 Obsidian → 标记对话链接
```

**给 Claude 大局观的方法：**

你说得对，这个项目需要给 AI 很强的项目大局观。实际操作就是——

在 Claude Project 的 **Project Knowledge**（项目知识库）中上传/粘贴以下内容：
- `项目总大纲.md`（让 Claude 知道整个项目的全貌）
- `AI安全术语表.md`（让 Claude 遵守你的术语规范）
- 当前正在工作的模块的架构设计文档

每当架构文档有更新，同步更新 Project Knowledge 中的版本。

### 4. Cursor vs PyCharm — 编码分工

**核心原则：不重叠。同一个文件不要两个编辑器同时打开。**

| 维度 | Cursor | PyCharm |
|------|--------|---------|
| 适合做 | AI 辅助写新代码、快速原型、Lua 脚本、前端/轻量代码 | Python 大型项目、调试、重构、数据库工具 |
| 用在哪个子项目 | 安卓脚本框架（Lua）、小工具、快速实验 | PC中控框架（Python）、网络验证系统（后端）、官网监控 |
| 为什么 | Cursor 的 AI 能力适合"从0写"和脚本类工作 | PyCharm 的调试器、数据库工具、项目级重构更强 |

**具体分工：**

```
Cursor 负责打开的项目：
  - HiveGreatSage-AndroidScript（Lua 脚本）
  - HiveGreatSage-Tools（小工具）
  - 任何需要"用 AI 帮写"的新代码

PyCharm 负责打开的项目：
  - HiveGreatSage-Verify（Python 后端 + 数据库）
  - HiveGreatSage-PCControl（Python GUI）
  - HiveGreatSage-Monitor（Python 爬虫/监控）

规则：不要两个编辑器打开同一个仓库。选定了就固定下来。
```

### 5. Google Drive — 文档镜像层（不是真相源）

**它是什么：** 对外共享和协作的"展示窗口"。

**它负责：**
- 需要分享给别人的正式文档（如产品说明、用户手册）
- Claude 的 Google Drive Connector 读取参考资料
- 备份（Obsidian 库可以定期导出一份到 Drive）

**它不负责：**
- 不是你日常编辑的地方（日常编辑在 Obsidian）
- 不是代码存储（代码在 GitHub）

**数据流向：** Obsidian 定稿 → 导出/复制到 Google Drive（单向）

### 6. Excalidraw + draw.io — 只画两类图

**核心原则：只画有用的图，不画好看但没人看的图。**

| 工具 | 画什么 | 为什么用它 |
|------|--------|-----------|
| Excalidraw | 快速草图、头脑风暴、架构初始构想、流程草稿 | 手绘风格，低摩擦，随想随画，嵌入 Obsidian |
| draw.io | 正式系统架构图、数据库ER图、通信时序图 | 结构化、可导出、适合技术文档 |

**只画这两类图：**

```
类型A — 系统架构图（draw.io）
  用途：各子系统之间如何连接、数据怎么流
  什么时候画：架构设计定稿时
  存放：导出 PNG/SVG → Obsidian 对应模块文件夹

类型B — 思维草图（Excalidraw）
  用途：讨论时快速画关系、流程、备选方案对比
  什么时候画：和 Claude 讨论时，或自己理思路时
  存放：Excalidraw 文件直接存在 Obsidian 库里（有插件支持）

不画的东西：
  ✗ 纯装饰性的流程图
  ✗ 已经在代码/文档里说清楚的东西
  ✗ 给自己看的没必要正式化的图
```

## 三、日常工作循环（可持续节奏）

### 一个典型工作单元（2-4小时）

```
┌─────────────────────────────────────────────────────┐
│  Phase 1：准备（15分钟）                              │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 打开 Obsidian → 看项目总大纲 → 确定今天做什么  │ │
│  │ 2. 看对应模块文档的当前状态                       │ │
│  │ 3. 写下今天的具体目标（1-2个，不要贪多）           │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 2：推演（30-60分钟）                            │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 开 Claude 对话，贴入相关文档 + 今天的目标      │ │
│  │ 2. 和 Claude 讨论方案、排查问题、写代码草稿       │ │
│  │ 3. 对话中如果发现 Claude 输出可疑 → 用术语表判断  │ │
│  │    是 hallucination 还是 sycophancy？标记存疑     │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 3：执行（60-120分钟）                           │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. 在 Cursor/PyCharm 中写代码                     │ │
│  │ 2. 测试、调试                                     │ │
│  │ 3. 遇到卡住的问题 → 回 Claude 讨论（带错误信息）  │ │
│  └─────────────────────────────────────────────────┘ │
│                                                       │
│  Phase 4：收尾（15-30分钟）← 最重要但最容易跳过的！    │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 1. Git commit + push（写清楚做了什么）             │ │
│  │ 2. Obsidian 更新：                                │ │
│  │    - 对话结论 → 99-Claude对话提炼/                │ │
│  │    - 架构变更 → 对应模块文档                      │ │
│  │    - 新问题 → 问题追踪清单                        │ │
│  │    - 重要决策 → 决策日志                          │ │
│  │ 3. 标记模块文档状态（草稿/进行中/定稿）            │ │
│  └─────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

### 为什么 Phase 4 是最重要的

你过去5个月"修修补补"的核心问题，大概率不是代码写得不好，而是**知识没有沉淀**。

每次和 Claude 对话完，有价值的结论留在了对话窗口里，下次开新对话又要从头来。这就像开了100次会但从来不写会议纪要——每次都在重新讨论同样的问题。

Phase 4 就是"写会议纪要"的环节。哪怕只花10分钟，写3条要点，也比不写强100倍。

## 四、你可能真正需要补充的（实用，不花哨）

### 需要补充的

| 工具/习惯 | 为什么需要 | 具体建议 |
|-----------|-----------|---------|
| **任务追踪方式** | 你有那么多子模块，需要知道"哪个在做、哪个卡住" | 不需要买 Jira。在 Obsidian 里建一个 `任务看板.md`，用 Kanban 插件或者纯文本表格。字段只需要：任务名、状态（待做/进行/卡住/完成）、所属模块、下一步 |
| **Obsidian Git 插件** | 给你的知识库也加版本控制，防止误改 | 安装 Obsidian Git 插件，设置每天自动提交到一个私有 GitHub 仓库 |
| **固定的对话提炼模板** | 让 Phase 4 变成机械化动作，降低阻力 | 见下方模板 |

### 不需要补充的

| 经常被推荐但你不需要的 | 为什么不需要 |
|----------------------|-------------|
| Notion / Confluence | 你已经有 Obsidian，再加一个知识库只会分散内容 |
| Trello / Linear / Jira | 个人项目用 Obsidian 看板足够，上系统反而增加维护成本 |
| Docker（现阶段） | 等代码稳定后再容器化，现在加只增加复杂度 |
| CI/CD（现阶段） | 等有稳定的测试用例后再搭，现阶段手动部署够了 |
| 更多 AI 工具 | 你已经有 Claude + Cursor，再加 Copilot/GPT 会增加信息冲突风险 |

## 五、对话提炼模板

每次 Claude 对话结束后，在 Obsidian 的 `99-Claude对话提炼/` 下新建文件，用以下模板：

```markdown
# [日期] [模块] [主题]

## 对话目标
<!-- 这次对话想解决什么 -->

## 关键结论
<!-- 3-5 条最重要的结论，用自己的话写 -->
1.
2.
3.

## 代码产出
<!-- 这次对话产出了什么代码？已经提交到哪个仓库的哪个分支？ -->

## 存疑/待验证
<!-- Claude 的输出中哪些你不确定是否正确？用 AI 安全术语标注 -->
<!-- 例如：Claude 说 XX 方案性能更好 → 可能是 sycophancy，需要实测验证 -->

## 下一步
<!-- 基于这次对话，下一步要做什么 -->

## 对话链接
<!-- Claude 对话的 URL -->
```

## 八、常见问题预判

**Q: Claude 对话中产出的代码，直接复制到编辑器吗？**
A: 复制后必须自己读一遍、理解、测试。Claude 产出的代码可能存在 hallucination（编造不存在的API）或 specification gaming（字面上满足了你的要求但实际逻辑不对）。特别是懒人精灵相关的 API，Claude 的训练数据中可能不包含准确信息。

**Q: Obsidian 和 Google Drive 的内容重复了怎么办？**
A: 以 Obsidian 为准。Google Drive 里的版本如果更旧，覆盖掉。不要出现两个地方各存一半的情况。

**Q: 一个问题涉及多个模块怎么办？**
A: 在 Claude 里可以讨论跨模块问题（比如 PC中控和安卓脚本的通信协议），但提炼结论时要分别写入两个模块的文档，并用 `[[双链]]` 互相指向。

**Q: 画图到底什么时候画？**
A: 只在两种情况下画：(1) 你自己理不清关系需要可视化思考时 → Excalidraw 草图；(2) 架构设计定稿需要一张正式参考图时 → draw.io。其他时候用文字描述就够了。

**Q: 这个工作流本身需要多久适应？**
A: 给自己两周时间。前几天会觉得"收尾阶段写 Obsidian 好麻烦"，但坚持一周后你会发现之前讨论过的东西再也不用重新问了。

## 九、这套工作流解决了什么

| 过去的问题 | 现在的解法 |
|-----------|-----------|
| 修修补补写了5个月的垃圾代码 | 先在 Claude 推演清楚方案 → 再写代码，不边写边改 |
| AI 产出了大量错误内容，分不清真假 | AI安全术语表 + 对话提炼时标注存疑点 |

| 和 Claude 对话的结论丢失 | Phase 4 强制提炼 + 对话提炼模板 |
| 工具太多不知道用哪个 | 每个工具固定职责，不重叠 |
| 缺乏项目大局观 | 项目总大纲在 Obsidian + Claude Project Knowledge 都有 |
