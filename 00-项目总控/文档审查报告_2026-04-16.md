## 二、P0 严重问题（必须立即修复）

### P0-1: 项目总大纲的"版本号"自相矛盾

**问题：**
标题写的是 `v2`，副标题又写 `v3`：
```
# 蜂巢·大圣 (Hive-GreatSage) 平台级项目 — 全新重构版总大纲 v2
> 文档状态：草稿 v3（已纳入第二轮决策确认）
```
这是典型的**文档漂移**——版本升级时只改了一处没改另一处。

**影响：**
新人看到会困惑到底是哪版；后续再迭代时不知道 v4 应该从哪里改起。

**修复方案：**
```markdown
# 蜂巢·大圣 (Hive-GreatSage) 平台级项目总大纲

> 文档状态: 草稿 v3
> 最后更新: 2026-04-16
> 变更记录:
>   - v3: 锁定核心决策（Redis / Vue / 双模式存储 / 独立仓库）
>   - v2: 纳入关键修正（独立中控脚本 / 云端数据流 / 多级代理 / PySide 6）
>   - v1: 初始推演版
```
**把版本号从标题移到元信息区**，以后只改一处。

### P0-3: 编码规范文档自己不符合编码规范

**问题：**
`编码规范.md` 自己的文件头是：
```markdown
# 蜂巢·大圣 (Hive-GreatSage) 编码规范
> **文档性质：** 项目级编码规范...
```

但根据编码规范第一节"文件头规范（强制）"，每个文件都必须有标准文件头。虽然规范里讨论的是"代码文件"，但 Markdown 文档是否也要遵守？目前项目里的文档有两种风格并存：

**风格 A（多数文档）：** `> 文档性质 / 状态 / 更新时间` 块引用
**风格 B（我最近产出的）：** `r""" ... """` Python docstring 式

这种不一致会让 Claude 在产出新文档时随机选一种，造成持续漂移。

**影响：**
规范的权威性受损；文档头信息缺失关键字段（版本、作者）。

**修复方案：**
在编码规范中新增一节"Markdown 文档头规范"，明确约定：
```markdown
## 一点五、Markdown 文档头规范（强制）

所有 Obsidian 文档统一使用 YAML frontmatter 格式：

---
文件位置: 路径/文件名.md
名称: 文档显示名称
作者: 蜂巢·大圣 (Hive-GreatSage)
时间: 2026-04-16
版本: V1.0.1
状态: 草稿 / 定稿 / 过时
关联文档: [[文档A]], [[文档B]]
变更记录:
  - V1.0.1: ...
  - V1.0.0: 初始版本
---

# 标题
正文...
```

理由：
1. YAML frontmatter 是 Obsidian 原生支持的元数据格式
2. Dataview 插件可以跨文档查询这些字段
3. 字段顺序和内容与代码文件头对齐
4. 既不用代码风格的 `r"""`（Markdown 里视觉违和），也不用简化的 `>` 块（缺字段）

### P1-7: "工作流方案"目录规划中的一个矛盾

**问题：**
工作流方案里写：
```
├── 02-PC中控框架/
│   ├── 架构设计.md
│   ├── 通信协议.md           ← 这个名字
```

但我们实际产出的是 `PC安卓局域网通信方案.md`，不叫"通信协议"。

一个是"方案"（怎么做），一个是"协议"（消息格式定义）——其实这是两个不同层级的文档：
- **方案**：整体策略、技术选型、架构理由
- **协议**：消息格式、字段定义、API 规范

**修复方案：**
两个都保留，分工如下：
```
02-PC中控框架/
├── 架构设计.md                  ← 整体架构
├── PC安卓局域网通信方案.md      ← 方案层（为什么这么做）
├── 通信协议.md                  ← 协议层（具体消息格式，从方案抽出）
└── 跨账号智能调度分析.md        ← 功能分析
```

把方案文档里"五、消息协议"那一节抽出来成为独立的 `通信协议.md`，方案文档里保留链接即可。

### P2-2: 时间格式不统一

文档里出现过：
- `2026-04-13`（ISO 8601，推荐）
- `2026年4月13日`（中文写法）
- `2026/04/13`（反斜杠）

**修复方案：**
编码规范已规定为 `YYYY-MM-DD`，但 Markdown 文档元数据里没强调。把这条扩展到文档规范。

### P2-3: 表情符号不一致

文档里混用：
- ✓ / ✗
- ✅ / ❌
- □ / ✓（已确认）

**修复方案：**
编码规范里统一：
- `✓` 表示确认/完成
- `✗` 表示禁止/错误
- `□` 表示待做
- `⚠️` 表示警告
- 不使用 emoji（避免渲染差异）

## 六、给未来对话的建议

为了防止漂移再次积累，建议每次结束重要对话后：

1. **强制更新总大纲**
   总大纲是唯一真相源，任何架构级变动必须反映在这里。

2. **强制登记决策日志**
   哪怕只有一行：`D0XX — 日期 — 什么决策 — 为什么`。

3. **用 Claude 做月度审查**
   每个月让 Claude 做一次像本次这样的审查，主动发现漂移。

4. **新文档必须有双链**
   至少有一个 `[[父文档]]` 和一个 `[[兄弟文档]]` 引用，否则是孤岛。
