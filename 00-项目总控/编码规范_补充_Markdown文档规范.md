# 编码规范补充 — Markdown 文档规范

> **说明：** 这是对 [[编码规范]] 的补充章节，专门规范 Obsidian 知识库中 Markdown 文档的格式。
> **应用范围：** 所有存放在 Obsidian Vault 内的 `.md` 文件。
> **合并时机：** 下一次修订 [[编码规范]] 时，将本文内容并入主规范作为"第一点五节"。

---

## 一、文档头规范（强制）

所有 Obsidian 文档统一使用 **YAML Frontmatter 格式**（Obsidian 原生支持）。

### 标准模板

```markdown
---
文件位置: 00-项目总控/文档名.md
名称: 文档的中文显示名称
作者: 蜂巢·大圣 (Hive-GreatSage)
时间: 2026-04-16
版本: V1.0.1
状态: 草稿 / 定稿 / 过时
关联文档: "`文档A`, `文档B`"
变更记录:
  - V1.0.1: 新增XX内容
  - V1.0.0: 初版
---

# 文档标题

正文从这里开始...
```

### 字段说明

| 字段 | 必填 | 说明 | 示例 |
|------|------|------|------|
| 文件位置 | 是 | 相对于 Obsidian Vault 根目录的路径 | `00-项目总控/决策日志.md` |
| 名称 | 是 | 中文显示名称（可与标题不同） | `决策日志` |
| 作者 | 是 | 统一为项目名 | `蜂巢·大圣 (Hive-GreatSage)` |
| 时间 | 是 | 文档创建日期 `YYYY-MM-DD` | `2026-04-16` |
| 版本 | 是 | 文档版本号 `V主.次.修` | `V1.0.1` |
| 状态 | 是 | 草稿 / 定稿 / 过时 | `定稿` |
| 关联文档 | 否 | 相关文档的双链列表 | `"[[项目总大纲]], [[决策日志]]"` |
| 变更记录 | 是 | 按版本倒序列出 | 见模板 |

### 为什么不用 `r"""` docstring 格式

在 Markdown 文档里用 `r"""` 会有三个问题：

1. **视觉违和** — Markdown 环境里 `r"""` 是 Python 语法，会让人误以为这是代码块
2. **Obsidian 不支持** — Obsidian 的 YAML frontmatter 提供原生的元数据支持，被 Dataview 等插件识别
3. **无法跨文档查询** — Dataview 插件可以查询所有带 YAML frontmatter 的文档（如"列出所有状态为草稿的文档"），`r"""` 格式做不到

### 为什么不用简单的 `> 引用块` 格式

虽然简洁但有三个问题：

1. **字段不完整** — 经常缺失版本号、作者、变更记录
2. **格式不统一** — 每个文档写法略有差异
3. **无法被工具识别** — 纯引用块对 Obsidian 插件来说就是普通文本

## 二、文档正文规范

### 标题层级

```
# 一级标题（每个文档只有一个，作为文档标题）
## 二级标题（章节）
### 三级标题（小节）
#### 四级标题（谨慎使用，过深说明结构需要拆分）
```

### 段落与空行

- 段落之间用一个空行分隔
- 标题前后各留一个空行
- 列表与段落之间留一个空行

### 代码块语言标注（强制）

所有代码块必须标注语言：

````markdown
```python
def hello():
    pass
```

```lua
function hello()
end
```

```sql
SELECT * FROM user;
```

```text
纯文本内容（ASCII 图、目录结构等）
```

```bash
ls -la
```
````

不写语言的 ` ``` ` 代码块在 Obsidian 里会失去语法高亮，也会在未来 Dataview 查询时被识别为"未分类内容"。

### 表格规范

- 表格前后各留一个空行
- 列头和数据行的分隔线必须对齐
- 单元格内不要换行（如有多行内容用 `<br>`）

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 值1 | 值2 | 值3 |
```

## 三、双链规范（P1-6 修复）

### 每个文档必须至少有：

1. **一个父节点引用**（在 YAML 的 `关联文档` 字段或正文第一段）
2. **一个相关文档引用**（兄弟文档或被引用的参考资料）

孤岛文档（没有任何双链）是禁止的。

### 双链写法

```markdown
- [[项目总大纲]]              ← 默认显示为"项目总大纲"
- [[项目总大纲|总大纲]]       ← 显示为"总大纲"
- [[01-网络验证系统/架构设计]] ← 跨文件夹引用完整路径
```

### 面包屑（推荐）

在子文档正文第一行加返回链接：

```markdown
← 返回 [[项目总大纲]] | 父节点: [[02-PC中控框架/架构设计]]
```

## 四、文档状态流转

```
草稿 ──(内容稳定后)──> 定稿 ──(架构调整导致内容失效)──> 过时
                            │
                            └─(内容需要修订)──> 草稿（新版本号）
```

**状态变更规则：**

- 从草稿转定稿：内容已经在团队内达成共识，不会频繁改动
- 从定稿转过时：架构调整导致本文档的方案已被替换，但历史意义仍需保留
- 过时文档不删除，在顶部注明 `已被 `新文档` 替代`

## 五、文件命名规范

### 目录命名

```
00-项目总控/      ← 两位数字前缀 + 短横线 + 中文名
01-网络验证系统/
02-PC中控框架/
```

### 文件命名

```
项目总大纲.md          ← 中文名，简洁明了
决策日志.md
PC安卓局域网通信方案.md ← 可包含简短的英文标识
跨账号智能调度分析.md
```

**禁止：**
- 文件名包含空格
- 文件名包含版本号（版本号写在 YAML 里）
- 文件名过长（超过 20 个汉字）
- 文件名带日期（除非是会议纪要等时间相关文档）

## 六、符号使用规范（P2-3 修复）

| 符号 | 含义 | 用法 |
|------|------|------|
| `✓` | 确认 / 完成 | 决策已确认、任务已完成 |
| `✗` | 禁止 / 错误 | 反面示例、禁止的做法 |
| `□` | 待做 | 未完成的任务 |
| `⚠️` | 警告 / 注意 | 特别重要的提醒 |
| `→` | 流向 | 数据流或步骤顺序（A → B） |
| `←` | 引用关系 | A ← B 表示 A 被 B 引用 |
| `◄──` | 单向连接 | WebSocket 客户端连接服务端 |
| `◄─►` | 双向连接 | 对等通信（谨慎使用） |

**禁止使用的符号：**
- 彩色 emoji（✅ ❌ 🔴 🟢）— 不同平台渲染差异大
- 表情符号装饰（🎉 💡 🔥）— 降低专业性
- 特殊字符（⭐ ♥ ☆）— 字体依赖问题

## 七、时间格式（P2-2 修复）

**唯一允许的日期格式：** `YYYY-MM-DD`

```
✓ 2026-04-16
✗ 2026/04/16
✗ 2026年4月16日
✗ 16/04/2026
✗ Apr 16, 2026
```

时间戳格式（带时间）：`YYYY-MM-DD HH:MM:SS`（如日志）

## 八、文档的审查机制

为了防止文档漂移：

1. **每次架构级变更必须同步更新总大纲**
   子系统文档改了方案，必须立刻反映到 [[项目总大纲]]

2. **每月全局审查**
   每月末让 Claude 做一次全局文档审查，识别漂移

3. **决策必须登记**
   任何架构/选型决定，当天登记到 [[决策日志]]

4. **孤岛文档警报**
   创建新文档时必须加至少一个双链，否则视为不合规

## 九、检查清单

提交/保存新 Markdown 文档前对照检查：

```text
YAML frontmatter:
  □ 文件位置、名称、作者、时间、版本、状态 都填了
  □ 关联文档 至少一个双链
  □ 变更记录 按版本倒序
  □ 作者统一为 "蜂巢·大圣 (Hive-GreatSage)"
  □ 时间是 YYYY-MM-DD 格式
  □ 版本是 V主.次.修 格式

正文:
  □ 一级标题只有一个
  □ 代码块都标注了语言
  □ 表格格式对齐
  □ 双链引用路径正确
  □ 符号使用符合规范（✓ ✗ □ ⚠️）

文件:
  □ 文件名不含空格、版本号、日期
  □ 在正确的子目录下
```
