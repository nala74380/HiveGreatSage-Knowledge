### 目录结构设计理由

```
核心原则：game/ 目录是唯一的游戏定制区

  fork 框架基座后，开发者需要修改的文件主要集中在 game/ 目录。
  core/ 和 ui/widgets/ 一般不需要改动。
  ui/main_window.py 可能需要微调（如添加游戏特有的菜单项）。

  这样做的好处：
  1. 框架基座更新时，cherry-pick 冲突概率最低
     （因为游戏定制代码和框架代码在不同目录）
  2. 新开发者 fork 后能快速定位"我需要改哪里"
  3. 代码审查时能清楚区分"框架代码"和"游戏代码"
```

## 四、线程模型（关键设计）

### 为什么需要多线程

```
PySide 6 的黄金规则：不能在 UI 线程（主线程）做耗时操作。
  · HTTP 请求（云端 API 调用）是 I/O 操作，可能耗时几百毫秒到几秒
  · 如果在主线程做 HTTP 请求，界面会卡住（用户以为程序死了）
  · 所以所有网络操作必须放在工作线程中

PySide 6 的另一个黄金规则：不能在工作线程直接操作 UI。
  · 工作线程拿到数据后，必须通过信号（Signal）通知主线程
  · 主线程收到信号后更新 UI
```

### 线程架构

```
┌──────────────────────────────────────────────────────────┐
│  主线程（UI 线程）                                         │
│                                                          │
│  · Qt 事件循环                                            │
│  · 所有 UI 操作（创建/更新/销毁 widget）                   │
│  · 接收工作线程的信号 → 更新 UI                            │
│  · 用户交互处理（点击按钮 → 发起异步操作）                  │
│                                                          │
│  连接的信号：                                              │
│    sync_worker.data_received   → device_list_widget.refresh│
│    sync_worker.error_occurred  → status_bar.show_error    │
│    auth_manager.token_expired  → 弹出重新登录窗口          │
│    update_checker.update_found → 弹出更新对话框            │
└──────────────────────────────────────────────────────────┘
         ▲ Signal（线程安全）  │ 触发异步操作
         │                    ▼
┌──────────────────────────────────────────────────────────┐
│  同步工作线程（QThread）                                    │
│                                                          │
│  · SyncWorker 运行在这个线程                               │
│  · 定时循环（如每 10 秒一次）：                             │
│      1. 调用 device_api.get_device_list()                 │
│      2. 调用 device_api.get_device_data()                 │
│      3. 发射 data_received 信号（携带数据）                 │
│  · 遇到网络错误 → 发射 error_occurred 信号                 │
│  · 检测到 Token 过期 → 发射 token_expired 信号             │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│  按需工作线程（QRunnable / QThreadPool）                    │
│                                                          │
│  · 用户点击"保存参数" → 提交到线程池 → HTTP 推送到云端      │
│  · 用户点击"检查更新" → 提交到线程池 → HTTP 请求检查        │
│  · 热更新下载 → 提交到线程池 → 下载文件 + 校验              │
│                                                          │
│  每个任务完成后通过信号回调主线程                            │
└──────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────┐
│  组队通信线程（QThread）— 可选，仅局域网组队时启用           │
│                                                          │
│  · 运行局域网 Socket 服务                                  │
│  · 接收安卓设备的组队响应                                   │
│  · 发送组队指令                                            │
│  · 通过信号与主线程交互                                     │
└──────────────────────────────────────────────────────────┘
```

### 信号槽连接示意（核心）

```python

    """运行在同步工作线程中"""
    data_received = Signal(dict)        # 设备数据到达
    error_occurred = Signal(str)        # 网络错误
    token_expired = Signal()            # Token 过期

class DeviceListWidget(QWidget):
    """设备列表 UI 组件"""
    def refresh(self, data: dict):
        # 用新数据刷新设备列表显示
        ...

class MainWindow(QMainWindow):
    def __init__(self):
        # 创建组件
        self.sync_worker = SyncWorker()
        self.device_list = DeviceListWidget()
        self.status_bar = StatusBarWidget()

        # 连接信号槽 — 这是核心接线
        self.sync_worker.data_received.connect(self.device_list.refresh)
        self.sync_worker.error_occurred.connect(self.status_bar.show_error)
        self.sync_worker.token_expired.connect(self.on_token_expired)

    def on_token_expired(self):
        # 弹出重新登录窗口
        ...
```

## 五、模块详细设计

### 5.1 登录/验证模块（core/auth/）

```
职责：
  · 管理用户登录流程
  · 存储和管理 Token（内存 + 本地加密存储）
  · Token 过期检测和刷新
  · 提供当前用户信息给其他模块

核心类：AuthManager

  属性：
    · token: str | None           ← 当前有效 Token
    · user_info: UserInfo | None  ← 当前用户信息
    · is_logged_in: bool

  方法：
    · login(username, password) → LoginResult
        调用 auth_api.login()
        成功 → 保存 Token + user_info → 返回成功
        失败 → 返回失败原因

    · logout()
        清除 Token 和 user_info
        通知其他模块停止工作

    · verify_token() → bool
        检查 Token 是否仍有效
        如果过期 → 尝试刷新 → 刷新失败 → 发射 token_expired 信号

    · get_token() → str
        供 API 客户端获取当前 Token（每个请求都需要）

  信号：
    · login_success(UserInfo)
    · login_failed(str)         ← 错误原因
    · token_expired()
    · logged_out()

数据模型（core/auth/models.py）：

  class UserInfo:
      user_id: str
      username: str
      user_level: str           ← trial/normal/vip/svip/tester
      game_project_id: str      ← 当前授权的游戏项目（中控是单游戏的）
      authorized_until: datetime

  class LoginResult:
      success: bool
      token: str | None
      user_info: UserInfo | None
      error_message: str | None
```

### 5.2 云端 API 客户端（core/api_client/）

```
职责：
  · 封装所有与网络验证系统的 HTTP 通信
  · 统一处理请求头（Token）、错误处理、重试逻辑、超时
  · 为上层模块提供干净的 Python 方法调用

核心类：BaseClient（base_client.py）

  属性：
    · base_url: str              ← 云端 API 地址（从配置读取）
    · timeout: int               ← 请求超时（秒）
    · max_retries: int           ← 最大重试次数
    · auth_manager: AuthManager  ← 用于获取 Token

  方法：
    · get(endpoint, params) → Response
    · post(endpoint, data) → Response
    · _build_headers() → dict
        自动附加 Authorization: Bearer {token}
        自动附加 X-Game-Project-Id: {game_project_id}
    · _handle_response(response) → data | raise
        200 → 返回 JSON 数据
        401 → 通知 auth_manager Token 可能过期
        429 → 等待后重试
        5xx → 重试 / 抛出服务器错误
        网络超时 → 重试 / 抛出连接错误
```

### 中控需要的 API 接口契约（中控侧定义）

```
以下是 PC 中控需要调用的接口，定义的是"中控需要什么"。
等设计网络验证系统时，以此作为需求输入来设计实际 API。

═══════════════════════════════════════════════════════
auth_api.py — 认证相关
═══════════════════════════════════════════════════════

POST /auth/login
  请求：{ username, password, device_fingerprint, client_type: "pc" }
  响应：{ token, refresh_token, user_info, expires_in }
  说明：PC 中控登录

POST /auth/refresh-token
  请求：{ refresh_token }
  响应：{ token, expires_in }
  说明：Token 刷新

POST /auth/logout
  请求：{ token }
  说明：主动登出

═══════════════════════════════════════════════════════
device_api.py — 设备数据相关
═══════════════════════════════════════════════════════

GET /device/list?user_id={uid}
  响应：{
    devices: [
      {
        device_id, device_name, status(online/offline/running/error),
        last_heartbeat, current_task, ip_address
      }
    ]
  }
  说明：获取当前用户所有绑定设备的状态列表

GET /device/data?user_id={uid}&device_id={did}
  响应：{
    device_id, last_update,
    runtime_data: { ... }     ← JSONB，内容因游戏而异
  }
  说明：获取单个设备的运行数据详情

GET /device/data/batch?user_id={uid}
  响应：{
    devices: [
      { device_id, last_update, runtime_data: { ... } }
    ]
  }
  说明：批量获取所有设备的运行数据（减少请求次数）

GET /device/logs?user_id={uid}&device_id={did}&limit=100&offset=0
  响应：{
    logs: [ { timestamp, level, message } ]
  }
  说明：获取设备日志

═══════════════════════════════════════════════════════
params_api.py — 参数设置相关
═══════════════════════════════════════════════════════

GET /params/schema
  响应：{
    params_schema: [
      {
        key: "auto_heal_threshold",
        label: "自动回血阈值",
        type: "int",
        default: 50,
        min: 10, max: 90,
        group: "战斗设置"
      },
      {
        key: "task_priority",
        label: "任务优先级",
        type: "select",
        options: ["日常优先", "副本优先", "采集优先"],
        default: "日常优先",
        group: "任务设置"
      }
    ]
  }
  说明：获取当前游戏项目的参数 schema
       PC 中控根据 schema 动态生成参数设置面板
       ⚠️ 虽然每个游戏独立打包，但参数面板仍可由 schema 驱动
       这样修改参数项不需要重新打包中控，只需更新服务端 schema

GET /params/values?user_id={uid}
  响应：{
    params: { "auto_heal_threshold": 60, "task_priority": "副本优先" }
  }
  说明：获取用户当前的参数设置值

POST /params/values
  请求：{ user_id, params: { "auto_heal_threshold": 70 } }
  响应：{ success: true }
  说明：保存用户修改的参数（安卓脚本下次同步时拉取新值）

═══════════════════════════════════════════════════════
update_api.py — 热更新相关
═══════════════════════════════════════════════════════

GET /update/check?client_type=pc&current_version={ver}
  响应：{
    has_update: true,
    latest_version: "1.2.0",
    release_notes: "修复了XX问题",
    download_url: "https://...",

    checksum_sha256: "abc123..."

  说明：检查 PC 端是否有新版本

GET /update/download?version={ver}&client_type=pc
  响应：文件流（二进制下载）
  说明：下载更新包
```

### 5.3 设备管理模块（core/device/）

### 5.4 数据同步模块（core/sync/）

```
职责：
  · 管理定时从云端拉取数据的节奏
  · 在工作线程中执行 HTTP 请求
  · 将数据传递给 DeviceManager 处理

核心类：SyncManager

  属性：
    · sync_interval: int             ← 同步间隔（秒），默认 10
    · is_running: bool
    · sync_thread: QThread
    · sync_worker: SyncWorker

  方法：
    · start()   → 启动同步线程
    · stop()    → 停止同步线程
    · force_sync() → 立即触发一次同步（用户手动刷新时）
    · set_interval(seconds) → 修改同步间隔

核心类：SyncWorker（运行在 QThread 中）

  工作循环：
    while self.is_running:
        try:
            # 1. 批量拉取设备数据
            data = device_api.get_device_data_batch(user_id)
            # 2. 通过信号传递数据
            self.data_received.emit(data)
        except TokenExpiredError:
            self.token_expired.emit()
            break  # 停止同步，等待重新登录
        except NetworkError as e:
            self.error_occurred.emit(str(e))
            # 不退出循环，下次重试
        
        # 3. 等待下次同步
        self.sleep(self.sync_interval)

  信号：
    · data_received(dict)
    · error_occurred(str)
    · token_expired()

同步间隔策略：
  · 默认 10 秒拉取一次
  · 网络错误时自动退避（10s → 20s → 30s → 最大 60s）
  · 恢复正常后回到默认间隔
  · 用户可在设置中调整间隔（配置持久化到 local.yaml）
```

### 参数面板的 Schema 驱动机制

```
即使每个游戏独立打包，参数面板仍然建议 schema 驱动。

原因：
  · 修改参数项（新增/删除/改默认值）不需要重新打包中控 exe
  · 只需要在服务端更新 params_schema → 中控拉取后自动重新渲染
  · 减少热更新频率 = 减少用户打扰

schema 到 UI 的自动渲染逻辑：

  schema 中 type 字段    →    自动生成的 PySide 6 组件
  ─────────────────────────────────────────────────
  "int"                  →    QSpinBox（带 min/max）
  "float"                →    QDoubleSpinBox
  "bool"                 →    QCheckBox
  "str"                  →    QLineEdit
  "select"               →    QComboBox（带 options）
  "multi_select"         →    QListWidget（多选）
  "slider"               →    QSlider（带 min/max/step）

  schema 中 group 字段   →    QGroupBox 分组显示

  如果 schema 驱动不够灵活（某些游戏需要非常特殊的参数 UI），
  可以在 game/params_widget.py 中完全自定义覆盖。
```

## 八、本地缓存策略

```
为什么需要本地缓存：
  · 减少云端请求次数（省流量、降服务器压力）
  · 程序启动时快速显示上次的数据（不用等网络响应）
  · 网络短暂断开时仍能显示最近数据

缓存内容和策略：

  | 缓存项 | 存储位置 | 过期策略 |
  |--------|---------|---------|
  | 上次登录的用户名 | local.yaml | 永不过期 |
  | Token | 加密文件 | 按 Token 有效期 |
  | 设备列表 + 最新数据 | JSON 文件 | 每次同步覆盖 |
  | 参数 schema | JSON 文件 | 每次启动拉新 |
  | 用户参数值 | JSON 文件 | 修改时立即更新 |

缓存目录：
  Windows: %APPDATA%/HiveGreatSage/GameA/cache/
  （不放在程序安装目录，避免权限问题和更新覆盖）
```

## 九、错误处理策略

```
错误分级：

  Level 1 — 致命错误（必须停止运行）
    · 登录失败（凭证错误）
    · 授权过期
    · Token 无法刷新
    处理：弹窗提示 → 返回登录页面

  Level 2 — 功能受损（部分功能不可用）
    · 云端 API 持续不可达
    · 热更新下载失败
    处理：状态栏显示错误 → 自动重试 → 使用缓存数据

  Level 3 — 暂时异常（自动恢复）
    · 单次 HTTP 请求超时
    · 单次同步失败
    处理：静默重试 → 记录日志 → 不打扰用户

  Level 4 — 信息提示
    · 有新版本可用
    · 某设备离线
    处理：状态栏或通知提示

所有错误写入 logs/ 目录，包含时间戳和堆栈信息。
```
