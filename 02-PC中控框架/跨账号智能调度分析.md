## 一、你的四个功能场景，翻译成架构语言

### 场景1：老号带新号（轮转带队）

```
你的描述：
  同区服高等级/高战力账号，带新号组队过主线
  一组新号到了指定章节 → 换下组 → 循环

翻译成系统行为：

  参与者：
    · 1 个"队长"账号（高战力，跑在设备A上）
    · N 组"新号"（每组若干个账号，分别跑在设备B/C/D...上）

  流程：
    1. 队长账号和新号第1组 组队
    2. 队长带着第1组跑主线 → 到达指定章节
    3. 队长退组 → 与新号第2组 组队
    4. 队长带着第2组跑主线 → 到达指定章节
    5. 循环直到所有组都完成
    6. 新号完成后可能进入其他任务（如日常/打工）

  这意味着系统需要：
    ✦ 账号角色定义：谁是"队长"、谁是"新号"
    ✦ 分组管理：新号怎么分组、每组几个
    ✦ 任务排队：下一组自动接上
    ✦ 进度追踪：当前第几组、到第几章了
    ✦ 游戏内操作：组队邀请、接受组队、跟随移动、对话/战斗
    ✦ 完成判定：怎么判断"到了指定章节"（OCR/UI识别）
```

### 场景2：账号性格系统（角色分工）

```
你的描述：
  模拟真人性格——部分号优先提升战力，部分号负责打工

翻译成系统行为：

  每个账号有一个"角色定位"，决定它的行为优先级：

  角色A — 战力号（主力号）：
    · 优先做提升战力的任务（副本、强化、升级）
    · 需要什么物品 → 发布"需求单"
    · 从交易所购买打工号上架的物品

  角色B — 打工号（资源号）：
    · 接收战力号的"需求单"
    · 去指定地图刷怪/采集获取目标物品
    · 获取到物品后上架交易所（智能定价）

  角色C — 新号（待培养号）：
    · 等待队长带过主线
    · 主线完成后转为战力号或打工号

  这意味着系统需要：
    ✦ 账号角色定义：战力号 / 打工号 / 新号 / 队长
    ✦ 角色可切换：今天是打工号，主线完了可能变战力号
    ✦ 行为优先级：不同角色执行不同的任务链
```

### 场景2.1：供需任务系统（核心难点）

```
你的描述：
  战力号发布需要什么物品 → 打工号接单 → 去指定地图获取 →
  上架交易所 → 价格不低于最低价也不高于最高价 → 价格不重复

翻译成系统行为：

  这是一个完整的"供需调度 + 交易所交互"系统：

  供给端（战力号/PC中控）：
    1. 战力号分析自身装备/物品缺口 → 生成"需求清单"
       （或用户在PC中控手动指定"我需要XX物品"）
    2. 需求清单上传到云端

  调度层（云端）：
    3. 云端收到需求清单
    4. 分配给空闲的打工号："你去地图X刷怪获取物品Y"
    5. 考虑因素：
       · 哪个打工号离目标地图最近/等级最合适
       · 当前有多少打工号空闲
       · 优先级排序（哪个物品最急需）

  执行端（打工号/安卓脚本）：
    6. 打工号收到任务指令 → 前往指定地图
    7. 执行刷怪/采集逻辑 → 获取目标物品
    8. 获取成功 → 前往交易所

  交易所交互（安卓脚本）：
    9. OCR 识别交易所当前价格列表
    10. 获取该物品的最低价和最高价
    11. 计算上架价格：
        · >= 最低价
        · <= 最高价
        · 不等于任何已存在的价格（避免重复）
    12. 上架物品
    13. 上报结果到云端

  ⚠️ 这里有一个重要的信息流问题：
     交易所价格是"安卓脚本在游戏内 OCR 识别"得到的
     不是从外部 API 获取的（游戏通常没有公开的交易所 API）
```

### 场景3：跨区服价格监控

```
你的描述：
  关注某些高价物品 → 在不同区服的交易所获取最低价 →
  上传到云端 → 在PC中控开辟一个页面显示

翻译成系统行为：

  采集端（各区服的安卓脚本）：
    1. 在不同区服运行的脚本，定时进入交易所
    2. OCR 识别指定物品的价格
    3. 上报数据：{ 区服, 物品名, 最低价, 时间 }
    4. 上报到云端

  云端（网络验证系统/游戏项目库）：
    5. 收集各区服价格数据 → 存入数据库
    6. 提供 API：按物品查询各区服最低价

  显示端（PC中控）：
    7. PC中控新增一个"跨服价格监控"面板
    8. 定时从云端拉取价格数据
    9. 以表格/图表形式展示：
       物品名 | 区服1最低价 | 区服2最低价 | 区服3最低价 | 更新时间

  ⚠️ 这个场景相对独立，是一个"数据采集→聚合→展示"的链路
     技术上比场景2.1简单很多
```

## 三、实现可行性评估（诚实分析）

### 场景1（老号带新号）— 可行，但游戏操作部分复杂度高

```
可行性：✅ 可以实现

容易的部分：
  · 账号角色定义和分组管理 → 在云端和PC中控都是数据管理
  · 轮转调度逻辑 → 云端的任务队列
  · 进度追踪 → 安卓脚本上报进度到云端

困难的部分：
  · 游戏内组队操作（找到组队按钮、发邀请、确认）→ 高度依赖 OCR/找图
  · "跟随"队长移动 → 不同游戏的跟随机制不同
  · 主线对话/战斗的自动化 → 游戏定制层的核心工作量
  · 判断"到了第X章" → 需要可靠的进度识别

关键依赖：
  · 组队功能需要局域网通信（或云端中转）来协调"现在组队/现在退组"
  · 如果不在同一局域网 → 需要云端消息队列来协调（增加延迟和复杂度）
```

### 场景2/2.1（角色分工 + 供需系统）— 可行，是最有价值也最复杂的

```
可行性：✅ 可以实现，但这是整个项目中复杂度最高的部分

容易的部分：
  · 账号角色定义 → PC中控设置
  · 需求清单管理 → 云端 CRUD
  · 任务分配逻辑 → 云端算法（简单版先来先服务，复杂版可以优化）

困难的部分：
  · 交易所 OCR 价格识别 → 数字识别精度要求高
  · 智能定价算法：
    - 获取当前所有挂单价格（OCR 整个价格列表）
    - 找到最低价和最高价
    - 生成一个不重复的价格
    - ⚠️ 如果价格列表很长需要翻页，OCR 工作量更大
  · "去指定地图" → 不同游戏的地图导航方式不同
  · 物品获取判定 → 怎么知道"已经拿到目标物品了"

这个场景的数据流极长：
  战力号→发布需求→云端→分配→打工号→前往地图→刷怪→获得物品
  →前往交易所→OCR价格→计算定价→上架→上报结果→云端→通知战力号

任何一个环节失败都需要异常处理和重试逻辑。
```

### 场景3（跨服价格监控）— 最容易实现

```
可行性：✅ 完全可以实现，且相对独立

容易的部分：
  · 定时进入交易所 → 安卓脚本定时任务
  · OCR 识别价格 → 和场景2.1共用交易所 OCR 模块
  · 上报数据 → 现有的数据上报通道
  · 云端存储和聚合 → 数据库新增价格表
  · PC中控新面板 → game/custom_widgets/ 里新增一个价格面板

困难的部分：
  · 价格 OCR 的准确率（数字识别比文字识别容易，但仍需调试）
  · 不同区服可能 UI 布局略有不同

建议作为最先实现的场景（验证 OCR + 数据上报链路）。
```

## 四、架构层面需要新增的概念

### 4.1 账号角色系统

```
在哪里定义：PC中控 + 云端

数据结构（云端 — 游戏项目库新增表）：

  account_role（账号角色配置表）
    id
    user_id           ← 哪个用户
    device_id         ← 哪台设备上的哪个游戏账号
    game_account_name ← 游戏内角色名
    role_type         ← 枚举：captain / power / farmer / newbie
    server_id         ← 区服标识
    level             ← 游戏等级（安卓脚本定时上报）
    combat_power      ← 战力（安卓脚本定时上报）
    current_task      ← 当前正在执行的任务
    status            ← idle / busy / offline / error

PC中控面板：
  · 用户在"账号管理"面板中为每个设备上的账号设定角色
  · 可以拖拽调整分组（场景1的新号分组）
  · 显示每个账号的当前状态和任务

安卓脚本行为：
  · 启动时从云端拉取自己的角色定位
  · 根据角色执行不同的任务链：
    if role == "captain":
        run_captain_carry_task()
    elif role == "power":
        run_power_leveling_task()
    elif role == "farmer":
        run_farming_task()
    elif role == "newbie":
        run_follow_captain_task()
```

### 4.2 任务调度引擎（云端新增）

```
这是最核心的新增模块。

在哪里运行：云端（网络验证系统中新增的服务模块）

功能：
  1. 接收"需求单"（来自 PC 中控或战力号的自动分析）
  2. 将需求单拆解为具体任务
  3. 将任务分配给合适的打工号
  4. 追踪任务进度
  5. 处理任务失败/重试

数据结构：

  task_demand（需求单表）
    id
    requester_account_id   ← 谁提的需求（战力号）
    item_name              ← 需要什么物品
    item_quantity           ← 需要多少
    fulfilled_quantity      ← 已完成多少
    priority               ← 优先级
    status                 ← pending / in_progress / fulfilled / cancelled
    created_at

  task_assignment（任务分配表）
    id
    demand_id              ← 关联需求单
    assignee_account_id    ← 分配给谁（打工号）
    target_map             ← 去哪个地图
    target_item            ← 获取什么物品
    quantity               ← 目标数量
    acquired_quantity      ← 已获取数量
    listing_status         ← 上架状态
    listing_price          ← 上架价格
    status                 ← assigned / traveling / farming / listing / completed / failed
    assigned_at
    completed_at

调度算法（初始版本可以很简单）：
  1. 按优先级排序未分配的需求
  2. 查找 status=idle 且 role=farmer 的账号
  3. 优先选择等级合适的、离目标地图近的
  4. 分配任务 → 更新账号状态为 busy
  5. 打工号上报完成 → 更新需求单进度 → 如果全部完成标记 fulfilled
```

### 4.3 交易所交互模块（安卓脚本新增）

```
在哪里运行：安卓脚本（游戏定制层）

模块：game/tasks/marketplace.lua

功能：
  · navigate_to_marketplace()    ← 导航到交易所
  · search_item(item_name)       ← 搜索物品
  · read_price_list()            ← OCR 读取价格列表
    返回：[{price: 1200, seller: "xxx"}, {price: 1350, ...}, ...]
  · calculate_listing_price(price_list)
    逻辑：
      min_price = min(所有价格)
      max_price = max(所有价格)
      existing_prices = set(所有价格)
      
      # 在 [min_price, max_price] 范围内找一个不重复的价格
      candidate = min_price
      while candidate in existing_prices:
          candidate = candidate + 1   # 或更智能的步长
      if candidate > max_price:
          candidate = max_price - 1   # 回退
      return candidate
      
  · list_item(item_name, price, quantity)  ← 上架物品
  · report_listing_result()               ← 上报结果到云端

⚠️ OCR 价格识别的精度是这个模块成败的关键
   数字的 OCR 准确率通常比文字高，但需要：
   · 训练/调试适合该游戏字体的 OCR 参数
   · 处理 K/W（千/万）等缩写单位
   · 处理翻页场景
```

### 4.4 价格监控数据链路（场景3）

```
新增数据结构（云端 — 游戏项目库）：

  market_price_snapshot（交易所价格快照表）
    id
    server_id         ← 区服
    item_name         ← 物品名
    min_price         ← 该区服最低价
    max_price         ← 该区服最高价（可选）
    sample_count      ← 本次识别的挂单数量
    collected_by      ← 哪个设备采集的
    collected_at      ← 采集时间

新增 API（云端）：

  POST /market/price-report
    请求：{ server_id, item_name, min_price, max_price, collected_at }
    说明：安卓脚本上报价格数据

  GET /market/price-overview?item_name=XX
    响应：{
      item_name: "XX",
      servers: [
        { server_id: "S1", min_price: 1200, updated_at: "..." },
        { server_id: "S2", min_price: 980, updated_at: "..." },

    }
    说明：PC中控拉取跨服价格数据

新增 PC 中控面板：
  game/custom_widgets/market_monitor_widget.py

  UI 形式：
  ┌──────────────────────────────────────────────────────┐
  │  跨服价格监控                              [刷新]     │
  ├──────────┬────────┬────────┬────────┬────────────────┤
  │  物品     │ 区服1  │ 区服2  │ 区服3  │ 最后更新       │
  ├──────────┼────────┼────────┼────────┼────────────────┤
  │  物品A    │ ¥1200  │ ¥980↓  │ ¥1100  │ 5分钟前        │

  │  物品C    │ ¥500   │ ¥520   │ ¥480↓  │ 3分钟前        │
  └──────────┴────────┴────────┴────────┴────────────────┘
  
  ↓ 箭头标记各行最低价的区服
  可选：点击某个物品展开历史价格趋势图
```

## 五、实现路线建议（不是一口气做完）

```
这四个场景按复杂度排序：

  场景3（价格监控）<<<< 场景1（带队轮转）< 场景2（角色分工）<< 场景2.1（供需系统）

建议的实现顺序：

Phase 1（MVP阶段）：
  不做这四个场景。先把基础框架跑通（登录、设备管理、参数同步、基本任务自动化）。

Phase 2：
  ✦ 先做场景3（价格监控）
    原因：
    · 它最独立，不依赖其他场景
    · 它验证了"OCR交易所价格"的技术可行性
    · 它验证了"安卓采集→云端存储→PC展示"的完整链路
    · 做完后，OCR 价格模块可以复用到场景2.1

  ✦ 再做账号角色系统的基础定义
    · PC中控上能设置账号角色
    · 安卓脚本能根据角色选择不同任务链
    · 不需要复杂调度，只是"静态配置→执行"

Phase 3：
  ✦ 做场景1（带队轮转）
    · 需要组队的局域网通信先可用
    · 需要游戏内组队操作的自动化先实现
    · 轮转调度逻辑本身不复杂

  ✦ 做场景2.1 的简化版
    · 手动在PC中控指定"打工号去地图X刷物品Y"
    · 不做自动供需匹配，先做手动分配
    · 交易所上架用已验证的 OCR + 定价模块

Phase 4（远期）：
  ✦ 完整的自动供需系统
    · 战力号自动分析缺口 → 自动发布需求
    · 云端自动分配打工号
    · 全自动闭环
```

## 六、关键风险标注

```
⚠️ OCR 精度风险（影响场景2.1和3）
  交易所价格识别如果错了，可能导致：
  · 上架价格低于市场价 → 亏损
```
