## 一、系统定位

网络验证系统是整个蜂巢·大圣平台的**中枢**，所有其他子系统都围绕它运转。

### 三类核心职责

|职责|说明|依赖方|
|---|---|---|
|**身份验证（门卫）**|校验用户是谁、有没有资格用、有哪个游戏的授权|PC 中控 / 安卓脚本|
|**数据管理（账本）**|管理员、多级代理、用户、授权关系、设备绑定的唯一权威存储|管理后台|
|**数据中转（枢纽）**|安卓设备上报运行数据 → 云端存储 → PC 中控拉取显示|PC 中控 / 安卓脚本|

### 规模要求

```
单个游戏项目：  10,000+ 台安卓设备同时运行脚本
单个 SVIP 客户：3,000+ 台设备
多游戏并行：    总设备数可能远超 10,000

并发压力估算：
  · 登录验证：脚本启动时段集中突发（可能数千台同时）
  · 心跳上报：10,000 台设备 × 每 30 秒一次 ≈ 333 次/秒持续写入
  · 数据查询：PC 中控定期拉取（频率较低，但聚合数据量大）
  · 热更新下载：版本发布时大量设备同时请求
```

## 五、数据库设计

### 5.1 数据库拆分策略

```
hive_main           所有游戏共用的用户管理数据（管理员、代理、用户、授权）
hive_game_001       游戏项目 A 的独立数据（设备数据、脚本参数、版本记录）
hive_game_002       游戏项目 B 的独立数据
...

拆分原因：
  · 每个游戏的 device_runtime 数据结构完全不同（JSONB 字段内容）
  · 游戏项目下线时可以直接归档/删除该库，不影响其他项目
  · 单游戏大客户（3000+ 设备）可以独立部署自己的 hive_game_xxx
```

### 5.2 主库（hive_main）

```sql
-- 管理员表
CREATE TABLE admin (
    id            SERIAL PRIMARY KEY,
    username      VARCHAR(64)  UNIQUE NOT NULL,
    password_hash VARCHAR(128) NOT NULL,
    status        VARCHAR(16)  DEFAULT 'active',  -- active / suspended
    created_at    TIMESTAMPTZ  DEFAULT NOW()
);

-- 代理表（自引用外键，支持无限多级）
CREATE TABLE agent (
    id                  SERIAL PRIMARY KEY,
    username            VARCHAR(64)  UNIQUE NOT NULL,
    password_hash       VARCHAR(128) NOT NULL,
    parent_agent_id     INTEGER      REFERENCES agent(id) ON DELETE RESTRICT,
    -- NULL = 由管理员直接创建的顶级代理
    level               INTEGER      NOT NULL DEFAULT 1,
    -- 由应用层在创建时自动计算：parent.level + 1
    created_by_admin_id INTEGER      REFERENCES admin(id),
    max_users           INTEGER      DEFAULT 100,
    -- 上级分配给该代理可创建的用户数上限
    status              VARCHAR(16)  DEFAULT 'active',
    created_at          TIMESTAMPTZ  DEFAULT NOW(),
    CONSTRAINT chk_agent_creator CHECK (
        (parent_agent_id IS NULL AND created_by_admin_id IS NOT NULL) OR
        (parent_agent_id IS NOT NULL)
    )
);

-- 用户表
CREATE TABLE "user" (
    id                  SERIAL PRIMARY KEY,
    username            VARCHAR(64)  UNIQUE NOT NULL,
    password_hash       VARCHAR(128) NOT NULL,
    user_level          VARCHAR(16)  NOT NULL
                        CHECK (user_level IN ('trial','normal','vip','svip','tester')),
    created_by_agent_id INTEGER      REFERENCES agent(id),
    created_by_admin    BOOLEAN      DEFAULT FALSE,
    -- tester 级别时 created_by_admin 必须为 true（应用层强制）
    status              VARCHAR(16)  DEFAULT 'active',  -- active / suspended / expired
    created_at          TIMESTAMPTZ  DEFAULT NOW(),
    expired_at          TIMESTAMPTZ,
    CONSTRAINT chk_tester_creator CHECK (
        user_level != 'tester' OR created_by_admin = TRUE
    )
);

-- 游戏项目注册表
CREATE TABLE game_project (
    id           SERIAL PRIMARY KEY,
    code_name    VARCHAR(64)  UNIQUE NOT NULL,   -- 如 "game_001"（同时是数据库名后缀）
    display_name VARCHAR(128) NOT NULL,
    db_name      VARCHAR(64)  UNIQUE NOT NULL,   -- 对应的 PostgreSQL 库名
    is_active    BOOLEAN      DEFAULT TRUE,
    created_at   TIMESTAMPTZ  DEFAULT NOW()
);

-- 授权表（用户 × 游戏项目）
CREATE TABLE authorization (
    id              SERIAL PRIMARY KEY,
    user_id         INTEGER     NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    game_project_id INTEGER     NOT NULL REFERENCES game_project(id),
    valid_from      TIMESTAMPTZ DEFAULT NOW(),
    valid_until     TIMESTAMPTZ,                 -- NULL = 永久有效
    status          VARCHAR(16) DEFAULT 'active',
    UNIQUE (user_id, game_project_id)            -- 一用户对一游戏只有一条记录
);

-- 设备绑定表
CREATE TABLE device_binding (
    id                 SERIAL PRIMARY KEY,
    user_id            INTEGER      NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    device_fingerprint VARCHAR(256) NOT NULL,
    bound_at           TIMESTAMPTZ  DEFAULT NOW(),
    last_seen_at       TIMESTAMPTZ,
    status             VARCHAR(16)  DEFAULT 'active',
    UNIQUE (user_id, device_fingerprint)
);

-- 登录日志（只写不删，用于审计）
CREATE TABLE login_log (
    id                 BIGSERIAL   PRIMARY KEY,
    user_id            INTEGER     REFERENCES "user"(id),
    device_fingerprint VARCHAR(256),
    ip                 VARCHAR(64),
    client_type        VARCHAR(16),  -- 'pc' / 'android'
    game_project_id    INTEGER,
    result             VARCHAR(32),  -- 'success' / 'fail_auth' / 'fail_device' / 'fail_expired' 等
    login_at           TIMESTAMPTZ  DEFAULT NOW()
);
CREATE INDEX idx_login_log_user ON login_log (user_id, login_at DESC);
```

## 六、认证机制

### 6.1 登录验证流程（10 步检查链）

```
POST /api/auth/login
  请求体：{ username, password, device_fingerprint, client_type, game_project_id }

服务端按序执行：
  Step 1   username 在数据库中存在？
  Step 2   password 与 password_hash 匹配？
  Step 3   user.status == 'active'？
  Step 4   user.expired_at 未到期（或为 NULL）？
  Step 5   authorization 表中存在 (user_id, game_project_id) 的有效授权？
  Step 6   authorization.status == 'active'？
  Step 7   authorization.valid_until 未到期（或为 NULL）？
  Step 8   device_fingerprint 检查：
             (a) 在 device_binding 中存在 → 更新 last_seen_at，通过
             (b) 不存在 → 查询该用户已绑定设备数 < 上限（按 user_level 决定上限）
                           → 未超上限：插入新绑定，通过
                           → 超出上限：拒绝，返回错误码 DEVICE_LIMIT_EXCEEDED
  Step 9   所有检查通过 → 写 login_log（result='success'）
  Step 10  签发 Access Token + Refresh Token，返回

  任一步骤失败 → 写 login_log（result='fail_xxx'）→ 返回对应错误码
```

### 6.2 JWT 设计

```python
# Access Token（短期，每次请求携带）
payload = {
    "sub": user_id,             # 用户 ID
    "level": "vip",             # 用户级别
    "game": "game_001",         # 当前登录的游戏项目
    "type": "access",
    "exp": now + timedelta(hours=2)   # 2 小时有效期
}

# Refresh Token（长期，只用于换新 Access Token）
payload = {
    "sub": user_id,
    "jti": uuid4(),             # 唯一 ID，存入 Redis，支持单独吊销
    "type": "refresh",
    "exp": now + timedelta(days=30)   # 30 天有效期
}
```

### 6.3 强制下线机制

```
触发场景：管理员/代理封禁用户、用户账号过期

实现：
  1. 将该用户所有在用 Refresh Token 的 jti 写入 Redis 黑名单
     Key：token:blacklist:{jti}，TTL = token 剩余有效时长
  2. 客户端用 Refresh Token 换 Access Token 时，先查黑名单
     → 命中黑名单：拒绝，强制客户端重新登录
  3. Access Token 自然过期（2 小时内失效，无需主动吊销）
```

## 七、Redis 使用策略

### 7.1 心跳缓冲（核心设计，支撑 10,000+ 设备）

```
问题：
  10,000 台设备 × 每 30 秒上报一次 ≈ 333 次/秒持续写入
  device_runtime 只需要保留每台设备的最新值
  直接写 PostgreSQL 压力过大

方案：Redis 作写缓冲层

上报时（FastAPI 处理）：
  1. HSET device:runtime:{game_id}:{user_id}:{device_fp} data <json_payload>
  2. EXPIRE device:runtime:... 120   ← 2 分钟 TTL，超时视为设备离线
  3. 立即返回 200 OK（不等待落库）

批量落库（Celery 定时任务，每 30 秒执行）：
  4. SCAN device:runtime:{game_id}:* 获取所有 key
  5. 批量 UPSERT 到 PostgreSQL device_runtime 表
     ON CONFLICT (user_id, device_fingerprint) DO UPDATE SET ...

PC 中控拉取设备列表时（每 10 秒轮询）：
  优先查 Redis（最新实时数据）
  Redis 无数据时回落到 PostgreSQL（设备可能刚启动或已离线）
```

### 7.2 Redis Key 规划

|Key 格式|用途|TTL|
|---|---|---|
|`device:runtime:{game_id}:{user_id}:{device_fp}`|设备心跳缓冲|120s|
|`token:blacklist:{jti}`|Token 黑名单|token 剩余有效期|
|`ratelimit:{ip}:{endpoint}`|限流计数（滑动窗口）|60s|
|`update:latest:{game_id}:{client_type}`|最新版本信息缓存|300s|

## 八、多级代理权限查询

多级代理使用 PostgreSQL 递归 CTE 实现，代理层级无硬性上限。

```sql
-- 查询代理 ID=5 能管理的所有下级代理（含自身）
WITH RECURSIVE agent_tree AS (
    SELECT id, username, parent_agent_id, level
    FROM agent
    WHERE id = 5

    UNION ALL

    SELECT a.id, a.username, a.parent_agent_id, a.level
    FROM agent a
    INNER JOIN agent_tree at ON a.parent_agent_id = at.id
)
SELECT * FROM agent_tree;

-- 查询代理 ID=5 能看到的所有用户（自己 + 所有下级代理创建的）
WITH RECURSIVE agent_tree AS (
    SELECT id FROM agent WHERE id = 5
    UNION ALL
    SELECT a.id FROM agent a
    INNER JOIN agent_tree at ON a.parent_agent_id = at.id
)
SELECT u.*
FROM "user" u
WHERE u.created_by_agent_id IN (SELECT id FROM agent_tree);
```

**权限规则（应用层强制）：**

- 每个代理只能看到：自己直属用户 + 所有下级代理 + 下级代理的用户
- 代理不能看到：同级代理的数据、上级代理的数据
- 管理员能看到全部
- 代理能操作的范围由上级设定（最多创建多少用户、可创建什么级别）
- `tester` 级别用户只有管理员能创建

## 十、热更新机制

### 10.1 文件目录结构

```
存储根目录
  本地模式：/var/www/hive-updates/
  对象存储：s3://bucket/hive-updates/

hive-updates/
├── game_001/
│   ├── pc/
│   │   ├── latest.json              ← 客户端轮询此文件
│   │   └── packages/
│   │       ├── v1.0.0/update.zip
│   │       └── v1.0.1/update.zip
│   └── android/
│       ├── latest.json
│       └── packages/
│           └── v1.0.1.lrj           ← 懒人精灵热更新包
└── game_002/
    └── ...
```

### 10.2 latest.json 格式

```json
{
  "version": "1.0.1",
  "min_compatible": "1.0.0",
  "checksum_sha256": "abc123...",
  "force_update": false,
  "package_size_bytes": 2048000,
  "release_notes": "修复找图精度问题",
  "published_at": "2026-04-13T12:00:00Z"
}
```

### 10.3 安卓端更新流程

```
脚本启动
  → 登录验证通过
  → GET /api/update/check?game=game_001&type=android&current=1.0.0
  → 服务端比对版本号，返回 { need_update: true, ...latest.json 内容 }
  → 客户端请求签名下载 URL
  → 下载 .lrj 包
  → 校验 checksum_sha256
  → 调用 installLRPkg 安装
  → 重载脚本
```

### 10.4 下载安全

下载 URL 由服务端动态签名生成，有效期 10 分钟，防止链接被传播滥用。对象存储模式下直接使用 S3 Presigned URL；本地模式下服务端生成带 token 的临时 URL。

## 十一、部署方案

### 11.1 两种部署模式

**模式 A — VPS 自建（家庭宽带场景）**

```
[家庭 VPS / 本地服务器]
  └── nginx + uvicorn + PostgreSQL + Redis（本机运行）
         ↕ FRP / 内网穿透
[云端中转服务器（最低配 1C1G）]
  └── 只做流量转发，不跑业务
         ↑
[公网域名]   ← PC 中控 / 安卓脚本 连接
```

**模式 B — 云服务器（大客户独立部署）**

```
[云服务器 2-4C 4-8G]
  └── nginx + uvicorn + PostgreSQL + Redis（单机部署）
  └── 对象存储（云厂商 OSS / 自建 MinIO）
         ↑
[公网域名]   ← 该客户的 PC 中控 / 安卓脚本 连接
```

两种模式的代码完全相同，仅通过环境变量切换配置。

### 11.2 环境变量配置

```bash
# .env — 通用配置（两种模式共用）
SECRET_KEY=<随机生成的强密钥，至少 32 字节>
ACCESS_TOKEN_EXPIRE_HOURS=2
REFRESH_TOKEN_EXPIRE_DAYS=30

DATABASE_URL=postgresql+asyncpg://user:pass@localhost:5432/hive_main
REDIS_URL=redis://localhost:6379/0

DOMAIN=your-domain.com
ENVIRONMENT=production   # development / production

# --- 文件存储（选其一）---

# 模式 A：本地文件系统
STORAGE_MODE=local
STORAGE_LOCAL_ROOT=/var/www/hive-updates

# 模式 B：S3 / MinIO
STORAGE_MODE=s3
S3_ENDPOINT=https://your-minio-or-s3-endpoint
S3_BUCKET=hive-updates
S3_ACCESS_KEY=...
S3_SECRET_KEY=...
```

### 11.3 nginx 核心配置

```nginx
# 热更新静态文件：nginx 直接服务（不过 FastAPI，性能更好）
location /updates/ {
    alias /var/www/hive-updates/;
    expires 1h;
    add_header Cache-Control "public";
}

# FastAPI 应用（通过 unix socket，比 TCP 快）
location /api/ {
    proxy_pass http://unix:/var/run/hive-verify.sock;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}

# Vue 3 管理后台（SPA）
location /admin/ {
    root /var/www/hive-admin/dist;
    try_files $uri /admin/index.html;
}
```

### D2 — 心跳与刷新间隔

```
安卓脚本上报心跳：每 30 秒一次  →  写入 Redis（TTL=120s）
Celery 批量落库：  每 30 秒执行  →  UPSERT 到 PostgreSQL
PC 中控刷新显示：  每 10 秒一次  →  从 Redis 读取（回落 PostgreSQL）

推算：10,000 台设备 × 1次/30s ≈ 333 次/秒写 Redis（内存操作，微秒级）
```

### D3 — Vue UI 组件库

**Element Plus**（锁定）

理由：中文文档完整、社区活跃、Table/Form/Tree 组件覆盖管理后台全部需求、与 Vue 3 原生兼容。

### D4 — 文件存储选型

|部署场景|存储方案|配置方式|
|---|---|---|
|VPS 自建|本地文件系统 + nginx 直接服务|`STORAGE_MODE=local`|
|云服务器（大客户）|自建 MinIO 或云厂商 OSS|`STORAGE_MODE=s3` + S3 环境变量|

代码通过 `app/core/storage/base.py` 抽象接口统一调用，切换存储模式只改环境变量，业务代码不变。

### D5 — 限流策略

|接口|规则|Key 设计|超限响应|
|---|---|---|---|
|`POST /api/auth/login`|同 IP 每分钟 ≤ 10 次|`ratelimit:login:{ip}` TTL=60s|HTTP 429|
|`POST /api/device/heartbeat`|同 Token 每分钟 ≤ 4 次|`ratelimit:heartbeat:{user_id}` TTL=60s|HTTP 429|
|其他接口|暂不限流（MVP 阶段）|—|—|

实现位置：`app/core/rate_limiter.py`，基于 Redis INCR + EXPIRE 的滑动窗口，作为 FastAPI 中间件挂载。

### D6 — 日志与监控

|组件|用途|配置|
|---|---|---|
|**loguru**|结构化日志，写本地文件，按天轮转|生产环境输出 JSON 格式，保留 30 天|
|**Sentry**|异常捕获与告警，实时通知 500 错误|FastAPI 通过 `sentry-sdk` 集成，DSN 写入环境变量|

Phase 2 扩展预留：日志文件格式为 JSON，后期可直接接 Filebeat → ELK，不需要改代码。
