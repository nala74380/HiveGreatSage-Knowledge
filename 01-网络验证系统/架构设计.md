## 一、系统定位

网络验证系统是整个蜂巢·大圣平台的**中枢**，所有其他子系统都围绕它运转。

### 三类核心职责

|职责|说明|依赖方|
|---|---|---|
|**身份验证（门卫）**|校验用户是谁、有没有资格用、有哪个游戏的授权|PC 中控 / 安卓脚本|
|**数据管理（账本）**|管理员、多级代理、用户、授权关系、设备绑定的唯一权威存储|管理后台|
|**数据中转（枢纽）**|安卓设备上报运行数据 → 云端存储 → PC 中控拉取显示|PC 中控 / 安卓脚本|

### 规模要求

```
单个游戏项目：  10,000+ 台安卓设备同时运行脚本
单个 SVIP 客户：3,000+ 台设备
多游戏并行：    总设备数可能远超 10,000

并发压力估算：
  · 登录验证：脚本启动时段集中突发（可能数千台同时）
  · 心跳上报：10,000 台设备 × 每 30 秒一次 ≈ 333 次/秒持续写入
  · 数据查询：PC 中控定期拉取（频率较低，但聚合数据量大）
  · 热更新下载：版本发布时大量设备同时请求
```


## 五、数据库设计

> 完整字段规范见 [[数据库设计]]。本节保留 SQL 建表语句作为实现参考，两份文档保持同步。

### 5.1 数据库拆分策略

```
hive_platform       所有游戏共用的用户管理数据（管理员、代理、用户、授权）
hive_game_001       游戏项目 A 的独立数据（设备数据、脚本参数、版本记录）
hive_game_002       游戏项目 B 的独立数据
...

拆分原因：
  · 每个游戏的 device_runtime 数据结构完全不同（JSONB 字段内容）
  · 游戏项目下线时可以直接归档/删除该库，不影响其他项目
  · 单游戏大客户（3000+ 设备）可以独立部署自己的 hive_game_xxx
```

### 5.2 主库（hive_platform）

```sql
-- 管理员表
CREATE TABLE admin (
    id            SERIAL       PRIMARY KEY,
    username      VARCHAR(64)  UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    email         VARCHAR(128),                    -- 可选，用于找回密码
    status        VARCHAR(16)  DEFAULT 'active',   -- active / suspended
    created_at    TIMESTAMPTZ  DEFAULT NOW(),
    updated_at    TIMESTAMPTZ  DEFAULT NOW()
);

-- 代理表（自引用外键，支持无限多级）
CREATE TABLE agent (
    id                  SERIAL       PRIMARY KEY,
    username            VARCHAR(64)  UNIQUE NOT NULL,
    password_hash       VARCHAR(255) NOT NULL,
    parent_agent_id     INTEGER      REFERENCES agent(id) ON DELETE RESTRICT,
    -- NULL = 由管理员直接创建的顶级代理
    level               INTEGER      NOT NULL DEFAULT 1,
    -- 由应用层在创建时自动计算：parent.level + 1
    created_by_admin_id INTEGER      REFERENCES admin(id),
    max_users           INTEGER      DEFAULT 0,    -- 0 = 无限制
    commission_rate     DECIMAL(5,2),              -- 佣金比例（百分比，可留空）
    status              VARCHAR(16)  DEFAULT 'active',   -- active / suspended
    created_at          TIMESTAMPTZ  DEFAULT NOW(),
    updated_at          TIMESTAMPTZ  DEFAULT NOW(),
    CONSTRAINT chk_agent_creator CHECK (
        (parent_agent_id IS NULL AND created_by_admin_id IS NOT NULL) OR
        (parent_agent_id IS NOT NULL)
    )
);

-- 用户表
CREATE TABLE "user" (
    id                  SERIAL       PRIMARY KEY,
    username            VARCHAR(64)  UNIQUE NOT NULL,
    password_hash       VARCHAR(255) NOT NULL,
    user_level          VARCHAR(20)  NOT NULL
                        CHECK (user_level IN ('trial','normal','vip','svip','tester')),
    created_by_agent_id INTEGER      REFERENCES agent(id),
    created_by_admin_id INTEGER      REFERENCES admin(id),
    -- tester 级别时 created_by_admin_id 必须非空（应用层强制）
    status              VARCHAR(16)  DEFAULT 'active',   -- active / suspended / expired
    created_at          TIMESTAMPTZ  DEFAULT NOW(),
    updated_at          TIMESTAMPTZ  DEFAULT NOW(),
    expired_at          TIMESTAMPTZ,
    CONSTRAINT chk_user_creator CHECK (
        (created_by_agent_id IS NULL) != (created_by_admin_id IS NULL)
    ),
    CONSTRAINT chk_tester_creator CHECK (
        user_level != 'tester' OR created_by_admin_id IS NOT NULL
    )
);

-- 游戏项目注册表
CREATE TABLE game_project (
    id           SERIAL       PRIMARY KEY,
    project_uuid UUID         UNIQUE NOT NULL DEFAULT gen_random_uuid(),
    -- 客户端（PC中控/安卓脚本）登录时传 project_uuid；内部外键引用用 INTEGER id
    code_name    VARCHAR(64)  UNIQUE NOT NULL,   -- 如 "game_001"（同时是数据库名后缀）
    display_name VARCHAR(128) NOT NULL,
    db_name      VARCHAR(64)  UNIQUE NOT NULL,   -- 对应的 PostgreSQL 库名
    is_active    BOOLEAN      DEFAULT TRUE,
    created_at   TIMESTAMPTZ  DEFAULT NOW()
);

-- 授权表（用户 × 游戏项目）
CREATE TABLE authorization (
    id              SERIAL      PRIMARY KEY,
    user_id         INTEGER     NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    game_project_id INTEGER     NOT NULL REFERENCES game_project(id),
    valid_from      TIMESTAMPTZ DEFAULT NOW(),
    valid_until     TIMESTAMPTZ,                 -- NULL = 永久有效
    status          VARCHAR(16) DEFAULT 'active',
    UNIQUE (user_id, game_project_id)            -- 一用户对一游戏只有一条记录
);

-- 设备绑定表
CREATE TABLE device_binding (
    id                 SERIAL       PRIMARY KEY,
    user_id            INTEGER      NOT NULL REFERENCES "user"(id) ON DELETE CASCADE,
    device_fingerprint VARCHAR(255) NOT NULL,
    bound_at           TIMESTAMPTZ  DEFAULT NOW(),
    last_seen_at       TIMESTAMPTZ,
    status             VARCHAR(16)  DEFAULT 'active',  -- active / unbound
    UNIQUE (user_id, device_fingerprint)
);
CREATE UNIQUE INDEX idx_device_user_fingerprint
    ON device_binding(user_id, device_fingerprint) WHERE status = 'active';

-- 登录日志（只写不删，用于审计）
CREATE TABLE login_log (
    id                 BIGSERIAL    PRIMARY KEY,
    user_id            INTEGER      REFERENCES "user"(id),
    device_fingerprint VARCHAR(255),
    ip_address         INET,                     -- 使用 PostgreSQL 原生 INET 类型
    client_type        VARCHAR(20),              -- 'pc' / 'android'
    game_project_id    INTEGER,                  -- 登录时关联的游戏项目（审计用）
    success            BOOLEAN      NOT NULL,
    fail_reason        VARCHAR(32),              -- NULL=成功；'fail_auth'/'fail_device_limit'/'fail_expired' 等
    login_at           TIMESTAMPTZ  DEFAULT NOW()
);
CREATE INDEX idx_login_log_user ON login_log (user_id, login_at DESC);
CREATE INDEX idx_login_log_ip   ON login_log (ip_address);
```

### 5.3 游戏项目库（hive_game_xxx）

```sql
-- 项目配置（KV 结构，灵活存各种全局配置）
CREATE TABLE project_config (
    config_key   VARCHAR(128) PRIMARY KEY,
    config_value TEXT,
    description  VARCHAR(255),
    updated_at   TIMESTAMPTZ  DEFAULT NOW()
);

-- 设备实时运行数据（安卓端上报 → PC 端拉取）
CREATE TABLE device_runtime (
    device_id          VARCHAR(128) PRIMARY KEY,  -- 对应 device_binding.device_fingerprint
    user_id            INTEGER      NOT NULL,      -- 对应 hive_platform.user.id（跨库，应用层维护）
    status             VARCHAR(20),               -- 'running' / 'idle' / 'error' / 'offline'
    last_seen          TIMESTAMPTZ  NOT NULL,
    game_data          JSONB,                      -- 游戏自定义字段，格式由各游戏定义
    updated_at         TIMESTAMPTZ  DEFAULT NOW()
);
CREATE INDEX idx_runtime_user      ON device_runtime (user_id);
```

## 六、认证机制

### 6.1 登录验证流程（10 步检查链）

登录接口接收[[接入契约]]中定义的请求体（含 `project_uuid`、`hardware_serial` 等字段），服务端按序执行：

```
POST /api/auth/login

服务端按序执行：
  Step 1   username 在数据库中存在？
  Step 2   password 与 password_hash 匹配？
  Step 3   user.status == 'active'？
  Step 4   user.expired_at 未到期（或为 NULL）？
  Step 5   通过 project_uuid 找到 game_project 记录？
  Step 6   authorization 表中存在 (user_id, game_project.id) 的有效授权？
  Step 7   authorization.status == 'active' 且 valid_until 未到期？
  Step 8   device_fingerprint（= hardware_serial）检查：
             (a) 在 device_binding 中存在且 status='active' → 更新 last_seen_at，通过
             (b) 不存在 → 查询该用户已绑定设备数 < 上限（按 user_level 决定上限）
                           → 未超上限：插入新绑定，通过
                           → 超出上限：拒绝，fail_reason='fail_device_limit'
  Step 9   所有检查通过 → 写 login_log（success=TRUE）
  Step 10  签发 Access Token + Refresh Token，返回

  任一步骤失败 → 写 login_log（success=FALSE，fail_reason='fail_auth'/'fail_expired' 等）→ 返回对应错误码
```

### 6.2 JWT 设计

```python
# Access Token（短期，每次请求携带）
payload = {
    "sub": user_id,             # 用户 ID
    "level": "vip",             # 用户级别
    "project": "game_001",      # 当前登录的游戏项目（code_name）
    "jti": str(uuid4()),        # 唯一 ID，用于黑名单校验
    "iat": now,
    "exp": now + timedelta(minutes=15)   # 15 分钟有效期
}

# Refresh Token（长期，只用于换新 Access Token）
payload = {
    "sub": user_id,
    "jti": str(uuid4()),        # 唯一 ID，存入 Redis，支持单独吊销
    "exp": now + timedelta(days=7)    # 7 天有效期（可配置，记住我延长至 30 天）
}
```

### 6.3 强制下线机制

```
触发场景：管理员/代理封禁用户、用户账号过期

实现：
  1. 将该用户所有在用 Refresh Token 的 jti 写入 Redis 黑名单
     Key：token:blacklist:{jti}，TTL = token 剩余有效时长
  2. 客户端用 Refresh Token 换 Access Token 时，先查黑名单
     → 命中黑名单：拒绝，强制客户端重新登录
  3. Access Token 自然过期（15 分钟内失效，无需主动吊销）
```

## 七、Redis 使用策略

### 7.1 心跳缓冲（核心设计，支撑 10,000+ 设备）

```
问题：
  10,000 台设备 × 每 30 秒上报一次 ≈ 333 次/秒持续写入
  device_runtime 只需要保留每台设备的最新值
  直接写 PostgreSQL 压力过大

方案：Redis 作写缓冲层

上报时（FastAPI 处理）：
  1. HSET device:runtime:{game_id}:{user_id}:{device_fp} data <json_payload>
  2. EXPIRE device:runtime:... 120   ← 2 分钟 TTL，超时视为设备离线
  3. 立即返回 200 OK（不等待落库）

批量落库（Celery 定时任务，每 30 秒执行）：
  4. SCAN device:runtime:{game_id}:* 获取所有 key
  5. 批量 UPSERT 到 PostgreSQL device_runtime 表
     ON CONFLICT (device_id) DO UPDATE SET ...

PC 中控拉取设备列表时（每 10 秒轮询）：
  优先查 Redis（最新实时数据）
  Redis 无数据时回落到 PostgreSQL（设备可能刚启动或已离线）
```

### 7.2 Redis Key 规划

|Key 格式|用途|TTL|
|---|---|---|
|`device:runtime:{game_id}:{user_id}:{device_fp}`|设备心跳缓冲|120s|
|`token:blacklist:{jti}`|Token 黑名单|token 剩余有效期|
|`refresh:{user_id}:{jti}`|Refresh Token 存储|7天（或30天）|
|`ratelimit:{ip}:{endpoint}`|限流计数（滑动窗口）|60s|
|`update:latest:{game_id}:{client_type}`|最新版本信息缓存|300s|

## 八、多级代理权限查询

多级代理使用 PostgreSQL 递归 CTE 实现，代理层级无硬性上限。

```sql
-- 查询代理 ID=5 能管理的所有下级代理（含自身）
WITH RECURSIVE agent_tree AS (
    SELECT id, username, parent_agent_id, level
    FROM agent
    WHERE id = 5

    UNION ALL

    SELECT a.id, a.username, a.parent_agent_id, a.level
    FROM agent a
    INNER JOIN agent_tree at ON a.parent_agent_id = at.id
)
SELECT * FROM agent_tree;

-- 查询代理 ID=5 能看到的所有用户
WITH RECURSIVE agent_tree AS (
    SELECT id FROM agent WHERE id = 5
    UNION ALL
    SELECT a.id FROM agent a
    INNER JOIN agent_tree at ON a.parent_agent_id = at.id
)
SELECT u.*
FROM "user" u
WHERE u.created_by_agent_id IN (SELECT id FROM agent_tree);
```

**权限规则（应用层强制）：**

- 每个代理只能看到：自己直属用户 + 所有下级代理 + 下级代理的用户
- 代理不能看到：同级代理的数据、上级代理的数据
- 管理员能看到全部
- `tester` 级别用户只有管理员能创建
