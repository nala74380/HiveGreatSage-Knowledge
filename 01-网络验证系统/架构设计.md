## 三、Verify 在平台中的定位

### 3.1 Verify 是什么

Verify 是蜂巢·大圣平台的 **云端中枢系统**。

它不是普通登录接口，而是整个平台的中心控制面，承担：

```text
1. 用户身份认证。
2. 代理体系管理。
3. 游戏项目管理。
4. 用户项目授权校验。
5. 设备绑定与在线状态管理。
6. 安卓脚本心跳数据接收。
7. PC 中控设备数据拉取。
8. PC 中控脚本参数下发。
9. 安卓脚本参数拉取。
10. PC / 安卓热更新管理。
11. 管理员后台 API。
12. 代理后台 API。
13. 点数计费与流水。
14. 项目准入策略。
15. 多游戏数据隔离。
```

### 3.2 Verify 不是什么

Verify 不负责以下事情：

```text
1. 不直接执行游戏自动化。
2. 不直接控制安卓屏幕。
3. 不直接做 OCR、找图、找色、YOLO。
4. 不直接执行 PC 本地 ADB 操作。
5. 不直接内置某一款游戏的任务逻辑。
6. 不代替 PCControl 的 GUI。
7. 不代替 AndroidScript 的任务状态机。
8. 不代替官网监控系统。
9. 不代替 ADB+CV 素材提取工具。
```

### 3.3 Verify 的核心价值

Verify 的长期价值不是“登录验证”，而是：

```text
1. 统一身份。
2. 统一授权。
3. 统一代理商业体系。
4. 统一设备运行数据。
5. 统一参数与热更新。
6. 支撑多游戏独立发布。
7. 支撑多客户独立部署。
8. 支撑后续跨账号调度。
```

## 五、部署视角架构

### 5.1 开发环境

```text
Windows / 本地开发机
├── PyCharm
├── Python 3.11+
├── FastAPI / Uvicorn
├── PostgreSQL
├── Redis
├── Celery Worker
├── Celery Beat
└── Vue 前端开发服务
```

开发启动方式：

```bash
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

Celery 开发环境：

```bash
celery -A app.core.celery_app worker --pool=solo --loglevel=info
celery -A app.core.celery_app beat --loglevel=info
```

说明：

```text
Windows 开发环境使用 --pool=solo，是因为 Windows 不支持 Linux fork 模式。
生产环境不应使用 --pool=solo。
```

### 5.2 生产环境

```text
Linux VPS / 云服务器
├── Nginx
├── Uvicorn + Gunicorn
├── PostgreSQL
├── Redis
├── Celery Worker
├── Celery Beat
├── 前端 dist 静态资源
├── 本地文件存储或对象存储
└── systemd 服务
```

生产环境原则：

```text
1. DEBUG 必须为 False。
2. SECRET_KEY 必须替换为强随机密钥。
3. docs_url / redoc_url 生产环境默认关闭。
4. 数据库密码、Redis 密码、S3 凭证必须从环境变量读取。
5. Sentry 可选开启。
6. 日志必须轮转。
7. 热更新包必须可备份、可回滚。
```

### 5.3 双模式部署

Verify 必须兼容两种部署模式：

```text
模式 A：家庭 VPS + FRP / 内网穿透
模式 B：云服务商独立部署
```

两种模式的代码必须一致，差异只能来自配置：

```text
DATABASE_MAIN_URL
REDIS_URL
SECRET_KEY
ALLOWED_ORIGINS
STORAGE_BACKEND
LOCAL_STORAGE_ROOT
S3_ENDPOINT
S3_BUCKET
S3_ACCESS_KEY
S3_SECRET_KEY
```

禁止：

```text
1. 在源码中写死域名。
2. 在源码中写死数据库地址。
3. 在源码中写死云存储密钥。
4. 在源码中写死客户专属配置。
```

## 六、目录结构与职责边界

### 6.1 后端目录结构

```text
HiveGreatSage-Verify/
├── app/
│   ├── main.py                 FastAPI 应用入口
│   ├── config.py               配置读取与 Settings
│   ├── database.py             主库与游戏库连接管理
│   ├── models/                 ORM 模型
│   ├── schemas/                Pydantic 输入输出模型
│   ├── routers/                API 路由层
│   ├── services/               业务服务层
│   ├── core/                   基础设施：安全、Redis、Celery、存储等
│   ├── dependencies/           FastAPI 依赖注入
│   └── tasks/                  Celery 任务
│
├── frontend/                   Vue 3 管理后台
├── migrations/                 Alembic 数据库迁移
├── scripts/                    初始化、部署、清理脚本
├── tests/                      测试
├── deploy/                     nginx / systemd 等部署配置
└── docs/                       仓库内文档或历史报告
```

### 6.2 app/main.py 职责

`app/main.py` 是 Verify 的应用入口，职责是：

```text
1. 初始化日志。
2. 初始化 Sentry。
3. 执行生产环境安全检查。
4. 注册生命周期 lifespan。
5. 创建 FastAPI app。
6. 注册 CORS。
7. 注册所有 router。
8. 提供 /health 健康检查。
9. 应用关闭时释放数据库连接池。
```

禁止在 `main.py` 中堆业务逻辑。

不应放入：

```text
1. 登录验证细节。
2. 数据库查询细节。
3. 点数扣费逻辑。
4. 热更新业务逻辑。
5. 设备心跳业务逻辑。
```

### 6.3 routers/ 职责

`routers/` 是 API 层。

它只负责：

```text
1. 定义 HTTP 方法与路径。
2. 接收请求参数。
3. 注入当前用户 / 当前管理员 / 当前代理。
4. 调用 service。
5. 返回 response schema。
6. 把业务异常转成 HTTPException。
```

它不应负责：

```text
1. 复杂业务规则。
2. 多表事务决策。
3. 点数财务计算。
4. Redis Key 拼接细节。
5. 文件存储底层实现。
```

### 6.4 services/ 职责

`services/` 是业务规则层。

它负责：

```text
1. 登录验证规则。
2. 用户授权规则。
3. 代理层级规则。
4. 点数扣费与返点规则。
5. 项目准入规则。
6. 设备绑定与心跳规则。
7. 参数保存与校验规则。
8. 热更新版本管理规则。
```

长期要求：

```text
1. 业务规则必须优先沉淀在 service。
2. router 不允许无限变厚。
3. 重复查询逻辑后续可下沉到 repository。
4. service 中涉及事务的逻辑必须清晰标注。
```

### 6.5 models/ 职责

`models/` 是 ORM 层。

它负责：

```text
1. 定义数据库表。
2. 定义字段类型。
3. 定义外键关系。
4. 定义索引与唯一约束。
5. 定义枚举字段。
6. 定义关系 relationship。
```

它不负责：

```text
1. 业务规则。
2. HTTP 输入校验。
3. 返回给前端的字段裁剪。
4. 权限判断。
```

### 6.6 schemas/ 职责

`schemas/` 是 Pydantic 数据契约层。

它负责：

```text
1. 请求体校验。
2. 响应体结构。
3. 字段类型约束。
4. 默认值。
5. API 文档展示。
```

长期要求：

```text
1. 对外 API 必须返回 schema，不直接返回 ORM 对象。
2. PCControl / AndroidScript 依赖的字段必须稳定。
3. 管理后台字段可扩展，但不能破坏客户端契约。
4. 涉及金额、点数、授权时间的字段必须明确单位。
```

### 6.7 core/ 职责

`core/` 是基础设施层。

它负责：

```text
1. JWT / 密码哈希 / Token 安全。
2. Redis 客户端。
3. Celery 应用。
4. 文件存储抽象。
5. 日志或底层工具。
6. 跨模块通用能力。
```

禁止：

```text

2. 把具体代理业务写进 core。
3. 把某一款游戏逻辑写进 core。
```

### 6.8 tasks/ 职责

`tasks/` 是异步任务层。

当前核心任务：

```text
flush_device_heartbeats
```

职责：

```text
1. 从 Redis 读取心跳缓冲。
2. 聚合设备心跳数据。
3. 批量写入游戏库 device_runtime。
4. 降低接口请求对数据库的直接写入压力。
```

后续可扩展：

```text
1. 清理过期 Refresh Token。
2. 清理过期在线设备缓存。
3. 统计日报。
4. 热更新包归档清理。
5. 对账任务。

主库特点：

游戏库特点：

```text

3. 降低单表膨胀风险。

```text
1. 主库固定连接。

```text
1. 平台级身份与授权永远进主库。

4. 设备运行状态在游戏库。

## 八、Redis 架构

### 8.1 Redis 的定位

Redis 是 Verify 的缓存层和任务中转层。

它不是主数据源。

Redis 用于：

```text
1. Refresh Token 存储。
2. Access Token 黑名单。
3. 设备心跳实时缓存。
4. 心跳批量落库缓冲。
5. 热更新 active 版本缓存。
6. Celery Broker。
7. Celery Result Backend。
```

### 8.2 Redis 使用原则

```text
1. Redis 中的数据必须可以从数据库重建，除非明确设计为短期状态。
2. Redis Key 必须统一命名。
3. Redis TTL 必须明确。
4. 不允许将长期业务真相只存在 Redis。
5. 设备在线态可使用 Redis，但历史运行数据必须落库。
```

### 8.3 设备心跳缓存

设备心跳的设计目标：

```text
1. 安卓端高频上报时不直接打爆 PostgreSQL。
2. 接口层快速写 Redis。
3. Celery 每 30 秒批量落库。
4. PC 中控可以读取较新的在线状态。
```

设计流向：

```text
AndroidScript
  → POST /api/device/heartbeat
  → Verify router
  → DeviceService
  → Redis heartbeat buffer
  → Celery flush_device_heartbeats
  → hive_game_xxx.device_runtime
  → PCControl GET /api/device/list
```

### 8.4 待补充 Redis Key 规范

建议新增独立文档：

```text

```

建议 Key 格式：

```text
verify:refresh_token:{user_id}:{token_id}
verify:token_blacklist:{jti}
verify:heartbeat:{project_code}:{device_id}
verify:heartbeat_flush:{project_code}
verify:update:active:{project_code}:{client_type}
verify:device_online:{project_code}:{device_id}
```

待确认：

```text
1. 当前源码实际 Key 是否与上述建议一致。
2. 是否存在 Key 命名不统一。
3. 是否存在未设置 TTL 的短期缓存。
4. 是否需要区分 PC / Android client_type。
```

### 9.1 Celery 的定位

Celery 是 Verify 的后台任务系统。

当前主要用于：

  app.tasks.heartbeat_flush.flush_device_heartbeats

  每 30 秒执行一次

Broker:
  Redis

  Redis

```bash

## 十一、鉴权架构

### 11.1 身份类型

Verify 当前至少存在三类身份：

### 11.2 用户 Token

用户 Token 用于：

```text

2. AndroidScript 登录。
3. PCControl 拉设备数据。
4. AndroidScript 上报心跳。
5. 参数与热更新接口。
```

### 11.3 管理员 Token

管理员 Token 用于：

```text
1. 管理后台登录。
2. 用户管理。
3. 代理管理。
4. 项目管理。
5. 热更新上传。
6. 点数管理。
7. 设备监控。
```

### 11.4 代理 Token

代理 Token 用于：

```text
1. 代理后台登录。
2. 查看自身余额。
3. 查看流水。
4. 查看项目目录。
5. 提交项目申请。
6. 管理自身权限范围。
```

### 11.5 Refresh Token

Refresh Token 设计目标：

```text
1. 减少用户频繁登录。
2. 支持 Access Token 短有效期。
3. 支持单设备登出。
4. 支持批量踢出设备。
```

当前已确认有：

```text
1. Access Token。
2. Refresh Token。
3. Token 黑名单。
4. revoke-all 踢出所有设备接口。
```

### 11.6 高风险点：踢出所有设备

当前需要特别注意：

```text
删除 Refresh Token ≠ 立即让所有已签发 Access Token 失效。
```

如果 Access Token 是无状态 JWT，则已经签发、尚未过期的 Access Token 可能继续可用。

建议后续引入：

```text
user.token_version
agent.token_version
admin.token_version
```

Token payload 增加：

```text
token_version
```

每次鉴权时校验：

```text
payload.token_version == db.token_version
```

踢出所有设备时：

```text
token_version + 1
删除所有 Refresh Token
把当前 jti 加入黑名单
```

这样才能实现更严格的即时失效。

```text

```

### 12.3 PC 与安卓区别

```text
PCControl:
  不占设备授权名额。
  只作为控制端、展示端、参数配置端。

AndroidScript:
  占设备授权名额。
  是实际运行自动化脚本的设备。
```

### 12.4 心跳数据

安卓心跳应包含：

```text

project_code
status
script_version
app_version
game_data
error_code
error_message
timestamp
```

### 12.5 心跳数据流

```text

  → POST /api/device/heartbeat

  → 校验授权
  → 校验设备绑定

## D019 — Verify 心跳采用 Redis 缓冲 + Celery 批量落库

**日期：** 2026-04-29  
**状态：** 已确认  
**关联文档：** [[01-网络验证系统/架构设计]], [[01-网络验证系统/Redis心跳落库策略]]

### 背景

单个游戏可能存在 10,000+ 安卓设备，每 30 秒一次心跳约等于 333 次/秒持续写入。如果每次心跳都直接写 PostgreSQL，会导致数据库压力过高。

### 决策

设备心跳采用：

```text
AndroidScript → Verify API → Redis 缓冲 → Celery 每 30 秒批量落库 → PostgreSQL 游戏库
```text

### 理由

1. 降低数据库直接写压力。
2. 提升心跳接口响应速度。
3. 允许后续做批量合并。
4. 方便在线态缓存。
5. 方便设备监控读取实时状态。

### 风险

1. Redis 异常时心跳缓冲受影响。
2. Celery Worker / Beat 未启动时无法落库。
3. 批量落库失败需要重试与报警。
4. Redis Key 与 TTL 必须规范。

### 后续动作

```text
□ 建立 Redis Key 规范。
□ 建立 Celery 任务监控。
□ 建立心跳压测方案。
□ 验证 Redis 异常和 Celery 异常场景。
