# Verify 热更新端到端测试清单

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于沉淀 `HiveGreatSage-Verify` 热更新链路的端到端测试标准。

本文不是单个接口测试文档，而是用于验证：

```text
1. 管理后台能上传 PC / Android 热更新包。
2. Verify 能保存文件并生成 SHA-256。
3. Verify 能正确归档旧版本并激活新版本。
4. Verify 能清理 Redis 版本缓存。
5. PCControl 能检查、下载、校验、安装 PC 更新包。
6. AndroidScript 能检查、下载、校验、安装 lrj 更新包。
7. 强制更新 force_update 能在客户端正确阻断旧版本继续运行。
8. 热更新失败时能回退、能排查、不会产生半更新状态。
```

## 二、当前总判断

### 2.1 已确认

当前源码中已经存在热更新相关模块：

```text
Verify 管理端:
  app/routers/update_admin.py
  POST /admin/api/updates/{project_id}/{client_type}
  GET  /admin/api/updates/{project_id}/{client_type}/latest
  GET  /admin/api/updates/{project_id}/{client_type}/history

Verify 客户端:
  app/routers/update.py
  GET /api/update/check
  GET /api/update/download

Verify 服务层:
  app/services/update_service.py

Verify Schema:
  app/schemas/update.py

PCControl:
  core/api_client/update_api.py
  core/updater/update_checker.py
  core/updater/update_downloader.py
  core/updater/update_installer.py

AndroidScript:
  脚本/framework/updater.lua
```

### 2.2 已确认能力

源码层面已确认存在：

```text
1. client_type 限制为 pc / android。
2. current_version 格式要求 MAJOR.MINOR.PATCH。
3. 管理端上传文件支持 .lrj / .zip / .apk / .exe。
4. 管理端文件大小限制 500 MB。
5. 上传时计算 SHA-256。
6. 上传后保存文件到存储抽象层。
7. 上传后归档旧 active 版本。
8. 同版本重复上传会更新已有记录。
9. 上传后删除 Redis 版本缓存。
10. 客户端 check 接口返回 need_update。
11. 客户端 download 接口返回 download_url、expires_at、version、checksum_sha256。
12. PCControl 下载时已经有 SHA-256 校验。
13. AndroidScript 已经调用 installLrPkg，但当前需补 SHA-256 校验和失败回滚。
```

### 2.3 当前最高风险

当前热更新链路存在一个 P0 级待确认问题：

```text
管理员上传端 V2:
  版本记录写入主库 version_record。

客户端检查/下载服务层:
  当前 update_service.py 已统一从主库 VersionRecord 读取 active 版本（V2 迁移已完成）。

旧测试:
  test_update_admin.py 仍存在 /admin/api/updates/{game_project_code}/{client_type} 旧路径口径。
```

这意味着可能出现：

```text
管理员上传成功
→ 主库 version_record 有新版本
→ Redis 缓存被清理
→ PC / Android 调用 /api/update/check
→ 客户端服务层去主库 VersionRecord 查 active 版本（已收敛）
→ 查不到新版本
→ 客户端认为无需更新
```

### 2.4 当前状态结论

```text
热更新源码已经具备主要模块；
主库/游戏库 version_record 真相源已统一（2026-05-09 审查确认）；
在统一版本真相源前，不应标记热更新端到端可用。
```

## 四、热更新角色边界

## 4.1 Verify 管理端负责什么

Verify 管理端负责：

```text
1. 管理员鉴权。
2. 校验项目是否存在。
3. 校验 client_type。
4. 校验版本号格式。
5. 校验文件类型。
6. 校验文件大小。
7. 读取文件内容。
8. 计算 SHA-256。
9. 保存文件到存储系统。
10. 归档旧 active 版本。
11. 写入新版本记录。
12. 删除 Redis 版本缓存。
13. 返回版本记录。
```

Verify 管理端不负责：

```text
1. 直接安装 PC 更新。
2. 直接安装 Android lrj。
3. 判断用户是否愿意更新。
4. 客户端本地回滚。
```

---

## 4.2 Verify 客户端接口负责什么

Verify 客户端接口负责：

```text
1. 校验 User Token。
2. 校验当前用户项目授权。
3. 根据当前项目和 client_type 查找 active 版本。
4. 比较 current_version 和 server_version。
5. 返回是否需要更新。
6. 返回 force_update。
7. 返回 release_notes。
8. 返回 checksum_sha256。
9. 返回签名下载 URL。
10. 返回 URL 过期时间。
```

Verify 客户端接口不负责：

```text
1. 下载文件本身的本地保存。
2. 客户端安装。
3. 客户端重启。
4. 客户端回滚。
```

---

## 4.3 PCControl 负责什么

PCControl 负责：

```text
1. 启动时或手动触发检查 PC 更新。
2. 调用 /api/update/check?client_type=pc。
3. 弹窗显示 release_notes。
4. 判断 force_update。
5. 调用 /api/update/download?client_type=pc。
6. 下载 PC 更新包。
7. 校验 SHA-256。
8. 开发模式下解压覆盖并重启。
9. exe 模式下生成 updater.bat 处理外部覆盖。
10. 失败时提示用户并保留旧版本。
```

PCControl 不负责：

```text
1. 上传更新包。
2. 修改版本记录。
3. 管理 Android 包。
4. 绕过 Verify 直接下载存储路径。
```

---

## 4.4 AndroidScript 负责什么

AndroidScript 负责：

```text
1. 登录 Verify 后检查 Android 更新。
2. 调用 /api/update/check?client_type=android。
3. 判断 need_update。
4. 判断 force_update。
5. 调用 /api/update/download?client_type=android。
6. 下载 lrj / apk。
7. 校验 SHA-256。
8. 调用 installLrPkg。
9. 安装成功后重启脚本。
10. 强制更新失败时阻断继续运行。
```

AndroidScript 不负责：

```text
1. 上传更新包。
2. 修改版本记录。
3. 管理 PC 更新。
4. 管理代理和授权。
```

## 五、当前接口契约

## 5.1 管理端上传版本

接口：

```text
POST /admin/api/updates/{project_id}/{client_type}
```

鉴权：

```text
Admin Token
```

路径参数：

```text
project_id:
  主库 game_project.id。

client_type:
  pc
  android
```

Form 字段：

```text
version:
  必填，格式 MAJOR.MINOR.PATCH。

force_update:
  可选，bool，默认 false。

release_notes:
  可选，字符串。

file:
  必填，UploadFile。
```

允许扩展名：

```text
.lrj
.zip
.apk
.exe
```

最大文件：

```text
500 MB
```

响应字段：

```json
{
  "id": 1,
  "version": "1.0.1",
  "client_type": "android",
  "force_update": false,
  "release_notes": "更新说明",
  "checksum_sha256": "64位SHA256",
  "package_path": "game_001/android/packages/1.0.1/test.lrj",
  "released_at": "2026-05-10T00:00:00+00:00",
  "is_active": true,
  "game_project_code": "game_001",
  "message": "版本 1.0.1 发布成功"
}
```

接口：

```text

```

鉴权：

```text

```

Query：

```text
client_type:
  pc 或 android。

current_version:
  MAJOR.MINOR.PATCH。
```

响应：

```json
{
  "need_update": true,
  "current_version": "1.0.1",

## 六、P0 前置修订项

## 6.1 统一 version_record 真相源

### 当前问题

当前源码存在两套口径：

```text
管理端 V2:
  版本记录写入主库 version_record。

客户端服务层:
  check_update / get_download_url 当前从主库 VersionRecord 读取 active 版本（V2 已落地）。
```

### 风险

```text
后台上传成功，但 PC / Android 检查不到新版本。
```

### 必须决策

```text
T048:
热更新 version_record 的唯一真相源是否统一为主库？
```

### 建议方案

建议统一为：

```text
主库 version_record = 唯一热更新版本真相源。
游戏库 version_record = 已从 ORM 移除（V1.1.0）。
```

### 需要修改

```text
1. update_service._get_active_version 改为读主库 VersionRecord。
2. /api/update/check 不再依赖 get_game_db(game_project_code)。
3. /api/update/download 不再依赖 get_game_db(game_project_code)。
4. Redis Key 仍可使用 update:latest:{game_project_code}:{client_type}。
5. test_update.py 改为写主库 version_record。
6. test_update_admin.py 改为使用 project_id 路径。
7. 数据库设计文档中将游戏库 version_record 标为废弃。
```

### 临时回退方案

如果短期不改客户端服务层，则必须：

```text
1. 管理端上传同时写主库和游戏库 version_record。
2. 或管理端继续写游戏库 version_record。
```

不推荐该方案。

原因：

```text
双写会制造新的一致性风险。
```

### 关闭标准

```text
□ 管理端上传写入主库。
□ 客户端 check 从主库读取。
□ 客户端 download 从主库读取。
□ 管理端 latest 从主库读取。
□ 管理端 history 从主库读取。
□ test_update.py 通过。
□ test_update_admin.py 通过。
□ 手工端到端测试通过。
```

## 6.2 统一上传路径参数 project_id

### 当前问题

当前管理端 V2 路由：

```text
/admin/api/updates/{project_id}/{client_type}
```

旧测试和旧文档中可能仍存在：

```text
/admin/api/updates/{game_project_code}/{client_type}
```

### 风险

```text
1. 测试失败但不是业务失败。
2. 前端调用路径和后端路由不一致。
3. AI 或人工继续按旧路径写文档。
```

### 建议方案

正式统一为：

```text
管理端上传:
  使用 project_id。

客户端检查:
  使用 Token 中解析出的 game_project_code。
```

原因：

```text
1. 管理后台操作项目表，天然有 project_id。
2. 客户端不应传 project_id，应该由 Token 授权解析项目。
3. 避免客户端伪造项目。
```

### 关闭标准

```text
□ 前端 UploadVersionForm 使用 projectId。
□ test_update_admin.py 使用 project_id。
□ Obsidian 文档全部改为 project_id。
□ 旧 game_project_code 上传路径标记废弃。
```

## 6.3 统一客户端版本字段

### 当前问题

`UpdateCheckResponse.current_version` 当前实际含义更像：

```text
服务端 active 版本
```

而不是：

```text
客户端传入版本
```

### 风险

```text
1. PCControl 误读字段。
2. AndroidScript 误写本地版本。
3. 文档说明混乱。
```

### 建议方案

后续改为：

```text
client_version:
  客户端传入版本。

server_version:
  服务端 active 版本。

need_update:
  是否需要更新。
```

短期兼容：

```text
保留 current_version，但文档明确它当前代表服务端 active 版本。
```

### 关闭标准

```text
□ Schema 更新。
□ update_service 更新。
□ PCControl 更新。
□ AndroidScript 更新。
□ tests/test_update.py 更新。
```

## 十三、发布治理规则

## 13.1 同版本重复上传

当前源码行为：

```text
同版本重复上传会覆盖旧包，并更新 checksum / release_notes / force_update。
```

### 风险

```text
1. 同一个版本号对应不同文件。
2. 已下载旧包的客户端和后来下载的新包内容不一致。
3. 排查问题困难。
4. 发布不可追溯。
```

### 建议决策

```text
开发环境:
  允许同版本覆盖。

生产环境:
  禁止同版本覆盖。
```

生产规则：

```text
1. 如果版本号已存在，返回 409 Conflict。
2. 必须发布新版本号。
3. 回滚时通过激活旧版本，不通过覆盖旧版本。
```

### 待决策

```text
T049:
生产环境是否禁止同版本重复上传？
```

建议：

```text
必须禁止。
```

## 13.2 回滚规则

当前源码未确认存在专用回滚接口。

建议后续新增：

```text
POST /admin/api/updates/{version_id}/activate
```

或：

```text
POST /admin/api/updates/{project_id}/{client_type}/{version}/activate
```

回滚时：

```text
1. 将当前 active=false。
2. 将目标旧版本 active=true。
3. 删除 Redis 缓存。
4. 记录管理员操作日志。
5. 客户端下一次 check 能看到回滚版本。
```

### 注意

如果客户端当前版本高于回滚版本：

```text
当前 _version_gt(server, client) 会判断无需更新。
```

因此回滚场景需要单独策略：

```text
1. 支持 downgrade_allowed。
2. 或强制更新并允许服务端版本低于客户端版本时仍触发。
3. 或通过 emergency_build 机制。
```

### 待决策

```text

热更新是否支持降级回滚？
```

建议：

```text
必须支持，但放在基础端到端通过后实现。
```

## 13.3 强制更新规则

`force_update=true` 的语义：

```text
客户端不更新则不能继续运行。
```

客户端必须遵守：

```text
1. 检查到 force_update=true。
2. 必须进入下载。
3. 下载失败则阻断运行。
4. 校验失败则阻断运行。
5. 安装失败则阻断运行。
6. 不允许用户点击“跳过”。
```

PCControl 和 AndroidScript 必须分别验证。

## D024 — Verify 热更新版本记录统一以主库 version_record 为真相源

```markdown
## D025 — 生产环境禁止同版本热更新包覆盖

**日期：** 2026-05-10  
**状态：** 建议确认  
**关联文档：** [[01-网络验证系统/热更新端到端测试清单]]

### 背景

当前源码允许同版本重复上传并覆盖旧包。这对开发调试方便，但生产环境会导致同一版本号对应不同文件，影响追溯和回滚。

### 决策

开发环境允许同版本覆盖。

生产环境禁止同版本覆盖。

### 规则

```text
development:
  同版本上传允许覆盖。

production:
  同版本上传返回 409 Conflict。
  必须发布新版本号。
```text

### 理由

```text
1. 保证版本号和文件内容一一对应。
2. 方便排查问题。
3. 防止用户下载到不同内容的同版本包。
4. 符合长期发布治理。
```text

### 后续动作

```text
□ update_admin.py 根据 settings.ENVIRONMENT 判断是否允许覆盖。
□ 测试覆盖 development / production 两种行为。
□ 管理后台提示版本号重复。
```text
