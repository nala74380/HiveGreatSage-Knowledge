# Verify 设备绑定项目维度迁移方案

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/数据库设计]]

## 一、文档目的

本文用于沉淀 `HiveGreatSage-Verify` 当前 **设备绑定 DeviceBinding** 的真实状态、问题边界和后续迁移方案。

本文重点解决一个核心问题：

```text
设备绑定到底应该是“用户维度”，还是“用户 × 项目维度”？
```

当前源码中，用户项目授权已经进入新口径：

```text
用户等级:
  Authorization.user_level

授权设备数:
  Authorization.authorized_devices

授权到期:
  Authorization.valid_until
```

但设备绑定仍然是旧口径：

```text
DeviceBinding:
  user_id + device_fingerprint
```

这会导致多游戏项目接入后出现设备额度、设备归属、设备展示、设备解绑、设备统计的长期风险。

---

## 二、当前结论

### 2.1 已确认

当前源码已确认：

```text
1. DeviceBinding 表没有 game_project_id 字段。
2. DeviceBinding 唯一约束是 user_id + device_fingerprint。
3. PC 登录不创建设备绑定。
4. 安卓登录才创建设备绑定。
5. 安卓设备绑定上限当前仍按 User.user_level 查询 DEVICE_LIMIT_BY_LEVEL。
6. 新业务口径中授权设备数已经归属 Authorization.authorized_devices。
7. 当前用户列表中“项目激活设备数”因为 device_binding 无 project_id，只能通过 LoginLog 统计。
8. 心跳 Redis Key 已经包含 game_id。
9. 游戏库 device_runtime 按游戏库隔离。
10. 设备监控后台可按游戏项目查看设备，但主库绑定本身仍没有项目维度。
```

### 2.2 待确认

以下内容不能写成已完成：

```text
1. 是否已经有迁移脚本给 device_binding 增加 game_project_id。
2. 是否已有测试覆盖 authorized_devices 项目维度设备上限。
3. 是否已有测试证明 PC 登录不占设备额度。
4. 是否已有测试证明同一用户在不同项目下设备额度互不影响。
5. 是否已有前端页面支持按项目解绑设备。
6. 是否允许同一台安卓设备绑定同一用户的多个项目。
7. 是否允许同一设备在不同用户之间转移。
8. 解绑设备是否只解绑某项目，还是解绑该用户所有项目。
```

### 2.3 当前判断

当前状态：

```text
设备运行时链路已经按 game_id / game_project_code 走项目维度；
但设备绑定主表仍停留在用户维度。
```

这属于：

```text
P1 高风险架构债务。
```

原因：

```text
1. 第二游戏接入后风险会立即放大。
2. 同一用户不同项目授权设备数可能不同。
3. 当前绑定数量上限未使用 Authorization.authorized_devices。
4. 用户级设备绑定会造成项目之间互相占用设备额度。
5. 后续设备解绑、设备统计、设备审计都会被旧结构拖住。
```

## 三、术语定义

## 3.1 设备指纹 device_fingerprint

设备指纹是安卓端上报的设备唯一标识。

当前来源可能包括：

```text
1. 插件硬件序列号。
2. getSubscriberId。
3. WiFi MAC。
4. 本地持久化随机 ID。
```

设备指纹用于：

```text
1. 安卓登录绑定设备。
2. 安卓心跳上报。
3. Redis 心跳 Key。
4. 游戏库 device_runtime.device_id。
5. PCControl 展示设备。
```

---

## 3.2 设备绑定 DeviceBinding

设备绑定表示：

```text
某用户已经激活过某台安卓设备。
```

当前旧口径：

```text
user_id + device_fingerprint
```

建议新口径：

```text
user_id + game_project_id + device_fingerprint
```

---

## 3.3 授权设备数 authorized_devices

字段位置：

```text
authorization.authorized_devices
```

含义：

```text
某用户在某项目下允许绑定的安卓设备数量。
```

注意：

```text
authorized_devices 是项目维度。
不是用户全局维度。
```

示例：

```text
用户 user001:
  游戏 A 授权 3 台设备。
  游戏 B 授权 10 台设备。
```

如果 DeviceBinding 不带 project_id，就无法准确表达上面这个场景。

---

## 3.4 PC 登录设备

PCControl 登录 Verify 时：

```text
client_type = pc
```

设计口径：

```text
PC 登录不占安卓授权设备数。
PC 登录不应创建 DeviceBinding。
```

原因：

```text
1. PC 是中控端，不是脚本执行设备。
2. 授权设备数应该限制安卓脚本运行设备。
3. 一个用户换 PC 管理设备不应消耗安卓设备额度。
```

---

## 3.5 安卓登录设备

AndroidScript 登录 Verify 时：

```text
client_type = android
```

设计口径：

```text
安卓登录才占用授权设备数。
安卓登录才创建 DeviceBinding。
安卓心跳必须先有 DeviceBinding。
```

---

## 四、当前源码口径

## 4.1 DeviceBinding 当前模型

当前表：

```text
device_binding
```

当前字段：

```text
id
user_id
device_fingerprint
bound_at
last_seen_at
imsi
status
```

当前唯一约束：

```text
Unique(user_id, device_fingerprint)
```

当前缺少：

```text
game_project_id
```

### 当前问题

该表只能表达：

```text
某用户绑定了某设备。
```

不能表达：

```text
某用户在某项目下绑定了某设备。
```

---

## 4.2 登录服务当前行为

当前登录链路中已确认：

```text
Step 1:
  用户名存在。

Step 2:
  密码匹配。

Step 3:
  账号状态 active。

Step 4:
  账号未到期。

Step 5:
  project_uuid 对应游戏项目存在且激活。

Step 6:
  用户对该项目有 active Authorization。

Step 7:
  Authorization 未过期。

Step 8:
  设备绑定检查。

Step 9:
  写登录日志。

Step 10:
  签发 Access Token 和 Refresh Token。
```

当前 Step 8 具体行为：

```text
如果 client_type = pc:
  不创建设备绑定。
  不占用激活名额。

如果 client_type = android:
  查询 DeviceBinding(user_id, device_fingerprint, status=active)。
  如果已存在，更新 last_seen_at。
  如果不存在，检查绑定数量上限。
  未超限则创建 DeviceBinding。
```

### 当前高风险点

安卓设备上限当前按：

```text
DEVICE_LIMIT_BY_LEVEL[user.user_level]
```

而不是：

```text
authorization.authorized_devices
```

这与新业务口径冲突。

---

## 4.3 用户服务当前新口径

当前用户服务已经明确：

```text
1. User 是账号主体。
2. 用户等级、授权设备数、到期时间不再以 User 为主口径。
3. 项目内等级、项目授权设备数、项目到期时间统一归属 Authorization。
4. 一个用户可以在 A 项目是 normal，在 B 项目是 vip。
5. 一个用户可以在不同项目拥有不同授权设备数和不同到期时间。
```

因此设备绑定必须跟随这个口径。

---

## 4.4 当前激活设备数统计方式

因为 `device_binding` 没有 `game_project_id`，当前项目激活设备数统计使用：

```text
LoginLog:
  user_id
  game_project_id
  distinct device_fingerprint
  success = true
```

这只是临时替代方案。

### 临时方案问题

通过 LoginLog 统计有以下问题：

```text
1. 登录日志是行为日志，不是当前绑定状态表。
2. 已解绑设备如果历史登录成功过，可能仍被统计。
3. 登录成功不等于当前设备绑定 active。
4. 统计性能可能随日志增长下降。
5. 无法直接支持项目级解绑。
```

---

## 4.5 心跳服务当前行为

当前心跳服务已经具备项目维度：

```text
1. 通过 game_project_code 查询 GameProject。
2. Redis Key 使用 game_id。
3. Redis Key 格式:
   device:runtime:{game_id}:{user_id}:{device_fp}
4. 游戏库按 game_project_code 独立读取 device_runtime。
```

但心跳上报前的绑定校验仍是：

```text
user_id + device_fingerprint
```

未包含：

```text
game_project_id
```

### 当前风险

如果用户在游戏 A 绑定了设备 X，那么同一用户在游戏 B 上报设备 X 时，绑定校验可能误认为已经绑定。

---

## 五、为什么必须迁移到项目维度

## 5.1 多游戏授权设备数不同

示例：

```text
user001:
  游戏 A 授权 2 台设备。
  游戏 B 授权 10 台设备。
```

如果设备绑定是用户级：

```text
user001 绑定设备总数 = 10
```

系统无法判断：

```text
这 10 台设备到底属于游戏 A，还是游戏 B？
```

结果：

```text
游戏 A 可能被放行超过 2 台。
或游戏 B 被游戏 A 的设备绑定挤占额度。
```

---

## 5.2 同一设备可参与多个项目

一个安卓设备可能运行不同游戏脚本。

例如：

```text
设备 android_001:
  白天运行游戏 A。
  晚上运行游戏 B。
```

如果绑定是用户级：

```text
只记录一次 user_id + android_001。
```

如果绑定是项目级：

```text
user001 + game_A + android_001
user001 + game_B + android_001
```

后者更符合真实平台模型。

---

## 5.3 解绑必须支持项目维度

未来后台可能需要：

```text
只解绑用户在游戏 A 下的某台设备。
不影响该设备在游戏 B 下的绑定。
```

当前用户级绑定无法做到。

---

## 5.4 PCControl 按项目查看设备

PCControl 是某个游戏项目的中控端。

它应该看到：

```text
当前用户在当前项目下的设备。
```

不应该看到：

```text
当前用户在其他项目下的设备。
```

如果 DeviceBinding 没有 project_id，就只能通过 Redis/game_db/登录日志绕路判断。

---

## 5.5 审计与合规

设备绑定涉及：

```text
1. 用户。
2. 项目。
3. 设备指纹。
4. 登录时间。
5. 最后在线时间。
6. IMSI 可选敏感字段。
```

项目维度缺失会导致：

```text
无法回答某设备到底是因哪个项目被绑定。
```

---

## 六、建议目标设计

## 6.1 DeviceBinding 新模型

建议新增字段：

```text
game_project_id
```

新表结构：

```text
device_binding
├── id
├── user_id
├── game_project_id
├── device_fingerprint
├── bound_at
├── last_seen_at
├── imsi
├── status
```

新唯一约束：

```text
Unique(user_id, game_project_id, device_fingerprint)
```

建议索引：

```text
idx_device_binding_user_project(user_id, game_project_id)
idx_device_binding_project_last_seen(game_project_id, last_seen_at)
idx_device_binding_device(device_fingerprint)
```

---

## 6.2 新绑定口径

安卓登录时绑定判断：

```text
user_id
game_project_id
device_fingerprint
status = active
```

如果已存在：

```text
更新 last_seen_at。
```

如果不存在：

```text
检查该用户在该项目下 active 设备数。
```

设备数量统计：

```text
count(DeviceBinding.id)
where user_id = 当前用户
  and game_project_id = 当前项目
  and status = active
```

设备上限：

```text
Authorization.authorized_devices
```

---

## 6.3 authorized_devices 语义

建议语义：

```text
authorized_devices = 0:
  无限设备。

authorized_devices > 0:
  最多可绑定 N 台安卓设备。
```

当前模型注释中已有：

```text
authorized_devices: 0 表示无限制。
```

因此登录逻辑应按此执行。

---

## 6.4 PC 登录口径

PC 登录继续保持：

```text
不创建设备绑定。
不占用 authorized_devices。
```

但登录日志仍记录：

```text
client_type = pc
device_fingerprint = pc_control_xxx
game_project_id = 当前项目
```

用途：

```text
1. 安全审计。
2. 登录日志排查。
3. 未来 PC 设备信任管理。
```

---

## 6.5 安卓心跳口径

心跳绑定校验改为：

```text
DeviceBinding.user_id == current_user.id
DeviceBinding.game_project_id == current_game_project.id
DeviceBinding.device_fingerprint == body.device_fingerprint
DeviceBinding.status == "active"
```

心跳更新：

```text
只更新该项目下的 DeviceBinding.last_seen_at。
```

Redis Key 保持：

```text
device:runtime:{game_id}:{user_id}:{device_fp}
```

---

## 6.6 PC 设备列表口径

`GET /api/device/list` 应返回：

```text
当前用户 + 当前项目下的设备。
```

三层数据源也都按项目过滤：

```text
第 1 层 Redis:
  game_id + user_id

第 2 层 游戏库:
  当前游戏库 device_runtime

第 3 层 主库 DeviceBinding:
  user_id + game_project_id
```

---

## 6.7 后台设备监控口径

全局设备总览：

```text
GET /admin/api/devices/
```

建议支持筛选：

```text
project_id
game_project_code
user_id
online_only
status
```

项目设备监控：

```text
GET /admin/api/devices/{game_project_code}
```

必须只返回：

```text
该项目下的设备绑定与运行数据。
```

Agent Token：

```text
短期:
  只能看自己直接创建用户的设备。

长期:
  可扩展为递归子树范围。
```

---

## 七、迁移路线

## 7.1 总体原则

本迁移不能直接粗暴改表后上线。

必须按以下原则执行：

```text
1. 先加字段，后改逻辑。
2. 先兼容旧数据，后切换新约束。
3. 先测试单游戏，后测试多游戏。
4. 先内部联调，后生产数据迁移。
5. 保留回退方案。
```

---

## 7.2 Phase A：加字段兼容

新增迁移：

```text
device_binding.game_project_id nullable=True
```

新增索引：

```text
idx_device_binding_user_project
idx_device_binding_project_last_seen
```

暂不删除旧唯一约束。

目的：

```text
让新代码可以开始写 game_project_id。
旧数据仍可存在。
```

---

## 7.3 Phase B：回填历史数据

### 单游戏环境回填

如果当前只有一个游戏项目：

```sql
UPDATE device_binding
SET game_project_id = 目标项目ID
WHERE game_project_id IS NULL;
```

适用：

```text
当前开发环境 / 单游戏早期环境。
```

### 多游戏环境回填

如果已有多游戏历史数据，不能简单回填。

应按 LoginLog 推断：

```text
user_id
device_fingerprint
game_project_id
success = true
client_type = android
```

回填策略：

```text
1. 对每个 device_binding 查最近一次成功安卓登录日志。
2. 如果只对应一个 game_project_id，则回填该项目。
3. 如果对应多个 game_project_id，则需要拆分多条绑定记录。
4. 如果没有登录日志，标记为待人工确认。
```

---

## 7.4 Phase C：双写新逻辑

修改登录逻辑：

```text
安卓登录新建绑定时写 game_project_id。
安卓登录查绑定时优先查 user_id + game_project_id + device_fingerprint。
```

兼容旧数据：

```text
如果新查询找不到，但存在旧绑定 game_project_id IS NULL:
  可回填为当前项目后继续。
```

建议只在迁移过渡期保留该兼容。

---

## 7.5 Phase D：切换唯一约束

当所有 active 绑定都有 game_project_id 后：

```text
1. 删除旧唯一约束 uq_user_device。
2. 新增唯一约束 uq_user_project_device:
   user_id + game_project_id + device_fingerprint。
3. 将 game_project_id 改为 nullable=False。
```

---

## 7.6 Phase E：清理旧逻辑

清理以下旧逻辑：

```text
1. 设备上限按 User.user_level 判断。
2. 设备绑定按 user_id + device_fingerprint 判断。
3. 项目激活设备数通过 LoginLog 统计。
4. 后台设备监控通配 Redis 后不区分项目。
```

改为：

```text
1. 设备上限按 Authorization.authorized_devices。
2. 设备绑定按 user_id + game_project_id + device_fingerprint。
3. 项目激活设备数直接从 DeviceBinding 统计。
4. 后台设备监控按 project_id/game_project_code 精确过滤。
```

---

## 八、迁移 SQL 草案

> 注意：以下是设计草案，执行前必须根据当前 Alembic 版本生成正式迁移文件。

## 8.1 增加字段

```sql
ALTER TABLE device_binding
ADD COLUMN game_project_id INTEGER NULL;
```

## 8.2 增加外键

```sql
ALTER TABLE device_binding
ADD CONSTRAINT fk_device_binding_game_project
FOREIGN KEY (game_project_id)
REFERENCES game_project(id)
ON DELETE CASCADE;
```

## 8.3 增加索引

```sql
CREATE INDEX idx_device_binding_user_project
ON device_binding(user_id, game_project_id);

CREATE INDEX idx_device_binding_project_last_seen
ON device_binding(game_project_id, last_seen_at);

CREATE INDEX idx_device_binding_device
ON device_binding(device_fingerprint);
```

## 8.4 单游戏回填

```sql
UPDATE device_binding
SET game_project_id = 1
```

## 十七、结论

当前 Verify 已经在用户授权、Redis 心跳、游戏库运行数据上进入项目维度，但 DeviceBinding 仍是旧的用户级结构。

这会成为第二游戏接入前的关键阻塞点。

当前最准确结论：

```text
DeviceBinding 当前源码可支撑单游戏 MVP；
但不适合长期多游戏平台；
必须迁移为 user_id + game_project_id + device_fingerprint。
```

迁移后的目标状态：

```text
1. Authorization 控制项目授权和设备额度。
2. DeviceBinding 记录用户在某项目下绑定了哪些安卓设备。
3. PC 登录不占设备额度。
4. Android 登录占设备额度。
5. 心跳必须校验项目级设备绑定。
6. PCControl 只能看到当前项目设备。
7. 管理后台可以按项目审计和解绑设备。
```
