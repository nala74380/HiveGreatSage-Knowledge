# Verify 三端联调执行手册

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/三端联调测试清单]]

## 一、文档目的

本文用于指导 `HiveGreatSage-Verify`、`HiveGreatSage-PCControl`、`HiveGreatSage-AndroidScript` 三端联调。

本文不是接口设计文档，而是执行手册。

本文重点回答：

```text
1. 三端联调前必须准备什么？
2. Verify、PCControl、AndroidScript 的启动顺序是什么？
3. 哪些配置必须完全一致？
4. 哪些接口先测，哪些接口后测？
5. 如何判断 AndroidScript 心跳真的进入 Redis？
6. 如何判断 PCControl 看到的是真实设备，不是 mock 数据？
7. 如何验证参数从 PC 保存后，安卓能拉到？
8. 如何验证热更新 check / download 的端到端链路？
9. 如何验证 Token 过期、刷新、踢出后的三端行为？
10. 如何记录证据并生成联调报告？
```

## 二、当前结论

### 2.1 已确认

当前 Verify 后端认证 Schema 已确认登录字段：

```text
username
password
project_uuid
device_fingerprint
client_type
```

其中：

```text
client_type 只能是 pc 或 android。
```

登录成功响应字段：

```text
access_token
refresh_token
token_type
expires_in
user_id
username
user_level
game_project_code
```

### 2.2 已确认的设备链路

当前测试中已经覆盖：

```text
1. 创建用户。
2. 给用户授权项目。
3. Android client_type 登录。
4. 使用 device_fingerprint 创建设备绑定。
5. POST /api/device/heartbeat 上报心跳。
6. GET /api/device/list 能看到在线设备。
7. GET /api/device/data 能读到 Redis 中的 game_data。
```

### 2.3 已确认的 PCControl 同步机制

PCControl 当前 `SyncWorker` 已确认：

```text
1. 在独立 QThread 中运行。
2. 默认每 10 秒拉取 Verify 设备列表。
3. 成功后 emit devices_updated 给 UI。
4. 遇到 401 时调用 refresh_access_token。
5. 刷新成功后立即重试一次 fetch。
6. 刷新失败后 emit token_expired。
7. 开发模式下可能使用 mock fallback。
```

### 2.4 已确认的前端接入流程口径

当前 Verify 前端接入指南中已经写出两条完整流程：

```text
PC 中控:
  检查更新 → 登录 → 拉参数 → 拉设备列表 → 保存参数。

安卓脚本:
  检查更新 → 登录验证 + 设备绑定 → 上报 IMSI → 拉参数 → 每 30 秒心跳 → Token 过期刷新。
```

### 2.5 待修订口径

当前接入指南中仍存在旧口径示例：

```text
hardware_serial
project_uuid 作为 params/get 或 update/check 的 Query 参数
/api/update/download 携带 version 参数
/api/update/check 返回 version 字段
```

但当前正式后端 Schema 中登录字段是：

```text
device_fingerprint
```

热更新、参数接口是否需要 `project_uuid` Query，应以当前后端路由与 OpenAPI 快照为准。

建议：

```text
三端联调时不要直接照抄 GuideView 示例。
必须以当前后端 Schema、OpenAPI 快照和实际接口测试为准。
```

## 三、联调目标

## 3.1 第一阶段目标：最小闭环

最小闭环必须证明：

```text
1. Verify 可启动。
2. Redis 可用。
3. PostgreSQL 主库可用。
4. 游戏库可用。
5. AndroidScript 能登录 Verify。
6. AndroidScript 能上报心跳。
7. Redis 中出现 device:runtime key。
8. PCControl 能登录同一账号同一项目。
9. PCControl 能拉到 AndroidScript 的设备。
10. PCControl 显示的是真实 Verify 数据，不是 mock。
```

---

## 3.2 第二阶段目标：参数闭环

参数闭环必须证明：

```text
1. Verify 游戏库已有 script_param_def。
2. AndroidScript 能 GET /api/params/get。
3. PCControl 能 GET /api/params/get。
4. PCControl 修改参数后 POST /api/params/set。
5. AndroidScript 再次拉参数能看到修改后的值。
```

---

## 3.3 第三阶段目标：热更新闭环

热更新闭环必须证明：

```text
1. 管理后台上传 Android 包。
2. AndroidScript 能 check 到新版本。
3. AndroidScript 能 download。
4. AndroidScript 能校验 SHA-256。
5. AndroidScript 能 installLrPkg。
6. 管理后台上传 PC 包。
7. PCControl 能 check 到新版本。
8. PCControl 能 download。
9. PCControl 能校验 SHA-256。
10. PCControl 能安装或进入待重启状态。
```

---

## 3.4 第四阶段目标：异常与恢复闭环

异常闭环必须证明：

```text
1. Access Token 过期后，PCControl 能 refresh。
2. Access Token 过期后，AndroidScript 能 refresh 或重新登录。
3. Refresh Token 失效后，两端能进入重新登录。
4. revoke-all 后，两端行为符合当前 Token 方案。
5. 网络断开后，PCControl 不误显示真实在线。
6. AndroidScript 心跳失败后有明确日志。
```

## 五、三端角色边界

## 5.1 Verify

Verify 负责：

```text
1. 身份认证。
2. 项目授权校验。
3. 设备绑定。
4. Token 签发与刷新。
5. Redis 心跳缓存。
6. Celery 心跳落库。
7. 参数读取与保存。
8. 热更新版本记录与下载链接。
9. 管理后台和代理后台 API。
```

Verify 不负责：

```text
1. 直接操作安卓设备。
2. 直接修改 PCControl 本地 UI。
3. 替 AndroidScript 执行脚本任务。
4. 替 PCControl 安装更新。
```

---

## 5.2 AndroidScript

AndroidScript 负责：

```text
1. 读取本地 Config。
2. 获取安卓设备指纹。
3. 使用 client_type=android 登录 Verify。
4. 拉取脚本参数。
5. 周期性上报心跳。
6. 上报 game_data。
7. 检查 Android 热更新。
8. 下载并安装 lrj 包。
9. Token 失效后刷新或重新登录。
```

AndroidScript 不负责：

```text
1. 管理用户授权。
2. 管理代理点数。
3. 上传热更新包。
4. 代替 PCControl 展示设备列表。
```

---

## 5.3 PCControl

PCControl 负责：

```text
1. 读取本地 server.api_base_url。
2. 读取本地 server.project_uuid。
3. 使用 client_type=pc 登录 Verify。
4. 保存 Access Token。
5. 保存 Refresh Token。
6. 周期性拉取设备列表。
7. 展示 AndroidScript 上报的设备状态。
8. 编辑和保存脚本参数。
9. 检查 PC 热更新。
10. 下载并安装 PC 更新包。
```

PCControl 不负责：

```text
1. 上报安卓心跳。
2. 占用安卓设备额度。
3. 管理代理点数。
4. 上传热更新包。
5. 直接访问 PostgreSQL 或 Redis。
```

## 六、联调前置条件

## 6.1 源码状态

必须记录：

```text
Verify commit:
PCControl commit:
AndroidScript commit:
Knowledge commit:
```

要求：

```text
1. 三端都来自明确 commit。
2. 不使用未记录的临时代码。
3. 本地修改必须写入联调报告。
```

---

## 6.2 Verify 环境

必须确认：

```text
□ PostgreSQL 主库 hive_platform 可连接。
□ 游戏库 hive_game_001 或目标游戏库可连接。
□ Redis 可连接。
□ Alembic 已 upgrade head。
□ scripts/init_data.py 已执行。
□ 管理员账号可登录。
□ 目标 game_project 存在。
□ game_project.project_uuid 已记录。
□ game_project.code_name 已记录。
□ 游戏库中参数表存在。
□ 热更新存储目录可写。
```

---

## 6.3 Celery 环境

必须确认：

```text
□ Celery Worker 已启动。
□ Celery Beat 已启动。
□ Beat 每 30 秒触发 flush_device_heartbeats。
□ Worker 能连接 Redis。
□ Worker 能连接游戏库。
```

注意：

```text
只启动 Verify API 不等于心跳落库可用。
```

---

## 6.4 PCControl 配置

必须确认：

```yaml
server:
  api_base_url: "http://127.0.0.1:8000"
  project_uuid: "目标项目 UUID"

sync:
  interval: 10
```

检查文件：

```text
config/default.yaml
config/local.yaml
config/local.example.yaml
```

要求：

```text
1. api_base_url 指向当前 Verify。
2. project_uuid 与 Verify game_project.project_uuid 一致。
3. project_uuid 与 AndroidScript 配置一致。
4. 生产环境不能使用 localhost。
5. config/local.yaml 不应入库。
```

---

## 6.5 AndroidScript 配置

必须确认：

```text
API_BASE_URL:
  指向当前 Verify。

PROJECT_UUID:
  与 Verify game_project.project_uuid 一致。

SCRIPT_VERSION:
  当前脚本版本。

HEARTBEAT_INTERVAL:
  建议 30 秒。

DEVICE_FINGERPRINT:
  能稳定生成。
```

要求：

```text
1. API_BASE_URL 不要写错协议 http/https。
2. 安卓设备能访问 Verify 地址。
3. 模拟器或手机访问 127.0.0.1 时需特别注意网络映射。
4. PROJECT_UUID 与 PCControl 一致。
```

## 十、阶段二：AndroidScript 心跳闭环

## 10.1 Android 登录

执行：

```text

```

通过标准：

```text
□ AndroidScript 日志显示登录成功。
□ Verify 登录日志有 success=true。
□ Access Token 已保存到 AndroidScript 内存或本地安全位置。
```

---

## 10.2 Android 拉参数

执行：

```text
GET /api/params/get
Authorization: Bearer {android_access_token}
```

通过标准：

```text
□ 返回 200。
□ 返回 game_project_code。
□ 返回 params。
□ AndroidScript 能解析 bool / int / float / string / enum。
```

---

## 10.3 Android 上报心跳

请求体：

```json
{
  "device_fingerprint": "android_test_001",
  "status": "running",
  "game_data": {
    "current_task": "联调测试",
    "gold": 1024
  }
}
```

通过标准：

```text
□ 返回 200。
□ 返回 code=0。
□ Verify 日志无异常。
□ Redis 中出现 device:runtime key。
```

Redis 检查：

```bash
redis-cli -p 16379 keys "device:runtime:*"
redis-cli -p 16379 ttl "device:runtime:{game_id}:{user_id}:{device_fp}"
```

期望：

```text
□ TTL 接近 120 秒。
□ 重复心跳会刷新 TTL。
```

---

## 10.4 Celery 落库

等待：

```text
30 秒以上。
```

检查：

```text
□ Beat 日志触发 flush_device_heartbeats。
□ Worker 日志执行 flush_device_heartbeats。
□ 游戏库 device_runtime 有记录。
```

SQL 示例：

```sql
SELECT device_id, status, last_seen, game_data, updated_at
FROM device_runtime
WHERE device_id = 'android_test_001';
```

通过标准：

```text
□ 有记录。
□ status 正确。
□ game_data 正确。
□ updated_at 更新。
```

### 关键排除：不能是 mock 数据

必须确认：

## 14.3 revoke-all

当前实现状态：

```text
revoke-all 当前能删除所有 Refresh Token，并使当前 Access Token 失效。
其他已签发 Access Token 是否即时失效，取决于是否已完成 token_version 改造。
```

测试必须按当前实现记录，不得过度承诺。

通过标准分两种：

### 当前未完成 token_version

```text
□ 当前调用端 Access Token 失效。
□ 所有 Refresh Token 失效。
□ 其他端旧 Access Token 可能在过期前仍可访问。
□ 文档标记为当前限制。
```

### 完成 token_version 后

```text
□ PCControl 旧 Access Token 立即 401。
□ AndroidScript 旧 Access Token 立即 401。
□ 所有旧 Refresh Token refresh 失败。
□ 重新登录后恢复。
```
