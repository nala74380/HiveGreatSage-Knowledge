```text
6. SKIP、ERROR、FAIL 分别怎么判断？
7. 什么条件下允许进入三端联调？
8. 什么条件下允许进入生产部署测试？
9. 每次重构、迁移、接口变更后必须跑哪些测试？
10. 测试报告应该如何沉淀进 Obsidian？
```

### 2.4 当前质量状态判断

当前状态应标记为：

```text
已有集成测试基础；
已覆盖主链路的一部分；
但质量门禁尚未系统化；
财务、项目准入、设备项目维度、token_version、仓库安全、OpenAPI 快照、数据库迁移仍需补测试。
```

风险等级：

```text
P1 高风险。
```

原因：

```text
1. 当前测试复用开发库，环境漂移风险高。
2. 游戏库不可访问时部分测试会 SKIP，不能被误认为生产前通过。
3. 点数计费与返点属于财务模块，必须独立测试。
4. 项目准入决定代理商业边界，必须独立测试。
5. 设备绑定项目维度是多游戏前置条件，必须独立测试。
6. 热更新主库/游戏库真相源漂移风险必须通过端到端测试关闭。
```

## 三、测试分层

## 3.1 L0：静态检查

目标：

```text
在运行服务前发现明显错误。
```

包括：

```text
1. Python import 检查。
2. 配置加载检查。
3. OpenAPI 生成检查。
4. 仓库卫生检查。
5. .env 是否被跟踪检查。
6. 迁移文件链检查。
```

建议命令：

```bash
python -m compileall app tests
python scripts/export_openapi.py
python scripts/check_repo_hygiene.py
alembic heads
```

通过标准：

```text
□ Python 编译无语法错误。
□ OpenAPI paths 数量不为 0。
□ Git 未跟踪 .env / logs / celerybeat / dist。
□ Alembic heads 只有一个。
```

---

## 3.2 L1：后端集成测试

目标：

```text
验证 FastAPI 路由、依赖注入、服务层、数据库和 Redis 的集成行为。
```

当前主力：

```text
pytest tests
```

当前模式：

```text
httpx.AsyncClient + ASGITransport 直连 FastAPI app。
```

优点：

```text
1. 不需要真实启动 uvicorn。
2. 能覆盖 FastAPI 路由层。
3. 能使用真实数据库 session。
4. 速度比端到端联调快。
```

限制：

```text
1. 仍依赖真实 PostgreSQL / Redis。
2. 不等于浏览器前端通过。
3. 不等于 PCControl / AndroidScript 实机通过。
4. 不能证明 Celery Worker / Beat 真正运行。
```

## 四、当前 pytest 运行环境

## 4.1 主库

当前测试使用：

```text
settings.DATABASE_MAIN_URL
```

默认指向：

```text
hive_platform
```

注意：

```text
当前测试不是纯内存测试。
也不是临时数据库测试。
```

风险：

```text
1. 测试可能污染开发库。
2. 开发库历史数据可能影响测试。
3. 不同机器数据库状态不同，测试结果可能不同。
```

当前缓解方式：

```text
1. 测试用户名使用随机后缀。
2. 部分测试 teardown 会清理或挂起数据。
3. seed_data 幂等创建测试管理员和测试项目。
```

---

## 4.2 游戏库

当前测试游戏库：

```text
hive_game_001
```

推导来源：

```text
DATABASE_MAIN_URL 的 database 替换为 hive_game_001。
```

当前风险：

```text
hive_user 对 hive_game_001 可能无权限或密码不匹配。
```

当前处理：

```text
game_db_accessible 返回 True / False。
需要游戏库的 fixture 如果不可用则 pytest.skip。
```

修复命令：

```bash
psql -U postgres -p 15432 -f scripts/setup_game_db.sql
```

当前测试仍连接：

```text
settings.REDIS_URL
```

当前测试中：

## 五、测试结果语义

## 5.1 PASS

含义：

```text
测试按预期通过。
```

但需要区分：

```text
单个测试 PASS:
  只代表该用例通过。

全部 pytest PASS:
  代表当前后端集成测试通过。

全部 pytest PASS + frontend build PASS:
  才能进入 Web 联调。

全部 pytest PASS + frontend build PASS + 三端联调 PASS:
  才能认为主链路通过。
```

---

## 5.2 FAIL

含义：

```text
断言失败。
```

处理原则：

```text
1. 先判断是源码问题、测试问题，还是环境问题。
2. 不允许直接改测试迎合错误实现。
3. 不允许删除失败断言来让结果变绿。
4. 必须记录失败原因和修复方式。
```

---

## 5.3 ERROR

含义：

```text
测试执行过程异常，fixture、环境、连接或未捕获异常导致。
```

常见来源：

```text
1. PostgreSQL 连接失败。
2. Redis 连接失败。
3. Alembic 未迁移。
4. 依赖注入 patch 失效。
5. 事件循环冲突。
6. import 错误。
```

处理：

```text
ERROR 优先级高于 FAIL。
先修测试环境或基础设施。
```

---

## 5.4 SKIP

含义：

```text
测试主动跳过。
```

当前典型原因：

```text
hive_game_001 不可访问。
```

重要规则：

```text
开发阶段允许有明确原因的 SKIP。
生产前门禁不允许关键链路 SKIP。
```

生产前如果出现：

```text
热更新测试 SKIP。
参数测试 SKIP。
游戏库相关测试 SKIP。
```

结论必须是：

```text
不允许进入生产部署测试。
```

## 八、关键质量门禁

## 8.1 本地开发门禁

每次改后端代码，至少执行：

```bash
python -m pytest tests/test_auth.py tests/test_admin.py -v
```

如果涉及设备、参数、热更新，则补跑：

```bash
python -m pytest tests/test_device.py tests/test_params.py tests/test_update.py -v
```

通过标准：

```text
□ 无 ERROR。
□ 无 FAIL。
□ 允许非关键 SKIP，但必须记录原因。
```

---

## 8.2 合并前门禁

每次合并前必须执行：

```bash
python -m pytest -v
cd frontend && npm run build
python scripts/export_openapi.py
```

通过标准：

```text
□ pytest 无 FAIL。
□ pytest 无 ERROR。
□ 关键链路无 SKIP。
□ frontend build 成功。
□ OpenAPI 可生成。
□ 仓库不跟踪 .env / logs / dist。
```
