# Verify 日志监控与告警规范

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于规范 `HiveGreatSage-Verify` 的日志、监控、告警和排障体系。

本文重点回答：

```text
1. Verify 当前已经有哪些日志能力？
2. 哪些日志可以打印？
3. 哪些敏感信息绝对不能打印？
4. API / Redis / Celery / PostgreSQL / Nginx 分别要监控什么？
5. 哪些情况必须告警？
6. /health 只能说明什么，不能说明什么？
7. 心跳落库、热更新、点数、项目准入等高风险模块如何留痕？
8. 生产环境如何排查问题？
9. 日志文件如何轮转和归档？
10. 监控与告警状态如何沉淀进 Obsidian？
```

---

## 二、当前结论

### 2.1 已确认

当前 Verify 源码已经具备基础日志和监控入口。

已确认：

```text
1. app/main.py 中存在 _setup_logging()。
2. 日志使用 loguru。
3. 默认输出到 stderr。
4. 可通过 LOG_FILE 输出到文件。
5. 文件日志每日 00:00 轮转。
6. 文件日志保留 30 天。
7. 文件日志压缩为 zip。
8. production 环境下 serialize=True。
9. SQLAlchemy 日志级别可由 SQLALCHEMY_LOG_LEVEL 控制。
10. Sentry 可通过 SENTRY_DSN 开启。
11. Sentry 未配置时会跳过初始化。
12. /health 已存在。
13. /health 返回 status、environment、version。
```

### 2.2 已确认配置项

当前 `app/config.py` 中已存在：

```text
LOG_LEVEL
LOG_FILE
SQLALCHEMY_LOG_LEVEL
SENTRY_DSN
```

当前默认值：

```text
LOG_LEVEL = INFO
LOG_FILE = logs/app.log
SQLALCHEMY_LOG_LEVEL = WARNING
SENTRY_DSN = ""
```

### 2.3 待确认

以下内容不能写成已完成：

```text
1. 生产环境是否已经接入 Sentry。
2. 是否已有统一 request_id / trace_id。
3. 是否已有结构化 JSON 日志采集。
4. 是否已有日志脱敏中间件。
5. 是否已有 Prometheus / Grafana。
6. 是否已有 Redis / PostgreSQL / Celery 告警。
7. 是否已有 Nginx 访问日志分析。
8. 是否已有登录失败异常告警。
9. 是否已有热更新上传失败告警。
10. 是否已有点数扣费异常告警。
11. 是否已有 Celery 心跳落库延迟告警。
12. 是否已有日志轮转外部策略 logrotate。
```

### 2.4 当前风险判断

日志监控属于：

```text
P1 高风险基础设施。
```

原因：

```text
1. Verify 是 PCControl / AndroidScript / 管理后台 / 代理后台共同依赖的中枢。
2. 没有日志就无法排查授权、心跳、热更新、点数、项目准入问题。
3. 没有监控就无法知道 Redis / Celery / PostgreSQL 是否异常。
4. 日志如果不脱敏，会泄露 Token、密码、SECRET_KEY、数据库连接。
5. 点数和授权属于商业敏感链路，必须可审计。
```

## 三、日志分层

## 3.1 应用日志

来源：

```text
FastAPI / Verify 业务代码。
```

用途：

```text
1. 启动和关闭。
2. 接口异常。
3. 登录失败。
4. 授权失败。
5. 设备心跳异常。
6. 参数保存异常。
7. 热更新异常。
8. 点数扣费异常。
9. 项目准入审核异常。
```

当前入口：

```text
app/main.py::_setup_logging()
```

---

## 3.2 访问日志

来源：

```text
uvicorn / nginx。
```

用途：

```text
1. 查看接口访问量。
2. 查看 4xx / 5xx。
3. 查看来源 IP。
4. 查看上传包大小。
5. 查看请求耗时。
```

注意：

```text
访问日志中不能记录 Authorization Header。
```

---

## 3.3 数据库日志

来源：

```text
SQLAlchemy / PostgreSQL。
```

当前源码已经支持：

```text
SQLALCHEMY_LOG_LEVEL
```

建议生产默认：

```text
WARNING
```

不建议生产长期启用：

```text
DEBUG
```

原因：

```text
SQL DEBUG 可能刷屏，并可能暴露 SQL 参数、业务字段和性能问题。
```

---

## 3.4 Celery 日志

来源：

```text
hive-verify-worker.service
hive-verify-beat.service
```

用途：

```text
1. Worker 是否启动。
2. Beat 是否定时触发。
3. flush_device_heartbeats 是否执行。
4. 每批处理设备数。
5. 每批耗时。
6. 单个游戏库落库是否失败。
```

---

## 3.5 Nginx 日志

来源：

```text
/var/log/nginx/access.log
/var/log/nginx/error.log
```

用途：

```text
1. 502。
2. 504。
3. 413。
4. 静态资源 404。
5. 热更新包下载失败。
6. API 被异常频率访问。
```

---

## 四、日志级别规范

## 4.1 DEBUG

使用场景：

```text
1. 本地开发。
2. 临时排查。
3. 测试环境短期开启。
```

禁止：

```text
生产长期启用 DEBUG。
```

风险：

```text
1. 日志量过大。
2. 可能暴露 SQL。
3. 可能暴露请求体。
4. 影响性能。
```

---

## 4.2 INFO

使用场景：

```text
1. 服务启动。
2. 服务关闭。
3. 定时任务正常执行摘要。
4. 管理员关键操作成功。
5. 热更新上传成功。
6. 点数充值 / 授信成功摘要。
```

要求：

```text
INFO 只能记录摘要，不记录敏感明文。
```

---

## 4.3 WARNING

使用场景：

```text
1. 登录失败。
2. 授权不足。
3. 设备超限。
4. 心跳 401 / 403 / 429。
5. 参数部分保存失败。
6. 热更新检查异常但非阻塞。
7. Celery 单项目落库失败。
8. Redis 缓存读取失败但可回退。
```

---

## 4.4 ERROR

使用场景：

```text
1. 接口 500。
2. 数据库连接失败。
3. Redis 连接失败。
4. 热更新上传失败。
5. 点数扣费事务失败。
6. 返点事务失败。
7. 项目准入审核写库失败。
8. Celery 任务整体失败。
```

---

## 4.5 CRITICAL

使用场景：

```text
1. 生产环境安全检查失败。
2. SECRET_KEY 不符合要求。
3. 主库不可用。
4. Redis 完全不可用。
5. 数据库迁移版本不一致。
6. 无法启动 Verify。
```

---

## 五、敏感信息脱敏规则

## 5.1 绝对禁止打印

日志中绝对禁止出现：

```text
1. 明文密码。
2. SECRET_KEY。
3. 完整 Access Token。
4. 完整 Refresh Token。
5. Authorization Header。
6. 数据库完整连接串。
7. S3_SECRET_KEY。
8. S3_ACCESS_KEY。
9. Redis 密码。
10. 完整 IMSI。
11. 用户身份证、手机号等未来可能加入的敏感字段。
```

---

## 5.2 允许打印的脱敏信息

允许打印：

```text
user_id
agent_id
admin_id
project_id
game_project_code
client_type
device_fingerprint 后 6 位
token jti 前 8 位
request_id
HTTP status_code
error_code
```

设备指纹建议：

```text
android_device_123456 → ******123456
```

Token 建议：

```text
jti=8f12ab3c...
```

---

## 5.3 登录日志脱敏

登录成功可以记录：

```text
user_id
username 摘要
project_code
client_type
device_fp_tail
ip
```

登录失败可以记录：

```text
username 摘要
fail_reason
project_uuid 是否存在
client_type
ip
```

禁止记录：

```text
password
完整 token
完整 refresh_token
```

---

## 5.4 热更新日志脱敏

允许记录：

```text
project_id
project_code
client_type
version
file_size
checksum_sha256 前 12 位
admin_id
```

禁止记录：

```text
完整签名下载 URL
S3 密钥
内部存储凭据
```

---

## 5.5 点数日志脱敏

允许记录：

```text
agent_id
user_id
project_id
amount
tx_type
balance_type
charge_id
authorization_id
```

不建议打印：

```text
代理真实姓名
财务备注中的敏感内容
```

## 七、健康检查规范

```json
{
  "status": "ok",
  "environment": "...",
  "version": "0.1.0"
```

它能说明：

```text
1. FastAPI 应用能响应。
2. 进程还活着。
3. 路由可以访问。
```

当前 `/health` 不能证明：

```text
1. PostgreSQL 可用。
2. Redis 可用。
3. Celery Worker 可用。
4. Celery Beat 可用。
5. 游戏库可用。
6. 热更新存储可写。
7. 心跳落库正常。
```


---

## 九、告警规则

## 9.1 P0 告警

以下情况必须立即告警：

```text
1. Verify API /health 不通。
2. 主库不可连接。
3. Redis 不可连接。
4. Celery Worker 停止。
5. Celery Beat 停止。
6. 连续 2 个周期没有心跳落库。
7. 登录接口 5xx 激增。
8. 点数扣费事务失败。
9. 返点事务失败。
10. 热更新上传后客户端无法 check 到新版本。
```

---

## 9.2 P1 告警

以下情况应尽快处理：

```text
1. 5xx 持续超过阈值。
2. 429 大量出现。
3. Android 心跳 403 激增。
4. 热更新下载失败激增。
5. Redis 内存超过 80%。
6. PostgreSQL 连接数超过 80%。
7. Nginx 413 出现。
8. Sentry 错误集中爆发。
```

---

## 9.3 P2 告警

以下情况记录并观察：

```text
1. 单个用户登录失败多次。
2. 单个 IP 请求异常频繁。
3. 个别项目心跳落库失败。
4. 参数保存部分失败。
5. 代理项目申请被频繁拒绝。
```

## 十、Sentry 接入规范

## 10.1 当前源码能力

当前 `_setup_sentry()` 行为：

```text
1. SENTRY_DSN 为空时跳过。
2. SENTRY_DSN 不为空时尝试导入 sentry_sdk。
3. 集成 FastAPI。
4. 集成 SQLAlchemy。
5. 设置 environment。
6. traces_sample_rate = 0.1。
```

---

## 10.2 Sentry 应捕获什么

建议捕获：

```text
1. 未处理异常。
2. 500 错误。
3. 数据库异常。
4. Redis 异常。
5. Celery 任务异常。
6. 热更新上传异常。
7. 点数扣费异常。
```

---

## 10.3 Sentry 不应包含什么

必须脱敏：

```text
1. password。
2. access_token。
3. refresh_token。
4. Authorization。
5. SECRET_KEY。
6. DATABASE_URL。
7. S3_SECRET_KEY。
```

建议：

```text
配置 before_send 过滤敏感字段。
```

## 十一、生产配置风险

## 11.1 DEBUG 校验当前待修订

已确认：

```text
app/config.py 中 debug_must_be_false_in_production 当前只 return v。
```

这意味着：

```text
该 validator 名称表达了“生产必须 DEBUG=false”，但当前实现本身没有强制。
```

不过：

```text
app/main.py::_production_safety_check() 会在 ENVIRONMENT=production 且 DEBUG=true 时拒绝启动。
```

### 建议方案
