## 四、运行时总架构

### 4.1 请求分层

```text
客户端
  ├─ 安卓脚本
  ├─ PC 中控
  ├─ 管理后台 Web
  └─ 代理后台 Web
        │
        ▼
FastAPI Router 层
        │
        ▼
Service 业务层
        │
        ├─ 主库 hive_platform
        ├─ 游戏库 hive_{code_name}
        ├─ Redis
        ├─ Celery 定时任务
        └─ Storage 本地/S3
```

### 4.2 主库与游戏库分工

已确认当前实现采用双层数据库设计：

```text
主库 hive_platform:
  管理员
  代理
  用户账号主体
  游戏项目注册表
  用户项目授权
  设备绑定
  登录日志
  代理项目授权
  项目定价
  代理余额与点数流水
  项目准入策略
  代理业务档案

游戏库 hive_{code_name}:
  项目配置
  脚本参数定义
  用户脚本参数值
  设备运行状态
  项目版本记录
```

重要实现细节：

```text
1. database.py 中游戏库 URL 不再直接使用 DATABASE_GAME_PREFIX + code_name。
2. 当前逻辑通过主库连接串替换 database，得到 db_name = "hive_" + code_name。
3. Celery 心跳落库使用 project.db_name 构造连接，避免 hive_game_game_001 重复前缀问题。
```

### 4.3 Redis 分工

Redis 当前承担四类职责：

```text
1. Access Token 黑名单
   Key: token:blacklist:{jti}

2. Refresh Token 主数据
   Key: refresh:{user_id}:{jti}

3. Refresh Token O(1) 反查索引
   Key: rt_lookup:{sha256(rt_value)[:32]}

4. 设备心跳缓存
   Key: device:runtime:{game_id}:{user_id}:{device_fp}
   TTL: 120 秒

5. 简单限流计数
   Key: ratelimit:{endpoint_tag}:{identifier}
```

## 六、鉴权与会话系统

### 6.1 用户端 Token

用户登录成功后返回：

```text
access_token: JWT，默认 15 分钟
refresh_token: 不透明随机字符串，默认 7 天
expires_in: 秒数
user_id
username
user_level
游戏项目 code_name
```

Access Token payload：

```text
sub      用户 ID
level    用户等级
project  游戏项目 code_name
jti      UUID，用于黑名单
 iat     签发时间
exp      过期时间
type     access
```

Refresh Token 不携带业务信息，状态存 Redis。

### 6.2 Refresh Token v2 设计

已确认当前实现修复了早期刷新接口需要 Redis SCAN 的问题。当前 RT 存储写入两个 Key：

```text
refresh:{user_id}:{jti}       保存 RT 详情 JSON
rt_lookup:{hash(rt_value)}    保存 user_id:jti 指针
```

这样客户端刷新时只传 `refresh_token`，服务端可以 O(1) 找到对应主数据。

### 6.3 管理员与代理 Token

管理员和代理登录后使用独立 Token 类型：

```text
Admin Token:
  type = admin
  默认有效期 8 小时
  不放入 Redis 黑名单

Agent Token:
  type = agent
  默认有效期 8 小时
  不放入 Redis 黑名单
```

### 6.4 当前鉴权边界

```text
普通用户接口:
  Depends(get_current_user)
  校验 JWT、Redis 黑名单、用户 active 状态

管理员接口:
  Depends(get_current_admin)
  校验 admin token 与 Admin.status

代理接口:
  Depends(get_current_agent)
  校验 agent token 与 Agent.status

需要游戏库的接口:
  get_game_project_code 从用户 AT 中读取 project 字段
```

### 6.5 风险

用户登录中，设备上限仍使用 `user.user_level` 旧字段判断，而当前新业务口径已把项目内等级放到 `Authorization.user_level`。这会造成潜在不一致：

```text
用户在 A 项目 normal、B 项目 vip
但 login_user 设备上限按 User.user_level，而不是 Authorization.user_level
```

建议后续修正为：

```text
设备绑定上限依据 Authorization.user_level 或 Authorization.authorized_devices
其中 authorized_devices 应成为项目级授权设备数主口径
```

## 七、用户登录验证链

### 7.1 已确认登录流程

`auth_service.login_user()` 当前实现了 10 步验证链：

```text
Step 1  用户名存在
Step 2  密码匹配
Step 3  用户状态 active
Step 4  账号未过期（旧字段 User.expired_at）
Step 5  project_uuid 对应的游戏项目存在且 is_active
Step 6  用户对该项目存在 active 授权
Step 7  授权未过期
Step 8  安卓端设备绑定检查；PC 端不占设备名额
Step 9  写登录日志
Step 10 签发 Access Token 与 Refresh Token
```

### 7.2 设备绑定规则

```text
client_type == android:
  已绑定设备: 更新 last_seen_at
  新设备: 检查设备数量上限后创建 DeviceBinding

client_type == pc:
  不创建设备绑定
  不占用安卓授权设备名额
```

### 7.3 当前失败日志策略

失败登录会通过独立 Session 写 LoginLog，避免主事务回滚导致失败日志丢失。

### 7.4 当前限流策略

Router 层对登录做限流：

```text
登录接口: 同 IP 每分钟最多 10 次
心跳接口: 同用户/Token 每分钟最多 4 次
```

## 八、主库模型

### 8.1 主库表清单

```text
admin                         管理员
agent                         多级代理
user                          用户账号主体
game_project                  游戏项目注册表
agent_project_auth            代理项目授权
authorization                 用户项目授权
authorization_charge          授权扣点快照
device_binding                用户设备绑定
version_record                主库热更新版本记录
login_log                     登录日志
project_price                 项目定价
agent_balance                 代理余额
balance_transaction           点数流水
agent_level_policy            代理业务等级默认策略
project_access_policy         项目准入策略
agent_project_auth_request    代理项目授权申请
agent_project_access_invite   项目邀请可见关系
agent_business_profile        代理业务档案
```

### 8.2 关键模型含义

#### User

用户是账号主体。当前新口径中，用户等级、授权设备数、到期时间不再以 User 为主，而是迁移到 Authorization。

保留兼容字段：

```text
User.user_level
User.max_devices
User.expired_at
```

#### Authorization

用户与游戏项目之间的授权关系。当前新主口径字段：

```text
user_id
game_project_id
user_level
 authorized_devices
valid_from
valid_until
status
```

#### AuthorizationCharge

代理给用户开通项目授权时产生的扣点快照。用于删除用户时按剩余时间返点。

核心字段：

```text
authorization_id
agent_id
user_id
project_id
user_level
unit_price
authorized_devices
billing_period
paid_hours
valid_from
valid_until
original_cost
charged_consumed
credit_consumed
refunded_points
refund_status
```

#### Agent

代理组织树节点。`Agent.level` 表示组织层级/代理树深度，不再用于项目准入业务等级。

#### AgentBusinessProfile

代理业务档案。`tier_level` 表示代理业务等级 Lv.1 - Lv.4，项目准入策略必须基于此字段。

#### AgentProjectAuth

代理对某个项目的销售/授权资格，不是用户项目授权。

#### ProjectAccessPolicy

项目对代理的准入策略，控制可见性、申请、自动开通。

#### AgentBalance / BalanceTransaction

代理点数余额与流水。余额拆成充值点数、授信点数、冻结授信、累计消耗。

## 九、游戏库模型

### 9.1 游戏库表清单

```text
project_config       项目配置
script_param_def     脚本参数定义
user_script_param    用户脚本参数值
device_runtime       设备运行状态
version_record       游戏库热更新版本记录
```

### 9.2 游戏库边界

游戏库不直接存用户账号主体。跨库引用通过 `user_id`、`device_id` 等应用层字段维持，不依赖数据库外键。

### 9.3 设备运行表

`device_runtime` 以 `device_id` 为主键，Celery 落库时使用 PostgreSQL UPSERT：

```text
ON CONFLICT (device_id) DO UPDATE
```

同一设备只保留最新运行状态，不追加历史轨迹。

## 十、用户与授权服务

### 10.1 用户创建

当前 `create_user()` 规则：

```text
1. tester 级用户只能管理员创建。
2. 代理创建用户时受 max_users 限制。
3. 用户名必须唯一。
4. 密码使用 bcrypt hash。
5. 创建用户只创建账号主体。
6. 项目等级、设备数、到期时间在 grant_authorization 中设置。
```

### 10.2 用户列表

当前 `list_users()` 支持：

```text
按状态过滤: User.status
按等级过滤: Authorization.user_level
按项目过滤: Authorization.game_project_id
管理员按创建代理过滤: creator_agent_id
代理只能看自己创建的用户
```

用户列表返回授权明细，包括：

```text
授权项目
项目内用户等级
授权设备数
已激活设备数
未激活设备数
到期时间
是否过期
```

### 10.3 授权创建

`grant_authorization()` 当前规则：

```text
管理员:
  可以给用户授权项目
  可以设置 tester 授权
  可以覆盖已有授权

代理:
  只能给自己创建的用户授权
  只能授权自己已开通的项目
  必须设置 valid_until
  authorized_devices 必须大于 0
  tester 授权禁止代理设置

```

## 十一、点数与计费系统

### 11.1 定价

项目定价按项目与用户等级配置：

```text

```

### 11.2 计费周期

当前代码中存在按等级计费周期规则：

```text
trial   按周/设备
normal  按月/设备
vip     按月/设备
svip    按月/设备
```

### 11.3 扣点规则

代理授权用户项目时计算：

```text
总扣点 = 单价 × 授权设备数 × 计费周期数
```

扣点来源顺序：

```text
先扣 charged_points 充值点数
不足部分扣 credit_points 授信点数
frozen_credit 不可用
```

如果可用点数不足，返回 HTTP 402。

### 11.4 返点规则

删除用户时按 AuthorizationCharge 快照返点：

```text
1. 只处理 refund_status == none 且 valid_until > 删除时间的扣点快照。
2. 已过期授权不返点。
3. 不足 1 小时按 1 小时扣费。
4. 计算已用小时与已用点数。
5. 剩余点数按原始 charged/credit 扣点比例返还。
6. 写 refund 类型流水。
7. 更新 charge.refund_status、refunded_points、refunded_at 等审计字段。
```

### 11.5 风险

计费系统已经很复杂，应尽快建立单独文档：

```text
01-网络验证系统/代理点数计费与返点规则.md
```

并把以下规则锁定：

```text
1. trial/normal/vip/svip 的计费周期。
2. 授权修改是否补扣/退点。
3. 单独撤销授权是否退款。
4. 删除用户返点是否包含管理员免费授权。
5. 删除代理时余额与下级代理如何处理。
```

当前代码中“单独撤销授权不退款，删除用户才返点”已经实现，但需要产品层确认。

## 十六、心跳落库任务

### 16.1 Celery 调度

`app/tasks/heartbeat_flush.py` 实现：

```text
任务名: app.tasks.heartbeat_flush.flush_device_heartbeats
频率: 每 30 秒
失败重试: 最多 3 次，间隔 10 秒
```

### 16.2 落库流程

```text
1. 查询主库所有 active 游戏项目。
2. 对每个项目扫描 Redis: device:runtime:{game_id}:*
3. 构造 device_runtime upsert 记录。
4. 写入对应游戏库。
5. 单项目失败只记录日志，不影响其他项目。
```

### 16.3 NullPool 原因

Celery 使用 `asyncio.run()` 执行异步逻辑，每次会创建/销毁事件循环。普通连接池会缓存绑定旧事件循环的 asyncpg 连接，引发 `Event loop is closed`。当前实现使用 `NullPool` 避免跨事件循环复用连接。
