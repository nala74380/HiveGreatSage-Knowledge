本文用于沉淀 `HiveGreatSage-Verify` 当前测试体系与测试基线。

本文不是“测试已经全部通过”的证明，而是用于明确：

```text
1. 当前 Verify 仓库里已经有哪些测试。
2. 当前测试依赖什么环境。
3. 当前测试 fixture 如何工作。
4. 当前测试结果应该如何采信。
5. 当前哪些测试能代表源码健康。
6. 当前哪些测试不能代表三端真实闭环。
7. 下一轮必须怎么重跑基线。
8. 哪些测试缺口必须补齐。
```

### 2.2 待确认

原因：

```text
1. 本文是源码反向沉淀，不是本机实时运行结果。

```

### 2.3 当前测试口径

后续统一使用以下口径：

```text
源码存在:
  表示 tests/ 中有测试文件或测试函数。

pytest 通过:
  表示当前环境 pytest 无 failed / error。

pytest skip:
  表示该测试未执行成功也未失败，必须说明 skip 原因。

接口测试通过:
  表示 httpx / FastAPI ASGI 层可调用。

环境测试通过:
  表示 PostgreSQL / Redis / Celery 等依赖能正常启动和连接。

三端联调通过:
  表示 PCControl / AndroidScript / Verify 真实链路跑通。

生产验证通过:
  表示部署、压测、监控、备份、回滚、安全要求均已验证。
```

## 三、测试环境约定

### 3.1 当前测试数据库

当前测试直接复用开发库：

```text
主库:
  hive_platform

游戏库:
  hive_game_001
```

当前测试配置中：

```text
GAME_PROJECT_UUID = "00000000-0000-0000-0000-000000000001"
GAME_PROJECT_CODE = "game_001"
ADMIN_USERNAME = "test_admin"
ADMIN_PASSWORD = "TestAdmin@2026!"
```

### 3.2 已知游戏库权限问题

已知问题：

```text
hive_game_001 的 hive_user 认证可能失败。
错误类型可能表现为 InvalidPasswordError 或连接失败。
```

当前认定：

```text
这通常是 PostgreSQL 权限 / 密码 / 数据库初始化问题，不直接等同于 Python 代码 bug。
```

修复方式：

```bash
psql -U postgres -p 15432 -f scripts/setup_game_db.sql
```

### 3.3 游戏库不可访问时的处理

当前 `game_db_accessible` fixture 的设计：

```text
1. session 级别检测 hive_game_001 是否可访问。
2. 返回 True / False。
3. 不抛出异常。
4. 依赖游戏库的 fixture 如果发现不可访问，则 pytest.skip。
5. 这样 pytest 结果是 SKIP，而不是 ERROR。
```

这是一种测试降级策略。

注意：

```text
SKIP 只能说明测试环境缺失，不说明业务通过。
```

### 3.4 Redis 依赖

当前测试 client fixture 会覆盖 `get_redis`，并从 `settings.REDIS_URL` 创建 Redis 连接。

因此当前 pytest 仍依赖 Redis 可访问。

默认环境说明来自测试报告生成器：

```text
environment: development
PostgreSQL: 15432
Redis: 16379
```

### 3.5 Windows 事件循环处理

当前测试针对 Windows 做了两层事件循环处理：

```text
层1:
  模块顶部设置 WindowsSelectorEventLoopPolicy。

层2:
  显式 event_loop fixture，scope=session，同步 def。
```

目的：

```text
解决 pytest-asyncio / asyncpg / Windows ProactorEventLoop 相关问题。
```

脚本：

```text

```

运行命令：

```bash
python scripts/run_tests.py
```

该脚本会执行：

```bash
python -m pytest tests/ -v --tb=short --json-report --json-report-file=logs/test_result_raw.json --json-report-indent=2 -q
```

生成文件：

```text
logs/test_result_raw.json
logs/test_report_YYYYMMDD_HHMMSS.md
```

报告保留策略：

```text
只保留最近 3 份 test_report_*.md。
更早报告自动删除。
```

### 4.3 手动运行 pytest

也可直接运行：

```bash
python -m pytest tests/ -v --tb=short
```

仅运行某个文件：

```bash
python -m pytest tests/test_auth.py -v --tb=short
```

仅运行某个类：

```bash
python -m pytest tests/test_auth.py::TestLogin -v --tb=short
```

仅运行某个测试：

```bash
python -m pytest tests/test_auth.py::TestLogin::test_login_success -v --tb=short
```

## 五、测试 fixture 设计

## 5.1 `test_engine`

职责：

```text
创建主库 hive_platform 的 SQLAlchemy async engine。
使用 NullPool。
启动时执行 SELECT 1 确认主库可连接。
```

风险：

```text
如果主库不可连接，pytest 会直接失败。
```

## 5.2 `session_factory`

职责：

```text
基于 test_engine 创建主库 async_sessionmaker。
```

## 5.3 `game_test_engine`

职责：

```text
创建 hive_game_001 游戏库 engine。
不在创建时检测连通性。
```

原因：

```text
避免游戏库不可访问时导致所有测试 setup error。
```

## 5.4 `game_session_factory`

职责：

```text
基于 game_test_engine 创建游戏库 async_sessionmaker。
```

## 5.5 `game_db_accessible`

职责：

```text
一次性检测 hive_game_001 是否可访问。
返回 True / False。
不抛出异常。
```

使用原则：

```text
所有需要游戏库的 fixture 都必须先检查 game_db_accessible。
如果为 False，使用 pytest.skip。
```

## 5.6 `seed_data`

职责：

```text
幂等初始化测试管理员和测试游戏项目。
```

写入或修正：

```text
admin:
  username = test_admin
  password = TestAdmin@2026!
  status = active

game_project:
  project_uuid = 00000000-0000-0000-0000-000000000001
  code_name = game_001
  display_name = 测试游戏 001
  db_name = hive_game_001
  is_active = true
```

## 5.7 `client`

职责：

```text
使用 httpx.AsyncClient 直连 FastAPI ASGI app。
```

client fixture 做四层 patch：

```text
1. FastAPI dependency_overrides:
   覆盖 get_main_db / get_redis。

2. monkey-patch 主库 session factory:
   替换 app.database._main_session_factory。
   替换 auth_service._main_session_factory。

3. monkey-patch 限流:
   替换 auth_router.incr_rate_limit，避免测试受登录限流影响。

4. 注入游戏库:
   写入 app.database._game_engines[GAME_PROJECT_CODE]
   写入 app.database._game_session_factories[GAME_PROJECT_CODE]
```

测试结束后恢复：

```text
1. 清空 dependency_overrides。
2. 恢复主库 session factory。
3. 恢复 auth_service session factory。
4. 恢复限流函数。
5. 恢复游戏库 engine / session factory 缓存。
```

## 5.8 `admin_token`

职责：

```text
基于 seed_data 中的 admin_id 创建 Admin Token。
```

## 5.9 `admin_headers`

职责：

```text
返回:
  {"Authorization": "Bearer <admin_token>"}
```

## 5.10 `project_id`

职责：

```text
返回 seed_data 中测试项目 game_001 的 project_id。
```

## 十一、当前测试风险

### R001 SKIP 被误判为通过

风险：

```text
game_db_accessible=False 时，依赖游戏库的测试会 SKIP。
```

影响：

```text
测试报告可能显示 failed=0/error=0，但实际关键游戏库逻辑没有跑。
```

治理：

```text
1. 基线报告必须单独列 skipped 数量。
2. skipped > 0 时，必须说明原因。
3. 如果 skip 原因是游戏库不可访问，基线不能标记为完整通过。
```

---

### R002 使用开发库跑测试

风险：

```text
测试直接复用 hive_platform + hive_game_001 开发库。
```

影响：

```text
1. 可能污染开发数据。
2. 历史数据可能影响测试结果。
3. 并发运行测试有风险。
```

当前缓解：

```text
1. 测试用户名使用前缀和 UUID。
2. 部分测试有 teardown。
3. seed_data 幂等。
```

建议方案：

```text
中期改为独立测试库:
  hive_platform_test
  hive_game_test_001
```

---

### R003 Redis TTL 导致设备监控结果不稳定

风险：

```text
设备监控依赖 Redis 中在线数据，TTL=120s。
```

影响：

```text
跨 session 运行时，设备列表可能为空。
```

当前缓解：

```text
test_device_admin.py 只断言响应结构，不断言设备数量。
```

建议：

```text
1. 单独测试 Redis 在线态。
2. 单独测试 DB fallback。
3. 设备监控测试中主动制造心跳数据。
```

---

### R004 Celery 未纳入完整 pytest

风险：

```text
设备心跳写 Redis 后，是否由 Celery 批量落库未被完整覆盖。
```

影响：

```text
pytest 通过不等于心跳持久化链路通过。
```

建议：

```text
1. 单独写 Celery 任务单元测试。
2. 单独写 Redis → Celery → device_runtime 集成测试。
3. 手动启动 Worker/Beat 做环境级测试。
```

---

### R005 热更新测试与当前实现可能漂移

风险：

```text
热更新测试中可能仍依赖游戏库 version_record。
但当前热更新管理 V2 可能已经倾向主库 version_record。
```

影响：

```text
测试通过也可能覆盖的是旧路径，或测试失败来自测试落后于源码。
```

建议：

```text
1. 明确当前唯一版本真相源。
2. 同步 test_update.py。
3. 同步 test_update_admin.py。
4. 删除旧游戏库 version_record 依赖前先补迁移说明。
```
