本文用于沉淀 `HiveGreatSage-Verify` 当前 **代理点数计费、授权扣点、删除用户返点、余额流水** 的业务规则。

```text

4. 扣点优先从哪里扣？
5. 为什么要保存 AuthorizationCharge 快照？
6. 删除用户时如何返点？
7. 什么情况不返点？

## 二、当前结论

### 2.1 已确认

历史版本 Verify 曾实现点数系统的核心结构：

```text
1. ProjectPrice
   项目定价表。

2. AgentBalance
   代理余额表。

3. BalanceTransaction
   点数流水表。

4. AuthorizationCharge
   授权扣点快照表。

5. balance_service.py
   点数与返点核心服务层。

6. balance_admin.py
   管理员点数与定价接口。

7. balance_agent.py
   代理查看余额、流水、目录接口。

8. users.py / user_service.py
   用户授权时调用扣点逻辑，删除用户时调用返点逻辑。
```

历史点数体系已明确能力：

```text
1. 管理员设置项目定价。
2. 管理员给代理充值。
3. 管理员给代理授信。
4. 管理员冻结授信。
5. 管理员解冻授信。
6. 代理创建或授权用户时扣点。
7. 授权扣点时保存扣点快照。
8. 删除用户时按剩余未使用时间返点。
9. 点数流水返回完整业务上下文。
```

### 2.2 待确认 / 当前口径

当前不能直接按本文旧对象标记为“财务规则已验证”。

原因：

```text
1. 当前测试文件为 tests/test_accounting.py，而不是 tests/test_balance.py。
2. 充值、授信、冻结、解冻、授权扣点、删除用户返点已有基础自动化覆盖。
3. 并发扣点、删除单个授权返点、事务失败回滚仍需补充或实测。
4. 前端账务页面仍需浏览器联调。
5. 本文旧对象名需要逐步迁移为 AccountingWallet / AccountingLedgerEntry / AccountingDocument / AuthorizationChargeSnapshot。
```

### 2.3 当前风险等级

点数系统属于：

```text
P1 高风险模块。
```

原因：

```text
1. 涉及代理商业结算。
2. 涉及充值余额与授信额度。
3. 涉及用户授权扣费。
4. 涉及删除用户后的剩余时间返点。
5. 一旦出错会直接造成点数损失、代理纠纷和账务不一致。
```

## 三、术语定义

## 3.1 点数

点数是 Verify 平台内代理用于给用户开通项目授权的计费单位。

本文中：

```text
点数 ≠ 人民币。
点数 ≠ 游戏币。
点数 ≠ 设备数。
```

点数只表示平台内部的授权计费余额。

---

## 3.2 充值点数 charged_points

充值点数是管理员给代理充值的实际可消费点数。

来源：

```text
管理员执行:
POST /admin/api/agents/{agent_id}/recharge
```

特点：

```text
1. 属于代理实际可用余额。
2. 授权扣点时优先消耗。
3. 删除用户返点时，按原扣点来源比例返还。
4. 流水 balance_type = charged。
```

---

## 3.3 授信点数 credit_points

授信点数是管理员给代理开放的信用额度。

来源：

```text
管理员执行:
POST /admin/api/agents/{agent_id}/credit
```

特点：

```text
1. 属于代理可用额度。
2. 可被冻结。
3. 授权扣点时，在充值点数不足后消耗。
4. 删除用户返点时，按原扣点来源比例返还。
5. 流水 balance_type = credit。
```

---

## 3.4 冻结授信 frozen_credit

冻结授信是管理员临时冻结的授信额度。

来源：

```text
管理员执行:
POST /admin/api/agents/{agent_id}/freeze
```

解冻：

```text
管理员执行:
POST /admin/api/agents/{agent_id}/unfreeze
```

计算：

```text
可用授信 = credit_points - frozen_credit
```

---

## 3.5 可用总点数 available_total

计算：

```text
available_credit = credit_points - frozen_credit
available_total = charged_points + available_credit
```

授权扣点时检查：

```text
total_cost <= available_total
```

如果不满足：

```text
返回 402 Payment Required。
```

---

## 3.6 项目定价 ProjectPrice

项目定价表示：

```text
某项目某用户等级，每台设备每个计费周期需要多少点数。
```

当前定价维度：

```text
project_id
user_level
points_per_device
```

当前支持售卖等级：

```text
trial
normal
vip
svip
```

当前不进入代理售卖定价体系：

```text
tester
```

---

## 3.7 授权扣点快照 AuthorizationCharge

授权扣点快照是用户授权时保存的一次性财务快照。

保存内容包括：

```text
1. authorization_id。
2. agent_id。
3. user_id。
4. project_id。
5. user_level。
6. unit_price。
7. authorized_devices。
8. billing_period。
9. billing_period_hours。
10. period_count。
11. paid_hours。
12. valid_from。
13. valid_until。
14. original_cost。
15. charged_consumed。
16. credit_consumed。
17. refunded_points。
18. refund_status。
19. last_refund_*。
```

用途：

```text
1. 记录当时扣点时的价格。
2. 记录当时购买的设备数。
3. 记录当时授权时长。
4. 记录扣点来源：充值点数 / 授信点数。
5. 删除用户时按剩余时间返点。
6. 点数流水展示完整业务上下文。
```

## 四、数据模型边界

## 4.1 ProjectPrice

表名：

```text
project_price
```

职责：

```text
保存项目级定价。
```

唯一约束：

```text
project_id + user_level
```

字段：

```text
project_id
user_level
points_per_device
created_at
updated_at
```

设计口径：

```text
1. 修改 ProjectPrice 只影响后续授权。
2. 已经产生的 AuthorizationCharge 不受后续价格修改影响。
3. 返点必须按 AuthorizationCharge 中保存的 unit_price / original_cost 计算。
```

---

## 4.2 AgentBalance

表名：

```text
agent_balance
```

职责：

```text
保存代理当前点数余额状态。
```

核心字段：

```text
agent_id
charged_points
credit_points
frozen_credit
total_consumed
updated_at
```

计算字段：

```text
available_credit = credit_points - frozen_credit
available_total = charged_points + available_credit
```

设计口径：

```text
1. 每个代理只能有一条余额记录。
2. 如果不存在，服务层会自动创建。
3. 余额变化必须同时写 BalanceTransaction。
```

---

## 4.3 BalanceTransaction

表名：

```text
balance_transaction
```

职责：

```text
保存每一次点数变动流水。
```

核心字段：

```text
agent_id
tx_type
balance_type
amount
balance_before
balance_after
description
operated_by_admin_id
related_user_id
related_project_id
related_charge_id
created_at
```

tx_type：

```text
recharge    充值
credit      授信
consume     消费
refund      返点
freeze      冻结
unfreeze    解冻
adjust      调整
```

balance_type：

```text
charged     充值点数
credit      授信点数
```

amount 方向：

```text
正数:
  增加余额或增加冻结额度。

负数:
  减少余额或解冻冻结额度。
```

当前示例：

```text
充值:
  tx_type = recharge
  balance_type = charged
  amount > 0

授信:
  tx_type = credit
  balance_type = credit
  amount > 0

消费充值点数:
  tx_type = consume
  balance_type = charged
  amount < 0

消费授信点数:
  tx_type = consume
  balance_type = credit
  amount < 0

返点到充值点数:
  tx_type = refund
  balance_type = charged
  amount > 0

返点到授信点数:
  tx_type = refund
  balance_type = credit
  amount > 0

冻结授信:
  tx_type = freeze
  balance_type = credit
  amount > 0

解冻授信:
  tx_type = unfreeze
  balance_type = credit
  amount < 0
```

---

## 4.4 AuthorizationCharge

表名：

```text
authorization_charge
```

职责：

```text
保存授权扣点快照与返点状态。
```

关键约束：

```text
user_level IN ('trial', 'normal', 'vip', 'svip')
authorized_devices > 0
billing_period_hours > 0
period_count > 0
paid_hours > 0
original_cost >= 0
charged_consumed >= 0
credit_consumed >= 0
refunded_points >= 0
refund_status IN ('none', 'refunded', 'no_refund')
```

索引：

```text
authorization_id
agent_id + created_at
user_id + project_id
refund_status
```

返点状态：

```text
none:
  尚未返点。

refunded:
  已返点。

no_refund:
  经计算不需要返点。
```

## 五、计费周期规则

## 5.1 当前用户等级

当前可售卖等级：

```text
trial
normal
vip
svip
```

当前不售卖等级：

```text
tester
```

原因：

```text
tester 是测试用户等级，应由管理员控制，不进入代理售卖定价体系。
```

---

## 5.2 计费周期

当前源码规则：

```text
trial:
  按周计费。
  1 周 = 168 小时。
  unit_label = 每周/台。

normal:
  按月计费。
  1 月 = 720 小时。
  unit_label = 每月/台。

vip:
  按月计费。
  1 月 = 720 小时。
  unit_label = 每月/台。

svip:
  按月计费。
  1 月 = 720 小时。
  unit_label = 每月/台。
```

---

## 5.3 周期数量计算

计算逻辑：

```text
raw_hours = ceil((valid_until - valid_from) / 3600)

trial:
  period_count = ceil(raw_hours / 168)

normal/vip/svip:
  period_count = ceil(raw_hours / 720)
```

最小周期：

```text
至少 1 个周期。
```

示例：

```text
trial 授权 1 小时:
  period_count = 1 周。

normal 授权 1 小时:
  period_count = 1 月。

normal 授权 721 小时:
  period_count = 2 月。
```

注意：

```text
这不是按小时卖，而是按周/月周期卖。
```

---

## 5.4 总扣点计算

公式：

```text
total_cost = unit_price × authorized_devices × period_count
```

其中：

```text
unit_price:
  ProjectPrice.points_per_device。

authorized_devices:
  Authorization.authorized_devices。

period_count:
  按授权时长向上取整后的周期数。
```

示例：

```text
项目 normal 单价 = 100 点 / 每月 / 台
授权设备数 = 3
授权时长 = 30 天
period_count = 1

total_cost = 100 × 3 × 1 = 300 点
```

示例：

```text
项目 vip 单价 = 300 点 / 每月 / 台
授权设备数 = 2
授权时长 = 31 天
period_count = 2

total_cost = 300 × 2 × 2 = 1200 点
```

## 六、授权扣点规则

## 6.1 扣点触发时机

代理给用户开通项目授权时触发。

典型接口：

```text
POST /api/users/{user_id}/authorizations
```

可能也包括：

```text
PATCH /api/users/{user_id}/authorizations/{auth_id}
```

待确认：

```text
当前 update_auth 是否完全复用扣点快照逻辑，需要单独测试。
```

---

## 6.2 扣点前置条件

必须满足：

```text
1. 用户存在。
2. 项目存在。
3. 代理存在。
4. 代理有该项目准入或项目授权。
5. 项目已设置该 user_level 定价。
6. user_level 属于 trial/normal/vip/svip。
7. authorized_devices > 0。
8. valid_until 晚于 valid_from。
9. 代理可用总点数足够。
```

不满足时：

```text
1. 未设置定价 → 400。
2. 设备数 <= 0 → 400。
3. 到期时间不合法 → 400。
4. 点数不足 → 402。
5. tester 级别走管理员测试授权逻辑，不进入代理扣点。
```

---

## 6.3 扣点顺序

当前源码扣点顺序：

```text
1. 优先扣 charged_points。
2. charged_points 不足时，再扣 credit_points。
```

具体流程：

```text
remaining = total_cost

如果 charged_points > 0:
  consume_charged = min(charged_points, remaining)
  charged_points -= consume_charged
  remaining -= consume_charged
  写一条 consume / charged 流水

如果 remaining > 0:
  credit_points -= remaining
  写一条 consume / credit 流水
```

冻结授信影响：

```text
可用授信 = credit_points - frozen_credit
```

余额检查：

```text
charged_points + available_credit >= total_cost
```

---

## 6.4 扣点快照

扣点时必须保存 AuthorizationCharge。

原因：

```text
ProjectPrice 以后可能变化。
如果不保存快照，删除用户返点时就无法知道当时按什么价格扣点。
```

快照必须保存：

```text
1. 当时单价。
2. 当时设备数。
3. 当时购买周期。
4. 当时购买小时数。
5. 当时有效期。
6. 当时总扣点。
7. 当时从充值点数扣了多少。
8. 当时从授信点数扣了多少。
```

## 七、返点规则

## 7.1 返点触发时机

当前已确认：

```text
删除用户时会触发返点。
```

典型接口：

```text
DELETE /api/users/{user_id}
```

待确认：

```text
1. 删除单个授权 DELETE /api/users/{user_id}/authorizations/{auth_id} 是否也触发同等返点。
2. 授权延期或改设备数是否创建新的 AuthorizationCharge。
3. 授权降级是否需要部分返点。
```

---

## 7.2 返点适用范围

只处理：

```text
1. 存在 AuthorizationCharge 的代理扣点授权。
2. refund_status = none。
3. valid_until > delete_time。
```

不返点：

```text
1. 已过期授权。
2. 管理员免费授权。
3. original_cost <= 0。
4. paid_hours <= 0。
5. 已经返点过。
6. 经计算可返点数 <= 0。
```

---

## 7.3 已使用小时计算

规则：

```text
不足 1 小时按 1 小时计算。
```

计算：

```text
raw_used_hours = ceil((delete_time - valid_from) / 3600)
used_hours = min(max(raw_used_hours, 1), paid_hours)
```

示例：

```text
授权后 10 分钟删除:
  used_hours = 1

授权后 2.3 小时删除:
  used_hours = 3

授权已经超过购买时长:
  used_hours = paid_hours
```

---

## 7.4 每小时成本

公式：

```text
hourly_cost = original_cost / paid_hours
```

其中：

```text
original_cost:
  授权时总扣点。

paid_hours:
  授权时购买总小时数。
```

---

## 7.5 已使用成本

公式：

```text
used_cost = hourly_cost × used_hours
```

四舍五入：

```text
当前 _money 使用 Decimal("0.01")，ROUND_HALF_UP。
```

---

## 7.6 可返点数

公式：

```text
refund_points = original_cost - used_cost - already_refunded
```

如果：

```text
refund_points <= 0
```

则：

```text
不返点，refund_status = no_refund。
```

---

## 7.7 返点来源比例

返点必须按原扣点来源比例返还。

原扣点来源：

```text
charged_consumed
credit_consumed
```

计算：

```text
total_base = charged_consumed + credit_consumed

refund_charged = refund_points × charged_consumed / total_base
refund_credit = refund_points - refund_charged
```

示例：

```text
原扣点 100 点:
  charged_consumed = 60
  credit_consumed = 40

可返点 = 50

refund_charged = 50 × 60 / 100 = 30
refund_credit = 20
```

返还结果：

```text
charged_points += refund_charged
credit_points += refund_credit
```

并分别写入流水：

```text
refund / charged
refund / credit
```

---

## 7.8 返点状态更新

返点后必须更新：

```text
charge.refunded_points
charge.refund_status
charge.refunded_at
charge.last_refund_paid_hours
charge.last_refund_used_hours
charge.last_refund_used_cost
charge.last_refund_points
```

状态：

```text
有返点:
  refund_status = refunded

无返点:
  refund_status = no_refund
```

风险：

```text

```

结果可能：

```text
1. 余额透支。
2. 两个授权都成功。
3. 流水 balance_before / balance_after 不一致。
```

当前待确认：

```text
是否使用 SELECT FOR UPDATE 或其他行锁机制锁定 AgentBalance。
```

建议方案：

```text
1. 扣点时对 AgentBalance 行加 FOR UPDATE。
2. 所有余额修改必须在同一事务内完成。
3. 添加并发扣点测试。
```

测试：

```text
□ 代理可用余额只够一个授权。
□ 并发发起两个授权。

```

建议：

## D026 — Verify 代理点数计费采用授权扣点快照作为财务真相

### 背景

代理给用户开通授权时，项目定价可能在未来变更。如果删除用户返点时直接读取当前 ProjectPrice，会造成历史扣点和返点计算不一致。

### 决策

```text
1. 保证历史扣点可追溯。
2. 价格变更不影响旧授权返点。
3. 删除用户时可按剩余时长精确返点。
