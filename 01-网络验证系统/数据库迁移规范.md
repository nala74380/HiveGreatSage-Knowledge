# Verify 数据库迁移规范

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/数据库设计]]

## 一、文档目的

本文用于规范 `HiveGreatSage-Verify` 后续所有数据库结构变更。

本文重点解决：

```text
1. 哪些表由 Alembic 管理？
2. 哪些表不由 Alembic 管理？
3. 主库 hive_platform 和游戏库 hive_game_xxx 的迁移边界是什么？
4. 新增表、新增字段、回填数据、加约束应该怎么做？
5. 为什么不能直接手工改库？
6. 为什么不能只改 ORM 模型不写迁移？
7. 为什么不能把游戏库表误加入主库 Alembic？
8. 如何验证空库迁移能跑通？
9. 如何验证现有库升级不会破坏数据？
10. 后续 device_binding、token_version 等变更应该按什么迁移节奏推进？
```

---

## 二、当前总判断

### 2.1 已确认

当前 Verify 采用：

```text
主库:
  hive_platform

游戏库:
  hive_game_001
  hive_game_xxx
```

当前主库迁移方式：

```text
Alembic
```

当前游戏库建表方式：

```text
GameBase.metadata.create_all()
```

当前已确认：

```text
1. migrations/env.py 只迁移主库 Base.metadata。
2. migrations/env.py 明确不导入 GameBase。
3. 游戏库表不经过 Alembic。
4. 游戏项目数据库由 scripts/setup_game_db.py 创建。
5. 游戏库名称规则是 hive_{code_name}。
6. 游戏库建表使用 GameBase.metadata.create_all。
7. 游戏库权限需要授予 hive_user。
```

### 2.2 待确认

以下内容需要继续验证：

```text
1. 当前 Alembic heads 是否只有一个。
2. 当前 alembic upgrade head 是否能在空主库跑通。
3. 当前主库所有 ORM 模型是否都被 migrations/env.py 注册。
4. 当前新增的分模块模型是否全部进入 Base.metadata。
5. 当前游戏库 create_all 是否能在空游戏库跑通。
6. 当前迁移脚本 downgrade 是否真实可用。
7. 当前生产库是否存在手工改表导致的结构漂移。
8. 当前测试库和开发库是否和迁移脚本完全一致。
```

### 2.3 当前风险等级

数据库迁移属于：

```text
P0 阻塞级基础设施。
```

原因：

```text
1. Verify 后端、管理后台、PCControl、AndroidScript 都依赖数据库契约。
2. 数据库结构不一致会导致接口运行时崩溃。
3. 空库不能迁移成功，就不能部署新环境。
4. 现有库不能平滑升级，就不能安全迭代。
5. 多人协作下手工改库会造成不可复现问题。
```

## 三、迁移边界

## 3.1 主库 hive_platform

主库职责：

```text
1. 管理员。
2. 代理。
3. 用户账号。
4. 游戏项目。
5. 用户项目授权。
6. 代理项目授权。
7. 设备绑定。
8. 登录日志。
9. Refresh Token 关联逻辑。
10. 热更新版本记录。
11. 项目定价。
12. 代理余额。
13. 点数流水。
14. 授权扣点快照。
15. 项目准入策略。
16. 代理业务画像。
17. 后续 token_version。
18. 后续 device_binding.game_project_id。
```

主库必须由：

```text
Alembic
```

管理。

禁止：

```text
1. 手工在数据库里 CREATE TABLE 后不写迁移。
2. 只改 ORM 模型不写迁移。
3. 修改迁移历史文件后不重新验证空库。
4. 在生产库直接 ALTER TABLE 而不提交 migration。
```

---

## 3.2 游戏库 hive_game_xxx

游戏库职责：

```text
1. 脚本参数定义。
2. 用户脚本参数。
3. 设备运行状态。
4. 游戏侧统计数据。
5. 游戏项目私有运行数据。
```

当前游戏库建表方式：

```text
GameBase.metadata.create_all()
```

当前游戏库不由：

```text
Alembic
```

管理。

这意味着：

```text
1. 游戏库结构变化不会被 alembic upgrade head 自动处理。
2. 新游戏库创建时会按当前 GameBase 创建最新表。
3. 老游戏库结构升级需要单独治理。
```

---

## 3.3 主库和游戏库不能混用

禁止：

```text
1. 把游戏运行数据表放到主库 Alembic。
2. 把平台治理表放到游戏库 create_all。
3. 让一个表同时存在主库和游戏库并都作为真相源。
4. 主库 migration 中引用 GameBase.metadata。
5. 游戏库 create_all 中创建 Admin / User / Agent 等平台表。
```

## 四、当前迁移链现状

## 4.1 已知迁移主题

当前已确认部分迁移主题包括：

```text
0007_project_price_agent_balance:
  新增项目定价、代理余额、点数流水。

0008_authorization_level_devices:
  将用户等级、授权设备数迁移到 Authorization 项目授权维度。

0009_authorization_charge_refund:
  新增授权扣点快照，支持删除用户返点。

0010_agent_project_access_policy:
  新增项目准入策略、申请、邀请，并增强 agent_project_auth 溯源字段。

0011_agent_business_profile:
  新增代理业务画像，区分业务等级 tier_level 与组织层级 Agent.level。
```

### 4.1.1 迁移链价值

当前迁移脚本已经不只是“建表脚本”。

它们同时承担：

```text
1. 数据库结构变更。
2. 业务口径说明。
3. 历史数据回填。
4. 约束声明。
5. 风险说明。
6. 回退说明。
```

这是一种正确方向。

后续迁移也应保持这种风格。

---

## 4.2 当前迁移链高风险点

### R001 env.py 模型导入可能滞后

当前 `migrations/env.py` 已确认只导入部分主库模型。

如果后续 ORM 模型被拆到：

```text
app/models/main/project_access.py
app/models/main/balance.py
app/models/main/agent_profile.py
```

但没有在 env.py 或 app.models.main.__init__ 中确保导入，则：

```text
Base.metadata 可能不包含新模型。
```

影响：

```text
alembic revision --autogenerate 可能漏表、漏字段、漏约束。
```

建议：

```text
1. env.py 不应只导入旧核心模型。
2. 应统一从 app.models.main 导入所有主库模型。
3. app.models.main.__init__ 必须导出所有模型。
4. 每次新增模型后，运行 metadata 检查脚本。
```

---

### R002 游戏库没有迁移版本表

当前游戏库通过：

```text
GameBase.metadata.create_all()
```

建表。

优点：

```text
新建游戏库简单。
```

风险：

```text
1. 老游戏库不会自动升级。
2. create_all 不会修改已有字段。
3. create_all 不会删除废弃列。
4. create_all 不会新增约束到已存在表。
5. 多游戏库结构可能逐渐不一致。
```

建议：

```text
短期:
  游戏库结构保持稳定，变更必须极少。

中期:
  建立游戏库结构版本表 game_schema_version。

长期:
  如游戏库结构频繁变化，应引入独立 Alembic 或迁移脚本体系。
```

---

### R003 关键字表名需要双引号

当前已知：

```text
user
authorization
```

在 PostgreSQL 中属于敏感命名语义。

迁移中直接写 SQL 时必须使用：

```sql
"user"
"authorization"
```

错误写法：

```sql
UPDATE authorization ...
UPDATE user ...
```

正确写法：

```sql
UPDATE "authorization" ...
UPDATE "user" ...
```

---

### R004 迁移脚本不能只依赖 ORM 当前状态

迁移脚本必须是历史快照。

禁止：

```text
在迁移脚本里 import 当前业务模型后直接操作 ORM。
```

原因：

```text
未来 ORM 变化后，旧迁移可能无法运行。
```

建议：

```text
1. 结构迁移使用 op.create_table / op.add_column。
2. 数据回填使用 op.execute SQL。
3. 不在 migration 中依赖 service 层。
4. 不在 migration 中依赖当前 Pydantic Schema。
```

## 五、迁移文件规范

## 5.1 文件命名

建议格式：

```text
NNNN_short_topic.py
```

示例：

```text
0012_user_token_version.py
0013_device_binding_project_id.py
0014_update_version_record_main_source.py
```

命名要求：

```text
1. 使用连续编号。
2. 文件名必须表达业务主题。
3. 不要使用无意义名称。
4. 不要只写 add_column。
```

---

## 5.2 文件头规范

每个迁移文件建议包含：

```python
r"""
文件位置: migrations/versions/0012_user_token_version.py
文件名称: 0012_user_token_version.py
作者: 蜂巢·大圣 (Hive-GreatSage)
日期/时间: 2026-04-29
版本: V1.0.0
功能说明:
    为 user 表新增 token_version，实现用户全设备即时踢出。

设计原因:
    当前 revoke-all 只能删除 Refresh Token，无法让所有已签发 Access Token 即时失效。
    token_version 可让旧 Access Token 在鉴权阶段立即失效。

安全口径:
    - logout 不递增 token_version。
    - revoke-all 递增 token_version。
    - 旧 Token 未携带 token_version 时按 0 兼容。

回退说明:
    downgrade 会删除 token_version 字段。
    回退前必须确认没有业务依赖该字段。
"""
```

---

## 5.3 revision 信息

必须包含：

```python
revision = "0012_user_token_version"
down_revision = "0011_agent_business_profile"
branch_labels = None
depends_on = None
```

要求：

```text
1. down_revision 必须指向当前 head。
2. 不允许随意产生多个 head。
3. 如确实产生多 head，必须明确 merge migration。
```

---

## 5.4 upgrade / downgrade

必须同时写：

```python
def upgrade() -> None:
    ...

def downgrade() -> None:
    ...
```

要求：

```text
1. upgrade 可安全执行一次。
2. downgrade 至少能撤销结构变更。
3. 数据破坏性 downgrade 必须在注释中说明。
4. downgrade 不能留下孤儿约束或索引。
```

## 六、字段迁移规范

## 6.1 新增普通字段

如果字段允许为空：

```python
op.add_column(
    "table_name",
    sa.Column("new_field", sa.String(length=64), nullable=True),
)
```

如果字段不允许为空，不能直接：

```python
nullable=False
```

除非：

```text
1. 表为空。
2. 有 server_default。
3. 有回填。
```

推荐三步：

```text
Step 1:
  add nullable=True 或带 server_default。

Step 2:
  op.execute 回填旧数据。

Step 3:
  alter nullable=False。
```

---

## 6.2 新增非空字段推荐模式

示例：

```python
op.add_column(
    "user",
    sa.Column(
        "token_version",
        sa.Integer(),
        nullable=False,
        server_default="0",
        comment="用户 Token 版本号，用于全设备即时踢出",
    ),
)
```

适合：

```text
默认值清晰且所有旧数据都可用默认值。
```

---

## 6.3 新增需要复杂回填的字段

例如：

```text
device_binding.game_project_id
```

不应直接：

```text
nullable=False
```

推荐：

```text
Phase A:
  nullable=True 新增字段。

Phase B:
  按单游戏或 LoginLog 回填。

Phase C:
  代码双读 / 双写。

Phase D:
  确认无 NULL。

Phase E:
  nullable=False + 新唯一约束。
```

---

## 6.4 删除字段规范

删除字段必须满足：

```text
1. 字段已经不被任何代码读取。
2. 字段已经不被任何前端展示。
3. 字段已经不被任何测试依赖。
4. 字段有备份或可重建。
5. 已经过至少一个版本的废弃期。
```

禁止：

```text
刚迁移到新字段就立刻删除旧字段。
```

正确做法：

```text
1. 新字段上线。
2. 回填旧数据。
3. 业务切新字段。
4. 旧字段只读兼容。
5. 文档标记 deprecated。
6. 下一阶段再删除。
```

## 七、约束与索引规范

## 7.1 CheckConstraint

涉及枚举值必须加 CHECK。

示例：

```text
user_level IN ('trial', 'normal', 'vip', 'svip', 'tester')
```

好处：

```text
1. 防止脏数据进入数据库。
2. 防止绕过接口直接写错值。
3. 让数据库结构自解释。
```

---

## 7.2 UniqueConstraint

唯一约束必须表达业务唯一性。

示例：

```text
project_price:
  project_id + user_level

agent_project_access_invite:
  agent_id + project_id

device_binding 新口径:
  user_id + game_project_id + device_fingerprint
```

禁止：

```text
1. 仅靠 service 层防重复。
2. 只在前端防重复。
3. 业务唯一关系没有数据库约束。
```

---

## 7.3 部分唯一索引

适合状态型唯一规则。

示例：

```text
同一代理同一项目只能有一个 pending 申请。
```

建议使用：

```python
op.create_index(
    "uq_agent_project_pending_request",
    "agent_project_auth_request",
    ["agent_id", "project_id"],
    unique=True,
    postgresql_where=sa.text("status = 'pending'"),
)
```

---

## 7.4 外键 ondelete 规则

常用规则：

```text
CASCADE:
  父对象删除后子对象无意义。

SET NULL:
  子对象应保留审计记录，但引用对象可能删除。

RESTRICT:
  不允许删除被引用对象，保护审计数据。
```

建议：

```text
财务、流水、审计类数据优先 RESTRICT 或 SET NULL。
临时过程表可根据业务使用 CASCADE。
```

---

## 八、数据回填规范

## 8.1 回填必须幂等

回填 SQL 应尽量可重复执行。

示例：

```sql
UPDATE "authorization" AS a
SET
    user_level = COALESCE(u.user_level, 'normal'),
    authorized_devices = COALESCE(u.max_devices, 20),
    valid_until = COALESCE(a.valid_until, u.expired_at)
FROM "user" AS u
WHERE a.user_id = u.id;
```

幂等要求：

```text
重复执行不会制造重复数据。
重复执行不会覆盖新数据。
重复执行不会破坏人工修复的数据。
```

---

## 8.2 回填前必须确认数据来源

例如：

```text
authorization.user_level:
  可从 user.user_level 回填。

authorization.authorized_devices:
  可从 user.max_devices 回填。

authorization.valid_until:
  可从 authorization.valid_until 或 user.expired_at 回填。

device_binding.game_project_id:
  单游戏可统一回填。
  多游戏必须从 LoginLog 或人工确认。
```

---

## 8.3 回填后必须验证

每次回填后必须查询：

```sql
SELECT COUNT(*) FROM table WHERE new_field IS NULL;
```

如果目标字段最终要非空：

```text
COUNT 必须为 0。
```

## 九、主库迁移流程

## 9.1 开发迁移流程

```text
Step 1:
  修改 ORM 模型。

Step 2:
  确认模型已被 app.models.main 导入。

Step 3:
  生成迁移:
    alembic revision -m "short message"

Step 4:
  手写或审查 migration。

Step 5:
  在开发库执行:
    alembic upgrade head

Step 6:
  检查表结构。

Step 7:
  运行相关测试。

Step 8:
  复制关键变更进 Obsidian。
```

---

## 9.2 禁止盲目 autogenerate

可以使用：

```bash
alembic revision --autogenerate -m "message"
```

但必须人工审查。

原因：

```text
1. autogenerate 可能漏掉 CheckConstraint。
2. autogenerate 可能误删表。
3. autogenerate 可能受 env.py 导入影响。
4. autogenerate 不会自动写正确的数据回填。
5. autogenerate 不理解业务阶段迁移。
```

---

## 9.3 生产升级流程

```text
Step 1:
  备份主库。

Step 2:
  记录当前版本:
    alembic current

Step 3:
  检查目标版本:
    alembic heads

Step 4:
  低峰期执行:
    alembic upgrade head

Step 5:
  运行结构检查 SQL。

Step 6:
  启动 Verify。

Step 7:
  运行健康检查。

Step 8:
  运行关键接口冒烟测试。

Step 9:
  记录迁移报告。
```

---

## 9.4 主库迁移检查命令

```bash
alembic current
alembic heads
alembic history --verbose
alembic upgrade head
```

PostgreSQL 检查：

```bash
psql -U hive_user -p 15432 -d hive_platform -c "\dt"
psql -U hive_user -p 15432 -d hive_platform -c "SELECT version_num FROM alembic_version;"
```

## 十、游戏库建表与升级流程

## 10.1 新游戏库创建

脚本：

```bash
python scripts/setup_game_db.py --code yeya --name "椰芽"
```

执行内容：

```text
1. 创建数据库 hive_yeya。
2. 使用 GameBase.metadata.create_all 建表。
3. 授予 hive_user 权限。
4. 提示在管理后台创建项目记录。
```

---

## 10.2 新游戏项目注册

管理后台接口：

```text
POST /admin/api/projects/
```

请求示例：

```json
{
```

## 10.3 游戏库结构变更风险


## 十八、结论

当前 Verify 已经有 Alembic 迁移基础，并且迁移脚本中已经沉淀了较多业务口径。

但当前不能把数据库迁移视为“已经完全治理好”。

当前最准确状态：

```text
主库 Alembic 迁移体系已建立；
游戏库 create_all 初始化体系已建立；
但 env.py 模型注册、空库迁移、游戏库结构版本、未来 device_binding 复杂迁移仍需治理。
```

后续最重要的原则：

```text
所有平台级结构变更必须先写迁移；
所有迁移必须能空库运行；
所有迁移必须能现有库升级；
所有迁移必须有 Obsidian 报告；
主库和游戏库边界不能混淆。
```
