# Verify 服务层业务规则

← 返回 [[01-网络验证系统/当前实现反向沉淀_2026-04-29]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、认证与设备绑定规则

已确认：用户登录服务 `auth_service.login_user()` 是 10 步验证链：用户名存在、密码匹配、账号 active、账号未到期、项目存在且启用、用户对项目有 active 授权、授权未到期、安卓设备绑定检查、写登录日志、签发 Access Token 与 Refresh Token。

设备绑定规则：

```text
client_type = android: 参与设备绑定与数量限制
client_type = pc: 不创建设备绑定，不占用激活名额
trial: 100 台
normal: 500 台
vip: 1000 台
svip: 无限制
tester: 无限制
```

待确认：源码中设备绑定上限使用 `user.user_level` 兼容字段，但新业务主口径已经转为 `Authorization.user_level / authorized_devices`。建议后续把登录设备上限改为项目授权维度，避免同一个用户在不同游戏项目下无法设置不同设备数。

## 二、Refresh Token 与踢出全部设备

已确认：Refresh Token 存 Redis，并通过反查索引实现 O(1) 定位；Phase 1 不做滚动刷新。`revoke_all_devices()` 会吊销当前 Access Token，并删除该用户所有 Refresh Token。

重要边界：`revoke_all_devices()` 源码注释明确指出，Stateless JWT 下只有当前 AT 进黑名单，其他已签发 AT 在过期前仍可能有效，最多等待 Access Token 生命周期自然失效。如需即时全量失效，后续应引入 per-user token_version。

## 三、代理点数与项目定价

已确认：`balance_service.py` 负责项目定价、代理余额、授权扣点、删除用户返点、点数流水。

计费规则：

| 用户级别 | 是否售卖 | 周期 | 周期小时 |
|---|---|---|---|
| trial | 是 | 周 | 168 |
| normal | 是 | 月 | 720 |
| vip | 是 | 月 | 720 |
| svip | 是 | 月 | 720 |
| tester | 否 | 不进入代理售卖定价体系 | - |

扣点公式：

```text
period_count = ceil(授权小时数 / 计费周期小时数)，最小为 1
total_cost = 单价 × 授权设备数 × period_count
优先消耗 charged_points，不足时再消耗可用 credit_points
可用授信 = credit_points - frozen_credit
```

返点规则：

```text
只处理存在 AuthorizationCharge 快照的代理扣点授权
已过期授权不返点
不足 1 小时按 1 小时计算
每小时成本 = 原始扣点 / 已购买总小时数
返点 = 原始扣点 - 已用成本 - 已返还点数
返点按原扣点来源比例返还到 charged / credit
单独撤销授权不返点；删除用户时触发自动返点
```

## 四、项目准入规则

已确认：`project_access_service.py` 增加了代理项目目录、准入策略、代理申请、自动开通、管理员审核、邀请开通等能力。

准入核心字段：

```text
visibility_mode: 项目对代理是否可见
open_mode: 手动审核 / 自动开通等开通模式
min_visible_agent_level: 最低可见代理等级
min_apply_agent_level: 最低可申请代理等级
min_auto_open_agent_level: 最低自动开通代理等级
min_available_points: 最低可用点数
allow_apply: 是否允许申请
allow_auto_open: 是否允许自动开通
require_request_reason: 是否必须填写申请理由
cooldown_hours_after_reject: 拒绝后冷却小时数
```

待确认：准入策略与代理等级策略是 Phase 2 平台能力，当前源码已实现，但必须补一组完整测试：可见性、申请、冷却、自动开通、管理员审核、取消申请、重复申请拦截。

## 五、设备心跳与运行态

已确认：`device_service.process_heartbeat()` 接收安卓端心跳，写入 Redis 实时缓存，并由 Celery 任务批量落库到游戏库 `device_runtime`。PC/后台查询优先读取 Redis 在线态，再补 DB 离线态。

关键边界：设备运行数据 `game_data` 为 JSONB，具体字段由游戏适配层定义，Verify 不应内置某一款游戏字段。

## 六、热更新

已确认：`update_service.py` 支持版本检查、下载 URL、管理员上传新版本；上传时计算 SHA-256、保存文件、归档旧 active 版本、创建或更新同版本记录，并主动失效 Redis 版本缓存。

热更新缓存：活跃版本缓存 TTL 为 300 秒。发布新版本后会主动删除缓存。

待确认：PC exe 更新与安卓 lrj/installLrPkg 更新都需要端到端实测；静态源码不能证明客户端更新安装成功。
