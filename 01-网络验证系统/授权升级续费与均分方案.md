# 授权设备升级续费与时间均分方案


## 一、核心场景

用户已有某项目授权：20 台设备，30 天到期，已使用 7 天。用户想增加设备数。

当前 `Authorization` 是 `(user_id, game_project_id)` 唯一约束——一个项目只有一条授权记录。无法拆分多条。升级必须修改同一条记录。

## 二、两种模式

### 2.1 追加模式（默认）

**新增设备继承原有到期时间，按剩余时长比例付费。**

```
新到期 = 旧到期（不变）
扣点 = (新设备数 - 旧设备数) × (剩余小时 / 计费周期小时) × 单台单价
```

**推演（normal 级别，单价 100 点/月，计费周期 720 小时）：**

| 参数 | 值 |
|------|-----|
| 旧设备 | 20 台 |
| 新设备 | 30 台 |
| 已用 | 7 天 = 168 小时 |
| 剩余 | (30-7) × 24 = 552 小时 |
| 计费周期 | 720 小时 |

```
扣点 = (30 - 20) × ceil(552 / 720) 个周期 × 100
     = 10 × 1 × 100
     = 1000 点
```

周期数向上取整：剩余不足 1 小时也按 1 小时算 → 不足 1 周期也按 1 周期算。

> 这是保守计算（按整周期而非精算）。如需精算版本，可改为 `(30-20) × (552/720) × 100 = 766.7 → 767 点`。

### 2.2 平均模式

**旧剩余点数 + 新购点数 → 统一折算为所有设备的新到期时间。重点：不赠送、不免费延长——旧有多少剩余价值，新付多少点数，全部加总后均分。**

```
旧剩余价值 = 旧设备数 × (旧剩余小时 / 计费周期小时) × 单台单价
新购点数   = 新设备数 × 单台单价（完整周期）
总点数     = 旧剩余价值 + 新购点数
每台点数   = 总点数 / 总设备数
新到期     = now + (每台点数 / 单台单价) × 计费周期小时
代理扣点   = 新购点数
```

**推演（同上场景，normal 级别，单价 100 点/月，720 小时/周期）：**

```
旧剩余价值 = 20 × (552/720) × 100 = 20 × 0.7667 × 100 = 1533.3 点
新购点数   = 10 × 100 = 1000 点
总点数     = 1533.3 + 1000 = 2533.3 点
每台点数   = 2533.3 / 30 = 84.44 点
新到期     = now + (84.44/100) × 720 = now + 608 小时 ≈ 25.3 天

代理扣点 = 1000 点（新购部分）
```

**与精算版的差异：**

如果纯粹按剩余比例收新设备的费（追加模式），代理只能收 767 点。平均模式下代理收 1000 点，多出的 233 点对应旧设备的剩余价值被"买断"后重新均分——旧设备的到期时间从 23 天缩到 25.3 天，但用户付了完整月费买新设备。

这不是"免费赠送"——是用新设备的完整月费换取了所有设备的时间重新分配。旧设备原本享有的 23 天剩余时间被新设备的点数"稀释"了。

## 三、不足 1 小时按 1 小时计算

`_ceil_hours` 已有：`seconds <= 0 → 0, else max(1, ceil(seconds/3600))`。

升级时同样使用此规则：
- 剩余 0 秒 → 不可升级（已到期）
- 剩余 30 分钟 → 按 1 小时计费
- 剩余 1 小时 → 按 1 小时计费

## 四、等级变更

用户可以同时升级设备数和等级：

| 旧等级 | 新等级 | 处理 |
|--------|--------|------|
| normal | vip | 旧设备按旧等级剩余价值退款，新设备按新等级完整收费 |
| vip | normal | 不可降级（降级不退点） |

**简化版（MVP）：等级变更时，旧的不退，新的按新等级计费。**

```
同时升等级+升设备：
  扣点 = 升等级部分(旧设备数 × (新单价 - 旧单价) × 剩余比例)
       + 升设备部分(新设备数 × 新单价 × 剩余比例)
```

**更简版（MVP 推荐）：不支持同时升等级。升级只改设备数。等级变更单独操作（管理员）。**

## 五、代理扣点风险矩阵

| 操作 | 风险 | 对策 |
|------|------|------|
| 升级设备数 | 余额不足 | 升级前检查代理可用点数 |
| 平均模式 | 免费延长旧设备 | 新设备按整周期收费，补偿延长成本 |
| 降级设备数 | 已付费用不退 | 只改设备数，不退点 |
| 临近到期升级 | 剩余时间极短 | 不足 1 小时按 1 小时，不足 1 周期按 1 周期 |
| 频繁升降 | 刷点/价格套利 | 降级不退点，升级按剩余比例 |
| 非本人用户 | 权限越界 | 代理只能升级自己创建的用户 |
| 管理员升级 | 不走扣点 | 管理员操作不产生扣点流水 |

## 六、前后端实现

### 6.1 后端：新增端点

```
POST /api/users/{user_id}/authorizations/{auth_id}/upgrade
```

**请求体：**
```json
{
  "additional_devices": 10,
  "mode": "append",
  "new_user_level": null
}
```

- `additional_devices`: 要增加的设备数量（必须 > 0）
- `mode`: `"append"` | `"average"`
- `new_user_level`: 可选，`null` = 不变

**响应：**
```json
{
  "authorization": { ... },
  "consumed_points": 767.0,
  "mode": "append",
  "new_devices": 30,
  "new_expiry": "2026-06-04T00:00:00Z",
  "old_devices": 20
}
```

**服务层逻辑（`user_service.py`）：**

```python
async def upgrade_authorization(user_id, auth_id, body, db, agent=None):
    auth = await _get_authorization_or_404(user_id, auth_id, db)
    
    new_devices = auth.authorized_devices + body.additional_devices
    now = datetime.now(timezone.utc)
    remaining_hours = _ceil_hours(now, auth.valid_until)
    
    if body.mode == "average":
        new_expiry = _calc_average_expiry(auth, body.additional_devices, now)
    else:
        new_expiry = auth.valid_until  # 不变
    
    cost = _calc_upgrade_cost(
        old_devices=auth.authorized_devices,
        new_devices=new_devices,
        old_level=auth.user_level,
        remaining_hours=remaining_hours,
        mode=body.mode,
        db=db,
    )
    
    consume_agent_points(agent, cost)
    
    auth.authorized_devices = new_devices
    auth.valid_until = new_expiry
    await db.flush()
    
    return UpgradeResponse(...)
```

### 6.2 前端

**位置**：用户详情页 → 项目授权列表 → 操作列加"升级"按钮

**弹窗**：
- 选择模式：追加（默认）/ 平均
- 新增设备数（数字输入 + 快捷 10/20/50/100）
- 预览：新设备总数、新到期时间（平均模式时计算）、预计扣点
- 确认按钮

### 6.3 扣点快照

升级操作同样生成 `AuthorizationChargeSnapshot`，记录：
- `authorization_id`
- `old_devices` / `new_devices`
- `mode`（append / average）
- `remaining_hours`
- `unit_price`
- `original_cost`
- `charged_consumed` / `credit_consumed`

## 七、边界情况

| 情况 | 处理 |
|------|------|
| 已到期 | 拒绝，提示"授权已到期，请重新授权" |
| 剩余不足 1 周期 | 按 1 周期计费 |
| 设备数不变 | 拒绝，`additional_devices` 必须 > 0 |
| 管理端操作 | 不走扣点，直接修改 |
| 同时改等级 | MVP 不支持，需单独操作 |

## 八、推荐实施顺序

1. 后端：追加模式（默认，到期不变）
2. 后端：平均模式
3. 前端：升级弹窗
4. 前端：扣点预览
5. 测试：升级 → 降级 → 到期后升级 → 管理员直接改
