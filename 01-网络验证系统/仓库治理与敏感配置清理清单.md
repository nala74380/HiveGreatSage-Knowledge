# Verify 仓库治理与敏感配置清理清单

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于治理 `HiveGreatSage-Verify` 仓库中的：

```text
1. 敏感配置。
2. 运行态文件。
3. 本地 IDE 文件。
4. 构建产物。
5. 日志文件。
6. Celery 运行文件。
7. 本地上传文件。
8. 可能泄露的密钥。
9. 可能污染多人协作的本机配置。
```

本文不是普通 `.gitignore` 说明，而是 Verify 进入长期平台项目后必须执行的 **仓库卫生治理清单**。

```text
.env 文件绝不能提交到版本控制。
```

### 2.2 高风险已确认

当前 GitHub 检索结果中已出现真实 `.env` 文件。

该文件包含：

```text
1. SECRET_KEY。
2. 数据库连接地址。
3. 数据库用户名。
4. 数据库密码。
5. Redis URL。
6. 本地存储路径。
7. 日志路径。
```

本文不复制具体密钥值。

### 2.3 当前风险等级

该问题属于：

```text
P0 阻塞级。
```

原因：

```text
1. SECRET_KEY 泄露会影响 JWT Token 安全。
2. 数据库密码泄露会影响主库和游戏库安全。
3. Redis 地址泄露会影响 Token 黑名单、Refresh Token、心跳缓存。
4. S3 / MinIO 密钥如果配置后泄露，会影响热更新包存储。
5. 日志、上传目录、构建产物入库会污染仓库。
6. 多人协作时本地配置会互相覆盖。
7. Git 历史中出现过密钥后，仅删除当前文件不等于安全。
```

## 四、必须治理的文件类型

## 4.1 敏感配置文件

必须禁止入库：

```text
.env
.env.local
.env.development
.env.production
.env.test
*.env
```

唯一允许入库：

```text
.env.example
```

原因：

```text
.env.example 是模板。
.env 是真实环境配置。
```

---

## 4.2 日志文件

必须禁止入库：

```text
logs/
*.log
*.log.*
```

Verify 当前默认日志路径：

```text
logs/app.log
```

日志风险：

```text
1. 可能包含异常堆栈。
2. 可能包含接口路径。
3. 可能包含用户 ID。
4. 可能包含项目 code。
5. 如果日志等级配置错误，可能包含 SQL 或敏感请求信息。
```

---

## 4.3 Celery 运行态文件

必须禁止入库：

```text
celerybeat-schedule
celerybeat-schedule.*
```

原因：

```text
1. Celery Beat 本地调度状态不属于源码。
2. 不同机器状态不同。
3. 入库会造成协作冲突。
```

---

## 4.4 Python 缓存与测试产物

必须禁止入库：

```text
__pycache__/
*.py[cod]
.pytest_cache/
.coverage
htmlcov/
.mypy_cache/
.ruff_cache/
```

---

## 4.5 前端依赖与构建产物

必须禁止入库：

```text
frontend/node_modules/
frontend/dist/
frontend/.vite/
```

说明：

```text
frontend/dist 可以作为部署产物发布，但不建议进入源码仓库。
```

---

## 4.6 本地上传与存储目录

必须禁止入库：

```text
storage/
uploads/
var/
tmp/
```

Verify 热更新包如果使用本地存储，应该进入：

```text
STORAGE_LOCAL_ROOT
```

该目录不应位于 Git 仓库内部。

---

## 4.7 IDE 与系统文件

必须禁止入库：

```text
.idea/
.vscode/
.DS_Store
Thumbs.db
```

说明：

```text
多人协作时 IDE 配置会互相污染。
```

如果需要共享 VSCode 推荐配置，应该单独确认后只提交：

```text
.vscode/extensions.json
.vscode/settings.example.json
```

但当前阶段建议先全部忽略。

## 六、立即清理步骤

## 6.1 从 Git 当前索引移除 `.env`

如果 `.env` 已经被 Git 跟踪，不能只添加 `.gitignore`。

必须执行：

```bash
git rm --cached .env
```

然后确认：

```bash
git status
```

期望：

```text
.env 显示为 deleted from index。
本地文件仍保留。
```

提交：

```bash
git add .gitignore .env.example
git commit -m "chore: remove local env from repository"
```

注意：

```text
git rm --cached 只是不再跟踪文件。
不会从 Git 历史中删除已经提交过的密钥。
```

---

## 6.2 清理日志与运行态文件

执行：

```bash
git rm -r --cached logs 2>/dev/null || true
git rm --cached celerybeat-schedule 2>/dev/null || true
git rm --cached celerybeat-schedule.* 2>/dev/null || true
```

然后：

```bash
git status
```

---

## 6.3 清理 IDE 与构建产物

执行：

```bash
git rm -r --cached .idea 2>/dev/null || true
git rm -r --cached .vscode 2>/dev/null || true
git rm -r --cached frontend/dist 2>/dev/null || true
git rm -r --cached frontend/node_modules 2>/dev/null || true
```

---

## 6.4 确认是否还有敏感文件被跟踪

执行：

```bash
git ls-files | grep -E '(^|/)\.env($|\.)|logs/|celerybeat|\.idea/|\.vscode/|node_modules/|frontend/dist/|storage/|uploads/'
```

期望：

```text
无输出。
```

如果有输出：

```text
说明仍有运行态文件被 Git 跟踪。
必须继续清理。
```

## 七、密钥轮换策略

## 7.1 为什么必须轮换

如果真实 `.env` 曾经提交到 GitHub，即使之后删除：

```text
Git 历史仍可能保留旧密钥。
```

因此不能只删除文件。

必须轮换所有已暴露或疑似暴露的密钥。

---

## 7.2 必须轮换项

### SECRET_KEY

必须轮换。

原因：

```text
SECRET_KEY 用于 JWT 签名。
泄露后，旧 Token 安全性不可信。
```

轮换后影响：

```text
1. 所有旧 Access Token 失效。
2. 所有旧 Admin / Agent Token 失效。
3. 所有旧签名数据如果依赖 SECRET_KEY，也可能失效。
```

生成新值：

```bash
python -c "import secrets; print(secrets.token_hex(32))"
```

---

### PostgreSQL 密码

必须轮换：

```text
hive_user 密码。
```

原因：

```text
DATABASE_MAIN_URL 和 DATABASE_GAME_PREFIX 中包含数据库密码。
```

轮换后必须同步：

```text
1. PostgreSQL 用户密码。
2. .env 中 DATABASE_MAIN_URL。
3. .env 中 DATABASE_GAME_PREFIX。
4. 部署服务配置。
5. CI/CD secret。
```

---

### Redis 密码

当前 `.env` 中 Redis URL 未显示密码。

待确认：

```text
生产 Redis 是否设置密码。
```

如果生产 Redis 有密码并曾写入 `.env`：

```text
必须轮换。
```

---

### S3 / MinIO 密钥

当前 `.env.example` 包含 S3 配置模板。

待确认：

```text
真实 .env 是否配置 S3_ACCESS_KEY / S3_SECRET_KEY。
```

如果配置过：

```text
必须轮换。
```

---

### Sentry DSN

如果真实生产 `SENTRY_DSN` 曾入库：

```text
建议轮换或重新生成。
```

---

## 7.3 轮换后验证

执行：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

检查：

```text
GET /health
POST /api/auth/login
POST /admin/api/auth/login
Redis 连接
数据库连接
热更新上传
```

如果 SECRET_KEY 轮换：

```text
所有客户端都需要重新登录。
```

## 八、Git 历史处理

## 8.1 当前文件删除不等于历史删除

如果 `.env` 已经提交过：

```text
git rm --cached .env
```

只会从后续提交移除。

历史提交仍保留。

---

## 8.2 推荐优先级

### 第一优先级：轮换密钥

无论是否改 Git 历史，必须先轮换密钥。

原因：

```text
密钥一旦进入 Git 历史，就应视为泄露。
```

### 第二优先级：清理当前仓库

确保当前 HEAD 不再包含 `.env`。

### 第三优先级：评估是否重写历史

如果仓库尚未多人广泛拉取，可以考虑：

```bash
git filter-repo --path .env --invert-paths
```

或使用 BFG。

但注意：

```text
重写历史会影响所有协作者。
需要统一通知并重新 clone 或 reset。
```

---

## 8.3 不建议做的事

不建议：

```text
1. 只改 .gitignore，不移除已跟踪文件。
2. 只删当前 .env，不轮换密钥。
3. 直接在公开 issue 或聊天中粘贴旧密钥值。
4. 继续沿用已经提交过的 SECRET_KEY。
```

---

## 九、`.env.example` 规范

`.env.example` 应保留：

```text
1. 所有必要配置项。
2. 示例值。
3. 注释说明。
4. 安全提醒。
5. 生成命令。
```

`.env.example` 不应包含：

```text
1. 真实 SECRET_KEY。
2. 真实数据库密码。
3. 真实 Redis 密码。
4. 真实 S3 密钥。
5. 真实公网 IP。
6. 真实生产域名私密配置。
```

## 十、配置入口治理

## 10.1 单一入口

Verify 后端配置统一从：

```text
app/config.py
settings
```

读取。

禁止：

```text
1. 在业务代码中硬编码数据库 URL。
2. 在业务代码中硬编码 Redis URL。
3. 在业务代码中硬编码 SECRET_KEY。
4. 在业务代码中硬编码 S3 密钥。
5. 在测试里复制生产密钥。
```

---

## 10.2 生产环境校验

建议增强 `Settings` 校验。

生产环境中必须强制：

```text
ENVIRONMENT=production
DEBUG=false
SECRET_KEY 不等于 example 值
DATABASE_MAIN_URL 不使用默认 password
REDIS_URL 不使用无密码公网地址
STORAGE_MODE=s3 时 S3_* 必填
```

当前 `debug_must_be_false_in_production` 函数存在，但需要确认是否真正强制 `production` 下 `DEBUG=false`。

建议逻辑：

```python
@field_validator("DEBUG")
@classmethod
def debug_must_be_false_in_production(cls, v: bool, info) -> bool:
    if info.data.get("ENVIRONMENT") == "production" and v:
        raise ValueError("production 环境必须设置 DEBUG=false")
    return v
```

## 十三、PCControl 与 AndroidScript 扩展治理

## 13.1 PCControl 本地配置

PCControl 应禁止入库：

```text
config/local.yaml
config/device_id.txt
config/last_login.json
config/device_meta.json
logs/
```

原因：

```text
1. local.yaml 是本机服务器地址和项目 UUID。
2. device_id.txt 是本机设备指纹。
3. last_login.json 是本地登录状态。
4. device_meta.json 是本地设备备注。
5. logs/ 是运行日志。
```

允许入库：

```text
config/default.yaml
config/local.example.yaml
```

---

## 13.2 AndroidScript 本地敏感项

AndroidScript 应重点检查：

```text
1. OCR_LICENSE。
2. API_BASE_URL。
3. PROJECT_UUID。
4. 本地保存的账号密码。
5. 本地保存的 Token。
6. 插件授权信息。
```

原则：

```text
1. 项目 UUID 可以进入项目配置，但 fork 新游戏必须修改。
2. OCR_LICENSE 不应提交真实授权。
3. 用户 Token 和账号密码只允许存在设备本地。
4. 示例文件必须使用 placeholder。
```

## 十五、风险清单

## R001：真实 `.env` 已进入仓库

风险等级：

```text
P0
```

已确认：

```text
GitHub 检索结果中出现 .env。
```

影响：

```text
1. SECRET_KEY 可能泄露。
2. 数据库密码可能泄露。
3. Redis 连接可能泄露。
```

处理：

```text
1. 立即从 Git 跟踪中移除 .env。
2. 轮换 SECRET_KEY。
3. 轮换数据库密码。
4. 检查是否需要历史清理。
```

---

## R002：只添加 `.gitignore` 不能解决已跟踪文件

风险等级：

```text
P1
```

## D037 — 密钥一旦进入 Git 历史，必须视为泄露并轮换

**日期：** 2026-04-29

### 背景

Git 历史会保留已经提交过的文件内容。即使后续删除 .env，历史提交中仍可能存在 SECRET_KEY、数据库密码等敏感信息。

### 决策

凡是进入 Git 历史的密钥，统一视为已泄露。

必须轮换：

```text

2. 数据库密码。
3. Redis 密码，如果有。
4. S3 / MinIO 密钥，如果有。
5. Sentry DSN，如有必要。
```

### 理由

```text
1. 无法保证历史没有被拉取。
2. 无法保证 fork 或缓存中没有旧文件。
3. 轮换密钥比判断泄露范围更可靠。
```
