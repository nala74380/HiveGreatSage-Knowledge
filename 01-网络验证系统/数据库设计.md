## 一、数据库划分策略

网络验证系统使用**两个逻辑数据库**：

|数据库|用途|特点|
|---|---|---|
|`hive_platform`|用户管理库（全局共用）|存储管理员、代理（多级）、用户、授权、设备绑定等核心数据|
|`hive_game_{project_id}`|游戏项目库（每个游戏独立）|存储该游戏的运行时数据、脚本参数、版本记录等|

> **为何游戏项目独立数据库？**
> 
> - 不同游戏的数据结构差异极大，共库会导致表膨胀且难以维护。
> - 游戏下线时可直接归档/删除对应数据库，操作干净。
> - 未来若为大客户独立部署，可直接迁移单个游戏库。

## 二、数据库 `hive_platform` 表结构

### 2.1 管理员表 `admin`

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY|自增主键|
|`username`|`VARCHAR(64)`|NOT NULL, UNIQUE|登录名|
|`password_hash`|`VARCHAR(255)`|NOT NULL|密码哈希（bcrypt）|
|`email`|`VARCHAR(128)`||邮箱（可选，用于找回密码）|
|`status`|`VARCHAR(16)`|NOT NULL DEFAULT 'active'|'active' / 'suspended'|
|`created_at`|`TIMESTAMPTZ`|DEFAULT NOW()||
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `UNIQUE INDEX idx_admin_username ON admin(username);`

---

### 2.2 代理表 `agent`

支持**多级代理**（树形结构），通过 `parent_agent_id` 自关联。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`username`|`VARCHAR(64)`|NOT NULL, UNIQUE|登录名|
|`password_hash`|`VARCHAR(255)`|NOT NULL||
|`parent_agent_id`|`INTEGER`|REFERENCES `agent(id)` ON DELETE **RESTRICT**|上级代理，NULL 表示由管理员直接创建；RESTRICT 禁止删除有子代理的代理|
|`level`|`SMALLINT`|NOT NULL|代理层级（1,2,3...）|
|`created_by_admin_id`|`INTEGER`|REFERENCES `admin(id)`|若由管理员创建则记录|
|`max_users`|`INTEGER`|DEFAULT 0|可创建用户数量上限（**0=无限制**）|
|`commission_rate`|`DECIMAL(5,2)`||佣金比例（百分比，可留空）|
|`status`|`VARCHAR(16)`|NOT NULL DEFAULT 'active'|'active' / 'suspended'|
|`created_at`|`TIMESTAMPTZ`|DEFAULT NOW()||
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `UNIQUE INDEX idx_agent_username ON agent(username);`
- `CREATE INDEX idx_agent_parent ON agent(parent_agent_id);`
- `CREATE INDEX idx_agent_level ON agent(level);`

**树形查询支持：** PostgreSQL 提供 `WITH RECURSIVE` 查询下级所有代理：

```sql
WITH RECURSIVE subordinates AS (
    SELECT id, username, level FROM agent WHERE id = :agent_id
    UNION ALL
    SELECT a.id, a.username, a.level
    FROM agent a
    INNER JOIN subordinates s ON a.parent_agent_id = s.id
)
SELECT * FROM subordinates;
```

---

### 2.3 用户表 `user`

用户只能由代理或管理员创建，**无自主注册**。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`username`|`VARCHAR(64)`|NOT NULL, UNIQUE|登录名|
|`password_hash`|`VARCHAR(255)`|NOT NULL||
|`user_level`|`VARCHAR(20)`|NOT NULL|枚举：`trial`, `normal`, `vip`, `svip`, `tester`|
|`created_by_agent_id`|`INTEGER`|REFERENCES `agent(id)`|哪个代理创建（与下一字段二选一）|
|`created_by_admin_id`|`INTEGER`|REFERENCES `admin(id)`|哪个管理员创建（与上一字段二选一）|
|`status`|`VARCHAR(16)`|NOT NULL DEFAULT 'active'|'active' / 'suspended' / 'expired'|
|`expired_at`|`TIMESTAMPTZ`||账号过期时间（NULL=永久）|
|`created_at`|`TIMESTAMPTZ`|DEFAULT NOW()||
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**约束检查：**

- `user_level = 'tester'` 时，`created_by_admin_id` 必须非空（即测试用户只能由管理员创建）。
- `created_by_agent_id` 和 `created_by_admin_id` 必须有且仅有一个非空。

```sql
CONSTRAINT chk_user_creator CHECK (
    (created_by_agent_id IS NULL) != (created_by_admin_id IS NULL)
),
CONSTRAINT chk_tester_creator CHECK (
    user_level != 'tester' OR created_by_admin_id IS NOT NULL
)
```

**索引：**

- `UNIQUE INDEX idx_user_username ON user(username);`
- `CREATE INDEX idx_user_created_by_agent ON user(created_by_agent_id);`
- `CREATE INDEX idx_user_status_level ON user(status, user_level);`

---

### 2.4 授权表 `authorization`

记录用户对某个游戏项目的授权有效期。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`user_id`|`INTEGER`|NOT NULL REFERENCES `user(id)` ON DELETE CASCADE||
|`game_project_id`|`INTEGER`|NOT NULL REFERENCES `game_project(id)`||
|`valid_from`|`TIMESTAMPTZ`|DEFAULT NOW()|授权开始时间|
|`valid_until`|`TIMESTAMPTZ`||授权到期时间（NULL=永久）|
|`status`|`VARCHAR(16)`|DEFAULT 'active'|'active' / 'suspended'|

**约束：**

- `UNIQUE (user_id, game_project_id)` — 一用户对一游戏只有一条记录

---

### 2.5 游戏项目表 `game_project`

注册平台上的游戏项目。客户端登录时以 `project_uuid` 定位项目，内部外键引用继续使用 INTEGER `id`。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY|内部主键，用于外键引用|
|`project_uuid`|`UUID`|UNIQUE NOT NULL DEFAULT gen_random_uuid()|对外唯一标识，客户端登录时传此值|
|`code_name`|`VARCHAR(64)`|UNIQUE NOT NULL|内部代号，如 "game_001"（同时是数据库名后缀）|
|`display_name`|`VARCHAR(128)`|NOT NULL|显示名称|
|`db_name`|`VARCHAR(64)`|UNIQUE NOT NULL|对应的 PostgreSQL 库名|
|`is_active`|`BOOLEAN`|DEFAULT TRUE||
|`created_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `UNIQUE INDEX idx_game_project_uuid ON game_project(project_uuid);`
- `UNIQUE INDEX idx_game_project_code ON game_project(code_name);`

---

### 2.6 设备绑定表 `device_binding`

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`user_id`|`INTEGER`|NOT NULL REFERENCES `user(id)` ON DELETE CASCADE||
|`device_fingerprint`|`VARCHAR(255)`|NOT NULL|设备唯一标识（hardware_serial）|
|`bound_at`|`TIMESTAMPTZ`|DEFAULT NOW()|绑定时间|
|`last_seen_at`|`TIMESTAMPTZ`||最后一次登录时间|
|`status`|`VARCHAR(16)`|NOT NULL DEFAULT 'active'|'active' / 'unbound'|
|`unbound_at`|`TIMESTAMPTZ`||解绑时间（解绑时填写）|
|`unbind_reason`|`VARCHAR(64)`||解绑原因|
|`unbind_by_type`|`VARCHAR(20)`||操作方类型（'admin'/'agent'/'user'）|

**索引：**

- `UNIQUE INDEX idx_device_user_fingerprint ON device_binding(user_id, device_fingerprint) WHERE status = 'active';` （部分唯一索引，确保同一用户的同一设备只有一条活跃绑定）
- `CREATE INDEX idx_device_fingerprint ON device_binding(device_fingerprint);`

> **设备指纹生成策略**：以 `hardware_serial`（设备序列号）作为主设备指纹，`enhanced_device_id` 作为辅助字段。具体采集方案在安卓脚本框架中详细设计。

---

### 2.7 登录日志表 `login_log`

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`BIGSERIAL`|PRIMARY KEY||
|`user_id`|`INTEGER`|NOT NULL REFERENCES `user(id)` ON DELETE CASCADE||
|`device_fingerprint`|`VARCHAR(255)`||登录设备标识|
|`ip_address`|`INET`||客户端 IP（PostgreSQL 原生 INET 类型）|
|`client_type`|`VARCHAR(20)`|NOT NULL|`pc` 或 `android`|
|`game_project_id`|`INTEGER`||登录时关联的游戏项目（审计用）|
|`success`|`BOOLEAN`|NOT NULL|是否成功|
|`fail_reason`|`VARCHAR(32)`||NULL=成功；失败时填写原因枚举，如 `fail_auth` / `fail_device_limit` / `fail_expired` / `fail_no_auth`|
|`login_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `CREATE INDEX idx_login_log_user ON login_log(user_id, login_at DESC);`
- `CREATE INDEX idx_login_log_ip ON login_log(ip_address);`
- `CREATE INDEX idx_login_log_project ON login_log(game_project_id, login_at DESC);`

## 三、游戏项目库 `hive_game_{project_id}` 表结构

每个游戏项目拥有独立数据库，以下为**标准模板**，具体游戏可增加自定义表。

### 3.1 项目配置表 `project_config`

存储该游戏项目的全局配置。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`config_key`|`VARCHAR(128)`|NOT NULL, UNIQUE|配置键|
|`config_value`|`TEXT`||配置值（JSON 或纯文本）|
|`description`|`VARCHAR(255)`||说明|
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

---

### 3.2 脚本参数定义表 `script_param_def`

定义该游戏在 PC 中控上可配置的参数项（元数据）。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`param_key`|`VARCHAR(64)`|NOT NULL, UNIQUE|参数键（如 `farm_map_id`）|
|`param_type`|`VARCHAR(20)`|NOT NULL|`int`, `float`, `string`, `bool`, `enum`|
|`default_value`|`TEXT`||JSON 格式的默认值|
|`options`|`JSONB`||针对 enum 类型的可选项列表|
|`display_name`|`VARCHAR(128)`||在 PC 中控上显示的名称|
|`description`|`TEXT`||参数说明|
|`sort_order`|`INTEGER`|DEFAULT 0|显示排序|
|`created_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

---

### 3.3 用户脚本参数值表 `user_script_param`

存储特定用户为该游戏设置的参数值（覆盖默认值）。

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`user_id`|`INTEGER`|NOT NULL|来自平台库的 user.id（无外键约束，跨库）|
|`param_def_id`|`INTEGER`|NOT NULL REFERENCES `script_param_def(id)` ON DELETE CASCADE||
|`param_value`|`TEXT`|NOT NULL|用户设置的值（JSON 格式）|
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `UNIQUE INDEX idx_user_param ON user_script_param(user_id, param_def_id);`

> PC 中控修改参数 → 写入此表 → 安卓脚本启动时拉取并合并默认值。

---

### 3.4 设备运行时数据表 `device_runtime`

对应 Redis 心跳缓冲批量落库的目标表。

|字段|类型|约束|说明|
|---|---|---|---|
|`device_id`|`VARCHAR(128)`|PRIMARY KEY|设备唯一标识（与 device_binding.device_fingerprint 一致）|
|`user_id`|`INTEGER`|NOT NULL|来自平台库的 user.id（无外键约束，跨库）|
|`status`|`VARCHAR(20)`||`running` / `idle` / `error` / `offline`|
|`last_seen`|`TIMESTAMPTZ`|NOT NULL|最后一次心跳时间|
|`game_data`|`JSONB`||游戏自定义数据（如当前地图、金币数）|
|`updated_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `CREATE INDEX idx_device_runtime_user ON device_runtime(user_id);`
- `CREATE INDEX idx_device_runtime_last_seen ON device_runtime(last_seen);`

> 安卓脚本每 30 秒上报一次 → 写入 Redis → Celery 每 30 秒批量 UPSERT 至此表（ON CONFLICT (device_id) DO UPDATE）。

---

### 3.5 版本记录表 `version_record`

|字段|类型|约束|说明|
|---|---|---|---|
|`id`|`SERIAL`|PRIMARY KEY||
|`client_type`|`VARCHAR(20)`|NOT NULL|`pc` 或 `android`|
|`version`|`VARCHAR(32)`|NOT NULL|版本号（如 `1.0.1`）|
|`package_path`|`VARCHAR(512)`|NOT NULL|热更新包存储路径（本地路径或对象存储 key）|
|`checksum_sha256`|`VARCHAR(64)`||文件校验值（SHA256）|
|`release_notes`|`TEXT`||更新日志|
|`is_active`|`BOOLEAN`|DEFAULT TRUE|是否为当前最新版本|
|`force_update`|`BOOLEAN`|DEFAULT FALSE|是否强制更新|
|`released_at`|`TIMESTAMPTZ`|DEFAULT NOW()||

**索引：**

- `UNIQUE INDEX idx_version_active ON version_record(client_type) WHERE is_active = TRUE;` （每个客户端类型只能有一个活跃版本）

## 四、表关系图（文字版）

```
【hive_platform 库】

[admin] ────┐
            │ (created_by_admin_id)
            ▼
[agent] ────┼─── [agent] (parent_agent_id 自关联，ON DELETE RESTRICT)
            │
            │ (created_by_agent_id)
            ▼
          [user] ──── [authorization] ──── [game_project]
            │                                  │ (project_uuid 对客户端)
            ├── [device_binding]               │ (id 对内部引用)
            └── [login_log] ──────────────────┘ (game_project_id)

【hive_game_xxx 库】（每个游戏独立）

[script_param_def] ←── [user_script_param] (user_id 跨库引用)
[device_runtime]        (device_id = device_fingerprint，user_id 跨库引用)
[version_record]
[project_config]
```
