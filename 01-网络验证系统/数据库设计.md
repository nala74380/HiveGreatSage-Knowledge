- 本文是 Verify 数据库设计文档，不代表当前数据库已经完成运行验证。


## 三、数据库总体架构

### 3.1 总体原则

Verify 采用：

```text
一个平台主库 + 多个游戏项目库
```

结构如下：

```text
PostgreSQL
├── hive_platform
│   ├── admin
│   ├── agent
│   ├── user
│   ├── game_project
│   ├── authorization
│   ├── device_binding
│   ├── version_record
│   ├── login_log
│   ├── agent_balance
│   ├── balance_transaction
│   ├── project_price
│   ├── authorization_charge
│   ├── project_access_policy
│   ├── agent_project_auth_request
│   └── ...
│
├── hive_game_001
│   ├── project_config
│   ├── script_param_def
│   ├── user_script_param
│   ├── device_runtime
│   └── version_record
│
├── hive_game_002
│   ├── project_config
│   ├── script_param_def
│   ├── user_script_param
│   ├── device_runtime
│   └── version_record
│
└── hive_game_xxx
```

### 3.2 主库 hive_platform 定位

主库是平台级真相源。

主库保存：

```text
1. 管理员。
2. 代理。
3. 用户。
4. 游戏项目元信息。
5. 用户项目授权。
6. 安卓设备绑定。
7. 登录日志。
8. 热更新主记录。
9. 点数余额。
10. 点数流水。
11. 项目定价。
12. 授权扣点快照。
13. 项目准入策略。
14. 代理项目申请。
15. 代理业务画像。
```

主库不应保存：

```text
1. 某一款游戏的大量运行时数据。
2. 某一款游戏的脚本参数定义。
3. 某一款游戏的用户参数值。
4. 高频心跳历史明细。
```

### 3.3 游戏库 hive_game_xxx 定位

游戏库是游戏项目级数据真相源。

游戏库保存：

```text
1. 该游戏自己的项目配置。
2. 该游戏自己的脚本参数定义。
3. 该游戏下用户的脚本参数值。
4. 该游戏下设备运行数据。
5. 该游戏侧版本记录。
```

游戏库不应保存：

```text
1. 管理员。
2. 代理。
3. 用户密码。
4. 用户主账号。
5. 代理余额。
6. 点数流水。
7. 平台级授权主关系。
```

### 3.4 为什么主库和游戏库分离

已确认采用分离设计，原因如下：

```text
1. 多游戏数据隔离。
2. 避免所有游戏运行数据挤在同一组表。
3. 降低 device_runtime 单表膨胀风险。
4. 便于大客户独立迁移单个游戏。
5. 便于按游戏做备份、清理、归档。
6. 支撑“一个游戏 = 一套独立中控 + 一套独立脚本”的平台原则。
7. 避免某个游戏的数据爆量影响平台身份与授权系统。
```

### 3.5 跨库关系原则

PostgreSQL 不支持跨数据库外键，因此：

```text
游戏库中的 user_id、device_id 等字段不设置数据库外键。
这些字段必须由服务层写入前校验。
```

跨库字段示例：

```text
hive_game_xxx.user_script_param.user_id
  → 逻辑引用 hive_platform.user.id

hive_game_xxx.device_runtime.user_id
  → 逻辑引用 hive_platform.user.id

hive_game_xxx.device_runtime.device_id
  → 逻辑引用 hive_platform.device_binding.device_fingerprint
```

强制要求：

```text
1. 任何写入游戏库的 user_id，服务层必须先确认主库 user 存在。
2. 任何写入游戏库的 device_id，服务层必须先确认主库 device_binding 存在。
3. 任何写入游戏库的数据，必须基于当前用户已授权的 game_project。
```

## 四、数据库连接与迁移边界

### 4.1 主库连接

主库连接由 `app/database.py` 提供。

核心职责：

```text
1. 创建主库异步 engine。
2. 创建主库 AsyncSessionLocal。
3. 提供 get_db 依赖。
4. 定义主库 Base。
5. 应用关闭时释放连接池。
```

主库连接配置来自：

```text
DATABASE_MAIN_URL
```

### 4.2 游戏库连接

游戏库连接由 `app/database.py` 动态创建。

核心职责：

```text
1. 根据 game_project.code_name 推导游戏库名。
2. 动态创建游戏库 engine。
3. 缓存游戏库 sessionmaker。
4. 提供 get_game_db_session(code_name)。
5. 应用关闭时释放所有游戏库连接池。
```

建议命名规则：

```text
code_name = game_001
db_name   = hive_game_001
```

禁止出现：

```text
hive_game_game_001
```

### 4.3 Alembic 迁移边界

当前 Alembic 只管理主库：

```text
target_metadata = Base.metadata
```

游戏库模型 `GameBase.metadata` 不进入 Alembic。

原因：

```text
1. 主库结构是平台级结构，需要版本化迁移。
2. 游戏库结构是模板结构，由 GameBase.metadata.create_all() 初始化。
3. 游戏库数量可动态增加，不适合直接用同一个 Alembic target_metadata 混管。
4. 避免在主库误生成游戏表迁移。
```

### 4.4 当前迁移文件

当前识别到的迁移文件：

```text
migrations/versions/a458709e354f_initial_tables.py
migrations/versions/0010_agent_project_access_policy.py
migrations/versions/0011_agent_business_profile.py
```

### 4.5 迁移风险

⚠️ 当前需要重点检查：

```text
migrations/env.py 中导入的主库模型是否完整。
```

已观察到 env.py 导入了部分主模型，例如：

```text
Admin
Agent
Authorization
DeviceBinding
GameProject
LoginLog
User
```

但项目准入与代理业务画像等模型在其他文件中：

```text
app/models/main/project_access.py
app/models/main/agent_profile.py
```

风险：

```text
如果 env.py 未显式导入这些模型，Alembic autogenerate 可能无法识别新增表。
```

建议方案：

```text
1. 建立 app/models/main/__init__.py 统一导入全部主库模型。
2. migrations/env.py 只导入 app.models.main 中的完整模型集合。
3. 每次新增主库模型后，必须运行 alembic revision --autogenerate 并检查迁移内容。
4. 新增游戏库模型时，不通过主库 Alembic，而是更新游戏库初始化流程。
```

## 七、主库核心表设计

## 7.1 `admin` 管理员表

### 职责

`admin` 用于保存平台管理员账号。

管理员负责：

```text
1. 登录管理后台。
2. 创建代理。
3. 创建项目。
4. 管理用户。
5. 管理项目准入。
6. 管理热更新。
7. 管理点数与价格。
```

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 管理员 ID |
| `username` | 管理员用户名，唯一 |
| `password_hash` | 密码哈希 |
| `email` | 邮箱 |
| `status` | 管理员状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 关系

```text
Admin 1 → N Agent
```

通过：

```text
agent.created_by_admin_id
```

### 设计要求

```text
1. 不允许保存明文密码。
2. 管理员状态至少应支持 active / suspended。
3. 管理员关键操作必须进入审计日志。
```

## 7.2 `agent` 代理表

### 职责

`agent` 用于保存多级代理账号。

代理体系支持：

```text
管理员
  └── 一级代理
        └── 二级代理
              └── 三级代理
                    └── 用户
```

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 代理 ID |
| `username` | 代理用户名，唯一 |
| `password_hash` | 密码哈希 |
| `parent_agent_id` | 上级代理 ID |
| `level` | 代理组织层级 |
| `created_by_admin_id` | 创建该代理的管理员 |
| `max_users` | 最大可创建用户数 |
| `commission_rate` | 佣金比例 |
| `status` | 代理状态 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 表级约束

```text
chk_agent_creator:
  顶级代理必须由管理员创建；
  非顶级代理必须有 parent_agent_id。
```

### 关系

```text
Agent 1 → N Agent          通过 parent_agent_id 自关联
Admin 1 → N Agent          通过 created_by_admin_id
Agent 1 → N User           通过 user.created_by_agent_id
Agent 1 → 1 AgentBalance   通过 agent_balance.agent_id
Agent 1 → N AgentProjectAuth
Agent 1 → N AuthorizationCharge
```

### 风险点

```text
1. Agent.hierarchy_depth 是组织树层级，不等于业务等级。
2. AgentBusinessProfile.tier_level 才是业务等级 Lv.1 - Lv.4。
3. 两套 level 语义必须在 UI 和文档里明确区分。
```

---

## 7.3 `user` 用户表

### 职责

`user` 是平台用户主账号表。

用户由管理员或代理创建，不允许自行注册。

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 用户 ID |
| `username` | 用户名，唯一 |
| `password_hash` | 密码哈希 |
| `user_level` | 兼容旧字段：用户等级 |
| `created_by_agent_id` | 创建用户的代理 |
| `created_by_admin` | 是否由管理员创建 |
| `status` | 用户状态 |
| `is_deleted` | 软删除标记 |
| `max_devices` | 兼容旧字段：最大设备数 |
| `expired_at` | 兼容旧字段：账号到期时间 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 表级约束

```text
user_level IN ('trial', 'normal', 'vip', 'svip', 'tester')
tester 用户必须由管理员创建
```

### 关系

```text
User 1 → N Authorization
User 1 → N DeviceBinding
User 1 → N AuthorizationCharge
Agent 1 → N User
```

### 长期口径

⚠️ 当前 `user.user_level`、`user.max_devices`、`user.expired_at` 是兼容旧字段。

长期建议：

```text
用户等级:
  以 authorization.user_level 为准。

设备数:
  以 authorization.authorized_devices 为准。

授权到期:
  以 authorization.valid_until 为准。
```

原因：


## 7.6 `authorization` 用户项目授权表

### 职责

`authorization` 是用户在某个游戏项目下的授权记录。

这是长期核心表。

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 授权 ID |
| `user_id` | 用户 ID |
| `game_project_id` | 游戏项目 ID |

| `authorized_devices` | 该项目授权设备数 |
| `valid_from` | 授权开始时间 |
| `valid_until` | 授权到期时间 |
| `status` | 授权状态 |

### 表级约束

```text
Unique(user_id, game_project_id)
user_level IN ('trial', 'normal', 'vip', 'svip', 'tester')
authorized_devices >= 0
```

### 长期口径

```text
用户是否能运行某游戏:
  看 authorization 是否存在、是否 active、是否在有效期内。

用户在某游戏的等级:
  看 authorization.user_level。

用户在某游戏的设备数:
  看 authorization.authorized_devices。

用户在某游戏的到期时间:
  看 authorization.valid_until。
```

### 设计要求

```text
1. 登录 Verify 时必须校验 authorization。
2. AndroidScript 占用设备名额时必须校验 authorization.authorized_devices。
3. PCControl 登录不应占用设备名额。
4. tester 级别只能由管理员创建或授权。
```

## 7.7 `authorization_charge` 授权扣点快照表

### 职责

`authorization_charge` 记录每一次授权扣点的快照。

它不是普通日志，而是返点计算依据。

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 扣点快照 ID |
| `authorization_id` | 授权 ID |
| `agent_id` | 扣点代理 |
| `user_id` | 用户 ID |
| `project_id` | 项目 ID |
| `user_level` | 授权等级 |
| `unit_price` | 单价 |
| `authorized_devices` | 授权设备数 |
| `billing_period` | 计费周期 |
| `billing_period_hours` | 单周期小时数 |
| `period_count` | 周期数量 |
| `paid_hours` | 实付小时数 |
| `valid_from` | 生效开始 |
| `valid_until` | 生效结束 |
| `original_cost` | 原始费用 |
| `charged_consumed` | 充值余额消耗 |
| `credit_consumed` | 授信额度消耗 |
| `refunded_points` | 已返点数 |

```text
1. device_id 对应主库 device_binding.device_fingerprint。
2. user_id 对应主库 user.id。
3. game_data 允许各游戏自定义字段。
4. 高频写入必须走 Redis + Celery，避免直接压 PostgreSQL。
```

### 风险点

```text
1. device_id 作为主键可能无法区分同一设备在多个用户/项目下的历史情况。
2. 如果后续允许设备转移用户，需要定义历史保留策略。
3. 如果一个设备在多个游戏项目同时运行，需确认每个游戏库独立即可隔离。
```

## 8.5 `version_record` 游戏库版本记录表

### 职责

保存某游戏库内的热更新版本记录。

### 当前状态

当前源码中该表仍存在，但主库也已经有 `version_record`。

### 主要字段

| 字段 | 含义 |
|---|---|
| `id` | 版本 ID |
| `client_type` | pc / android |
| `version` | 版本号 |
| `package_path` | 包路径 |
| `checksum_sha256` | 文件 SHA-256 |
| `release_notes` | 更新说明 |
| `is_active` | 是否活跃 |
| `force_update` | 是否强制更新 |
| `released_at` | 发布时间 |

### 游戏库表级约束

```text
每种 client_type 只能有一个 is_active = TRUE
```

### 风险点

⚠️ 当前与主库 `version_record` 职责重复。

建议方案：

```text
1. 当前热更新 V2 以主库 version_record 为准。
2. 游戏库 version_record 暂时标记为兼容/待清理。
3. 不再新增对游戏库 version_record 的业务依赖。
4. 后续确认无调用后，制定迁移删除方案。
```

## 十、关键业务口径

### 10.1 用户等级口径

当前存在两处用户等级：

```text
user.user_level
authorization.user_level
```

已确认建议：

```text
长期以 authorization.user_level 为准。
user.user_level 作为兼容旧字段。
```

原因：

```text
同一用户可能在游戏 A 是 vip，在游戏 B 是 normal。
```

### 10.2 设备数量口径

当前存在两处设备数：

```text
user.max_devices
authorization.authorized_devices
```

已确认建议：

```text
长期以 authorization.authorized_devices 为准。
user.max_devices 作为兼容旧字段。
```

原因：

```text
不同游戏项目设备额度可能不同。
```

### 10.3 授权到期口径

当前存在两处到期时间：

```text
user.expired_at
authorization.valid_until
```

已确认建议：

```text
长期以 authorization.valid_until 为准。
user.expired_at 作为兼容旧字段。
```

原因：

```text
同一用户可能在不同项目下有不同到期时间。
```

### 10.4 设备绑定口径

当前唯一约束：

```text
user_id + device_fingerprint
```

建议修订：

```text
user_id + game_project_id + device_fingerprint
```

原因：

```text
设备授权是项目维度，不应只在用户维度。
```

### 10.5 热更新版本口径

当前存在两套 `version_record`：

```text
主库 version_record
游戏库 version_record
```

建议统一：

```text
主库 version_record 为长期版本记录表。
游戏库 version_record 标记为兼容/待清理。
```

## 十一、当前风险清单

### R001 Alembic 模型导入不完整风险

风险：

```text
migrations/env.py 如果没有导入所有主库模型，autogenerate 可能漏表。
```

影响：

```text
新增主库表无法自动生成迁移。
```

建议：

```text
统一在 app/models/main/__init__.py 导入全部主库模型。
env.py 只导入 app.models.main，确保 Base.metadata 完整。
```

---

### R002 主库与游戏库 version_record 重复

风险：

```text
主库和游戏库都有 version_record。
```

影响：

```text
热更新逻辑可能出现两个真相源。
```

建议：

```text
以主库 version_record 作为唯一热更新真相源。
游戏库 version_record 标记为待清理。
```

---

### R003 设备绑定缺少项目维度

风险：

```text
device_binding 唯一约束为 user_id + device_fingerprint。
```

影响：

```text
多游戏授权场景下，设备名额可能互相影响。
```

建议：

```text
增加 game_project_id。
唯一约束改为 user_id + game_project_id + device_fingerprint。
```

---

### R004 user 表兼容字段与 authorization 字段重复

风险字段：

```text
user.user_level
user.max_devices
user.expired_at
```

影响：

```text
后续开发者可能误用 user 表字段，绕过项目授权口径。
```

建议：

```text
1. 文档中明确标记为兼容字段。
2. Service 层统一以 authorization 字段为准。
3. 后续迁移时考虑废弃或只读兼容。
```

---

### R005 点数流水缺少强审计风险

风险：

```text
点数操作属于财务敏感行为。
```

需要确认：

```text
1. balance_transaction 是否记录操作者。
2. recharge / credit / freeze 是否记录 admin_id。
3. 返点是否记录触发原因。
```

建议：

```text
新增 operator_admin_id / operator_agent_id / operation_source。
```

---

### R006 IMSI 合规风险

风险：

```text
device_binding.imsi 保存敏感设备标识。
```

建议：

```text
1. 明确采集必要性。
2. 默认脱敏展示。
3. 限制访问权限。
4. 设置保留周期。
5. 加入隐私与合规说明。
```

---

### R007 跨库引用无数据库外键

风险：

```text
游戏库 user_id、device_id 无外键约束。
```

影响：

```text
服务层漏校验时可能写入孤儿数据。
```

建议：

```text
1. 所有跨库写入统一走 service。
2. 禁止 router 直接写游戏库。
3. 增加定期数据一致性检查脚本。
```
