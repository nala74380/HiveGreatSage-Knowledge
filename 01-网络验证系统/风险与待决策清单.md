← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于集中管理 `HiveGreatSage-Verify` 当前源码反向沉淀后暴露的风险点和待决策项。

本文不解决所有问题，而是用于：

```text
1. 防止风险散落在多个对话窗口。
2. 防止源码已实现但文档未同步。
3. 防止把“源码存在”误写成“已验证可用”。
4. 防止高风险模块在未测试前进入生产使用。
5. 防止后续继续在旧结构上修修补补。
6. 为下一步测试、重构、文档同步提供优先级。
```

## 三、风险分级规则

### 3.1 风险等级

```text
P0 阻塞级:
  不解决就不能进入三端联调或生产部署。

P1 高风险:
  不解决会导致权限、安全、财务、数据一致性或热更新事故。

P2 中风险:
  不解决会导致维护成本上升、接口漂移、测试不稳定。

P3 低风险:
  不影响当前主线，但应纳入后续治理。
```

### 3.2 状态标记

```text
未处理:
  已发现但尚未开始处理。

处理中:
  已经开始修复或推演。

待验证:
  已有修复方案或源码实现，但未测试通过。

已关闭:
  已完成修复、测试和文档同步。

暂缓:
  有风险，但当前阶段不是主线。
```

### 3.3 证据等级

```text
源码确认:
  当前源码明确存在相关实现或问题。

文档确认:
  当前 Obsidian / Knowledge 文档明确记录。

推断:
  基于源码结构或平台长期目标推断，需进一步验证。

未知:
  当前证据不足，不能写成事实。
```

## R-P1-001：设备绑定缺少项目维度

### 风险等级

```text
P1 高风险
```

### 当前状态

```text
待决策
```

### 已确认

当前 `DeviceBinding` 唯一约束是：

```text
Unique(user_id, device_fingerprint)
```

当前 `Authorization` 已经以：

```text
user_id + game_project_id
```

作为用户项目授权唯一口径。

### 风险说明

如果设备绑定不包含 `game_project_id`，会出现：

```text
1. 同一用户多个游戏项目的设备名额互相影响。
2. 游戏 A 的设备绑定可能占用游戏 B 的设备额度。
3. PCControl 按项目查看设备时边界不清。
4. AndroidScript 多游戏并行时设备归属不清。
```

### 建议方案

中期迁移：

```text
1. device_binding 增加 game_project_id。
2. 历史数据按登录时 project_id 回填。
3. 唯一约束改为 user_id + game_project_id + device_fingerprint。
4. 所有设备绑定查询增加 project_id。
5. PCControl / AndroidScript 联调确认。
```

### 待决策

```text
T032: device_binding 是否在 Phase 2 前迁移到 user_id + game_project_id + device_fingerprint 维度？
```

### 回退方案

如果短期不迁移：

```text
1. 文档中明确当前是用户级设备绑定。
2. 限制一个用户同一时间只测试一个游戏项目。
3. 第二游戏接入前必须完成迁移。
```

### 关闭标准

```text
1. 完成设计决策。
2. 完成数据库迁移。
3. 完成绑定逻辑修改。
4. 完成测试。
```

## R-P1-002：revoke-all 不能保证所有 Access Token 即时失效

### 风险等级

```text
P1 高风险
```

### 当前状态

```text
待决策
```

### 已确认

当前已有：

```text
1. Refresh Token。
2. Token blacklist。
3. POST /api/auth/revoke-all。
```

### 风险说明

如果 Access Token 是无状态 JWT，则：

```text
删除 Refresh Token ≠ 已签发 Access Token 立即失效。
```

因此：

```text
用户点击“踢出所有设备”后，其他设备的 Access Token 可能在过期前继续可用。
```

### 建议方案

引入：

```text
user.token_version
agent.token_version
admin.token_version
```

Token payload 增加：

```text
token_version
```

鉴权时校验：

```text
payload.token_version == db.token_version
```

踢出所有设备时：

```text
1. token_version + 1。
2. 删除所有 Refresh Token。
3. 当前 Access Token 加黑名单。
```

### 待决策

```text
T033: Verify 是否在 Phase 2 引入 token_version，实现真正的全设备即时踢出？
```

### 回退方案

如果暂不做：

```text
1. 将 Access Token 有效期保持 15 分钟。
2. 文档明确 revoke-all 只能保证 Refresh Token 失效。
3. UI 文案避免写“立即踢下线所有设备”。
```

### 关闭标准

```text
1. 完成 token_version 设计。
2. 完成数据库迁移。
3. 完成鉴权依赖修改。
4. 完成 revoke-all 测试。
5. 完成多设备登录实测。
```

## D020 — Verify 风险状态采用分级治理

**日期：** 2026-04-29  
**状态：** 建议确认  
**关联文档：** [[01-网络验证系统/风险与待决策清单]]

### 背景

Verify 当前源码已实现大量平台能力，但测试、三端联调、生产化验证尚未同步完成。继续只用“已完成 / 未完成”会造成误判。

### 决策

Verify 后续风险与实现状态统一分级：

```text
P0 阻塞级
P1 高风险
P2 中风险
P3 低风险
```

实现状态统一分为：

```text
源码已实现
待运行验证
待三端联调
待生产化
待修订
```

### 理由

```text
1. 防止把源码存在误写成生产可用。
2. 防止把未测试模块直接进入联调。
3. 防止高风险财务、热更新、设备绑定问题被忽略。
4. 便于后续多窗口协作。
```

### 后续动作

```text
1. 本文作为 Verify 风险真相源。
2. 每次修复风险后更新状态。
3. 重大决策同步到 [[决策日志]]。
```

## 十二、本文结论

当前 Verify 已经具备较完整的平台中枢源码，但风险治理必须跟上。

当前优先级不是继续堆新功能，而是：

```text
1. 建立当前测试基线。
2. 清理仓库运行态与敏感配置。

4. 跑通三端核心闭环。
5. 修订设备绑定和热更新真相源。
6. 把风险清单作为后续 Verify 推进的强制检查表。
```

只要这些风险不进入清单化治理，项目很容易重新回到过去的“边写边补、越补越乱”状态。
