## 1. 当前任务理解

本文档定义网络验证系统与客户端之间的接入契约，重点覆盖：

1. 安卓端脚本登录接入
2. 设备身份上报与编号同步
3. 心跳数据上报
4. 热更新检查与下载
5. 设备解绑语义

---

## 2. 已确认信息

### 2.1 平台接入原则

- 网络验证系统是中枢，客户端通过云端 API 接入
- 安卓脚本可打包为 APK 或 LRJ；LRJ 用于热更新
- 当前设备身份以"设备序列号"为主设备指纹
- 安卓端脚本与 PC 端中控统一采用 **账号 + 密码 + 编号 + 项目 UUIDv4 + 网址 + 端口** 完成首登接入
- 首登成功后，客户端获得 JWT Token 对（AT + RT），用于后续接口访问
- `imsi` 不属于首登必填字段，改为登录成功后的后续上传字段

### 2.2 设备接入口径

- 编号不作为设备身份，只作为展示编号
- 用户在安卓端脚本内修改编号后，登录时同步到网络验证系统
- 后台只保留当前编号，不做编号历史
- 前端"删设备"在服务端语义上定义为"解绑"

## 4. 风险点

### 4.1 禁止客户端把编号当设备身份

`device_alias_no` 只用于展示，服务端不得用于设备唯一性判定。

### 4.2 禁止客户端把"删除设备"理解为物理删除

客户端发起"删除设备"动作时，服务端统一处理为解绑，`device_binding.status = 'unbound'`。

### 4.3 v1 接口以"最小可用"优先

不追求一次性覆盖所有未来能力，优先固化当前已确认、已可落地的契约。

## 5. 登录接口契约

### 5.1 接口定义

```
POST /api/auth/login
Content-Type: application/json
鉴权: 无（公开接口）
```

### 5.2 请求体

```json
{
  "username": "登录账号",
  "password": "登录密码",
  "project_uuid": "项目 UUIDv4",
  "device_fingerprint": "hardware_serial（设备序列号）",
  "client_type": "android | pc",
  "device_alias_no": "A-001（展示编号，可选）"
}
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `username` | ✅ | 登录账号 |
| `password` | ✅ | 登录密码 |
| `project_uuid` | ✅ | 游戏项目 UUIDv4，用于定位游戏项目 |
| `device_fingerprint` | ✅ | 主设备指纹，当前使用 hardware_serial |
| `client_type` | ✅ | `android` 或 `pc` |
| `device_alias_no` | ❌ | 展示编号，登录时同步更新，不参与身份判定 |

### 5.3 登录成功响应

```json
{
  "access_token": "JWT AT（有效期 15 分钟）",
  "refresh_token": "JWT RT（有效期 7 天）",
  "token_type": "bearer",
  "user_id": 10086,
  "user_level": "vip",
  "game_project_code": "game_001"
}
```

### 5.4 登录失败响应

```json
{
  "detail": "用户名或密码错误 / 账号已禁用 / 无效授权 / 设备数量超限"
}
```

HTTP 状态码：`401 Unauthorized`

### 5.5 服务端 10 步验证链

```
Step 1  username 存在？
Step 2  password 匹配？
Step 3  user.status == 'active'？
Step 4  user.expired_at 未过期（或 NULL）？
Step 5  通过 project_uuid 找到 game_project？
Step 6  authorization (user_id, game_project.id) 存在？
Step 7  authorization.status == 'active' 且未过期？
Step 8  device_fingerprint 检查：
          (a) 已绑定且 status=active → 更新 last_seen_at，通过
          (b) 未绑定 → 查当前绑定数 < 级别上限 → 新增绑定，通过
                         超出上限 → 拒绝，fail_reason='fail_device_limit'
Step 9  写 login_log（success=TRUE）
Step 10 签发 AT + RT，返回
```

## 6. 心跳上报接口契约（v2 新增）

### 6.1 接口定义

```
POST /api/device/heartbeat
Content-Type: application/json
鉴权: Bearer AT（必须）
限流: 同 user_id 每分钟 ≤ 4 次（D5 决策，超出返回 429）
```

### 6.2 调用场景

安卓脚本启动后**每 30 秒**调用一次，上报当前运行状态和游戏数据。服务端**写入 Redis 后立即返回**，不等待落库。

### 6.3 请求体

```json
{
  "device_fingerprint": "hardware_serial（与登录时一致）",
  "status": "running | idle | error",
  "game_data": {
    "任意游戏自定义字段": "值，不在此层校验"
  }
}
```

**字段说明：**

| 字段 | 必填 | 说明 |
|------|------|------|
| `device_fingerprint` | ✅ | 必须与当前 AT 关联的登录设备一致 |
| `status` | ✅ | `running`（运行中）/ `idle`（空闲）/ `error`（异常） |
| `game_data` | ❌ | JSONB，各游戏自定义字段，如 `{"map": "北境", "gold": 1024}` |

### 6.4 成功响应

```json
{ "code": 0, "message": "ok" }
```

HTTP 状态码：`200 OK`

### 6.5 失败场景

| 场景 | HTTP 状态码 | detail |
|------|------------|--------|
| 无 Token | 403 | Forbidden |
| 设备未绑定当前账号 | 403 | 设备未绑定到当前账号 |
| 心跳频率超限 | 429 | 心跳上报过于频繁 |

### 6.6 服务端处理逻辑

```
1. 鉴权（get_current_user）获取 user_id 和 game_project_code
2. D5 限流：按 user_id 计数，每分钟 ≤ 4 次
3. 校验设备是否绑定到当前用户（查 device_binding 表）
4. 写入 Redis：
     Key: device:runtime:{game_project_id}:{user_id}:{device_fp}
     Fields: status, last_seen, game_data, user_id
     TTL: 120 秒（超时自动过期，视为离线）
5. 立即返回 200，不等待 Celery 落库
```

## 7. 热更新接口契约（v2 新增）

### 7.1 版本检查接口

```
GET /api/update/check
鉴权: Bearer AT（必须）
```

**Query 参数：**

| 参数 | 必填 | 说明 |
|------|------|------|
| `client_type` | ✅ | `pc` 或 `android` |
| `current_version` | ✅ | 客户端当前版本号，格式 `MAJOR.MINOR.PATCH`，如 `1.0.0` |

**调用示例（安卓脚本）：**

```
GET /api/update/check?client_type=android&current_version=1.0.0
Authorization: Bearer {access_token}
```

**成功响应：**

```json
{
  "need_update": true,
  "current_version": "1.0.1",
  "client_type": "android",
  "game_project_code": "game_001",
  "force_update": false,
  "release_notes": "修复找图精度问题",
  "checksum_sha256": "abc123..."
}
```

当 `need_update=false` 时，`force_update`、`release_notes`、`checksum_sha256` 均为 `null`。

**版本比较规则：**

| 场景 | 结果 |
|------|------|
| 客户端版本 < 服务端版本 | `need_update=true` |
| 客户端版本 = 服务端版本 | `need_update=false` |
| 客户端版本 > 服务端版本（灰度回滚） | `need_update=false` |
| 客户端版本格式非法 | 视为 `0.0.0`，触发更新 |

**Redis 缓存：**
服务端缓存最新版本信息，TTL=300 秒，减少游戏库查询压力。
发布新版本时主动删除缓存，下次查询自动刷新。

### 7.2 获取下载链接接口

```
GET /api/update/download
鉴权: Bearer AT（必须）
```

**Query 参数：**

| 参数 | 必填 | 说明 |
|------|------|------|
| `client_type` | ✅ | `pc` 或 `android` |

**成功响应：**

```json
{
  "download_url": "https://your-domain.com/updates/game_001/android/packages/v1.0.1.lrj?token=xxx",
  "expires_at": "2026-04-24T10:10:00Z",
  "version": "1.0.1",
  "checksum_sha256": "abc123..."
}
```

**安全机制：**
- 下载 URL 含签名 JWT token，有效期 10 分钟（`S3_URL_EXPIRE_SECONDS=600`）
- 本地模式：token 携带 game、client_type、version、exp，nginx 可选 auth_request 验证
- S3 模式（Phase 2）：直接使用 Presigned URL

**安卓脚本完整更新流程：**

```
1. 登录验证通过
2. GET /api/update/check?client_type=android&current_version={当前版本}
3. 若 need_update=true：
   a. 若 force_update=true：必须更新，否则阻止运行
   b. GET /api/update/download?client_type=android 获取签名 URL
   c. 下载更新包
   d. 校验 checksum_sha256（SHA-256 对比）
   e. 调用 installLRPkg 安装
   f. 重载脚本
4. 若 need_update=false：直接继续运行
```

### 7.3 失败场景

| 场景 | HTTP 状态码 | detail |
|------|------------|--------|
| 无 Token | 403 | Forbidden |
| client_type 非法 | 422 | Unprocessable Entity |
| current_version 格式非法 | 422 | Unprocessable Entity |
| 游戏项目暂无已发布版本 | 404 | 暂无可用版本 |

## 9. 设备解绑契约

### 9.1 语义定义

客户端发起"删除设备"动作时，服务端统一执行**解绑**，不物理删除 `device_binding` 记录：

```
device_binding.status = 'unbound'
device_binding.unbound_at = NOW()
device_binding.unbind_reason = '...'
device_binding.unbind_by_type = 'admin' | 'agent' | 'user'
```

### 9.2 当前版本不做

- 不做独立解绑历史接口
- 不做复杂审计轨迹
- 不做编号历史联动

---

## 10. 当前不纳入接入契约的字段

为保持当前版本极简，以下字段暂不纳入任何请求体：

`qaid`、`wifi_mac`、`brand`、`model`、`manufacturer_code`、`product_code`、`sdk_version`、`board_id`

`imsi` 已从首登字段移出，改为登录成功后单独上传（见 §8，Phase 2 实现）。
