# Verify 代理业务画像与等级治理规则

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于沉淀 `HiveGreatSage-Verify` 当前 **代理业务画像 AgentBusinessProfile** 与 **代理等级策略 AgentLevelPolicy** 的业务规则。

本文重点回答：

```text
1. Agent.hierarchy_depth 和 AgentBusinessProfile.tier_level 到底有什么区别？
2. 为什么必须新增代理业务画像？
3. Lv.1-Lv.4 代理等级分别控制什么？
4. 等级策略 AgentLevelPolicy 和单代理覆盖值 AgentBusinessProfile 如何合成最终权限？
5. 项目准入为什么必须使用 tier_level，而不是 Agent.hierarchy_depth？
6. 代理风险状态 risk_status 如何治理？
7. 管理员重置代理密码接口如何安全使用？
8. 当前源码已经实现什么？
9. 当前还缺哪些测试和前端联调？
10. 后续应该如何防止“组织层级”和“业务等级”再次混用？
```

```text

PATCH /admin/api/agents/{agent_id}/business-profile
```

已确认核心边界：

```text
Agent.hierarchy_depth:
  代理组织层级 / 代理树深度。

AgentBusinessProfile.tier_level:
  代理业务等级 Lv.1 - Lv.4。
```

已确认迁移行为：

```text
1. 新增 agent_business_profile 表。
2. tier_level 默认 1。
3. risk_status 默认 normal。
4. 为已有代理自动创建默认业务画像。
5. agent_level_policy 已在 0010 创建，0011 不重复创建。
```

已确认项目准入服务已明确：

```text
项目准入策略必须使用 AgentBusinessProfile.tier_level。
不再使用 Agent.hierarchy_depth。
```

### 2.3 当前风险判断

代理业务画像属于：

```text
P1 高风险治理模块。
```

原因：

```text
1. 它决定代理能否自动开通项目。
2. 它决定代理默认授信和最高授信边界。
3. 它决定代理是否可以创建下级代理。
4. 它决定代理下级数量上限。
5. 它与项目准入、点数、代理树、用户授权都有交叉。
6. 如果和 Agent.hierarchy_depth 混用，会造成严重权限错误。
```

当前最准确状态：

```text
代理业务画像源码已建立；
业务等级与组织层级边界已明确；
但风险状态、下级代理数量、自动开通数量、最大用户数等策略是否已被所有业务路径执行，仍需逐项验证。
```

## 三、术语定义

## 3.1 Agent.hierarchy_depth

字段位置：

```text
agent.level
```

含义：

```text
代理组织层级 / 代理树深度。
```

示例：

```text
总部代理:
  Agent.hierarchy_depth = 1

总部代理的下级:
  Agent.hierarchy_depth = 2

下级代理的下级:
  Agent.hierarchy_depth = 3
```

用途：

```text
1. 表达代理树结构。
2. 支持递归查询代理子树。
3. 支持代理上下级关系。
4. 支持权限范围裁剪。
```

禁止用于：

```text
1. 判断代理业务资质。
2. 判断项目是否可见。
3. 判断项目是否可自动开通。
4. 判断授信额度。
5. 判断是否高价值代理。
```

---

## 3.2 AgentBusinessProfile.tier_level

字段位置：

```text
agent_business_profile.tier_level
```

含义：

```text
代理业务等级 Lv.1 - Lv.4。
```

用途：

```text
1. 判断项目准入可见性。
2. 判断项目是否可申请。
3. 判断项目是否可自动开通。
4. 关联代理等级策略 AgentLevelPolicy。
5. 计算默认授信额度。
6. 计算最高授信额度。
7. 计算是否允许创建下级代理。
8. 计算下级代理数量上限。
```

---

## 3.3 AgentLevelPolicy

表名：

```text
agent_level_policy
```

职责：

```text
定义 Lv.1 - Lv.4 每个业务等级的默认策略。
```

它回答的问题：

```text
这个等级默认有哪些业务能力？
```

典型字段：

```text
level
level_name
description
default_credit_limit
max_credit_limit
max_users_default
can_create_sub_agents
max_sub_agents
can_auto_open_project
auto_open_project_limit
review_priority
is_active
```

---

## 3.4 AgentBusinessProfile

表名：

```text
agent_business_profile
```

职责：

```text
保存单个代理的业务画像和覆盖值。
```

它回答的问题：

```text
这个具体代理当前属于哪个业务等级？
有没有单独覆盖授信、下级代理、风险状态？
```

典型字段：

```text
agent_id
tier_level
credit_limit_override
max_credit_limit_override
can_create_sub_agents_override
max_sub_agents_override
risk_status
remark
```

---

## 3.5 覆盖值 override

覆盖值表示：

```text
针对单个代理覆盖等级策略的默认值。
```

例如：

```text
Lv.2 默认授信 = 1000
某代理 credit_limit_override = 1500
则该代理有效授信额度 = 1500
```

如果 override 为空：

```text
使用 AgentLevelPolicy 默认值。
```

---

## 3.6 risk_status

字段位置：

```text
agent_business_profile.risk_status
```

当前允许值：

```text
normal
watch
restricted
frozen
```

建议语义：

```text
normal:
  正常。

watch:
  观察中，仅用于管理提示，原则上不阻断业务。

restricted:
  限制中，应限制部分高风险操作。

frozen:
  冻结中，应阻断代理核心操作。
```

当前待确认：

```text
risk_status 是否已经在登录、创建用户、项目申请、用户授权等业务路径中生效。
```

## 四、数据模型边界

## 4.1 AgentBusinessProfile 表结构

当前模型字段：

```text
id
agent_id
tier_level
credit_limit_override
max_credit_limit_override
can_create_sub_agents_override
max_sub_agents_override
risk_status
remark
created_at
updated_at
```

约束：

```text
tier_level BETWEEN 1 AND 4

risk_status IN:
  normal
  watch
  restricted
  frozen

credit_limit_override IS NULL OR credit_limit_override >= 0

max_credit_limit_override IS NULL OR max_credit_limit_override >= 0

max_sub_agents_override IS NULL OR max_sub_agents_override >= 0
```

索引：

```text
idx_agent_business_profile_tier
idx_agent_business_profile_risk
```

外键：

```text
agent_id → agent.id
ondelete = CASCADE
```

唯一性：

```text
每个 agent 只能有一条 AgentBusinessProfile。
```

---

## 4.2 迁移 0011 的含义

迁移文件：

```text
migrations/versions/0011_agent_business_profile.py
```

设计目的：

```text
1. 保留 agent.level 作为组织层级。
2. 新增 agent_business_profile.tier_level 作为业务等级。
3. 避免组织层级和代理等级混用。
```

默认等级：

```text
Lv.1 新手代理
Lv.2 标准代理
Lv.3 核心代理
Lv.4 渠道代理
```

回填行为：

```sql
INSERT INTO agent_business_profile
    (agent_id, tier_level, risk_status)
SELECT
    id, 1, 'normal'
FROM agent
ON CONFLICT (agent_id) DO NOTHING;
```

含义：

```text
历史已有代理默认都是 Lv.1，risk_status=normal。
```

风险：

```text
历史代理如果实际应该是 Lv.2 / Lv.3 / Lv.4，需要管理员后续人工调整。
```

---

## 4.3 AgentLevelPolicy 与 AgentBusinessProfile 的关系

```text
AgentLevelPolicy:
  等级默认策略。

AgentBusinessProfile:
  单个代理当前等级 + 单代理覆盖值。
```

最终有效值计算：

```text
如果 AgentBusinessProfile.xxx_override 不为空:
  使用 override。

否则:
  使用 AgentLevelPolicy 默认值。
```

## 五、有效能力计算规则

## 5.1 credit_limit

字段来源：

```text
AgentBusinessProfile.credit_limit_override
AgentLevelPolicy.default_credit_limit
```

计算：

```text
credit_limit =
  credit_limit_override if not null
  else AgentLevelPolicy.default_credit_limit
```

含义：

```text
该代理建议默认授信额度。
```

注意：

```text
这不是当前余额。
当前余额仍在 AgentBalance。
```

---

## 5.2 max_credit_limit

字段来源：

```text
AgentBusinessProfile.max_credit_limit_override
AgentLevelPolicy.max_credit_limit
```

计算：

```text
max_credit_limit =
  max_credit_limit_override if not null
  else AgentLevelPolicy.max_credit_limit
```

含义：

```text
管理员给该代理授信时不应超过的最高额度。
```

待确认：

```text
充值 / 授信接口是否已经强制校验 max_credit_limit。
```

---

## 5.3 can_create_sub_agents

字段来源：

```text
AgentBusinessProfile.can_create_sub_agents_override
AgentLevelPolicy.can_create_sub_agents
```

计算：

```text
can_create_sub_agents =
  can_create_sub_agents_override if not null
  else AgentLevelPolicy.can_create_sub_agents
```

含义：

```text
该代理是否允许创建下级代理。
```

待确认：

```text
代理创建子代理接口是否已读取该有效值。
```

## 5.4 max_sub_agents

字段来源：

```text
AgentBusinessProfile.max_sub_agents_override
AgentLevelPolicy.max_sub_agents
```

计算：

```text
max_sub_agents =
  max_sub_agents_override if not null
  else AgentLevelPolicy.max_sub_agents
```

含义：

```text
该代理最多可创建多少个直接下级代理。
```

待确认：


## 5.5 can_auto_open_project

字段来源：

```text
AgentLevelPolicy.can_auto_open_project
```

当前响应中包含：

```text

```

## 六、管理员接口契约

## 6.1 查询代理等级策略列表

接口：

```text
GET /admin/api/agent-level-policies
```

鉴权：

```text
Admin Token
```

返回：

```json
[
  {
    "id": 1,
    "level": 1,
    "level_name": "新手代理",
    "description": "...",
    "default_credit_limit": 200,
    "max_credit_limit": 300,
    "max_users_default": 0,
    "can_create_sub_agents": false,
    "max_sub_agents": 0,
    "can_auto_open_project": false,
    "auto_open_project_limit": 0,
    "review_priority": 0,
    "is_active": true
  }
]
```

用途：

```text
1. 管理后台展示 Lv.1-Lv.4 等级策略。
2. 管理员调整默认授信、最高授信、下级代理、自动开通限制。
3. 前端编辑代理业务画像时展示等级说明。
```

---

## 6.2 更新代理等级策略

接口：

```text
PATCH /admin/api/agent-level-policies/{level}
```

请求字段：

```json
{
  "level_name": "标准代理",
  "description": "说明",
  "default_credit_limit": 1000,
  "max_credit_limit": 1500,
  "max_users_default": 100,
  "can_create_sub_agents": true,
  "max_sub_agents": 5,
  "can_auto_open_project": true,
  "auto_open_project_limit": 3,
  "review_priority": 10,
  "is_active": true
}
```

后端校验：

```text
1. level 必须在 1-4。
2. 等级策略必须存在。
3. max_credit_limit 不能低于 default_credit_limit。
```

风险：

```text
等级策略是全局默认值。
修改后会影响所有未设置 override 的代理。
```

建议：

```text
1. 前端保存前必须二次确认。
2. 管理端写操作日志。
3. 重大调整前导出当前策略。
```

## 6.4 更新代理业务画像

接口：

```text
PATCH /admin/api/agents/{agent_id}/business-profile
```

请求字段：

```json
{
  "tier_level": 2,
  "credit_limit_override": 1500,
  "max_credit_limit_override": 2000,
  "can_create_sub_agents_override": true,
  "max_sub_agents_override": 10,
  "risk_status": "watch",
  "remark": "重点观察"
}
```

后端校验：

```text
1. 代理必须存在。
2. tier_level 必须在 1-4。
3. max_credit_limit_override 不能低于 credit_limit_override。
4. risk_status 必须是 normal/watch/restricted/frozen。
5. extra 字段禁止。
```

建议：

```text
1. tier_level 调整应记录管理员操作日志。
2. risk_status 从 normal 调整为 restricted/frozen 时应要求填写 remark。
3. override 为空表示回退到等级默认值，不是 0。
```

## 6.5 管理员重置代理密码

接口：

```text
POST /admin/api/agents/{agent_id}/password
```

请求：

```json
{
  "auto_generate": true
}
```

或：

```json
{
  "auto_generate": false,
  "new_password": "new-password"
}
```

返回：

```json
{
  "agent_id": 1001,
  "username": "agent001",
  "generated_password": "Abc123Xyz789"
}
```

### 安全要求

该接口属于高风险接口。

必须遵守：

```text
1. 只能管理员调用。
2. generated_password 只展示一次。
3. 不写入日志。
4. 不写入 Obsidian。
5. 不通过不安全聊天工具传播。
6. 前端复制后立即提示管理员妥善保存。
7. 手动密码必须满足强度要求。
```

当前待确认：

```text
1. 是否已有操作审计日志。
2. 是否限制频繁重置。
3. 是否通知代理原 Token 失效。
4. 是否同步清理该代理旧 Token。
```

建议：

```text
重置代理密码后，应考虑让代理已有登录态失效。
```

## 七、项目准入联动规则

## 7.1 项目准入读取 tier_level

当前项目准入服务应读取：

```text
AgentBusinessProfile.tier_level
```

不读取：

```text
Agent.hierarchy_depth
```

如果代理没有业务画像：

```text
自动创建:
  tier_level = 1
  risk_status = normal
```

---

## 7.2 可见性联动

项目可见性：

```text
public:
  默认可见。

level_limited:
  tier_level >= min_visible_agent_level 才可见。

invite_only:
  必须有 active invite 才可见。

hidden:
  默认不可见。
```

其中 `level_limited` 使用：

```text
tier_level
```

不是：

```text
Agent.hierarchy_depth
```

---

## 7.3 申请资格联动

项目申请资格：

```text
tier_level >= ProjectAccessPolicy.min_apply_agent_level
```

同时还受：

