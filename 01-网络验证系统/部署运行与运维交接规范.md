← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于规范 `HiveGreatSage-Verify` 的部署、启动、运行、监控、回滚和交接。

本文重点回答：

```text
1. Verify 部署前必须具备哪些环境？
2. 开发环境、测试环境、生产环境分别怎么启动？
3. PostgreSQL / Redis / Celery / 前端 / Nginx 的边界是什么？
4. 为什么不能只启动 uvicorn 就认为系统可用？
5. Celery Worker 和 Celery Beat 为什么必须同时运行？
6. 热更新本地存储目录如何放置？
7. systemd / nginx 模板当前能不能直接生产使用？
8. 部署前后需要执行哪些检查？
9. 出现故障怎么排查？
10. 如何把部署状态沉淀到 Obsidian？
```

## 二、当前结论

### 2.1 已确认

当前 Verify 已有基础部署说明和模板。

已确认当前前端接入指南中列出的系统要求：

```text
Python:
  3.11+

PostgreSQL:
  14+

Redis:
  7.0+

Node.js:
  18+

操作系统:
  Linux / Windows
  推荐 Ubuntu 22.04 LTS

内存:
  ≥ 2GB
  适用于小规模部署，约 < 1000 设备
```

已确认当前快速部署流程包含：

```text
1. 克隆仓库。
2. 创建 Python 虚拟环境。
3. 安装 requirements.txt。
4. 复制 .env.example 为 .env。
5. 填写 SECRET_KEY / DATABASE_MAIN_URL / REDIS_URL。
6. 执行 alembic upgrade head。
7. 执行 scripts/init_data.py 初始化管理员账号。
8. 开发环境启动 uvicorn。
9. 生产环境可使用 uvicorn --workers。
10. 前端 npm install / npm run build。
```

已确认当前启动命令中包含：

```text
uvicorn app.main:app --reload --host 127.0.0.1 --port 8000
celery -A app.core.celery_app worker --loglevel=info
cd frontend && npm run dev
```

已确认仓库中存在：

```text
deploy/systemd/hive-verify.service
```

当前 systemd 模板使用：

```text
WorkingDirectory=/opt/HiveGreatSage-Verify
ExecStart=/opt/HiveGreatSage-Verify/venv/bin/uvicorn app.main:app --uds /var/run/hive-verify.sock
```

早期项目骨架中还包含：

```text
deploy/nginx.conf
deploy/systemd/
```

### 2.2 待确认

以下内容不能写成已完成：

```text
1. 当前 deploy/nginx.conf 是否真实存在于当前最新仓库。
2. 当前 hive-verify.service 是否已在真实服务器运行过。
3. 当前 systemd 是否已包含 EnvironmentFile=.env。
4. 当前 systemd 是否处理 /var/run/hive-verify.sock 权限。
5. 当前是否已有 Celery Worker systemd 服务。
6. 当前是否已有 Celery Beat systemd 服务。
7. 当前是否已有前端静态资源 Nginx 配置。
8. 当前是否已有 HTTPS / TLS 证书配置。
9. 当前是否已有日志轮转 logrotate。
10. 当前是否已有生产监控和告警。
11. 当前是否已有生产回滚脚本。
```

### 2.3 当前风险判断

部署运行属于：

```text
P0 生产前阻塞项。
```

原因：

```text
1. Verify 是 PCControl / AndroidScript / 管理后台共同依赖的中枢。
2. 只启动 uvicorn 不代表心跳落库可用。
3. Redis 不可用会影响 Token、限流、心跳、热更新缓存。
4. Celery Worker / Beat 任一缺失都会导致心跳不落库。
5. .env、SECRET_KEY、数据库密码若未治理，会造成安全风险。
6. 热更新本地存储目录若放错位置，可能导致上传成功但客户端下载失败。
7. 生产部署没有回滚方案会放大故障影响。
```

### 2.4 当前准确状态

当前最准确状态：

```text
Verify 已有开发启动说明和初始 systemd 模板；
但尚未形成完整生产部署闭环；
生产部署前必须补齐 Celery Worker / Beat 服务、Nginx、HTTPS、日志、监控、回滚和部署报告。
```

## 3.5 Celery Worker

职责：

```text
执行异步任务。
```

当前重点任务：

```text
app.tasks.heartbeat_flush.flush_device_heartbeats
```

作用：

```text
把 Redis 中的设备心跳批量落库到游戏库 device_runtime。
```

开发启动：

```bash
celery -A app.core.celery_app worker --pool=solo --loglevel=info
```

生产启动：

```bash
celery -A app.core.celery_app worker --concurrency=4 --loglevel=info
```

---

## 3.6 Celery Beat

职责：

```text
按计划触发周期任务。
```

当前关键调度：

```text
每 30 秒触发 flush_device_heartbeats。
```

启动命令：

```bash
celery -A app.core.celery_app beat --loglevel=info
```

### 重要说明

只启动 Celery Worker 不够。

必须同时启动：

```text
Celery Worker
Celery Beat
```

否则：

```text
Worker 存在，但没有 Beat 触发定时任务；
Redis 心跳不会定时落库到 device_runtime。
```

## 四、环境分类

## 4.1 开发环境

目标：

```text
本地开发、接口调试、pytest、前端联调。
```

特点：

```text
1. 可开启 DEBUG。
2. 可开放 /docs。
3. 可使用 uvicorn --reload。
4. 可直接运行 npm run dev。
5. Celery 可使用 --pool=solo。
6. 可使用本地 PostgreSQL / Redis。
```

禁止：

```text
1. 使用生产 SECRET_KEY。
2. 使用生产数据库。
3. 使用生产 Redis。
4. 将 .env 提交到 Git。
```

---

## 4.2 测试环境

目标：

```text
模拟生产部署，做三端联调、热更新、点数、项目准入和压测。
```

特点：

```text
1. 尽量接近生产。
2. DEBUG 可关闭。
3. /docs 可关闭。
4. 使用独立测试数据库。
5. 使用独立 Redis DB。
6. 使用 systemd 或等价守护方式。
7. PCControl / AndroidScript 指向该环境。
```

---

## 4.3 生产环境

目标：

```text
真实用户、真实代理、真实设备运行。
```

要求：

```text
1. ENVIRONMENT=production。
2. DEBUG=false。
3. SECRET_KEY 使用高强度随机值。
4. 数据库密码不可为默认值。
5. Redis 不允许无保护公网暴露。
6. /docs / /redoc 不直接开放。
7. HTTPS 必须启用。
8. 日志必须轮转。
9. Worker / Beat 必须守护。
10. 数据库必须备份。
11. 热更新存储路径必须稳定。
12. 部署必须有回滚方案。
```

## 八、systemd 服务设计

## 8.1 Verify API 服务

当前已有模板：

```text
deploy/systemd/hive-verify.service
```

当前模板核心：

```ini
[Service]
User=www-data
Group=www-data
WorkingDirectory=/opt/HiveGreatSage-Verify
Environment="PATH=/opt/HiveGreatSage-Verify/venv/bin"
ExecStart=/opt/HiveGreatSage-Verify/venv/bin/uvicorn app.main:app --uds /var/run/hive-verify.sock
```

### 当前模板风险

```text
1. 没有 EnvironmentFile 指向 .env。
2. 没有 Restart=always。
3. 没有 RuntimeDirectory。
4. /var/run/hive-verify.sock 权限待确认。
5. 没有明确日志输出策略。
6. 没有限制服务权限。
7. venv 路径使用 venv，而当前指南中建议 .venv，需统一。
```

### 建议版本

```ini
[Unit]
Description=HiveGreatSage Verify API
After=network.target postgresql.service redis.service
Wants=network.target

[Service]
Type=simple
User=hive
Group=hive
WorkingDirectory=/opt/HiveGreatSage-Verify
EnvironmentFile=/etc/hive-greatsage/verify.env
Environment="PATH=/opt/HiveGreatSage-Verify/.venv/bin"
RuntimeDirectory=hive-greatsage
RuntimeDirectoryMode=0755
ExecStart=/opt/HiveGreatSage-Verify/.venv/bin/uvicorn app.main:app --uds /run/hive-greatsage/verify.sock
Restart=always
RestartSec=3
KillSignal=SIGINT
TimeoutStopSec=30

[Install]
WantedBy=multi-user.target
```

待确认：

```text
uvicorn 使用 Unix Socket 时，Nginx upstream 配置必须匹配。
```

## 8.3 Celery Beat 服务

```text
deploy/systemd/hive-verify-beat.service
```

模板：

```ini
[Unit]

```

```bash

## 13.3 Redis 备份

Redis 主要存运行态：

```text
1. Token 黑名单。
2. Refresh Token。
3. 心跳缓存。
4. 热更新缓存。
5. 限流计数。
```

原则：

```text
Redis 可恢复性要求低于 PostgreSQL。
但生产中仍应启用 Redis 持久化或明确接受丢失。
```

Redis 丢失影响：

```text
1. 所有 Refresh Token 可能失效。
2. 黑名单丢失可能影响登出安全。
3. 在线设备态短暂丢失。
4. 热更新缓存重建。
```

```

如果 STORAGE_MODE=s3：

```text
1. API 服务是否启动。
2. systemd 状态。
3. Nginx 是否代理正确。

ls -l /run/hive-greatsage/verify.sock

## 15.2 登录接口 500

检查：

```text
1. SECRET_KEY 是否存在。
2. DATABASE_MAIN_URL 是否正确。
3. 主库迁移是否完成。
4. 用户表是否存在。
5. Redis 是否可用。
6. Token 生成是否异常。
```

---

## 15.3 心跳返回 200 但 device_runtime 不更新

检查：

```text
1. Redis 中是否有 device:runtime key。
2. Worker 是否启动。
3. Beat 是否启动。
4. Beat 是否触发 flush_device_heartbeats。
5. 游戏库是否可连接。
6. device_runtime 表是否存在。
```

命令：

```bash
redis-cli -p 16379 keys "device:runtime:*"
journalctl -u hive-verify-worker.service -f
journalctl -u hive-verify-beat.service -f
```

---

## 15.4 PCControl 看不到设备

检查：

```text
1. PCControl 是否登录同一 project_uuid。
2. AndroidScript 是否登录同一 project_uuid。
3. AndroidScript 心跳是否成功。
4. Redis Key 是否存在。
5. /api/device/list 是否返回设备。
6. PC Token 是否过期。
7. 设备绑定是否属于当前项目。
```

---

只有 Android 失败:
  查 AndroidScript base_url、project_uuid、设备绑定、授权状态。

只有 PC 失败:
  查 PCControl base_url、project_uuid、Token 刷新。

## 十九、建议新增决策日志

```markdown
## D042 — Verify 生产运行必须拆分 API、Celery Worker、Celery Beat 三类服务

**日期：** 2026-04-29  
**状态：** 建议确认  
**关联文档：** [[01-网络验证系统/部署运行与运维交接规范]]

### 背景

Verify 的设备心跳链路不是单一 API 服务。AndroidScript 心跳先写 Redis，再由 Celery Beat 定时触发 Celery Worker 批量落库到游戏库 device_runtime。

### 决策

生产运行必须至少包含：

```text
1. hive-verify.service
   FastAPI / Uvicorn API 服务。

2. hive-verify-worker.service
   Celery Worker，执行异步任务。

3. hive-verify-beat.service
   Celery Beat，触发周期任务。
```

### 理由

```text
1. 只启动 API，心跳不会落库。
2. 只启动 Worker，没有 Beat 也不会定时触发。
3. Worker / Beat 分离便于监控和重启。
4. 心跳落库故障可以独立排查。
```

### 后续动作

```text
□ 新增 worker systemd 模板。
□ 新增 beat systemd 模板。
□ 部署报告中加入三项服务状态。
□ 三端联调前检查 Worker / Beat。
```
```

```markdown
## D043 — Verify 热更新本地存储目录不得放在源码仓库内

**日期：** 2026-04-29  
**状态：** 建议确认  
**关联文档：** [[01-网络验证系统/部署运行与运维交接规范]], [[01-网络验证系统/仓库治理与敏感配置清理清单]]

### 背景

Verify 支持热更新包上传和客户端下载。如果 STORAGE_MODE=local，更新包会保存到本地文件系统。若将更新包放在源码仓库内，可能被 Git 清理、部署覆盖或误提交。

### 决策

生产环境本地热更新目录统一放到：

```text
/var/lib/hive-greatsage/updates
```

不得放在：

```text
/opt/HiveGreatSage-Verify/storage
/opt/HiveGreatSage-Verify/uploads
/opt/HiveGreatSage-Verify/frontend/dist
```

### 理由

```text
1. 源码和运行数据分离。
2. 避免部署覆盖更新包。
3. 避免更新包进入 Git。
4. 便于单独备份。
```

### 后续动作

```text
□ 修改生产 .env STORAGE_LOCAL_ROOT。
□ Nginx 或 StorageService 下载路径同步调整。
□ 部署报告记录存储目录。
```
```
