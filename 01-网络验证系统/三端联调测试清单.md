# Verify 三端联调测试清单

← 返回 [[项目总大纲]] | 父节点: [[01-网络验证系统/架构设计]]

## 一、文档目的

本文用于指导 **Verify + PCControl + AndroidScript** 的三端联调。

本文不是单元测试清单，也不是单独后端测试清单。

本文目标是验证：

```text
1. Verify 后端能作为平台中枢稳定运行。
2. PCControl 能登录 Verify、拉取设备数据、修改脚本参数、检查更新。
3. AndroidScript 能登录 Verify、上报心跳、拉取参数、检查并安装更新。
4. Redis / Celery / PostgreSQL 能完成心跳缓存与批量落库。
5. 管理后台能创建项目、代理、用户、授权、上传热更新包。
6. 代理后台能完成代理登录、查看余额、查看项目目录、申请项目准入。
7. 三端字段、接口、Token、项目口径一致。
```

## 三、三端角色定义

### 3.1 Verify

Verify 是云端中枢，负责：

```text
1. 用户身份认证。
2. 项目授权校验。
3. 设备绑定。
4. 心跳接收。
5. Redis 心跳缓存。
6. Celery 批量落库。
7. 参数存取。
8. 热更新元数据与包管理。
9. 管理员后台接口。
10. 代理后台接口。
```

### 3.2 PCControl

PCControl 是 PC 中控端，负责：

```text
1. 用户登录。
2. 展示账号项目授权状态。
3. 拉取安卓设备列表。
4. 展示设备在线状态和 game_data。
5. 修改脚本参数。
6. 检查 PC 中控更新。
7. 下载 PC 更新包。
8. 后续执行本地更新安装。
```

PCControl 不负责：

```text
1. 直接写数据库。
2. 直接写 Redis。
3. 代替安卓脚本上报心跳。
4. 绕过 Verify 读取设备数据。
```

### 3.3 AndroidScript

AndroidScript 是安卓端自动化脚本，负责：

```text
1. 启动时登录 Verify。
2. 校验项目授权。
3. 绑定当前安卓设备。
4. 定时上报心跳。
5. 上报 game_data。
6. 拉取脚本参数。
7. 检查安卓脚本更新。
8. 下载并 installLrPkg 安装 lrj。
9. 执行具体游戏自动化任务。
```

AndroidScript 不负责：

```text
1. 管理代理。
2. 创建用户。
3. 修改授权。
4. 上传热更新包。
5. 修改云端参数。
```

## 6.2 第 2 阶段：AndroidScript 登录 Verify

### 6.2.1 安卓端登录请求

接口：

```text
POST /api/auth/login
```

请求体：

```json
{
  "username": "user001",
  "password": "User@2026!",
  "project_uuid": "项目 UUID",
  "device_fingerprint": "android_device_001",
  "client_type": "android"
}
```

通过标准：

```text
1. HTTP 200。
2. 返回 access_token。
3. 返回 refresh_token。
4. 返回用户信息。
5. 返回项目授权信息。
6. device_fingerprint 被绑定。
7. login_log 写入 success=true。
```

数据库检查：

```sql
SELECT * FROM device_binding WHERE user_id = 目标用户ID;
SELECT * FROM login_log WHERE user_id = 目标用户ID ORDER BY login_at DESC LIMIT 5;
```

失败场景必须验证：

```text
□ 密码错误。
□ 项目 UUID 错误。
□ 项目未启用。
□ 用户无项目授权。
□ 授权已过期。
□ 设备数超限。
□ 用户被停用。
```

## 6.3 第 3 阶段：AndroidScript 上报心跳

### 6.3.1 心跳请求

接口：

```text
POST /api/device/heartbeat
```

请求体：

```json
{
  "device_fingerprint": "android_device_001",
  "status": "running",
  "game_data": {

    "map": "北境",
    "gold": 1024,
    "level": 35
  }

```

```text
Authorization: Bearer {android_access_token}
```

通过标准：

```text
1. HTTP 200。
2. code = 0。
3. message = ok。
4. Redis 中能看到在线态。
5. 同一 user_id 每分钟超过 4 次会返回 429。
```

```bash

```

通过标准：

```text
能看到该 device_fingerprint 的实时状态或心跳缓冲。
```

### 6.3.3 Celery 落库检查

等待至少 30 秒。

数据库检查：

```sql
SELECT * FROM device_runtime WHERE device_id = 'android_device_001';
```

通过标准：

```text
1. device_runtime 有记录。
2. status = running。
3. game_data 与上报内容一致。

```

| PC 登录 | client_type=pc | 不占安卓设备名额 |

## 7.2 心跳异常

| 场景 | 操作 | 期望 |
|---|---|---|
| 无 Token | POST `/api/device/heartbeat` | 401 或 403 |

## 十二、当前风险点

### R001 PC 登录是否占设备名额待确认

```text

4. 第二个安卓设备登录失败。
```

### R002 心跳只写 Redis 不代表已落库

必须确认：

```text
1. 心跳接口返回 200。
2. Redis 有数据。
3. Celery 触发。
4. device_runtime 有数据。
```

如果只完成 1 和 2：

```text
不能认为设备运行数据链路通过。
```

---

### R003 参数联调需要真实 script_param_def

如果游戏库没有参数定义：

```text
/api/params/get 可能返回空。
```

这不是接口失败。

联调前必须准备：

```text
script_param_def 测试数据。
```

---

### R004 热更新必须区分元数据通过和安装通过

热更新检查成功只代表：

```text
客户端知道有新版本。
```

不代表：

```text
1. 文件可下载。
2. SHA-256 正确。
3. 安卓能 installLrPkg。
4. PC 能完成更新安装。
```

---
